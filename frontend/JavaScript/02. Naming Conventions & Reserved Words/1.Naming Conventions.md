# JavaScript Naming Conventions

- Naming conventions in JavaScript are essential for writing clean, readable, and maintainable code. 
- They provide guidelines for naming variables, functions, constants, classes, and other identifiers.

## 1. General Naming Principles

### 1.1 Meaningful Names
Names should clearly describe the purpose of the variable or function.

```javascript
// Good
let userName = "John";

// Bad
let x = "John";
```

### 1.2 Avoid Reserved Keywords
Do not use JavaScript reserved keywords as names.

```javascript
// Invalid
let if = 10; // SyntaxError
```

### 1.3 No Special Characters
Names can only contain letters, digits, `$`, and `_`, and must not begin with a digit.

```javascript
// Valid
let _userName;
let $price;

// Invalid
let 1user; // SyntaxError
```

### 1.4 Case Sensitivity
JavaScript names are case-sensitive.

```javascript
let name = "Alice";
let Name = "Bob"; // Different variables
```

---

## 2. Variable and Function Naming

- Use **camelCase**: The first word is lowercase, and subsequent words are capitalized.

```javascript
let userAge = 25;
function calculateSum(a, b) {
  return a + b;
}
```

- **Descriptive Names**: Use descriptive and concise names.

```javascript
// Good
let isUserLoggedIn = true;

// Bad
let flag = true;
```

---

## 3. Constants

- Use **UPPERCASE with underscores**: Constants that never change should be named in uppercase.

```javascript
const PI = 3.14;
const MAX_LIMIT = 100;
```

- For constants with block scope (using `const`), camelCase is acceptable if they are not immutable.

```javascript
const userAge = 30; // Local constant, camelCase is okay
```

---

## 4. Class and Constructor Function Naming

- Use **PascalCase (TitleCase)**: Capitalize the first letter of each word.

```javascript
class UserAccount {
  constructor(name) {
    this.name = name;
  }
}

const user = new UserAccount("Alice");
```

---

## 5. Boolean Variables

- Prefix with `is`, `has`, or `can`: Use descriptive prefixes for variables that store boolean values.

```javascript
let isActive = true;
let hasItems = false;
let canEdit = true;
```

---

## 6. Object and Array Naming

- Use plural names for arrays:

```javascript
let users = ["Alice", "Bob"];
let items = [1, 2, 3];
```

- Use descriptive names for objects:

```javascript
let user = { name: "Alice", age: 25 };
let config = { timeout: 3000, retries: 3 };
```

---

## 7. Private Variables

- Prefix with `_` (convention, not enforced): Used for private properties in objects or classes.

```javascript
class User {
  constructor(name) {
    this._name = name; // Private convention
  }
  
  getName() {
    return this._name;
  }
}
```

**Note**: Modern JavaScript supports true private fields using `#` prefix:

```javascript
class User {
  #name; // Private field
  
  constructor(name) {
    this.#name = name;
  }
  
  getName() {
    return this.#name;
  }
}
```

---

## 8. File and Folder Naming

- **Kebab-case** or **snake_case**: Use lowercase words separated by dashes or underscores for file and folder names.

```plaintext
user-profile.js
user_profile.js
app_config.json
data-service.js
```

---

## 9. Event Handlers

- Prefix with `on`: For event handler functions, use the `on` prefix.

```javascript
function onButtonClick() {
  console.log("Button clicked");
}

function onUserLogin() {
  console.log("User logged in");
}

// Arrow function style
const onFormSubmit = (event) => {
  event.preventDefault();
  // Handle form submission
};
```

---

## 10. Abbreviations and Acronyms

- Treat abbreviations and acronyms as words:
  - First word lowercase in camelCase: `htmlParser`
  - Entirely uppercase in constants: `API_KEY`

```javascript
let apiResponse = fetch(...);
let xmlHttpRequest = new XMLHttpRequest();
const API_VERSION = "v1.0";
const HTTP_STATUS_OK = 200;
```

## 11. Reserved Words

Avoid using JavaScript reserved words as identifiers. These keywords have special meanings and cannot be used as variable names:

### ES5 Reserved Words
```plaintext
break, case, catch, class, const, continue, debugger, default, delete,
do, else, export, extends, finally, for, function, if, import, in,
instanceof, new, return, super, switch, this, throw, try, typeof,
var, void, while, with
```

### ES6+ Additional Keywords
```plaintext
let, static, yield, await, enum, implements, interface, package,
private, protected, public, async
```

### Future Reserved Words (Strict Mode)
```plaintext
implements, interface, let, package, private, protected, public, static, yield
```

**Example:**
```javascript
// Invalid
let class = "Math"; // SyntaxError
let return = 10;    // SyntaxError

// Valid alternatives
let className = "Math";
let returnValue = 10;
```

## 12. Getters and Setters

- **Prefix with `get` and `set`**: Use clear prefixes for getter and setter methods.

```javascript
class User {
  constructor(firstName, lastName) {
    this._firstName = firstName;
    this._lastName = lastName;
  }
  
  // Getter
  getFullName() {
    return `${this._firstName} ${this._lastName}`;
  }
  
  // Setter
  setFirstName(name) {
    this._firstName = name;
  }
  
  // Modern ES6 getters and setters
  get email() {
    return this._email;
  }
  
  set email(value) {
    if (!value.includes('@')) {
      throw new Error('Invalid email');
    }
    this._email = value;
  }
}

const user = new User("John", "Doe");
console.log(user.getFullName()); // "John Doe"
user.email = "john@example.com"; // Using setter
console.log(user.email); // Using getter
```

## 13. Async Functions

- **Descriptive names**: Use meaningful names that indicate async behavior, or prefix with `fetch`, `load`, `get`.

```javascript
// Good - clear async operations
async function fetchUserData(userId) {
  const response = await fetch(`/api/users/${userId}`);
  return response.json();
}

async function loadProducts() {
  // Load products from API
}

async function getUserProfile(id) {
  // Get user profile asynchronously
}

// Using promises
function getDataAsync() {
  return new Promise((resolve, reject) => {
    // Async operation
  });
}
```

**Note**: The `async` keyword already indicates the function is asynchronous, so additional prefixes like `asyncGetUser` are usually unnecessary and redundant.

## 14. Module and Export Naming

### 14.1 Default Exports
Use PascalCase for classes and camelCase for functions/objects:

```javascript
// userService.js
export default class UserService {
  // ...
}

// utils.js
export default function formatDate(date) {
  // ...
}
```

### 14.2 Named Exports
Use the same naming convention as the exported item:

```javascript
// constants.js
export const API_URL = "https://api.example.com";
export const MAX_RETRIES = 3;

// helpers.js
export function calculateTotal(items) {
  // ...
}

export class ValidationError extends Error {
  // ...
}
```

### 14.3 Namespace/Module Objects
Use camelCase for module objects:

```javascript
// mathUtils.js
const mathUtils = {
  add: (a, b) => a + b,
  subtract: (a, b) => a - b,
  multiply: (a, b) => a * b
};

export default mathUtils;

// Usage
import mathUtils from './mathUtils';
mathUtils.add(5, 3);
```

## 15. Component Naming (React/Vue)

### 15.1 React Components
- Use **PascalCase** for component names
- File names should match component names

```javascript
// UserProfile.js or UserProfile.jsx
function UserProfile({ name, email }) {
  return (
    <div>
      <h1>{name}</h1>
      <p>{email}</p>
    </div>
  );
}

export default UserProfile;

// Or as a class component
class UserDashboard extends React.Component {
  render() {
    return <div>Dashboard</div>;
  }
}
```

### 15.2 Vue Components
- Use **PascalCase** or **kebab-case** for component names

```javascript
// UserProfile.vue (PascalCase file name)
export default {
  name: 'UserProfile', // PascalCase
  props: {
    userName: String
  }
}

// Or kebab-case
// user-profile.vue
export default {
  name: 'user-profile', // kebab-case
}
```

### 15.3 Higher-Order Components (HOC)
Prefix with `with`:

```javascript
function withAuth(Component) {
  return function AuthComponent(props) {
    // Authentication logic
    return <Component {...props} />;
  };
}

// Usage
const ProtectedPage = withAuth(Dashboard);
```

## 16. Test File Naming

### 16.1 Test Files
Use `.test.js`, `.spec.js`, or `__tests__` folder:

```plaintext
// Adjacent to source file
user-service.js
user-service.test.js
user-service.spec.js

// Or in __tests__ folder
src/
  components/
    UserProfile.js
    __tests__/
      UserProfile.test.js
```

### 16.2 Test Naming Conventions
```javascript
// userService.test.js
describe('UserService', () => {
  describe('getUser', () => {
    it('should return user data when user exists', () => {
      // Test implementation
    });
    
    it('should throw error when user not found', () => {
      // Test implementation
    });
  });
});

// Or using test() instead of it()
test('calculates total correctly', () => {
  // Test implementation
});
```

## 17. Common Anti-Patterns to Avoid

### ‚ùå Don't Use:

#### 17.1 Hungarian Notation
```javascript
// Bad - Hungarian notation (rarely used in modern JS)
let strUserName = "John";
let intAge = 25;
let boolIsActive = true;

// Good - JavaScript is dynamically typed
let userName = "John";
let age = 25;
let isActive = true;
```

#### 17.2 Meaningless Names
```javascript
// Bad
let a = 10;
let temp = getData();
let data = something();
let thing = [];

// Good
let userAge = 10;
let userProfile = getData();
let productList = fetchProducts();
let activeUsers = [];
```

#### 17.3 Redundant Information
```javascript
// Bad - type info in name (redundant)
let userArray = [];
let userObject = {};
let nameString = "John";

// Good
let users = [];
let user = {};
let name = "John";
```

#### 17.4 Overly Long Names
```javascript
// Bad - too verbose
let thisIsTheUserAuthenticationTokenForTheCurrentSession = "abc123";

// Good - descriptive but concise
let authToken = "abc123";
let sessionToken = "abc123";
```

#### 17.5 Inconsistent Naming
```javascript
// Bad - inconsistent patterns
function getUser() { }
function user_delete() { }
function FetchUserData() { }

// Good - consistent camelCase
function getUser() { }
function deleteUser() { }
function fetchUserData() { }
```

#### 17.6 Acronyms and Abbreviations Without Context
```javascript
// Bad - unclear abbreviations
let usr = "John";
let calc = 0;
let msg = "";

// Good - clear names
let userName = "John";
let totalCalculation = 0;
let errorMessage = "";

// Exception: Well-known abbreviations are okay
let html = "<div></div>"; // HTML is widely understood
let api = "https://..."; // API is universally known
let url = "https://..."; // URL is clear
```

#### 17.7 Numbers in Names (Usually)
```javascript
// Bad - unclear purpose
let data1 = [];
let data2 = [];
let value3 = 0;

// Good - descriptive names
let activeUsers = [];
let pendingUsers = [];
let totalAmount = 0;

// Exception: Meaningful numbers
let base64Encoded = "...";
let sha256Hash = "...";
```

## 18. Best Practices

### Use Descriptive Length
- Short names (`i`, `j`, `k`) are acceptable for loop counters
- Longer, descriptive names for important variables

```javascript
// Loop counters - short names are fine
for (let i = 0; i < items.length; i++) {
  // ...
}

// Important variables - use descriptive names
let userAuthenticationToken = "abc123";
let customerShippingAddress = "123 Main St";
```

### Avoid Single Letter Variables
Except for:
- Loop counters (`i`, `j`, `k`)
- Mathematical formulas (`x`, `y`, `z`)
- Callback parameters when context is clear (`users.map(u => u.name)`)

### Consistency is Key
Maintain the same naming pattern throughout your codebase:

```javascript
// Good - consistent naming
getUserData()
setUserData()
deleteUserData()

// Bad - inconsistent naming
getUserData()
user_set()
removeUserData()
```
