## Advanced OOP & Design Patterns

### Q1. Explain the MRO (Method Resolution Order) in Python. How does C3 linearization work?

**Answer:**

Method Resolution Order defines the sequence in which base classes are searched when looking up an attribute on a class or instance. In Python 3, all classes inherit from `object` implicitly, and multiple inheritance is resolved deterministically so that subclasses precede their parents and the order respects the declaration order in each class’s `__bases__`.

C3 linearization produces a single linear order that preserves local precedence (left-to-right in the inheritance list) and monotonicity: if class A appears before class B in one branch, that ordering is not contradicted when merging with other branches. The algorithm merges the heads of several sequences (each class’s parents’ linearizations plus the direct bases list), repeatedly picking a head that does not appear in the tail of any other sequence; if no such head exists, the hierarchy is inconsistent and Python raises an error at class creation time.

```python
class A:
    def f(self):
        return "A"


class B(A):
    def f(self):
        return "B"


class C(A):
    def f(self):
        return "C"


class D(B, C):
    pass


# MRO: D -> B -> C -> A -> object
print(D.__mro__)
print(D().f())  # "B" — first match in MRO wins
```

### Q2. What are metaclasses in Python? When would you use them?

**Answer:**

A metaclass is the class of a class: it controls how `class` statements construct class objects (not instances). The default metaclass is `type`. By subclassing `type` and overriding `__new__` / `__init__`, you can inject hooks for validation, automatic registration, API schema generation, or ORM-style table mapping.

Use metclasses sparingly—most goals are achievable with class decorators or `__init_subclass__` (Python 3.6+). Reach for metaclasses when you need to coordinate subclasses globally, enforce invariants across a framework of classes, or customize class creation in ways decorators cannot express cleanly.

```python
class Registered(type):
    registry: dict[str, type] = {}

    def __new__(mcs, name, bases, namespace, **kwargs):
        cls = super().__new__(mcs, name, bases, namespace)
        if not name.startswith("_"):
            mcs.registry[name] = cls
        return cls


class Plugin(metaclass=Registered):
    pass


class EmailPlugin(Plugin):
    pass


assert Registered.registry["EmailPlugin"] is EmailPlugin
```

### Q3. Explain descriptors in Python. What are data and non-data descriptors?

**Answer:**

Descriptors are objects defining `__get__`, `__set__`, or `__delete__` and participate in attribute lookup on instances. When an attribute is accessed, if the value found on the class (or in the MRO) has these methods, Python invokes the descriptor protocol instead of returning the raw object.

A **data descriptor** implements `__set__` or `__delete__` (or both); it takes precedence over an instance’s `__dict__`. A **non-data descriptor** only implements `__get__`; if the instance has the same name in `__dict__`, the instance attribute wins. This distinction is why `property` (data descriptor) cannot be shadowed by assignment on the instance, while ordinary methods (non-data) can be replaced per instance.

```python
class NonData:
    def __get__(self, obj, owner=None):
        return "descriptor"


class Data:
    def __get__(self, obj, owner=None):
        return "descriptor"

    def __set__(self, obj, value):
        raise AttributeError("read-only")


class C:
    nd = NonData()
    d = Data()


c = C()
c.nd = "shadow"
assert c.nd == "shadow"  # instance dict wins

c.d = "try"  # raises — data descriptor controls storage
```

### Q4. What is the difference between `__new__` and `__init__`?

**Answer:**

`__new__` is a static method (conventionally `cls` is first) that **allocates** the object and returns it; it runs before the instance exists. `__init__` receives the already-created `self` and **initializes** it in place, returning `None`.

Immutable built-in subclasses (e.g., `int`, `tuple`) often require overriding `__new__` because you cannot mutate after creation. For typical mutable classes, `__init__` is enough. If `__new__` returns an instance of `cls`, `__init__` is called; if it returns a non-`cls` instance, `__init__` may be skipped.

```python
class OnlyPrime(int):
    def __new__(cls, value: int):
        if value < 2 or any(value % i == 0 for i in range(2, int(value**0.5) + 1)):
            raise ValueError("not prime")
        return int.__new__(cls, value)


class Resource:
    def __new__(cls, *args, **kwargs):
        instance = super().__new__(cls)
        instance._opened = False
        return instance

    def __init__(self, name: str) -> None:
        self.name = name
```

### Q5. What are abstract base classes (ABCs)? How do you implement them?

**Answer:**

ABCs define interfaces: which methods must exist on concrete subclasses. `abc.ABC` combined with `@abstractmethod` prevents instantiation of the base class until all abstract members are implemented, catching interface violations early.

You can also register virtual subclasses with `register()` for structural typing without inheritance, but `@abstractmethod` enforcement applies to subclasses of the ABC itself. ABCs are central to `collections.abc`, `typing.Protocol`-like patterns, and plugin APIs.

```python
from abc import ABC, abstractmethod


class Storage(ABC):
    @abstractmethod
    def get(self, key: str) -> bytes | None: ...

    @abstractmethod
    def put(self, key: str, value: bytes) -> None: ...


class MemoryStorage(Storage):
    def __init__(self) -> None:
        self._data: dict[str, bytes] = {}

    def get(self, key: str) -> bytes | None:
        return self._data.get(key)

    def put(self, key: str, value: bytes) -> None:
        self._data[key] = value
```

### Q6. Explain the Singleton pattern in Python. What are different ways to implement it?

**Answer:**

Singleton ensures one instance per process (for a given class). In Python, module-level objects are naturally singletons because modules are cached in `sys.modules`. Other approaches: override `__new__` to return a cached instance, use a metaclass, or use a decorator that wraps the class factory.

Thread safety matters if multiple threads might create the instance concurrently—use a lock around first creation or rely on import-time initialization. In async code, prefer explicit app-scoped objects over hidden globals.

```python
from threading import Lock
from typing import ClassVar, Self


class Settings:
    _instance: ClassVar["Settings | None"] = None
    _lock: ClassVar[Lock] = Lock()

    def __new__(cls) -> Self:
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._initialized = False
        return cls._instance

    def __init__(self) -> None:
        if getattr(self, "_initialized", False):
            return
        self.env = "production"
        self._initialized = True
```

### Q7. What is the Factory pattern? Implement it in Python.

**Answer:**

Factory encapsulates object construction: callers request a product by key or configuration without naming concrete classes. It reduces coupling when you add new implementations and centralizes validation and wiring (dependencies, connection strings).

Common variants: simple factory function, factory method on a class, or abstract factory returning families of related objects.

```python
from typing import Protocol


class Parser(Protocol):
    def parse(self, raw: str) -> dict: ...


class JsonParser:
    def parse(self, raw: str) -> dict:
        import json

        return json.loads(raw)


class YamlParser:
    def parse(self, raw: str) -> dict:
        import yaml  # type: ignore

        return yaml.safe_load(raw)


def get_parser(kind: str) -> Parser:
    factories = {"json": JsonParser, "yaml": YamlParser}
    try:
        return factories[kind]()
    except KeyError as e:
        raise ValueError(f"unknown parser: {kind}") from e
```

### Q8. What is the Observer pattern? Give a Python example.

**Answer:**

Observers subscribe to a subject; when the subject’s state changes, it notifies all observers. This decouples event producers from consumers and is the basis for pub/sub, GUI signals, and domain events.

In Python, you can implement with a list of callables, `weakref` to avoid retaining observers, or third-party libraries. Keep notification synchronous unless you explicitly queue work.

```python
from __future__ import annotations

from collections.abc import Callable


class EventSource:
    def __init__(self) -> None:
        self._observers: list[Callable[[str], None]] = []

    def subscribe(self, fn: Callable[[str], None]) -> None:
        self._observers.append(fn)

    def emit(self, message: str) -> None:
        for fn in list(self._observers):
            fn(message)


def log_event(msg: str) -> None:
    print("LOG:", msg)


bus = EventSource()
bus.subscribe(log_event)
bus.emit("order.created")
```

### Q9. Explain the Strategy pattern with a Python example.

**Answer:**

Strategy factors out a family of interchangeable algorithms behind a common interface. The context delegates behavior to a strategy object, enabling runtime selection and unit testing of each strategy in isolation.

Unlike simple `if/elif`, strategies scale when adding new behaviors without editing the context’s core logic (open/closed principle).

```python
from typing import Protocol


class DiscountStrategy(Protocol):
    def apply(self, amount: float) -> float: ...


class NoDiscount:
    def apply(self, amount: float) -> float:
        return amount


class PercentDiscount:
    def __init__(self, pct: float) -> None:
        self._pct = pct

    def apply(self, amount: float) -> float:
        return round(amount * (1 - self._pct), 2)


class Cart:
    def __init__(self, strategy: DiscountStrategy) -> None:
        self._strategy = strategy

    def total(self, subtotal: float) -> float:
        return self._strategy.apply(subtotal)


assert Cart(PercentDiscount(0.1)).total(100) == 90.0
```

### Q10. What are mixins in Python? How do they differ from regular inheritance?

**Answer:**

A mixin is a class intended to be combined with others to reuse orthogonal behavior (e.g., `JSONSerializableMixin`) without representing a full “is-a” domain concept. Mixins are usually abstract in the sense that they are not instantiated alone and often avoid deep hierarchies.

They differ from “regular” inheritance mainly by intent: mixins compose capabilities; base classes model taxonomy. Name mixins with `Mixin` suffix, keep them stateless or document required attributes, and watch MRO when combining multiple mixins.

```python
class TimestampMixin:
    def touch(self) -> None:
        from datetime import UTC, datetime

        self.updated_at = datetime.now(tz=UTC)


class Model(TimestampMixin):
    def __init__(self, name: str) -> None:
        self.name = name


m = Model("x")
m.touch()
assert hasattr(m, "updated_at")
```

### Q11. What is duck typing and how does Python use it?

**Answer:**

“If it walks like a duck…”—Python cares about **behavior** (methods and properties) more than nominal types. Protocols, ABCs, and structural checks (`hasattr`) express expected interfaces without requiring inheritance.

Static type checkers add nominal and structural rules on top; at runtime, Python still dispatches by available attributes. This enables flexible APIs but requires discipline: document contracts, use tests and type hints, and prefer `Protocol` for clear interfaces.

```python
from typing import Protocol


class Drawable(Protocol):
    def draw(self) -> None: ...


def render(obj: Drawable) -> None:
    obj.draw()


class Circle:
    def draw(self) -> None:
        print("circle")


render(Circle())  # OK — structurally compatible
```

### Q12. Explain the `__slots__` mechanism. When should you use it?

**Answer:**

`__slots__` replaces per-instance `__dict__` with fixed C-level slots, reducing memory and speeding attribute access. It also prevents arbitrary attribute assignment unless you include `__dict__` in slots or use a subclass strategy.

Use when you have many homogeneous instances (data records, nodes) and attribute names are fixed. Avoid if you need dynamic attributes, multiple inheritance complications, or weak references unless you add `"__weakref__"` to slots.

```python
class Point:
    __slots__ = ("x", "y", "__weakref__")

    def __init__(self, x: float, y: float) -> None:
        self.x = x
        self.y = y


p = Point(1, 2)
# p.z = 3  # AttributeError
```

### Q13. What are class methods vs static methods vs instance methods?

**Answer:**

**Instance methods** take `self` and operate on instance state; binding passes the instance automatically. **Class methods** take `cls` and receive the class object—useful for alternate constructors and hooks that should respect inheritance (`cls` is the actual subclass). **Static methods** are plain functions namespaced in the class without implicit first argument; use when the method logically belongs to the class but needs neither `self` nor `cls`.

```python
class User:
    def __init__(self, email: str) -> None:
        self.email = email

    @classmethod
    def from_record(cls, record: dict[str, str]) -> "User":
        return cls(record["email"])

    @staticmethod
    def normalize_email(email: str) -> str:
        return email.strip().lower()


u = User.from_record({"email": " A@B.COM "})
assert u.email == " A@B.COM "
assert User.normalize_email(" X@Y.Z ") == "x@y.z"
```

### Q14. What is the difference between composition and inheritance? When to use each?

**Answer:**

**Inheritance** models stable “is-a” relationships and shares interface through the MRO; overuse leads to fragile hierarchies. **Composition** builds objects from parts (“has-a” or “uses-a”), swapping collaborators without subclassing.

Favor composition for varying behavior and dependency injection; use inheritance for true taxonomies and when Liskov substitution holds. Many designs combine both: small ABC for interface, composition for implementation.

```python
class EmailNotifier:
    def send(self, msg: str) -> None:
        print("email:", msg)


class OrderService:
    def __init__(self, notifier: EmailNotifier) -> None:
        self._notifier = notifier

    def place(self) -> None:
        self._notifier.send("order placed")


OrderService(EmailNotifier()).place()
```

### Q15. Explain Python's data model and dunder (magic) methods.

**Answer:**

The data model specifies how objects interact with language syntax and builtins: `__str__` / `__repr__`, `__eq__` / `__hash__`, `__len__`, `__getitem__`, context managers, arithmetic, and more. Implementing these methods makes user-defined types behave like built-ins.

Coherence matters: e.g., if you define `__eq__` for mutable objects, often set `__hash__` to `None` to keep `dict`/`set` semantics safe. Follow documented invariants (Python data model docs) for edge cases like `NotImplemented`.

```python
from collections.abc import Iterator


class Deck:
    def __init__(self, cards: list[int]) -> None:
        self._cards = cards

    def __len__(self) -> int:
        return len(self._cards)

    def __iter__(self) -> Iterator[int]:
        return iter(self._cards)

    def __getitem__(self, idx: int) -> int:
        return self._cards[idx]


d = Deck([1, 2, 3])
assert len(d) == 3 and list(d) == [1, 2, 3]
```

## Decorators & Closures

### Q16. What is a closure in Python? How does it work?

**Answer:**

A closure is a function object that remembers bindings from enclosing lexical scopes, even after the outer function returns. Inner functions capture variables by reference; mutable objects can be updated, but rebinding the name in the inner function makes it local unless declared `nonlocal`.

Closures power decorators, factories, and callbacks. Late binding caveat: in loops, all closures may share the same variable unless you bind defaults (`lambda x=x: ...`).

```python
def make_multiplier(k: int):
    def inner(x: int) -> int:
        return k * x

    return inner


double = make_multiplier(2)
assert double(5) == 10


# Loop binding fix
fs = [(lambda i=i: i) for i in range(3)]
assert [f() for f in fs] == [0, 1, 2]
```

### Q17. Explain decorators in detail. How do you write a decorator with arguments?

**Answer:**

A decorator is a callable that takes a function (or class) and returns a replacement, usually wrapping it to add behavior. `@decorator` syntax applies `decorator` to the function object at **definition** time.

Parameterized decorators are implemented as a function returning the actual decorator: outer args configure, inner receives `f`, innermost returns wrapper. Preserve metadata with `functools.wraps`.

```python
from collections.abc import Callable
from functools import wraps
from typing import ParamSpec, TypeVar

P = ParamSpec("P")
R = TypeVar("R")


def retry(times: int) -> Callable[[Callable[P, R]], Callable[P, R]]:
    def decorator(f: Callable[P, R]) -> Callable[P, R]:
        @wraps(f)
        def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
            last: Exception | None = None
            for _ in range(times):
                try:
                    return f(*args, **kwargs)
                except Exception as e:  # noqa: BLE001 — demo only
                    last = e
            assert last is not None
            raise last

        return wrapper

    return decorator


@retry(3)
def flaky() -> str:
    return "ok"
```

### Q18. What is `functools.wraps` and why is it important?

**Answer:**

`wraps` copies `__module__`, `__name__`, `__qualname__`, `__doc__`, `__annotations__`, and `__wrapped__` from the original function onto the wrapper. Without it, stack traces, `help()`, and introspection show the wrapper’s name, breaking debugging and tooling.

It also sets `__wrapped__` for `inspect.signature` and similar utilities to traverse to the underlying callable.

```python
from functools import wraps


def shout(f):
    @wraps(f)
    def wrapper(*args, **kwargs):
        return f(*args, **kwargs).upper()

    return wrapper


@shout
def greet(name: str) -> str:
    """Say hi."""
    return f"hi {name}"


assert greet.__name__ == "greet"
assert greet.__doc__ == "Say hi."
```

### Q19. How do you chain multiple decorators?

**Answer:**

Decorators apply **bottom-up** nearest the `def`: `@a @b @c def f` is `f = a(b(c(f)))`. The innermost decorator sees the raw function first; each outer wraps the result of the inner.

Order matters for cross-cutting concerns: e.g., auth before caching, or logging outermost to capture total time including inner decorators.

```python
def trace(tag):
    def deco(f):
        def w(*a, **k):
            print(tag, "in")
            r = f(*a, **k)
            print(tag, "out")
            return r

        return w

    return deco


@trace("A")
@trace("B")
def work():
    print("body")


work()
# A in → B in → body → B out → A out (outer decorator runs first)
```

### Q20. Implement a retry decorator with exponential backoff.

**Answer:**

Exponential backoff spaces retries increasingly apart, reducing load on failing dependencies (network, databases). Combine with jitter to avoid thundering herds. Cap max delay and total attempts; log failures with context.

Production code should accept exceptions to catch, use monotonic clocks, and optionally integrate `tenacity` or similar libraries.

```python
from __future__ import annotations

import random
import time
from collections.abc import Callable
from functools import wraps
from typing import ParamSpec, TypeVar

P = ParamSpec("P")
T = TypeVar("T")


def retry_exponential(
    *,
    max_tries: int = 5,
    base: float = 0.1,
    max_delay: float = 10.0,
    jitter: float = 0.05,
) -> Callable[[Callable[P, T]], Callable[P, T]]:
    def decorator(fn: Callable[P, T]) -> Callable[P, T]:
        @wraps(fn)
        def wrapper(*args: P.args, **kwargs: P.kwargs) -> T:
            delay = base
            last: BaseException | None = None
            for attempt in range(max_tries):
                try:
                    return fn(*args, **kwargs)
                except Exception as e:  # noqa: BLE001 — narrow in real code
                    last = e
                    if attempt == max_tries - 1:
                        break
                    sleep_for = min(delay, max_delay) + random.uniform(0, jitter)
                    time.sleep(sleep_for)
                    delay *= 2
            assert last is not None
            raise last

        return wrapper

    return decorator


@retry_exponential(max_tries=4, base=0.05)
def call_external() -> str:
    return "ok"
```

### Q21. What is a class-based decorator?

**Answer:**

A class-based decorator implements `__call__` or uses `__get__` for descriptors. Stateful decorators map naturally to instances (`__init__` holds configuration). For functions, `__call__` wraps invocation; for methods, implement `__get__` to bind `self`.

Class decorators aid readability when the decorator maintains complex state or needs multiple methods.

```python
from collections.abc import Callable
from typing import Any


class CountCalls:
    def __init__(self, fn: Callable[..., Any]) -> None:
        self.fn = fn
        self.count = 0

    def __call__(self, *args: Any, **kwargs: Any) -> Any:
        self.count += 1
        return self.fn(*args, **kwargs)


@CountCalls
def add(a: int, b: int) -> int:
    return a + b


assert add(1, 2) == 3 and add.count == 1
```

### Q22. Explain `property` decorator and its getter/setter/deleter.

**Answer:**

`property` is a data descriptor providing computed attributes with optional validation. `@name.setter` and `@name.deleter` register additional aspects on the same property object.

Use properties to keep a simple attribute syntax while preserving invariants, lazy computation, or backward compatibility when refactoring plain attributes.

```python
class Temperature:
    def __init__(self, celsius: float) -> None:
        self._c = celsius

    @property
    def celsius(self) -> float:
        return self._c

    @celsius.setter
    def celsius(self, value: float) -> None:
        if value < -273.15:
            raise ValueError("below absolute zero")
        self._c = value

    @property
    def fahrenheit(self) -> float:
        return self._c * 9 / 5 + 32


t = Temperature(0)
assert t.fahrenheit == 32
```

### Q23. What is `functools.lru_cache`? How does it work?

**Answer:**

`lru_cache` memoizes a pure function’s results keyed by arguments (hashable), evicting least-recently-used entries when `maxsize` is set. It thread-safely caches in C for CPython and avoids repeated expensive work.

Use for idempotent calls with hashable args; avoid caching functions with unbounded distinct arguments or mutable defaults. `cache_info()` exposes hits/misses for tuning.

```python
from functools import lru_cache


@lru_cache(maxsize=256)
def fib(n: int) -> int:
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)


assert fib(40) == 102334155
print(fib.cache_info())
```

### Q24. Implement a timing decorator that logs execution time.

**Answer:**

Measure wall-clock time with `time.perf_counter()` for benchmarking. Log at `DEBUG` to avoid noise in production unless sampling; include function name and correlation IDs in real services.

For async functions, use a separate async wrapper or `inspect.iscoroutinefunction` to await properly.

```python
import logging
import time
from collections.abc import Callable
from functools import wraps
from typing import Any, ParamSpec, TypeVar

P = ParamSpec("P")
R = TypeVar("R")
log = logging.getLogger(__name__)


def timed(fn: Callable[P, R]) -> Callable[P, R]:
    @wraps(fn)
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        start = time.perf_counter()
        try:
            return fn(*args, **kwargs)
        finally:
            elapsed_ms = (time.perf_counter() - start) * 1000
            log.debug("%s took %.2f ms", fn.__qualname__, elapsed_ms)

    return wrapper


@timed
def work(n: int) -> int:
    return sum(range(n))
```

### Q25. What is `functools.singledispatch`? Give an example.

**Answer:**

`singledispatch` implements generic functions: one function name, behavior selected by the type of the **first** argument. Register implementations with `@func.register`. For methods, use `singledispatchmethod` (Python 3.8+).

This mirrors pattern matching / multimethods for open extension without giant `if isinstance` chains.

```python
from functools import singledispatch


@singledispatch
def serialize(obj: object) -> str:
    raise TypeError(f"unsupported {type(obj)!r}")


@serialize.register
def _(obj: int) -> str:
    return f"int:{obj}"


@serialize.register
def _(obj: dict) -> str:
    return "dict:" + ",".join(f"{k}={serialize(v)}" for k, v in obj.items())


assert serialize({"a": 1}) == "dict:a=int:1"
```

## Generators & Iterators

### Q26. What is the iterator protocol? Explain `__iter__` and `__next__`.

**Answer:**

Iterators expose `__next__` (raises `StopIteration` when exhausted) and are themselves iterable via `__iter__` returning `self`. An **iterable** implements `__iter__` returning a fresh iterator.

`for` loops call `iter()` then repeatedly `next()` until `StopIteration`. User-defined iterators must maintain internal state explicitly.

```python
class CountUp:
    def __init__(self, n: int) -> None:
        self._n = n
        self._i = 0

    def __iter__(self) -> "CountUp":
        return self

    def __next__(self) -> int:
        if self._i >= self._n:
            raise StopIteration
        self._i += 1
        return self._i - 1


assert list(CountUp(3)) == [0, 1, 2]
```

### Q27. What are generators and how do they differ from regular functions?

**Answer:**

Generator functions use `yield` to produce a sequence lazily, suspending between values and preserving stack state. They return a generator iterator object implementing the iterator protocol.

Unlike lists, generators are memory-efficient for large streams. They cannot be indexed or len’d without consuming.

```python
def squares(n: int):
    for i in range(n):
        yield i * i


g = squares(3)
assert next(g) == 0 and next(g) == 1
```

### Q28. What is `yield from` and when should you use it?

**Answer:**

`yield from subiter` delegates iteration to another iterable, forwarding `send`, `throw`, and `return` values (the sub-generator’s `return` becomes `StopIteration.value`). It simplifies flattening nested generators and composing pipelines.

Use when refactoring generators into smaller pieces or implementing recursive tree walks.

```python
def leaf():
    yield 1
    yield 2


def branch():
    yield from leaf()
    yield 3


assert list(branch()) == [1, 2, 3]
```

### Q29. Explain the difference between generator expressions and list comprehensions in terms of memory.

**Answer:**

List comprehensions build the **entire list** eagerly in memory. Generator expressions create an iterator that computes each item on demand—constant memory for the iterator object, streaming-friendly.

Choose genexprs for large or unbounded sequences; use lists when you need random access, length, or multiple passes without re-evaluation cost.

```python
import sys

lst = [x * x for x in range(10_000)]
gen = (x * x for x in range(10_000))
assert sys.getsizeof(lst) > sys.getsizeof(gen)
```

### Q30. How do you send values to a generator using `send()`?

**Answer:**

PEP 342 coroutine-style generators accept values from outside: `g.send(x)` resumes after `yield` and makes `yield` **expression** evaluate to `x`. The first call must be `next(g)` or `g.send(None)` to prime to the first `yield`.

This enables bidirectional communication before native `async`/`await` became standard for IO.

```python
def accumulator():
    total = 0
    while True:
        n = yield total
        if n is None:
            break
        total += n


acc = accumulator()
next(acc)  # prime
assert acc.send(2) == 2
assert acc.send(3) == 5
acc.send(None)
```

### Q31. What is `itertools`? Explain commonly used functions.

**Answer:**

`itertools` provides fast, memory-efficient iterator building blocks. `chain` flattens iterables; `cycle` repeats infinitely; `islice` slices iterators without materializing; `groupby` groups **consecutive** keys (sort first); `product`, `permutations`, `combinations` generate combinatorial sequences; `tee` splits iterators (uses memory—use carefully).

These primitives compose well with generator pipelines for ETL and parsing.

```python
from itertools import chain, groupby, islice

data = [("a", 1), ("a", 2), ("b", 3)]
groups = {k: list(v) for k, v in groupby(data, key=lambda x: x[0])}
assert groups["a"] == [("a", 1), ("a", 2)]

first_squares = list(islice((x * x for x in range(10**9)), 5))
assert first_squares == [0, 1, 4, 9, 16]
```

### Q32. How do you create an infinite iterator?

**Answer:**

Use `itertools.count`, `cycle`, `repeat`, or a generator with `while True`. Infinite iterators are valid but must be consumed with bounds (`islice`, `takewhile`) to avoid hanging.

They model streams, polling loops, and mathematical sequences.

```python
from itertools import count, islice

naturals = count(start=1, step=1)
```

### Q33. Explain coroutines vs generators.

**Answer:**

Historically, **generators** yielded values for iteration. **Generator-based coroutines** (PEP 342) used `yield`/`yield from` for cooperative multitasking. In modern Python (3.5+), **native coroutines** (`async def`) are the standard for asynchronous IO; they are distinct objects (`inspect.iscoroutinefunction`) and integrate with `asyncio`.

`async` coroutines are not iterators unless you explicitly `async for` over async iterables. Do not mix `yield` inside `async def` except for async generators (`async def` + `yield` = async generator).

```python
import asyncio


async def fetch(url: str) -> str:
    await asyncio.sleep(0)
    return f"body:{url}"


async def main() -> None:
    out = await fetch("https://ex")
    assert out.startswith("body:")


asyncio.run(main())
```

## Context Managers

### Q34. What is a context manager? How do you implement one using a class?

**Answer:**

Context managers encapsulate setup/teardown around a block (`with`), ensuring cleanup even on exceptions (similar to `try/finally`). Implement `__enter__` (return bound resource) and `__exit__(exc_type, exc, tb)` returning `True` to suppress exceptions.

They are ideal for files, locks, transactions, and temporary context changes.

```python
class Timer:
    def __enter__(self) -> "Timer":
        import time

        self._t0 = time.perf_counter()
        return self

    def __exit__(self, exc_type, exc, tb) -> None:
        import time

        self.elapsed = time.perf_counter() - self._t0


with Timer() as t:
    x = sum(range(1000))
assert t.elapsed >= 0
```

### Q35. How do you create a context manager using `contextlib.contextmanager`?

**Answer:**

`@contextmanager` turns a generator with a single `yield` into a context manager: code before `yield` runs on enter, code after runs on exit (including `finally` semantics). Exceptions propagate into the generator after `yield` where you can handle or re-raise.

Prefer this for lightweight resources; use classes when state or inheritance is complex.

```python
from contextlib import contextmanager


@contextmanager
def open_text(path: str):
    f = open(path, "w", encoding="utf-8")
    try:
        yield f
    finally:
        f.close()
```

### Q36. What is `contextlib.suppress`?

**Answer:**

`suppress(*exceptions)` is a context manager that catches and ignores listed exceptions inside the block, analogous to `try: ... except E: pass` but clearer and avoids empty `except` blocks.

Use for optional features or cleanup paths where failure is acceptable; do not blanket-suppress `Exception` in production without logging.

```python
from contextlib import suppress

with suppress(FileNotFoundError):
    open("/nonexistent", "rb").read()
```

### Q37. How do you handle multiple context managers?

**Answer:**

Python 3.1+ supports `with A() as a, B() as b:` which enters left-to-right and exits **reverse** order (like a stack). For dynamic lists, `contextlib.ExitStack` enters managers programmatically and guarantees exit in reverse.

Async code uses `async with` similarly; `AsyncExitStack` exists for dynamic async contexts.

```python
from contextlib import ExitStack, contextmanager


@contextmanager
def res(name: str):
    yield name


with ExitStack() as stack:
    a = stack.enter_context(res("a"))
    b = stack.enter_context(res("b"))
```

### Q38. Implement a custom database connection context manager.

**Answer:**

Wrap DB-API connections: commit on success, rollback on exception, always close. Use connection pools in production rather than raw connects per request. Thread/async pools differ by framework (`psycopg_pool`, SQLAlchemy `QueuePool`).

```python
from collections.abc import Iterator
from contextlib import contextmanager
from typing import Any


@contextmanager
def db_connection(dsn: str) -> Iterator[Any]:
    import psycopg

    conn = psycopg.connect(dsn)
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise
    finally:
        conn.close()
```

## Concurrency & Parallelism

### Q39. What is the GIL (Global Interpreter Lock)? How does it affect multithreading?

**Answer:**

The GIL is a mutex allowing only one thread to execute Python bytecode at a time per interpreter process (in CPython). It simplifies memory management (reference counting) but limits CPU-bound parallelism across threads.

Threads still help for **I/O-bound** work releasing the GIL around blocking syscalls and C extensions. For CPU-bound Python code, use **multiprocessing**, **C extensions**, or alternative interpreters (PyPy, sometimes nogil builds).

```python
import sys

# Standard CPython builds use a GIL; experimental free-threaded builds (3.13+)
# may expose sys._is_gil_enabled() — prefer measuring workload over introspection.
assert sys.implementation.name == "cpython"
```

### Q40. Explain the difference between threading, multiprocessing, and asyncio.

**Answer:**

**Threading** shares memory within a process; good for blocking I/O. **Multiprocessing** uses separate interpreters with separate memory; good for CPU-bound work, higher overhead. **Asyncio** cooperatively schedules tasks on one thread using `await` for non-blocking I/O—scalable for many connections, not for parallel CPU cores.

Choose based on workload: I/O concurrency vs CPU parallelism vs structured async network services.

```python
import asyncio


async def main():
    await asyncio.sleep(0)
```

---

### Q41. When should you use threading vs multiprocessing?

**Answer:**

Use **threading** when tasks spend most of their time waiting on I/O (network sockets, disk, databases) and you want lightweight concurrency with shared in-memory structures protected by locks or queues. The GIL is released around many blocking operations, so threads can overlap waiting time even though they do not parallelize pure Python bytecode.

Use **multiprocessing** when CPU-bound Python work dominates and you need multiple cores—each process has its own interpreter and GIL. Trade-offs include higher memory footprint, startup cost, and the need to pickle data sent across process boundaries (use shared memory or message queues for large payloads).

Avoid threads for tight numerical loops in pure Python; avoid processes for huge numbers of tiny tasks where scheduling overhead dominates. Hybrid models (asyncio + thread/process pools) are common in production services.

### Q42. What is `concurrent.futures`? Explain ThreadPoolExecutor and ProcessPoolExecutor.

**Answer:**

High-level APIs for submitting callables to thread or process pools. `Future` objects represent pending results; `as_completed` / `wait` help orchestration. `ProcessPoolExecutor` requires picklable top-level functions and data.

Use for batching independent tasks with a bounded worker count.

```python
from concurrent.futures import ProcessPoolExecutor, as_completed


def square(x: int) -> int:
    return x * x


with ProcessPoolExecutor(max_workers=4) as ex:
    futures = [ex.submit(square, i) for i in range(5)]
    results = [f.result() for f in as_completed(futures)]
assert sorted(results) == [0, 1, 4, 9, 16]
```

### Q43. What is asyncio? Explain async/await syntax.

**Answer:**

`asyncio` provides an event loop, transports, protocols, and task scheduling. `async def` defines a coroutine object; `await` suspends to the loop until the awaited awaitable completes, enabling concurrent I/O without threads.

It is cooperative: long CPU sections block the loop—offload with `asyncio.to_thread` or executors.

```python
import asyncio


async def fetch_one(i: int) -> int:
    await asyncio.sleep(0.01)
    return i


async def main():
    results = await asyncio.gather(*(fetch_one(i) for i in range(3)))
    assert results == [0, 1, 2]


asyncio.run(main())
```

### Q44. What are asyncio tasks and how do you manage them?

**Answer:**

Tasks wrap coroutines for concurrent scheduling: `asyncio.create_task` schedules immediately; `await task` retrieves result. `TaskGroup` (3.11+) manages structured concurrency—if one child fails, others cancel.

Use `asyncio.wait` with `FIRST_COMPLETED` for races; shield critical tasks with `asyncio.shield`.

```python
import asyncio


async def main():
    async with asyncio.TaskGroup() as tg:
        t1 = tg.create_task(asyncio.sleep(0, result=1))
        t2 = tg.create_task(asyncio.sleep(0, result=2))
    assert t1.result() == 1 and t2.result() == 2


asyncio.run(main())
```

### Q45. Explain `asyncio.gather()` vs `asyncio.wait()`.

**Answer:**

`gather` awaits multiple awaitables and returns results in order (or raises `ExceptionGroup` / first error depending on `return_exceptions`). `wait` returns two sets of done/pending tasks with flexible `return_when` policies (`ALL_COMPLETED`, `FIRST_COMPLETED`, `FIRST_EXCEPTION`).

Use `gather` for simple fan-out; `wait` for custom completion logic and timeouts per task.

```python
import asyncio


async def main():
    r1, r2 = await asyncio.gather(
        asyncio.sleep(0.01, result="a"),
        asyncio.sleep(0.01, result="b"),
    )
    assert (r1, r2) == ("a", "b")


asyncio.run(main())
```

### Q46. What is `aiohttp`? How does it compare to `requests`?

**Answer:**

`aiohttp` is an async HTTP client/server built on `asyncio`. Compared to **requests** (sync), it scales to many concurrent connections without thread explosion, but requires async/await throughout the call stack.

For simple scripts, `requests` is simpler; for high-concurrency services, `aiohttp` or `httpx` async mode fits better.

### Q47. How do you handle shared state in multithreaded programs?

**Answer:**

Prefer message passing (`queue.Queue`) over shared mutable state. When sharing, protect invariants with `threading.Lock`, `RLock`, or higher-level primitives; use immutable snapshots where possible.

For read-heavy data, consider `copy-on-write` patterns or external stores (Redis) with atomic operations.

```python
import threading
from threading import Lock

counter = 0
lock = Lock()


def bump():
    global counter
    for _ in range(1000):
        with lock:
            counter += 1


threads = [threading.Thread(target=bump) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
assert counter == 4000
```

### Q48. What are locks, semaphores, and events in threading?

**Answer:**

A **Lock** grants exclusive access to one thread at a time. A **Semaphore** limits N concurrent holders—useful for resource pools. An **Event** signals between threads (`wait` blocks until `set`).

`RLock` allows reentrant acquisition by the same thread. Choose the minimal synchronization primitive needed.

```python
import threading

sem = threading.Semaphore(2)


def work(i: int):
    with sem:
        print("enter", i)


threads = [threading.Thread(target=work, args=(i,)) for i in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

### Q49. What is a deadlock and how do you prevent it?

**Answer:**

Deadlock occurs when two or more threads (or tasks) each hold a resource and wait for another resource held by someone else, forming a cycle so no progress is possible. Classic example: thread 1 locks `A` then waits on `B` while thread 2 locks `B` then waits on `A`.

Prevention strategies include: acquiring multiple locks in a single global order across the codebase; using `Lock.acquire(timeout=...)` and backing off; minimizing lock scope and never holding locks across I/O or RPC calls; preferring higher-level primitives (`queue.Queue`, `asyncio`-friendly patterns) over ad-hoc lock nests; and using deadlock detection in databases where applicable.

In async code, blocking the event loop while waiting on a threading lock can stall all coroutines—use `asyncio.Lock` within the same loop or offload blocking work with `asyncio.to_thread`.

### Q50. Explain the Producer-Consumer pattern with queues.

**Answer:**

Producers enqueue work; consumers dequeue and process. `queue.Queue` is thread-safe and blocks with optional maxsize for backpressure. For asyncio, use `asyncio.Queue`.

This decouples rates and simplifies shutdown via `sentinel` values or `join` with `task_done`.

```python
import queue
import threading

q: queue.Queue[int | None] = queue.Queue()


def producer():
    for i in range(3):
        q.put(i)
    q.put(None)


def consumer():
    while True:
        item = q.get()
        if item is None:
            q.task_done()
            break
        print(item)
        q.task_done()


threading.Thread(target=producer).start()
threading.Thread(target=consumer).start()
```

## Python Internals & Memory

### Q51. How does Python's memory allocation work internally?

**Answer:**

CPython layers memory management from the operating system down to per-type optimizations. Large allocations may go straight to `malloc`; **pymalloc** handles small objects (typically ≤512 bytes on 64-bit builds) using arenas divided into pools and blocks to reduce fragmentation and speed allocation/deallocation of short-lived objects common in Python programs.

Immutable and numeric types may use specialized allocators and internal free lists. Reference counting triggers immediate deallocation when possible; the cyclic GC handles reference cycles separately.

Understanding this stack explains why reducing allocation churn (reuse buffers, use `__slots__`, vectorize with NumPy) wins in hot paths, and why micro-benchmarks can be noisy due to allocator warmup and OS paging.

---

### Q52. What is reference counting and how does Python's garbage collector work?

**Answer:**

CPython primarily uses **reference counting**: each object stores a count; when it hits zero, memory is reclaimed immediately. Cyclic references (e.g., mutually pointing dicts) require a **cycle detector** (generational GC) that runs periodically to break unreachable cycles.

You can tune thresholds with `gc.set_threshold`; `gc.disable()` is rarely advisable in production.

```python
import gc

gc.collect()
print(gc.get_stats())
```

---

### Q53. What are weak references and when should you use them?

**Answer:**

`weakref.ref` points to an object without increasing its refcount, allowing GC when only weak refs remain. Useful for caches, observers, and avoiding cycles without `gc` involvement.

Not all objects are weak-referenceable (e.g., some builtins).

```python
import weakref


class Node:
    pass


a = Node()
b = Node()
a.other = weakref.ref(b)
assert a.other() is b
del b
assert a.other() is None
```

### Q54. Explain Python's string interning mechanism.

**Answer:**

CPython interns some strings (identifiers, small compile-time constants) so identical strings share one object (`is` may be true). Do not rely on interning for correctness—use `==` for value equality.

Interning reduces memory and speeds dict lookups for repeated keys in some scenarios.

```python
a = "hello"
b = "hello"
assert a == b
# Interning may make `a is b` true for some literals; not guaranteed for all constructions.
```

### Q55. What is the `__dict__` attribute and how does attribute lookup work?

**Answer:**

Instances store attributes in `__dict__` (unless `__slots__`). Lookup: data descriptors on the class, then instance `__dict__`, then non-data descriptors / functions, then class `__dict__` along MRO.

`__getattr__` runs only after normal lookup fails; `__getattribute__` intercepts everything.

```python
class C:
    class_var = 1

    def __init__(self):
        self.inst = 2


c = C()
assert c.__dict__ == {"inst": 2}
```

---

### Q56. How does Python's import system work internally?

**Answer:**

When `import foo` runs, the import machinery consults `sys.meta_path` importers (including the built-in finder for frozen modules and `PathFinder` for filesystem paths), locates `foo.py` or `foo/__init__.py`, compiles to bytecode if needed, and executes the module body **once**. Subsequent imports retrieve the module object from `sys.modules`.

Namespace packages (PEP 420) can span multiple directories without an `__init__.py`. Import hooks and `importlib.util` allow dynamic loading for plugins. Relative imports inside packages depend on `__package__` and the importing context.

Circular imports often fail with `ImportError` or partial initialization when module A imports B which imports A before names exist—fix by restructuring dependencies, lazy imports inside functions, or moving shared symbols to a third module.

### Q57. What is bytecode? How do you inspect it using `dis` module?

**Answer:**

CPython compiles source to stack-based bytecode executed by the evaluation loop. `dis.dis` prints opcodes for a function, helping understand performance characteristics and optimization limits.

JITs (PyPy) differ; this is CPython-specific.

```python
import dis


def add(a, b):
    return a + b


dis.dis(add)
```

### Q58. What is the difference between CPython, PyPy, Cython, and Jython?

**Answer:**

**CPython** is the reference interpreter written in C; it defines the language semantics most developers expect and has the broadest ecosystem for C extensions (NumPy, many database drivers).

**PyPy** implements Python with an RPython toolchain and a tracing JIT—often faster for long-running pure-Python workloads, but can lag on C-extension compatibility and per-release support.

**Cython** is a distinct language/compiler that accepts Python-like syntax annotated with C types to generate C extensions—use to accelerate tight loops or wrap C libraries, not as a drop-in interpreter.

**Jython** targets the JVM for Java interop; **IronPython** targets .NET similarly—both trade CPython C-API compatibility for platform integration.

Pick CPython for default deployments; PyPy after validating extension support; Cython for targeted extension builds; JVM/.NET stacks when those runtimes are fixed requirements.

## Error Handling & Logging

### Q59. How do you create a custom exception hierarchy?

**Answer:**

Subclass `Exception` (not `BaseException`) for recoverable errors; group domain errors under a base like `AppError` for selective handling. Add attributes for context (codes, fields).

Keep hierarchies shallow; document which exceptions API boundaries raise.

```python
class AppError(Exception):
    pass


class ValidationError(AppError):
    def __init__(self, field: str, message: str) -> None:
        super().__init__(f"{field}: {message}")
        self.field = field
```

### Q60. What are exception groups and `except*` (Python 3.11+)?

**Answer:**

`ExceptionGroup` bundles multiple exceptions from concurrent operations (`asyncio.TaskGroup`, `asyncio.gather` with multiple). `except*` matches sub-exceptions by type inside the group, enabling partial handling—unlike a single `except` that matches the group as a whole.

This improves structured error reporting in concurrent code.

```python
try:
    raise ExceptionGroup(
        "multi",
        [ValueError("a"), TypeError("b")],
    )
except* ValueError as eg:
    print("handled values", eg.exceptions)
```

### Q61. Explain the `logging` module. How do you configure it properly?

**Answer:**

Use loggers hierarchically (`logging.getLogger(__name__)`), handlers (stdout, files, syslog), formatters, and levels. Configure once at startup (`dictConfig` or `basicConfig` only for small scripts). Include `exc_info=True` on errors.

In services, use JSON formatters and correlation IDs; avoid `print` for operational logs.

```python
import logging.config

logging.config.dictConfig(
    {
        "version": 1,
        "handlers": {
            "console": {
                "class": "logging.StreamHandler",
                "formatter": "std",
            }
        },
        "formatters": {
            "std": {"format": "%(levelname)s %(name)s %(message)s"}
        },
        "root": {"handlers": ["console"], "level": "INFO"},
    }
)
log = logging.getLogger(__name__)
log.info("started")
```

### Q62. What is structured logging and why is it important?

**Answer:**

Structured logs emit machine-parseable fields (JSON) instead of free-form strings, enabling queries, aggregations, and alerts in ELK/Datadog/Splunk. Include stable keys: `request_id`, `user_id`, `error_code`.

Pass `extra={...}` to `logging` or use `structlog` for ergonomic APIs.


### Q63. How do you handle exceptions in async code?

**Answer:**

Awaited coroutines propagate exceptions to the caller. With `gather`, use `return_exceptions=True` to collect errors. `TaskGroup` raises `ExceptionGroup` on child failures—handle with `except*` or inspect `exc.exceptions`.

Never swallow `asyncio.CancelledError` in library code without re-raising.

```python
import asyncio


async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            tg.create_task(asyncio.sleep(0))
```

### Q69. What is TDD (Test Driven Development)?

**Answer:**

Test-Driven Development is a discipline where you write an automated test that fails first, write the smallest amount of production code to make it pass, then refactor while keeping tests green. The loop (red → green → refactor) tightens feedback and forces testable seams: dependency injection, pure functions, and clear module boundaries emerge because untestable code is painful early.

TDD is not a silver bullet: poor tests can ossify bad APIs, and exploratory spikes sometimes need freedom before tests. Many teams use TDD selectively—for example on core domain logic—while relying on broader integration tests for boundaries. Pairing and code review keep tests maintainable.

### Q71. What is property-based testing with Hypothesis?

**Answer:**

```python
from hypothesis import given
```

## Data Structures & Algorithms

### Q73. Explain `collections` module - Counter, defaultdict, OrderedDict, deque, namedtuple.

**Answer:**

`Counter` counts hashable items with multiset operations. `defaultdict` supplies factory for missing keys—avoid `KeyError`. `OrderedDict` remembers insertion order (dicts do this since 3.7+ as implementation detail; `OrderedDict` adds `move_to_end`). `deque` is O(1) append/pop from both ends. `namedtuple`/`NamedTuple` gives tuple subclasses with named fields.

```python
from collections import Counter, defaultdict, deque

cnt = Counter("abracadabra")
assert cnt["a"] == 5

dd = defaultdict(list)
dd["k"].append(1)

dq = deque([1, 2])
dq.appendleft(0)
assert list(dq) == [0, 1, 2]
```

### Q74. What is the time complexity of common Python operations (list, dict, set)?

**Answer:**

Average case: dict/set get/insert/delete **O(1)** amortized; list index **O(1)**; list append **O(1)** amortized; list insert/delete middle **O(n)**. Sorting is **O(n log n)**.

Worst-case hash collisions can degrade to **O(n)**—mitigate with good `__hash__` and randomized hash seeds (CPython default).

### Q75. How do you implement a stack and queue in Python?

**Answer:**

Use `list` as stack with `append`/`pop`. For queues, prefer `collections.deque` for O(1) pops from left; `list.pop(0)` is O(n).

For priority queues, `heapq`.

```python
from collections import deque

stack: list[int] = []
stack.append(1)
stack.pop()

q: deque[int] = deque()
q.append(1)
q.popleft()
```

### Q76. What is `heapq` and how do you use it?

**Answer:**

`heapq` implements a binary min-heap in-place on a list. `heappush`/`heappop` maintain invariant. For max-heap or custom ordering, negate values or push tuples `(priority, item)`.

`heapq.merge` merges sorted iterables lazily.

```python
import heapq

h: list[int] = []
heapq.heappush(h, 3)
heapq.heappush(h, 1)
assert heapq.heappop(h) == 1
```

### Q77. Explain `bisect` module for binary search.

**Answer:**

`bisect_left` / `bisect_right` find insertion points in **sorted** lists maintaining order. Useful for ranges, schedules, and static sorted data.

Binary search is O(log n) but list insertion is O(n)—`bisect` for read-heavy workloads.

```python
import bisect

xs = [1, 3, 3, 5]
i = bisect.bisect_left(xs, 3)
assert i == 1
```

### Q78. How do you implement a custom hash for objects?

**Answer:**

Define `__hash__` consistent with `__eq__`: equal objects must have equal hashes. If `__eq__` is customized, often set `__hash__ = None` for mutables to prevent dict corruption.

Immutable value objects should hash components immutably.

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class Point:
    x: int
    y: int

    def __hash__(self) -> int:
        return hash((self.x, self.y))
```

## Database & ORM

### Q79. How do you connect to databases using Python? Explain DB-API 2.0.

**Answer:**

PEP 249 defines DB-API 2.0: `connect()`, `Connection`, `Cursor`, parameter styles (`qmark`, `numeric`, `named`, etc.), exceptions hierarchy, and transaction semantics (`commit`/`rollback`). Drivers like `psycopg`, `sqlite3`, `mysqlclient` comply.

Always use bound parameters—never string interpolation for SQL values.

```python
import sqlite3

with sqlite3.connect(":memory:") as conn:
    conn.execute("CREATE TABLE t (id INTEGER PRIMARY KEY, name TEXT)")
    conn.execute("INSERT INTO t (name) VALUES (?)", ("ada",))
    row = conn.execute("SELECT name FROM t WHERE id = ?", (1,)).fetchone()
    assert row[0] == "ada"
```

### Q80. What is SQLAlchemy? Explain Session, Engine, and ORM patterns.

**Answer:**

SQLAlchemy Core provides composable SQL expressions; ORM maps classes to tables with `Session` managing unit-of-work (identity map, dirty tracking). `Engine` pools connections and executes SQL.

Use `sessionmaker` for sessions; prefer explicit transactions and `expire_on_commit=False` when returning detached DTOs.

```python
from sqlalchemy import Integer, String, create_engine, select
from sqlalchemy.orm import DeclarativeBase, Mapped, Session, mapped_column


class Base(DeclarativeBase):
    pass


class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    name: Mapped[str] = mapped_column(String(64))


engine = create_engine("sqlite:///:memory:", echo=False)
Base.metadata.create_all(engine)

with Session(engine) as session:
    session.add(User(name="ada"))
    session.commit()
    u = session.scalars(select(User).where(User.name == "ada")).one()
    assert u.name == "ada"
```

### Q81. How do you handle database migrations?

**Answer:**

Schema migrations should be versioned, repeatable, and applied in order through the same pipeline that deploys application code. For SQLAlchemy, **Alembic** autogenerates revisions from model metadata; Django ships its own migration framework; teams using raw SQL sometimes adopt **Flyway** or **Liquibase** for vendor-neutral scripts.

Operational practices matter: review autogenerated diffs for destructive changes; prefer additive changes (new columns nullable first, backfill, then constrain) to avoid long locks; test up and down migrations against production-like data volumes; and take backups or use online DDL features when supported.

Never rewrite migration history that has already run in shared environments—append a corrective migration instead so all environments stay convergent.

### Q82. What is connection pooling and why is it important?

**Answer:**

Pools reuse TCP connections and DB sessions, amortizing handshake/auth costs and bounding concurrency. SQLAlchemy `QueuePool` manages checkout/checkin; serverless may need smaller pools or external poolers (PgBouncer).

Misconfigured pools cause exhaustion or too many idle connections.

```python
from sqlalchemy import create_engine

engine = create_engine(
    "postgresql+psycopg://user:pass@host/db",
    pool_size=10,
    max_overflow=20,
    pool_pre_ping=True,
)
```

### Q83. How do you prevent SQL injection in Python?

**Answer:**

SQL injection happens when untrusted input is interpreted as SQL syntax. The fix is **parameterized queries**: placeholders (`?`, `%s`, or named params) bind values separately from the statement so the database never treats user data as keywords or operators. Every major Python DB driver supports this via DB-API 2.0.

ORMs (SQLAlchemy, Django ORM) generate parameterized statements when you use their expression APIs; danger returns when developers build SQL with f-strings or `%` formatting for convenience—especially for `ORDER BY` columns or table names where you must validate against an allowlist, not free text.

Defense in depth includes least-privilege DB roles, stored procedures where appropriate, and Web Application Firewalls for edge cases—but correct parameterization is the primary control.

## Web Development & APIs

### Q84. Explain WSGI and ASGI. What is the difference?

**Answer:**

**WSGI** (PEP 3333) standardizes how synchronous Python web apps plug into servers: a callable `app(environ, start_response)` handles one request at a time per worker process/thread. It maps cleanly to traditional HTTP request/response cycles and underpins Gunicorn+uWSGI deployments with Flask or Django’s synchronous stack.

**ASGI** generalizes this to async: an async callable receives `scope`, `receive`, and `send`—enabling HTTP, WebSockets, HTTP/2 features, and long-lived streams in one process. Servers like Uvicorn and Hypercorn drive Starlette/FastAPI and Django’s async views.

Choose ASGI when you need high-concurrency IO, WebSockets, or async middleware; WSGI remains simpler for purely synchronous workloads and has a long deployment history—though many teams standardize on ASGI for future flexibility.

---

### Q85. What is Flask vs FastAPI vs Django? When to use each?

**Answer:**

**Django** ships an opinionated stack: ORM, migrations, admin, auth, forms, and templating—excellent for content-heavy sites, internal tools, and teams that want conventions over assembly. You trade flexibility in low-level HTTP handling for productivity and a vast plugin ecosystem.

**Flask** is a microframework: routing and WSGI integration are core; everything else is extensions. It fits small services, prototypes, and teams that want full control over components and synchronous simplicity (with async extensions available).

**FastAPI** (ASGI) emphasizes type-annotated request/response models, dependency injection, and automatic OpenAPI—strong for JSON APIs, ML model serving, and async IO-heavy workloads. It pairs well with Pydantic v2 validation.

Choose Django for full-stack CRUD apps; Flask for lightweight services; FastAPI for async-first APIs and rapid typed contracts—always validate operational needs (ORM depth, auth, deployment) before picking.

### Q86. How do you implement REST APIs with proper error handling?

**Answer:**

Return consistent JSON error bodies with `code`, `message`, `details`, and HTTP status semantics (400 validation, 401 auth, 403 forbidden, 404, 409 conflict, 500). Map domain exceptions to handlers; log server errors with trace IDs; never leak stack traces to clients in production.

Use RFC 7807 `problem+json` for structured errors when appropriate.

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()


class ErrorBody(BaseModel):
    code: str
    message: str


@app.get("/items/{item_id}")
def read_item(item_id: str):
    if not item_id.isdigit():
        raise HTTPException(status_code=400, detail={"code": "bad_id", "message": "invalid"})
    return {"id": item_id}
```

### Q87. What are middleware in web frameworks?

**Answer:**

Middleware wraps request/response processing: logging, auth, CORS, gzip, tracing. In ASGI, middleware is an async callable wrapping inner apps; order matters.

Keep middleware pure and fast; offload heavy work to background tasks.

```python
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response


class AddHeaderMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        response: Response = await call_next(request)
        response.headers["X-Frame-Options"] = "DENY"
        return response
```

### Q88. How do you handle authentication and authorization in Python web apps?

**Answer:**

**Authentication** establishes *who* the caller is: session cookies with server-side storage, signed tokens (JWTs/PASETO), or delegated login (OAuth2/OIDC). Prefer established libraries (Authlib, Django auth, FastAPI security utilities) over rolling crypto.

**Authorization** decides *what* they may do: role-based (RBAC), attribute-based (ABAC), or resource-specific policies. Enforce checks at a single layer (dependencies/middleware) and again on sensitive operations (defense in depth).

Operational hygiene: store secrets in vaults or sealed env injection; rotate signing keys; scope JWT claims narrowly; use HttpOnly/SameSite cookies for browser sessions to reduce XSS token theft; audit logs for admin actions.

### Q89. What is serialization and deserialization? Explain Pydantic.

**Answer:**

Serialization converts objects to wire formats (JSON); deserialization parses and validates inbound data. **Pydantic v2** uses Rust core for fast validation, coercion, and schema generation—ideal for API boundaries.

Define models with types; catch `ValidationError` and return 422 responses.

```python
from pydantic import BaseModel, Field, ValidationError
```

### Q91. What are generic types in Python?

**Answer:**

Generics parameterize containers (`list[int]`, `dict[str, float]`). User-defined generics subclass `Generic[T]` or use class syntax in 3.12+ (`class Box[T]: ...`). `TypeVar` bounds and variance enable precise static checking.

```python
class Stack[T]:
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()
```

### Q93. Explain `Protocol` classes and structural subtyping.

**Answer:**

`Protocol` defines interfaces structurally: a class satisfies the protocol if it has the required methods without inheritance (static duck typing). `runtime_checkable` enables `isinstance` checks.

Use for test doubles and decoupling from concrete classes.

```python
from typing import Protocol, runtime_checkable


@runtime_checkable
class Readable(Protocol):
    def read(self, n: int = -1) -> bytes: ...


class MemoryReadable:
    def read(self, n: int = -1) -> bytes:
        return b"data"


assert isinstance(MemoryReadable(), Readable)
```

## Performance & Optimization

### Q97. How do you profile Python code? Explain `cProfile` and `line_profiler`.

**Answer:**

`cProfile` records function call counts and cumulative time—use `python -m cProfile -s cumtime script.py` or programmatically. **line_profiler** (`kernprof -l -v`) annotates line-by-line hot paths.

Profile representative workloads; optimize only proven bottlenecks.

```python
import cProfile
import pstats


def main():
    sum(range(10_000))


profiler = cProfile.Profile()
profiler.enable()
main()
profiler.disable()
stats = pstats.Stats(profiler).sort_stats("cumtime")
stats.print_stats(10)
```

### Q98. What are common performance pitfalls in Python and how do you avoid them?

**Answer:**

Accidental O(n²) patterns (string concatenation in loops, repeated `list` scans), excessive attribute lookup in tight loops, GIL-bound CPU in threads, sync I/O in async code, and N+1 ORM queries. Mitigate with better algorithms, local variables, `multiprocessing`, async drivers, and eager/joined loads.

Measure before micro-optimizing; often IO/network dominates.

### Q99. When and how should you use NumPy over pure Python?

**Answer:**

Use **NumPy** for numeric arrays, vectorized math, and interoperability with SciPy/pandas—C-level loops and SIMD beat pure Python loops by orders of magnitude. Avoid Python object arrays; prefer contiguous dtypes and broadcasting.

For small `n`, pure Python may be faster due to array setup overhead.

```python
import numpy as np

a = np.arange(1_000_000, dtype=np.float64)
assert float((a * a).sum()) > 0
```

### Q100. What is Cython and when should you use it for performance?

**Answer:**

Cython translates Python-like code to C/Cython extensions with optional static typing and `cdef` for C-speed loops and C library calls. Use when profiling shows hot loops unsuitable for NumPy vectorization or when wrapping C libraries.

Trade-offs: build complexity, distribution of wheels per platform, maintenance burden—prefer algorithmic fixes and NumPy first.

```cython
# example.pyx - illustrative
def sum_ints(long n):
    cdef long i, s = 0
    for i in range(n):
        s += i
    return s
```
