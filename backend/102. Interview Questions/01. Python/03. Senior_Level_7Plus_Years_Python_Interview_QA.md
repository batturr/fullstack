## Python Internals & CPython Deep Dive

### Q1. Explain how CPython's evaluation loop works. What is the role of `ceval.c`?

**Answer:**

CPython's evaluation loop is the heart of the interpreter, implemented in `Python/ceval.c`. When Python source is compiled to bytecode, the evaluation loop (function `_PyEval_EvalFrameDefault`) fetches opcodes from the code object's `co_code` (or `co_code_adaptive` in 3.11+), decodes arguments, and dispatches to handler code for each instruction.

The loop maintains a **value stack** for operands, a **frame object** holding locals, globals, and execution state, and interacts with the **GIL** to allow thread switching at safe points (after every N bytecodes, or on blocking calls). Since Python 3.11, the **adaptive interpreter** (PEP 659) specializes frequently executed bytecodes at runtime — for example, `LOAD_ATTR` may become `LOAD_ATTR_INSTANCE_VALUE` when the attribute layout is predictable, skipping dictionary lookups.

Key concepts for interviews:
- **Opcode dispatch**: Traditionally a giant `switch` statement; computed goto (`DISPATCH()` macro) on supported compilers for ~15-20% speedup.
- **Frame objects**: In 3.11+, frames are allocated on the C stack rather than the heap for efficiency; `sys._getframe()` materializes them lazily.
- **Quickening**: Bytecode is rewritten in place as type patterns stabilize; inline caches store metadata for specialization.
- **Signal handling**: Checked between opcodes via pending calls mechanism (`_Py_CheckFunctionResult`).

Understanding `ceval.c` helps explain why certain Python patterns are slower (e.g., heavy attribute access vs local variables) and why CPython-specific optimizations (like the adaptive specializer) can make a measurable difference.

```python
import dis
import sys

def example(x):
    return x.upper()

dis.dis(example)
print(f"Python {sys.version}")
```

### Q2. What is PEP 659 (Specializing Adaptive Interpreter)? How does it improve performance in Python 3.11+?

**Answer:**

PEP 659 introduced a mechanism where CPython replaces generic bytecode instructions with **type-specialized versions** after observing runtime behavior. This is a form of inline caching inspired by techniques in V8 and LuaJIT, but applied conservatively to maintain CPython's simplicity.

When a generic instruction like `BINARY_OP` executes, it checks if the operands match a fast path (e.g., both `int`). If so, subsequent executions use `BINARY_OP_ADD_INT`, which bypasses the full `__add__` protocol. If the specialization fails (type mismatch), it "de-optimizes" back to the generic form with a counter before retrying.

Key mechanisms:
- **Quickening**: Bytecode is copied to a mutable form on first execution; the original stays for `dis`/tracing.
- **Inline caches**: Extra bytes after specialized opcodes store type version tags, attribute offsets, or function pointers.
- **Adaptive warmup**: Instructions start generic, specialize after a threshold of consistent type observations, and de-specialize with backoff on mismatches.

Performance impact: Python 3.11 was roughly 10-25% faster than 3.10 on common benchmarks (pyperformance), with 3.12-3.13 continuing incremental gains. The approach is complementary to potential future JIT compilation (PEP 744 in 3.13+).

For seniors, the key insight is that **stable type patterns** in hot code benefit most—polymorphic call sites with varying types will not specialize effectively.

### Q3. Explain the CPython object model. What is `PyObject`, `ob_refcnt`, and `ob_type`?

**Answer:**

Every Python object in CPython is a C struct beginning with `PyObject_HEAD`, which contains two fields: **`ob_refcnt`** (reference count, `Py_ssize_t`) and **`ob_type`** (pointer to the `PyTypeObject` describing the object's type). Variable-size objects (lists, tuples) use `PyObject_VAR_HEAD`, adding `ob_size`.

The type object (`ob_type`) is itself a `PyObject` whose type is `PyType_Type` (i.e., `type`). This recursive structure bottoms out at `type` being its own metatype. The type object holds **tp_slots**: function pointers for protocols (`tp_hash`, `tp_richcompare`, `tp_as_number`, `tp_as_sequence`, etc.). When Python code calls `len(x)`, CPython invokes `x->ob_type->tp_as_sequence->sq_length` at the C level.

Important details:
- **Small integer caching**: Integers -5 to 256 are pre-allocated singletons; `id()` reflects memory address, so `a = 5; b = 5; a is b` is True due to caching, not interning.
- **Type flags**: `tp_flags` encode capabilities (GC trackable, `await`-able, immutable). Checking flags is O(1).
- **Immortal objects** (PEP 683, Python 3.12): `None`, `True`, `False`, small ints have `ob_refcnt` set to a special value that never decrements—eliminating refcount contention for the free-threaded build.

Understanding the object model explains why Python operations have inherent overhead (every operation goes through type dispatch) and why C extensions can be faster (they skip the protocol when types are known).

### Q4. What is PEP 703 (Free-Threaded CPython / no-GIL)? What are its implications?

**Answer:**

PEP 703 proposes making the GIL optional in CPython, allowing true multi-threaded parallelism for CPU-bound Python code. An experimental free-threaded build was introduced in Python 3.13 behind the `--disable-gil` build flag (later exposed as `python3.13t`).

The implementation replaces the GIL's protection with finer-grained mechanisms:
- **Biased reference counting**: Objects have a "biasing thread" that can increment/decrement refcounts without atomic operations; other threads use atomic ops. This keeps the common single-threaded case fast.
- **Immortal objects** (PEP 683): Shared constants skip refcounting entirely.
- **Per-object locks**: Deferred reference counting for container objects; lightweight mutexes on dicts and lists for concurrent access.
- **Stop-the-world GC**: Cyclic GC pauses all threads briefly, similar to generational collectors in Java/.NET.
- **Thread-safe allocator**: `mimalloc` replaces pymalloc for thread-safe small-object allocation.

Implications for senior engineers:
- **C extensions** must be audited: any code relying on the GIL for thread safety (common in NumPy, database drivers) needs explicit locking or adaptation.
- **Performance tradeoffs**: Single-threaded code may be ~5-10% slower due to atomic operations overhead; multi-threaded CPU-bound code can scale across cores.
- **Migration path**: The GIL remains the default; free-threaded is opt-in per interpreter build. Expect a multi-year transition with dual support.
- **Library ecosystem**: PyPI packages will need to declare GIL compatibility; wheels may ship both variants.

This is the most significant architectural change in CPython's history and will reshape how Python handles CPU parallelism.

### Q5. How does CPython's garbage collector handle reference cycles? Explain the generational approach.

**Answer:**

CPython's primary memory reclamation is **reference counting**: when `ob_refcnt` reaches zero, the object is deallocated immediately. However, reference cycles (A→B→A) prevent counts from reaching zero. The **cyclic garbage collector** (module `gc`) breaks these cycles using a **generational** strategy.

Three generations (0, 1, 2) partition tracked container objects by age. New objects start in generation 0. When allocation count minus deallocation count exceeds a threshold (default 700 for gen 0), a collection runs. Surviving objects are promoted to the next generation. Higher generations are collected less frequently (triggered when lower-gen collections exceed their thresholds: 10 for gen 1 and 2).

The collection algorithm:
1. **Build a reference graph** among tracked objects in the target generation (and all younger generations).
2. **Compute tentative reference counts** by subtracting internal references — any object with a non-zero external reference count is **reachable** from outside the cycle.
3. **Move reachable objects** to the next generation (they survive).
4. **Remaining objects** are unreachable — invoke `tp_finalize` (finalizers), then deallocate. If finalizers resurrect objects, they are moved to a special list and not collected.

Key details for seniors:
- **`__del__` and finalizers**: Objects with `__del__` in cycles used to be uncollectable (pre-3.4); PEP 442 fixed this with safe finalization ordering.
- **`gc.freeze()`** (3.7+): Moves all tracked objects to a permanent generation, useful before `fork()` to avoid copy-on-write refcount churn.
- **Performance**: GC pauses are proportional to the number of tracked objects in the collected generation. In latency-sensitive services, tune thresholds or disable GC temporarily for batch operations.

```python
import gc

gc.set_threshold(700, 10, 10)
print(gc.get_stats())
gc.collect()
```

### Q6. What are Python's `__init_subclass__` and `__set_name__` hooks? How do they reduce the need for metaclasses?

**Answer:**

**`__init_subclass__`** (PEP 487, Python 3.6) is called on the **parent class** whenever a new subclass is created. It receives keyword arguments passed in the class statement and can validate, register, or modify the subclass. This replaces many metaclass use cases — plugin registration, interface enforcement, and automatic attribute injection — without the complexity of `type.__new__`.

**`__set_name__`** (also PEP 487) is called on descriptor objects when the owning class is created, passing the owner class and the attribute name. This lets descriptors discover their name automatically instead of requiring redundant declarations like `field = Field("field")`.

Together, these hooks cover 80%+ of common metaclass patterns with simpler, composable code:

```python
class Serializable:
    _registry: dict[str, type] = {}

    def __init_subclass__(cls, *, type_key: str = "", **kwargs):
        super().__init_subclass__(**kwargs)
        if type_key:
            Serializable._registry[type_key] = cls


class Validated:
    def __init__(self, *, min_val: float = float("-inf")):
        self._min = min_val
        self._name = ""

    def __set_name__(self, owner, name):
        self._name = name

    def __set__(self, obj, value):
        if value < self._min:
            raise ValueError(f"{self._name} must be >= {self._min}")
        obj.__dict__[self._name] = value

    def __get__(self, obj, owner=None):
        if obj is None:
            return self
        return obj.__dict__.get(self._name)


class Temperature(Serializable, type_key="temp"):
    kelvin = Validated(min_val=0)

    def __init__(self, kelvin: float):
        self.kelvin = kelvin


t = Temperature(300)
assert Serializable._registry["temp"] is Temperature
assert t.kelvin == 300
```

When to still use metaclasses: when you need to control `__new__` (class allocation), prevent certain inheritance patterns, or deeply modify the namespace before class creation (e.g., `__prepare__` for ordered namespaces or DSL builders).

### Q7. Explain Python's `__prepare__` method and custom class namespaces.

**Answer:**

`__prepare__` is a classmethod on metaclasses that returns the namespace object (typically a `dict`) used during class body execution. By returning a custom mapping, you can track attribute definition order, validate names, or build DSLs.

Before PEP 3115, class namespaces were always regular dicts. With `__prepare__`, metaclasses like `enum.EnumMeta` use custom mappings to detect duplicate names and auto-assign values.

```python
from collections import OrderedDict


class RecordMeta(type):
    @classmethod
    def __prepare__(mcs, name, bases, **kwargs):
        return OrderedDict()

    def __new__(mcs, name, bases, namespace, **kwargs):
        cls = super().__new__(mcs, name, bases, dict(namespace))
        cls._field_order = [
            k for k in namespace if not k.startswith("_")
        ]
        return cls


class Config(metaclass=RecordMeta):
    host = "localhost"
    port = 8080
    debug = False


assert Config._field_order == ["host", "port", "debug"]
```

In modern Python (3.7+), regular `dict` preserves insertion order, reducing the need for `OrderedDict` in `__prepare__`. However, custom namespaces remain useful for validation (disallowing overwrites), auto-numbering (enums), and intercepting attribute assignment during class definition.

### Q8. What is `sys.settrace` and how do debuggers/profilers hook into CPython?

**Answer:**

`sys.settrace(tracefunc)` registers a per-thread trace function called by the evaluation loop at specific events: `call` (function entry), `line` (new source line), `return` (function exit), `exception` (exception raised), and `opcode` (each bytecode, if enabled). The trace function receives `(frame, event, arg)` and returns itself (or `None` to stop tracing that scope).

This is how debuggers (`pdb`, PyCharm, VS Code debugpy) implement breakpoints, step-through, and watch expressions. Profilers like `coverage.py` use it to track line execution.

Performance implications are significant: **tracing slows execution 2-10x** because the evaluation loop calls into Python for every event. Python 3.12 introduced **monitoring** (`sys.monitoring`, PEP 669) as a more efficient alternative:

```python
import sys

def simple_tracer(frame, event, arg):
    if event == "call":
        print(f"Calling {frame.f_code.co_qualname}")
    return simple_tracer

sys.settrace(simple_tracer)

def greet(name):
    return f"Hello, {name}"

greet("Ada")
sys.settrace(None)
```

**PEP 669 (`sys.monitoring`)** in 3.12+ provides event-based monitoring with **zero overhead when disabled** and minimal overhead when active — critical for always-on production profiling and coverage collection. Events are registered per tool ID, and callbacks are only invoked for subscribed events.

Senior-level insight: When building custom profiling or instrumentation, prefer `sys.monitoring` over `sys.settrace` in 3.12+ for dramatically lower overhead in production.

### Q9. How does Python's `match` statement (PEP 634) work internally? What are its optimization characteristics?

**Answer:**

Structural pattern matching (`match`/`case`, PEP 634-636, Python 3.10) compiles patterns into a decision tree of bytecode checks. Unlike a simple `switch` on values, the compiler analyzes patterns for:

- **Literal comparisons**: `case 200` generates `COMPARE_OP ==`.
- **Sequence unpacking**: `case [x, y, *rest]` checks `isinstance(subject, Sequence)`, length bounds, and assigns captures.
- **Mapping patterns**: `case {"status": s}` checks keys and extracts values.
- **Class patterns**: `case Point(x=1, y=y)` uses `__match_args__` for positional, then attribute access.
- **OR patterns**: `case 401 | 403` merges branches.
- **Guards**: `case x if x > 0` adds conditional filtering after structural match.

The compiler produces sequential checks, not a jump table — so worst-case is O(cases). However, the decision tree collapses common prefixes and avoids redundant type checks:

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

def classify(shape):
    match shape:
        case Point(x=0, y=0):
            return "origin"
        case Point(x=x, y=0):
            return f"x-axis at {x}"
        case Point(x=0, y=y):
            return f"y-axis at {y}"
        case Point(x=x, y=y) if x == y:
            return f"diagonal at {x}"
        case Point(x=x, y=y):
            return f"({x}, {y})"
        case _:
            return "unknown"

assert classify(Point(0, 0)) == "origin"
assert classify(Point(3, 3)) == "diagonal at 3"
```

Performance considerations: `match` is not faster than equivalent `if/elif` chains — it is primarily a **readability and correctness** improvement for destructuring complex data. The bytecode generated is functionally equivalent to manual isinstance + attribute checks, but the compiler can optimize better than hand-written chains in some layouts.

### Q10. What is `__class_getitem__` and how does it enable generic syntax (`list[int]`) for user-defined classes?

**Answer:**

`__class_getitem__` (PEP 560, Python 3.7) is a classmethod called when a class is subscripted with `[]`. It enables `MyClass[int]` syntax without requiring a metaclass or `__getitem__` on instances. The standard library uses it extensively for `list[int]`, `dict[str, Any]`, `type[T]`, etc.

For user-defined generics, you typically inherit from `Generic[T]` which provides `__class_getitem__`. Since Python 3.9, built-in types (`list`, `dict`, `tuple`, `set`) support `[]` natively for type hints without importing from `typing`. Python 3.12 introduced the `class Foo[T]:` syntax (PEP 695) that makes generic class definitions even simpler.

```python
from typing import TypeVar, Generic

T = TypeVar("T")

class Registry(Generic[T]):
    def __init__(self) -> None:
        self._items: dict[str, T] = {}

    def register(self, key: str, item: T) -> None:
        self._items[key] = item

    def get(self, key: str) -> T | None:
        return self._items.get(key)


r: Registry[int] = Registry()
r.register("a", 42)

# Python 3.12+ syntax
class Stack[T]:
    def __init__(self) -> None:
        self._data: list[T] = []

    def push(self, item: T) -> None:
        self._data.append(item)

    def pop(self) -> T:
        return self._data.pop()
```

At runtime, `Registry[int]` returns a `types.GenericAlias` (or `_GenericAlias` from `typing`), which is primarily a hint container — it does not enforce types at runtime unless you use libraries like Pydantic or beartype for runtime checking.

## Advanced Metaclasses & Descriptors

### Q11. How would you implement a frozen (immutable) class without `dataclasses`?

**Answer:**

Immutability in Python requires intercepting attribute assignment after `__init__`. The cleanest approach combines `__setattr__` / `__delattr__` overrides with a flag indicating initialization is complete. Using `object.__setattr__` during `__init__` bypasses the custom guard.

```python
class Frozen:
    __slots__ = ("_frozen",)

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        orig_init = cls.__init__

        def guarded_init(self, *args, **kw):
            object.__setattr__(self, "_frozen", False)
            orig_init(self, *args, **kw)
            object.__setattr__(self, "_frozen", True)

        cls.__init__ = guarded_init

    def __setattr__(self, name, value):
        if getattr(self, "_frozen", False):
            raise AttributeError(f"Cannot set {name!r} on frozen instance")
        object.__setattr__(self, name, value)

    def __delattr__(self, name):
        if getattr(self, "_frozen", False):
            raise AttributeError(f"Cannot delete {name!r} on frozen instance")
        object.__delattr__(self, name)


class Point(Frozen):
    __slots__ = ("x", "y")

    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y


p = Point(1, 2)
try:
    p.x = 3
except AttributeError as e:
    assert "frozen" in str(e)
```

`dataclasses(frozen=True)` does something similar internally. For production code, prefer `@dataclass(frozen=True)` or `NamedTuple` — the manual approach is useful to understand the mechanism and for custom ORM or configuration frameworks.

### Q12. How do you implement a descriptor that validates types at attribute assignment time?

**Answer:**

A **data descriptor** (defining `__set__`) intercepts attribute assignment. Combine with `__set_name__` for automatic name discovery and type annotations for validation:

```python
from typing import Any, get_type_hints


class TypeChecked:
    def __set_name__(self, owner: type, name: str) -> None:
        self._name = name
        hints = get_type_hints(owner)
        self._expected_type = hints.get(name, object)

    def __get__(self, obj: Any, owner: type | None = None) -> Any:
        if obj is None:
            return self
        return obj.__dict__.get(self._name)

    def __set__(self, obj: Any, value: Any) -> None:
        if not isinstance(value, self._expected_type):
            raise TypeError(
                f"{self._name} expects {self._expected_type.__name__}, "
                f"got {type(value).__name__}"
            )
        obj.__dict__[self._name] = value


class User:
    name: str = TypeChecked()
    age: int = TypeChecked()

    def __init__(self, name: str, age: int) -> None:
        self.name = name
        self.age = age


u = User("Ada", 30)
try:
    u.age = "thirty"
except TypeError as e:
    assert "expects int" in str(e)
```

This pattern powers validation frameworks (Pydantic fields, Django model fields, SQLAlchemy columns). Production descriptors should handle `Optional`, `Union`, generic types, and provide meaningful error messages with field paths.

### Q13. Explain the descriptor priority chain in Python's attribute lookup.

**Answer:**

Python's attribute lookup follows a well-defined priority order, governed by the descriptor protocol:

1. **Data descriptors on the type** (MRO): Objects in the class hierarchy implementing `__set__` or `__delete__`. These take highest priority — even over the instance's `__dict__`. This is why `property` setters work.

2. **Instance `__dict__`**: If no data descriptor matches, the instance's own `__dict__` is checked.

3. **Non-data descriptors and class attributes** (MRO): Objects implementing only `__get__` (like plain functions, which are non-data descriptors that return bound methods). Class variables without descriptors also fall here.

4. **`__getattr__`**: Called as a fallback when normal lookup fails entirely. Not called if the attribute is found anywhere above.

5. **`__getattribute__`**: Overriding this method intercepts **all** attribute access (both found and not-found). Custom `__getattribute__` can bypass the entire chain.

```python
class DataDesc:
    def __get__(self, obj, owner=None):
        return "data-desc-get"
    def __set__(self, obj, value):
        pass

class NonDataDesc:
    def __get__(self, obj, owner=None):
        return "non-data-desc-get"

class C:
    data = DataDesc()
    non_data = NonDataDesc()

c = C()

c.__dict__["data"] = "instance-val"
assert c.data == "data-desc-get"  # data descriptor wins

c.__dict__["non_data"] = "instance-val"
assert c.non_data == "instance-val"  # instance dict wins over non-data
```

Understanding this chain is essential for debugging frameworks that use descriptors (Django models, SQLAlchemy, dataclasses) and for building correct custom attribute machinery.

### Q14. How would you implement an Aspect-Oriented Programming (AOP) framework using metaclasses?

**Answer:**

AOP separates cross-cutting concerns (logging, authorization, caching) from business logic. A metaclass can automatically wrap methods matching a pattern with advice (before/after/around):

```python
from collections.abc import Callable
from functools import wraps
from typing import Any


def log_calls(method: Callable) -> Callable:
    @wraps(method)
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        print(f"CALL {method.__qualname__}")
        result = method(*args, **kwargs)
        print(f"RETURN {method.__qualname__} -> {result!r}")
        return result
    return wrapper


class AOPMeta(type):
    def __new__(
        mcs,
        name: str,
        bases: tuple[type, ...],
        namespace: dict[str, Any],
        aspects: dict[str, Callable] | None = None,
        **kwargs: Any,
    ):
        aspects = aspects or {}
        for attr_name, attr_value in list(namespace.items()):
            if callable(attr_value) and not attr_name.startswith("_"):
                for pattern, advice in aspects.items():
                    if pattern == "*" or attr_name.startswith(pattern):
                        namespace[attr_name] = advice(attr_value)
        return super().__new__(mcs, name, bases, namespace, **kwargs)


class OrderService(metaclass=AOPMeta, aspects={"*": log_calls}):
    def create_order(self, item: str) -> str:
        return f"order:{item}"

    def cancel_order(self, order_id: str) -> bool:
        return True


svc = OrderService()
svc.create_order("widget")
```

In practice, most teams prefer **decorators** or **middleware** over metaclass-based AOP for clarity. However, metaclass AOP is valuable when you need to **enforce** cross-cutting concerns automatically across all methods of all subclasses without relying on developers to remember decorators.

### Q15. What is `__del__` and what are its pitfalls? When should you use weak references instead?

**Answer:**

`__del__` is a finalizer called when an object's reference count reaches zero (or during cyclic GC). It is **not** a destructor in the C++ sense — you cannot rely on its timing, order, or even that it runs at all (interpreter shutdown, circular references pre-3.4).

Pitfalls:
- **Resurrection**: If `__del__` stores `self` somewhere reachable, the object is resurrected, and `__del__` may be called again.
- **Ordering**: During interpreter shutdown, globals may already be `None` when `__del__` runs, causing `AttributeError`.
- **Cyclic GC interaction**: Before PEP 442 (3.4), objects with `__del__` in cycles were moved to `gc.garbage` and never collected.
- **Thread safety**: `__del__` may be called from any thread in CPython.

Prefer `with` statements and context managers for deterministic cleanup. Use `weakref.finalize` for reliable destructor-like behavior:

```python
import weakref


class Connection:
    def __init__(self, name: str) -> None:
        self.name = name
        self._finalizer = weakref.finalize(self, self._cleanup, name)

    @staticmethod
    def _cleanup(name: str) -> None:
        print(f"Cleaning up connection: {name}")

    def close(self) -> None:
        self._finalizer()

    @property
    def alive(self) -> bool:
        return self._finalizer.alive


c = Connection("db")
c.close()
assert not c.alive
```

`weakref.finalize` survives circular references, runs at most once, and works correctly during interpreter shutdown. Use `weakref.ref` (weak references) when you need to observe an object without preventing its collection — common in caches, observer patterns, and parent-child bidirectional relationships.

## Concurrency, Parallelism & the GIL

### Q16. How does the GIL interact with C extensions? When is the GIL released?

**Answer:**

The GIL is released whenever CPython calls into C code that does not need to interact with Python objects. The macros `Py_BEGIN_ALLOW_THREADS` and `Py_END_ALLOW_THREADS` bracket regions where the GIL is released. This is why I/O operations (file reads, network calls, `time.sleep`) allow other threads to run — the underlying C library calls release the GIL.

Extensions like **NumPy** release the GIL during heavy computation on C arrays, enabling real multi-threaded parallelism for numeric workloads even under the GIL. Similarly, database drivers release the GIL while waiting for query results.

However, any C code that creates, inspects, or modifies Python objects **must** hold the GIL. If an extension forgets to re-acquire the GIL before touching `PyObject*`, you get undefined behavior (segfaults, corruption).

```c
// C extension pattern (conceptual)
static PyObject* compute(PyObject* self, PyObject* args) {
    double* data;
    Py_ssize_t len;
    // ... parse args, extract buffer ...

    Py_BEGIN_ALLOW_THREADS
    // Pure C computation - no Python objects touched
    heavy_math(data, len);
    Py_END_ALLOW_THREADS

    Py_RETURN_NONE;
}
```

For the free-threaded build (PEP 703), extensions must use `PyMutex` or atomic operations instead of relying on the GIL — this is the major migration challenge for the C extension ecosystem.

### Q17. Explain `asyncio.to_thread` and when you should offload work from the event loop.

**Answer:**

`asyncio.to_thread(func, *args, **kwargs)` (Python 3.9+) runs a synchronous function in a thread pool executor, returning an awaitable. It is the clean way to call blocking code from async contexts without stalling the event loop.

Use it for:
- **Blocking I/O** that lacks an async API (legacy database drivers, file operations on some systems).
- **CPU-bound functions** that are short enough that thread overhead is acceptable, but long enough to starve the loop if run inline.
- **Third-party library calls** not designed for async (e.g., `Pillow` image processing, PDF generation).

Do **not** use it for:
- **Already async operations** — double-wrapping adds overhead.
- **Very short functions** where the thread scheduling cost exceeds the work.
- **Heavy CPU work** — use `ProcessPoolExecutor` via `loop.run_in_executor` to bypass the GIL.

```python
import asyncio
import time


def blocking_io() -> str:
    time.sleep(1)
    return "done"


async def main():
    result = await asyncio.to_thread(blocking_io)
    assert result == "done"


asyncio.run(main())
```

Architecture pattern: In a FastAPI service, use `asyncio.to_thread` for legacy integrations and `ProcessPoolExecutor` for CPU-intensive tasks. Set explicit `max_workers` to prevent thread explosion.

### Q18. How do you implement structured concurrency in Python?

**Answer:**

Structured concurrency ensures that concurrent tasks have well-defined lifetimes tied to a scope — when the scope exits, all child tasks are complete or cancelled. This prevents "fire and forget" task leaks that can cause silent errors or resource leaks.

Python 3.11 introduced **`asyncio.TaskGroup`** (inspired by Trio's nurseries):

```python
import asyncio


async def fetch(url: str) -> str:
    await asyncio.sleep(0.1)
    if "fail" in url:
        raise ValueError(f"bad url: {url}")
    return f"body:{url}"


async def main():
    try:
        async with asyncio.TaskGroup() as tg:
            t1 = tg.create_task(fetch("https://api/a"))
            t2 = tg.create_task(fetch("https://api/b"))
            t3 = tg.create_task(fetch("https://api/fail"))
    except* ValueError as eg:
        for exc in eg.exceptions:
            print(f"Handled: {exc}")
    else:
        print(t1.result(), t2.result())


asyncio.run(main())
```

Key semantics:
- If any task raises, **all remaining tasks are cancelled** and their exceptions are collected into an `ExceptionGroup`.
- The `async with` block only exits after **all tasks** have completed or been cancelled.
- `except*` (PEP 654) handles sub-exceptions by type.
- Tasks cannot escape the group's lifetime — no orphaned background work.

For pre-3.11, libraries like **Trio** and **AnyIO** provide similar patterns. In production, structured concurrency simplifies error handling, resource cleanup, and prevents subtle bugs from fire-and-forget `create_task` usage.

### Q19. What is `multiprocessing.shared_memory` and when should you use it?

**Answer:**

`multiprocessing.shared_memory` (Python 3.8+) provides OS-level shared memory segments accessible from multiple processes without serialization overhead. Unlike `multiprocessing.Queue` or `Pipe` (which pickle data across process boundaries), shared memory maps the same physical pages into each process's address space.

Use cases:
- **Large numeric arrays** shared between worker processes (avoid copying gigabytes of data).
- **Low-latency IPC** where pickle serialization is too slow.
- **Shared state** that is read-heavy with infrequent updates (with explicit synchronization).

```python
from multiprocessing import shared_memory, Process
import numpy as np


def worker(shm_name: str, shape: tuple, dtype: str) -> None:
    existing = shared_memory.SharedMemory(name=shm_name)
    arr = np.ndarray(shape, dtype=dtype, buffer=existing.buf)
    arr[:] = arr * 2
    existing.close()


def main():
    data = np.array([1, 2, 3, 4, 5], dtype=np.float64)
    shm = shared_memory.SharedMemory(create=True, size=data.nbytes)
    shared_arr = np.ndarray(data.shape, dtype=data.dtype, buffer=shm.buf)
    shared_arr[:] = data

    p = Process(target=worker, args=(shm.name, data.shape, str(data.dtype)))
    p.start()
    p.join()

    assert list(shared_arr) == [2, 4, 6, 8, 10]
    shm.close()
    shm.unlink()


if __name__ == "__main__":
    main()
```

Caveats: You are responsible for synchronization (use `multiprocessing.Lock` or `Barrier`), proper cleanup (`unlink`), and handling process crashes that might leave orphaned shared memory segments.

### Q20. How do you prevent and detect race conditions in Python applications?

**Answer:**

Race conditions occur when the outcome of concurrent operations depends on scheduling order. Despite the GIL making individual bytecodes thread-safe, compound operations (check-then-act, read-modify-write) are not atomic:

```python
import threading

counter = 0

def increment():
    global counter
    for _ in range(100_000):
        counter += 1  # NOT atomic: LOAD_GLOBAL, BINARY_OP, STORE_GLOBAL

threads = [threading.Thread(target=increment) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
# counter is likely != 400_000 due to interleaving
```

Prevention strategies:
1. **Locks/mutexes**: Wrap critical sections with `threading.Lock()`. Use `RLock` only when reentrant locking is truly needed.
2. **Atomic data structures**: `queue.Queue`, `collections.deque` (append/pop are GIL-atomic), `multiprocessing.Value` with locks.
3. **Immutability**: Share immutable data; create new versions instead of mutating in place.
4. **Message passing**: Use queues between threads/processes instead of shared mutable state.
5. **Async single-threaded**: `asyncio` avoids data races by running on one thread — but `await` points are still preemption points for task interleaving.

Detection tools:
- **Thread Sanitizer (TSan)**: Compile CPython with `-fsanitize=thread` (experimental).
- **`concurrent.futures` exceptions**: Unhandled exceptions in thread pools silently disappear — always check `Future.result()`.
- **Stress testing**: Run concurrent tests with high thread counts and short sleeps at contention points.
- **Code review**: Look for shared mutable state accessed without locks, especially `global` variables modified by multiple threads.

## Async Architecture & Event Loops

### Q21. How does the asyncio event loop work internally?

**Answer:**

The event loop is a single-threaded scheduler that multiplexes I/O readiness and timer callbacks. Internally, it uses OS-level I/O multiplexing (`epoll` on Linux, `kqueue` on macOS, `IOCP` on Windows) to watch file descriptors.

The loop cycle:
1. **Run all ready callbacks** from the ready queue.
2. **Poll for I/O events** with a timeout derived from the nearest scheduled timer.
3. **Process I/O events**: mark associated futures as done, schedule their callbacks.
4. **Process timers**: run `call_later`/`call_at` callbacks whose deadlines have passed.
5. Repeat.

Coroutines are wrapped in `Task` objects that call `__next__` on the coroutine, advancing it until the next `await`. When a coroutine awaits a `Future`, the task registers a callback on that future and suspends. When the future resolves (by I/O completion or another task setting its result), the callback re-schedules the task.

Key architectural insights:
- **Never block the loop**: Any synchronous call longer than ~1ms stalls all other tasks. Use `asyncio.to_thread` for blocking work.
- **Task cancellation**: `task.cancel()` injects `CancelledError` at the next `await` point — the coroutine can catch it for cleanup.
- **Custom event loops**: `uvloop` replaces the default loop with a libuv-based implementation for ~2-4x throughput on I/O benchmarks.

### Q23. How do you implement graceful shutdown in an async Python service?

**Answer:**

Graceful shutdown ensures in-flight requests complete, background tasks clean up, and resources are released before the process exits. This requires signal handling, task cancellation with timeouts, and ordered teardown.

```python
import asyncio
import signal
import logging

log = logging.getLogger(__name__)


class Server:
    def __init__(self) -> None:
        self._shutdown_event = asyncio.Event()
        self._background_tasks: set[asyncio.Task] = set()

    async def handle_request(self, data: str) -> str:
        await asyncio.sleep(0.1)
        return f"processed:{data}"

    async def _background_worker(self) -> None:
        while not self._shutdown_event.is_set():
            try:
                await asyncio.wait_for(
                    self._shutdown_event.wait(), timeout=5.0
                )
            except asyncio.TimeoutError:
                log.debug("background tick")

    async def run(self) -> None:
        loop = asyncio.get_running_loop()
        for sig in (signal.SIGTERM, signal.SIGINT):
            loop.add_signal_handler(sig, self._signal_handler)

        task = asyncio.create_task(self._background_worker())
        self._background_tasks.add(task)
        task.add_done_callback(self._background_tasks.discard)

        log.info("Server running")
        await self._shutdown_event.wait()
        log.info("Shutting down...")

        for t in self._background_tasks:
            t.cancel()

        await asyncio.gather(*self._background_tasks, return_exceptions=True)
        log.info("Shutdown complete")

    def _signal_handler(self) -> None:
        self._shutdown_event.set()
```

Production checklist:
- Set a **shutdown timeout** (e.g., 30 seconds) after which remaining tasks are forcibly cancelled.
- **Drain connection pools** (DB, HTTP clients) during shutdown.
- **Health check endpoints** should start returning unhealthy immediately on signal receipt.
- Use **readiness probes** in Kubernetes to stop routing traffic before shutdown begins.

### Q24. What are async generators and async context managers? How do they differ from their sync counterparts?

**Answer:**

**Async generators** use `async def` with `yield` to produce values consumable via `async for`. They can `await` inside the body, making them suitable for streaming data from async sources:

```python
import asyncio

async def stream_data(n: int):
    for i in range(n):
        await asyncio.sleep(0.01)
        yield i

async def main():
    results = []
    async for item in stream_data(5):
        results.append(item)
    assert results == [0, 1, 2, 3, 4]

asyncio.run(main())
```

**Async context managers** implement `__aenter__` and `__aexit__` (both async), used with `async with`. They enable async setup/teardown for resources like database connections and HTTP sessions:

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def managed_connection(dsn: str):
    conn = await connect(dsn)
    try:
        yield conn
    finally:
        await conn.close()
```

Key differences from sync counterparts:
- Async generators support `asend()`, `athrow()`, and `aclose()` — all coroutines requiring `await`.
- `async for` and `async with` can only be used inside `async def` functions.
- Finalization: async generators need explicit `aclose()` or `async for` consumption; abandoned async generators may not clean up resources.

### Q25. How do you handle backpressure in async Python applications?

**Answer:**

Backpressure occurs when a producer generates work faster than consumers can process it. Without control, queues grow unboundedly, consuming memory until the process crashes.

Strategies:
1. **Bounded queues**: `asyncio.Queue(maxsize=N)` blocks the producer's `put()` when the queue is full.
2. **Semaphores**: Limit concurrent in-flight operations with `asyncio.Semaphore(N)`.
3. **Flow control in protocols**: `asyncio.Transport.pause_reading()` / `resume_reading()` for TCP streams.
4. **Rate limiting**: Token bucket or leaky bucket algorithms controlling request throughput.

```python
import asyncio


async def producer(queue: asyncio.Queue[int]) -> None:
    for i in range(100):
        await queue.put(i)
        print(f"Produced {i}, queue size: {queue.qsize()}")


async def consumer(queue: asyncio.Queue[int]) -> None:
    while True:
        item = await queue.get()
        await asyncio.sleep(0.1)
        queue.task_done()
        print(f"Consumed {item}")


async def main():
    queue: asyncio.Queue[int] = asyncio.Queue(maxsize=5)

    async with asyncio.TaskGroup() as tg:
        tg.create_task(producer(queue))
        for _ in range(3):
            tg.create_task(consumer(queue))

        await queue.join()
        # Cancel consumers after all items are processed
        for task in tg._tasks:
            task.cancel()


asyncio.run(main())
```

In production, combine bounded queues with monitoring (queue depth metrics) and circuit breakers (reject new work when downstream is overloaded). For HTTP APIs, return 429 (Too Many Requests) with `Retry-After` headers.

## Performance Engineering & Optimization

### Q26. How do you systematically optimize a Python application?

**Answer:**

Follow a disciplined process — never optimize without measurement:

1. **Define targets**: Set measurable goals (p99 latency < 100ms, throughput > 1000 RPS) tied to business requirements.

2. **Profile before optimizing**:
   - **CPU profiling**: `cProfile` (function-level), `py-spy` (sampling, low-overhead, production-safe), `Scalene` (CPU+memory+GPU).
   - **Memory profiling**: `tracemalloc`, `memray`, `objgraph` for leak detection.
   - **I/O profiling**: `strace`/`dtrace` for syscalls, database query logs for N+1 patterns.

3. **Address algorithmic issues first**: O(n²) → O(n log n) beats any micro-optimization. Common wins: hash-based lookups instead of linear scans, batch database queries, cached computations.

4. **Reduce Python overhead in hot paths**:
   - Local variable access is faster than global/attribute lookups.
   - List comprehensions and generators vs manual loops.
   - `__slots__` for many small objects.
   - String joining vs concatenation.

5. **Leverage C-speed libraries**: NumPy for numeric arrays, `ujson`/`orjson` for JSON, `re2` for regex, compiled Cython/Rust extensions for critical sections.

6. **Architectural optimization**: Caching (Redis, `lru_cache`), connection pooling, async I/O, pre-computation, and lazy loading.

7. **Validate improvements**: Re-profile after each change. Use benchmarking frameworks (`pytest-benchmark`, `asv`) with statistical significance testing.

```python
import cProfile
import pstats
from io import StringIO

def profile(func):
    def wrapper(*args, **kwargs):
        pr = cProfile.Profile()
        pr.enable()
        result = func(*args, **kwargs)
        pr.disable()
        s = StringIO()
        ps = pstats.Stats(pr, stream=s).sort_stats("cumulative")
        ps.print_stats(20)
        print(s.getvalue())
        return result
    return wrapper
```

### Q27. What is `__slots__` optimization and what are its real-world performance implications?

**Answer:**

`__slots__` replaces the per-instance `__dict__` (a hash table) with fixed-offset C-level storage, yielding:
- **30-50% memory reduction** per instance (no dict overhead per object).
- **10-20% faster attribute access** (direct offset lookup vs dict hash).
- **Prevents dynamic attribute creation** (catches typos).

Real-world impact is most significant when creating millions of small objects (data processing, graph nodes, game entities):

```python
import sys

class WithDict:
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

class WithSlots:
    __slots__ = ("x", "y", "z")
    def __init__(self, x, y, z):
        self.x = x
        self.y = y
        self.z = z

d = WithDict(1, 2, 3)
s = WithSlots(1, 2, 3)

print(f"dict: {sys.getsizeof(d) + sys.getsizeof(d.__dict__)} bytes")
print(f"slots: {sys.getsizeof(s)} bytes")
```

Caveats:
- No `__dict__` means no dynamic attributes — add `"__dict__"` to slots if you need both.
- No `__weakref__` by default — add to slots if needed.
- Inheritance: all classes in the hierarchy should use slots for full benefit. A slotted class inheriting from a non-slotted class still has `__dict__` from the parent.
- `dataclasses(slots=True)` (3.10+) generates slots automatically.

### Q28. How do you detect and fix memory leaks in Python?

**Answer:**

Python memory leaks typically come from objects being held alive unintentionally — reference cycles, growing caches, forgotten callbacks, or global collections that never shrink.

Detection toolkit:
1. **`tracemalloc`** (stdlib): Tracks allocation sites; compare snapshots to find growing allocations.
2. **`memray`**: Full heap profiler with flame graph visualization — excellent for production analysis.
3. **`objgraph`**: Visualizes reference graphs; `show_growth()` identifies increasing object types.
4. **`gc.get_referrers()`**: Traces who holds a reference to a suspect object.
5. **Process metrics**: Monitor RSS/VMS over time with `psutil` or container metrics.

```python
import tracemalloc

tracemalloc.start()

cache = {}
for i in range(10000):
    cache[i] = [0] * 1000

snapshot = tracemalloc.take_snapshot()
top = snapshot.statistics("lineno")
for stat in top[:5]:
    print(stat)
```

Common leak patterns and fixes:
- **Unbounded caches**: Use `functools.lru_cache(maxsize=N)` or `cachetools.TTLCache`.
- **Closures capturing large objects**: Extract needed data instead of closing over entire objects.
- **Observer/callback patterns**: Use `weakref` for registrations.
- **Circular references with `__del__`**: Prefer `weakref.finalize`.
- **Thread-local storage**: Clean up on thread exit.
- **Module-level lists/dicts**: Audit for growth; add eviction policies.

### Q33. Explain copy-on-write (COW) behavior in `multiprocessing.fork()` and how to optimize for it.

**Answer:**

When `os.fork()` creates a child process, the OS shares parent pages as read-only via COW: pages are only physically copied when either process writes to them. This is memory-efficient until CPython's reference counting triggers writes to `ob_refcnt` on shared objects, dirtying pages that would otherwise remain shared.

Optimization strategies:
1. **`gc.freeze()`** (Python 3.7+): Moves all tracked objects to the permanent generation before forking. The GC won't traverse them, avoiding refcount churn on generation-tracking structures.
2. **Minimize pre-fork allocations**: Load data after fork or use `multiprocessing.spawn` start method (no COW benefits but no COW penalty either).
3. **Use shared memory** (`multiprocessing.shared_memory`, `mmap`) for large read-only datasets instead of relying on COW.
4. **Immortal objects** (Python 3.12+): PEP 683 makes certain objects immortal (refcount never changes), preserving COW for `None`, `True`, small ints, and interned strings.

```python
import gc
import os
import multiprocessing

def worker(data_ref):
    # data is COW-shared from parent
    print(f"Worker {os.getpid()}: data[0]={data_ref[0]}")

large_data = list(range(10_000_000))
gc.freeze()  # prevent GC from dirtying pages

processes = []
for _ in range(4):
    p = multiprocessing.Process(target=worker, args=(large_data,))
    p.start()
    processes.append(p)

for p in processes:
    p.join()
```

In containerized environments, this matters significantly: forked workers that dirty shared pages through refcount changes can double or triple actual memory usage compared to the theoretical COW savings.

## System Design & Architecture

### Q34. How do you design a plugin/extension system in Python?

**Answer:**

Plugin systems allow third-party code to extend an application without modifying its core. Key design decisions: discovery mechanism, API contract, isolation, and lifecycle management.

```python
from abc import ABC, abstractmethod
from importlib.metadata import entry_points
from typing import Any


class PluginBase(ABC):
    @abstractmethod
    def name(self) -> str: ...

    @abstractmethod
    def execute(self, context: dict[str, Any]) -> Any: ...


class PluginManager:
    def __init__(self) -> None:
        self._plugins: dict[str, PluginBase] = {}

    def discover(self, group: str = "myapp.plugins") -> None:
        eps = entry_points()
        for ep in eps.get(group, []):
            plugin_class = ep.load()
            if issubclass(plugin_class, PluginBase):
                instance = plugin_class()
                self._plugins[instance.name()] = instance

    def register(self, plugin: PluginBase) -> None:
        self._plugins[plugin.name()] = plugin

    def execute_all(self, context: dict[str, Any]) -> dict[str, Any]:
        results = {}
        for name, plugin in self._plugins.items():
            try:
                results[name] = plugin.execute(context)
            except Exception as e:
                results[name] = {"error": str(e)}
        return results
```

Architecture considerations:
- **Discovery**: `importlib.metadata.entry_points()` (declared in `pyproject.toml`), directory scanning, or explicit configuration.
- **Contracts**: ABC or `Protocol` for type-safe interfaces; versioned APIs for backward compatibility.

### Q35. How do you implement the Repository pattern in Python?

**Answer:**

The Repository pattern abstracts data persistence behind a collection-like interface, decoupling domain logic from storage details. It enables testing with in-memory implementations and swapping backends.

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Protocol
from uuid import UUID, uuid4


@dataclass
class Order:
    id: UUID = field(default_factory=uuid4)
    customer: str = ""
    total: float = 0.0


class OrderRepository(Protocol):
    def save(self, order: Order) -> None: ...
    def get(self, order_id: UUID) -> Order | None: ...
    def find_by_customer(self, customer: str) -> list[Order]: ...
    def delete(self, order_id: UUID) -> None: ...


class InMemoryOrderRepository:
    def __init__(self) -> None:
        self._store: dict[UUID, Order] = {}

    def save(self, order: Order) -> None:
        self._store[order.id] = order

    def get(self, order_id: UUID) -> Order | None:
        return self._store.get(order_id)

    def find_by_customer(self, customer: str) -> list[Order]:
        return [o for o in self._store.values() if o.customer == customer]

    def delete(self, order_id: UUID) -> None:
        self._store.pop(order_id, None)


class SQLAlchemyOrderRepository:
    def __init__(self, session) -> None:
        self._session = session

    def save(self, order: Order) -> None:
        self._session.merge(order)
        self._session.flush()

    def get(self, order_id: UUID) -> Order | None:
        return self._session.get(Order, order_id)

    # ... other methods using SQLAlchemy queries
```

Use `Protocol` (not ABC) so implementations don't need inheritance — structural subtyping is sufficient. This pairs well with dependency injection frameworks (dependency-injector, FastAPI's `Depends`).

### Q36. How do you implement the Unit of Work pattern in Python?

**Answer:**

Unit of Work tracks changes to domain objects during a business transaction and coordinates writing them to the database in a single atomic operation. SQLAlchemy's `Session` is essentially a built-in UoW, but understanding the pattern helps when building custom persistence or working with multiple data stores.

```python
from __future__ import annotations
from abc import ABC, abstractmethod
from contextlib import contextmanager
from typing import Any


class UnitOfWork(ABC):
    @abstractmethod
    def commit(self) -> None: ...

    @abstractmethod
    def rollback(self) -> None: ...

    def __enter__(self) -> UnitOfWork:
        return self

    def __exit__(self, exc_type, exc, tb) -> None:
        if exc_type:
            self.rollback()
        self.rollback()


class SQLAlchemyUnitOfWork(UnitOfWork):
    def __init__(self, session_factory) -> None:
        self._session_factory = session_factory
        self._session = None

    def __enter__(self) -> SQLAlchemyUnitOfWork:
        self._session = self._session_factory()
        return self

    @property
    def session(self):
        return self._session

    def commit(self) -> None:
        self._session.commit()

    def rollback(self) -> None:
        self._session.rollback()

    def __exit__(self, exc_type, exc, tb) -> None:
        if exc_type:
            self.rollback()
        if self._session:
            self._session.close()


def create_order(uow: UnitOfWork, customer: str, total: float) -> None:
    with uow:
        # ... create order using uow.session
        uow.commit()
```

The UoW pattern is especially valuable when:
- You need to coordinate writes across multiple repositories in one transaction.
- You want to decouple service logic from session lifecycle management.
- You need testable services (inject an in-memory UoW for tests).

### Q37. How do you implement CQRS (Command Query Responsibility Segregation) in Python?

**Answer:**

CQRS separates read and write models: commands mutate state through a write model (domain entities, repositories), while queries read from an optimized read model (denormalized views, caches, search indices).

```python
from dataclasses import dataclass
from typing import Protocol
from uuid import UUID


@dataclass
class CreateOrderCommand:
    customer_id: UUID
    items: list[dict]


@dataclass
class OrderSummaryQuery:
    customer_id: UUID
    page: int = 1
    size: int = 20


@dataclass
class OrderSummary:
    order_id: UUID
    total: float
    status: str


class CommandHandler(Protocol):
    def handle(self, command: CreateOrderCommand) -> UUID: ...


class QueryHandler(Protocol):
    def handle(self, query: OrderSummaryQuery) -> list[OrderSummary]: ...


class CreateOrderHandler:
    def __init__(self, repo, event_bus) -> None:
        self._repo = repo
        self._bus = event_bus

    def handle(self, command: CreateOrderCommand) -> UUID:
        order = Order.create(command.customer_id, command.items)
        self._repo.save(order)
        self._bus.publish(OrderCreated(order.id))

    def __init__(self, read_db) -> None:
```

- **Complex domains**: Write model uses rich domain objects; read model uses flat DTOs.
- **Scalability**: Read and write databases can scale independently.

When NOT to use:
- Simple CRUD applications where the complexity overhead isn't justified.

### Q38. How do you implement event-driven architecture in Python?

**Answer:**

Event-driven architecture decouples components through asynchronous event publication and subscription. In Python, this spans in-process event buses and distributed message systems.

```python
from __future__ import annotations
from collections import defaultdict
from collections.abc import Callable
from dataclasses import dataclass, field
from datetime import datetime, timezone
from typing import Any
from uuid import UUID, uuid4


@dataclass(frozen=True)
class DomainEvent:
    event_id: UUID = field(default_factory=uuid4)
    timestamp: datetime = field(default_factory=lambda: datetime.now(tz=timezone.utc))


@dataclass(frozen=True)
class OrderPlaced(DomainEvent):
    order_id: UUID = field(default_factory=uuid4)
    customer_id: str = ""
    total: float = 0.0


class EventBus:

        self._handlers: dict[type, list[Callable]] = defaultdict(list)

    def subscribe(self, event_type: type, handler: Callable) -> None:
        self._handlers[event_type].append(handler)

    def publish(self, event: DomainEvent) -> None:
        for handler in self._handlers.get(type(event), []):
            handler(event)


class NotificationService:
    def on_order_placed(self, event: OrderPlaced) -> None:

inventory = InventoryService()
```

### Q39. How do you implement the Circuit Breaker pattern in Python?

**Answer:**

Circuit breakers prevent cascading failures by monitoring call success rates and short-circuiting requests to failing dependencies. States: **Closed** (normal), **Open** (failing, reject immediately), **Half-Open** (test recovery with limited traffic).

```python
import time
from enum import Enum
from threading import Lock


class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"


class CircuitBreaker:
    def __init__(
        self,
        failure_threshold: int = 5,
        recovery_timeout: float = 30.0,
        half_open_max_calls: int = 1,
    ) -> None:
        self._failure_threshold = failure_threshold
        self._recovery_timeout = recovery_timeout
        self._half_open_max_calls = half_open_max_calls
        self._state = CircuitState.CLOSED
        self._failure_count = 0
        self._last_failure_time = 0.0
        self._half_open_calls = 0
        self._lock = Lock()

    @property
    def state(self) -> CircuitState:
        with self._lock:
            if self._state == CircuitState.OPEN:
                if time.monotonic() - self._last_failure_time >= self._recovery_timeout:
                    self._state = CircuitState.HALF_OPEN
                    self._half_open_calls = 0
            return self._state

    def call(self, func, *args, **kwargs):
        state = self.state

        if state == CircuitState.OPEN:
            raise CircuitOpenError("Circuit is open")

        if state == CircuitState.HALF_OPEN:
            with self._lock:
                if self._half_open_calls >= self._half_open_max_calls:
                    raise CircuitOpenError("Half-open limit reached")
                self._half_open_calls += 1

        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception:
            self._on_failure()
            raise

    def _on_success(self) -> None:
        with self._lock:
            self._failure_count = 0
            self._state = CircuitState.CLOSED

    def _on_failure(self) -> None:
        with self._lock:
            self._failure_count += 1
            self._last_failure_time = time.monotonic()
            if self._failure_count >= self._failure_threshold:
                self._state = CircuitState.OPEN


class CircuitOpenError(Exception):
    pass
```

In production, use libraries like `pybreaker` or `tenacity` with circuit breaker policies. Integrate with metrics (Prometheus counters for state transitions) and alerting.

### Q40. How do you implement idempotent APIs in Python?

**Answer:**

Idempotency ensures that retrying the same request produces the same result without side effects. This is critical for payment processing, order creation, and any state-changing operation in distributed systems where network failures cause retries.

Implementation strategy using an idempotency key:

```python
from datetime import datetime, timezone
from uuid import UUID


class IdempotencyStore:
    def __init__(self) -> None:
        self._store: dict[str, dict] = {}

    def get(self, key: str) -> dict | None:
        entry = self._store.get(key)
        if entry and self._is_expired(entry):
            del self._store[key]
            return None
        return entry

    def set(self, key: str, response: dict, ttl_hours: int = 24) -> None:
        self._store[key] = {
            "response": response,
            "created_at": datetime.now(tz=timezone.utc),
            "ttl_hours": ttl_hours,
        }

    def _is_expired(self, entry: dict) -> bool:
        from datetime import timedelta
        age = datetime.now(tz=timezone.utc) - entry["created_at"]
        return age > timedelta(hours=entry["ttl_hours"])


async def create_order(
    request: dict,
    idempotency_key: str,
    store: IdempotencyStore,
):
    existing = store.get(idempotency_key)
    if existing:
        return existing["response"]

    order = process_order(request)
    response = {"order_id": str(order.id), "status": "created"}

    store.set(idempotency_key, response)
    return response
```

Best practices:
- Use client-generated UUIDs as idempotency keys.
- Store the complete response, not just status.
- Set TTL based on business requirements (24-72 hours typical).
- Use database-level uniqueness constraints as the ultimate safety net.
- In Redis: `SET key value NX EX ttl` for atomic check-and-set.

### Q42. How do you implement the Saga pattern for distributed transactions?

**Answer:**

Sagas manage multi-service transactions through a sequence of local transactions with compensating actions for rollback. Two approaches: **choreography** (event-driven, each service reacts to events) and **orchestration** (a central coordinator directs the flow).

```python
from dataclasses import dataclass
from enum import Enum
from typing import Callable


class StepStatus(Enum):
    PENDING = "pending"
    COMPLETED = "completed"
    COMPENSATED = "compensated"
    FAILED = "failed"


@dataclass
class SagaStep:
    name: str
    action: Callable
    compensate: Callable
    status: StepStatus = StepStatus.PENDING


class SagaOrchestrator:
    def __init__(self, steps: list[SagaStep]) -> None:
        self._steps = steps
        self._completed: list[SagaStep] = []

    def execute(self, context: dict) -> dict:
        for step in self._steps:
            try:
                step.action(context)
                step.status = StepStatus.COMPLETED
                self._completed.append(step)
            except Exception as e:
                step.status = StepStatus.FAILED
                self._compensate(context)
                raise SagaFailed(f"Step {step.name} failed: {e}") from e
        return context

    def _compensate(self, context: dict) -> None:
        for step in reversed(self._completed):
            try:
                step.compensate(context)
                step.status = StepStatus.COMPENSATED
            except Exception:
                pass  # log and continue compensating


class SagaFailed(Exception):
    pass
```

Production considerations:
- **Idempotent steps**: Both action and compensation must be idempotent.
- **Persistence**: Store saga state in a database for recovery after crashes.
- **Timeout handling**: Set timeouts on each step; trigger compensation on timeout.
- **Dead letter queues**: Capture failed compensations for manual resolution.

### Q43. How do you implement rate limiting in a Python service?

**Answer:**

Rate limiting controls request throughput to protect services from overload and abuse. Common algorithms: token bucket, leaky bucket, sliding window, and fixed window.

```python
import time
from threading import Lock


class TokenBucket:
    def __init__(self, rate: float, capacity: int) -> None:
        self._rate = rate
        self._capacity = capacity
        self._tokens = float(capacity)
        self._last_refill = time.monotonic()
        self._lock = Lock()

    def acquire(self, tokens: int = 1) -> bool:
        with self._lock:
            now = time.monotonic()
            elapsed = now - self._last_refill
            self._tokens = min(
                self._capacity,
                self._tokens + elapsed * self._rate,
            )
            self._last_refill = now

            if self._tokens >= tokens:
                self._tokens -= tokens
                return True
            return False


limiter = TokenBucket(rate=10, capacity=20)

for i in range(25):
    if limiter.acquire():
        print(f"Request {i}: allowed")
    else:
        print(f"Request {i}: rate limited")
```

For distributed rate limiting, use Redis with Lua scripts for atomic token bucket operations, or use a dedicated service like Envoy's rate limiting filter.

### Q46. How do you implement secure password hashing in Python?

**Answer:**

Use **Argon2** (preferred) or **bcrypt** with unique per-user salts. Never use MD5, SHA-1, or SHA-256 alone for password storage.

```python
import argon2

hasher = argon2.PasswordHasher(
    time_cost=3,
    memory_cost=65536,
    parallelism=4,
    hash_len=32,
    salt_len=16,
)

hashed = hasher.hash("user_password_here")

try:
    hasher.verify(hashed, "user_password_here")
    if hasher.check_needs_rehash(hashed):
        hashed = hasher.hash("user_password_here")
except argon2.exceptions.VerifyMismatchError:
    print("Invalid username or password")
```

Key practices:
- **Constant-time comparison**: `argon2` and `bcrypt` handle this internally.
- **Rehashing**: Check `needs_rehash()` on login to upgrade parameters over time.
- **Error messages**: Return "Invalid username or password" — never reveal which part failed.
- **Rate limiting**: Limit login attempts per IP and account.

### Q56. How do you implement ETL pipelines in Python?

**Answer:**

ETL (Extract, Transform, Load) pipelines process data from sources to destinations. For production-grade pipelines:

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any, Iterator


class Extractor(ABC):
    @abstractmethod
    def extract(self) -> Iterator[dict[str, Any]]: ...


class Transformer(ABC):
    @abstractmethod
    def transform(self, record: dict[str, Any]) -> dict[str, Any] | None: ...


class Loader(ABC):
    @abstractmethod
    def load(self, records: list[dict[str, Any]]) -> None: ...


class Pipeline:
    def __init__(
        self,
        extractor: Extractor,
        transformers: list[Transformer],
        loader: Loader,
        batch_size: int = 1000,
    ) -> None:
        self._extractor = extractor
        self._transformers = transformers
        self._loader = loader
        self._batch_size = batch_size

    def run(self) -> dict[str, int]:
        stats = {"extracted": 0, "transformed": 0, "loaded": 0, "errors": 0}
        batch: list[dict] = []

        for record in self._extractor.extract():
            stats["extracted"] += 1
            transformed = record
            try:
                for t in self._transformers:
                    transformed = t.transform(transformed)
                    if transformed is None:
                        break
                if transformed is not None:
                    batch.append(transformed)
                    stats["transformed"] += 1
            except Exception:
                stats["errors"] += 1

            if len(batch) >= self._batch_size:
                self._loader.load(batch)
                stats["loaded"] += len(batch)
                batch = []

        if batch:
            self._loader.load(batch)
            stats["loaded"] += len(batch)

        return stats
```

For orchestration at scale, use **Airflow**, **Dagster**, or **Prefect** for scheduling, retries, monitoring, and dependency management.

### Q59. How do you design idempotent and safe HTTP APIs?

**Answer:**

HTTP methods have defined safety and idempotency semantics:

| Method | Safe | Idempotent | Use |
|--------|------|------------|-----|
| GET | Yes | Yes | Read resources |
| HEAD | Yes | Yes | Check existence/headers |
| PUT | No | Yes | Full replacement |
| DELETE | No | Yes | Remove resource |
| POST | No | No | Create, actions |
| PATCH | No | No | Partial update |

- **GET** must never cause side effects (no writes, no state changes).

### Q60. How do you implement API pagination correctly?

**Answer:**

Pagination strategies:

1. **Offset-based** (`?offset=20&limit=10`): Simple but degrades at large offsets (DB must skip rows).
2. **Cursor-based** (`?cursor=eyJpZCI6MTAwfQ&limit=10`): Stable under insertions/deletions; opaque cursor encodes position.
3. **Keyset/seek** (`?after_id=100&limit=10`): Uses indexed column for efficient seeking.

```python
import base64
import json
from dataclasses import dataclass


@dataclass
class Page:
    items: list[dict]
    next_cursor: str | None
    has_more: bool


def paginate_cursor(
    query_fn,
    cursor: str | None,
    limit: int = 20,
) -> Page:
    after_id = None
    if cursor:
        after_id = json.loads(base64.b64decode(cursor))["id"]

    items = query_fn(after_id=after_id, limit=limit + 1)
    has_more = len(items) > limit
    items = items[:limit]

    next_cursor = None
    if has_more and items:
        next_cursor = base64.b64encode(
            json.dumps({"id": items[-1]["id"]}).encode()
        ).decode()

    return Page(items=items, next_cursor=next_cursor, has_more=has_more)
```

Response format following JSON:API or custom envelope:
```json
{
  "data": [...],
  "pagination": {
    "next_cursor": "eyJpZCI6MTAwfQ",
    "has_more": true
  }
}
```

### Q67. How do you implement the Specification pattern in Python?

**Answer:**

The Specification pattern encapsulates business rules as composable, reusable objects. Each specification answers "does this object satisfy the rule?" and can be combined with AND, OR, NOT:

```python
from abc import ABC, abstractmethod
from typing import Generic, TypeVar

T = TypeVar("T")


class Specification(ABC, Generic[T]):
    @abstractmethod
    def is_satisfied_by(self, candidate: T) -> bool: ...

    def __and__(self, other: "Specification[T]") -> "Specification[T]":
        return AndSpec(self, other)

    def __or__(self, other: "Specification[T]") -> "Specification[T]":
        return OrSpec(self, other)

    def __invert__(self) -> "Specification[T]":
        return NotSpec(self)


class AndSpec(Specification[T]):
    def __init__(self, a: Specification[T], b: Specification[T]) -> None:
        self._a, self._b = a, b

    def is_satisfied_by(self, candidate: T) -> bool:
        return self._a.is_satisfied_by(candidate) and self._b.is_satisfied_by(candidate)


class OrSpec(Specification[T]):
    def __init__(self, a: Specification[T], b: Specification[T]) -> None:
        self._a, self._b = a, b

    def is_satisfied_by(self, candidate: T) -> bool:
        return self._a.is_satisfied_by(candidate) or self._b.is_satisfied_by(candidate)


class NotSpec(Specification[T]):
    def __init__(self, spec: Specification[T]) -> None:
        self._spec = spec

    def is_satisfied_by(self, candidate: T) -> bool:
        return not self._spec.is_satisfied_by(candidate)


class MinAge(Specification):
    def __init__(self, age: int) -> None:
        self._age = age

    def is_satisfied_by(self, candidate) -> bool:
        return candidate.age >= self._age


class HasVerifiedEmail(Specification):
    def is_satisfied_by(self, candidate) -> bool:
        return candidate.email_verified


eligible = MinAge(18) & HasVerifiedEmail()
```

### Q68. How do you implement lazy loading and the Proxy pattern in Python?

**Answer:**

Lazy loading defers expensive initialization until first use. The Proxy pattern wraps an object and controls access — lazy initialization is one common application:

```python
from typing import TypeVar, Generic, Callable

T = TypeVar("T")


class Lazy(Generic[T]):
    def __init__(self, factory: Callable[[], T]) -> None:
        self._factory = factory
        self._instance: T | None = None
        self._initialized = False

    def __call__(self) -> T:
        if not self._initialized:
            self._instance = self._factory()
            self._initialized = True
        return self._instance


class LazyProxy:
    def __init__(self, factory: Callable) -> None:
        object.__setattr__(self, "_factory", factory)
        object.__setattr__(self, "_instance", None)
        object.__setattr__(self, "_initialized", False)

    def _ensure_initialized(self):
        if not object.__getattribute__(self, "_initialized"):
            instance = object.__getattribute__(self, "_factory")()
            object.__setattr__(self, "_instance", instance)
            object.__setattr__(self, "_initialized", True)

    def __getattr__(self, name):
        self._ensure_initialized()
        return getattr(object.__getattribute__(self, "_instance"), name)


db = LazyProxy(lambda: create_database_connection())
# Connection created only when first attribute is accessed
```

Use cases: database connections, large model loading (ML), configuration that depends on runtime state.

### Q69. How do you implement event sourcing in Python?

**Answer:**

Event sourcing persists all state changes as an append-only sequence of events. Current state is derived by replaying events. This provides a complete audit trail and enables temporal queries.

```python
from dataclasses import dataclass, field
from datetime import datetime, timezone
from uuid import UUID, uuid4


@dataclass(frozen=True)
class Event:
    aggregate_id: UUID
    timestamp: datetime = field(default_factory=lambda: datetime.now(tz=timezone.utc))
    version: int = 0


@dataclass(frozen=True)
class AccountCreated(Event):
    owner: str = ""


@dataclass(frozen=True)
class MoneyDeposited(Event):
    amount: float = 0.0


@dataclass(frozen=True)
class MoneyWithdrawn(Event):
    amount: float = 0.0


class BankAccount:
    def __init__(self) -> None:
        self.id: UUID | None = None
        self.owner: str = ""
        self.balance: float = 0.0
        self.version: int = 0
        self._pending_events: list[Event] = []

    def apply(self, event: Event) -> None:
        match event:
            case AccountCreated(aggregate_id=aid, owner=owner):
                self.id = aid
                self.owner = owner
            case MoneyDeposited(amount=amount):
                self.balance += amount
            case MoneyWithdrawn(amount=amount):
                self.balance -= amount
        self.version = event.version

    @classmethod
    def from_events(cls, events: list[Event]) -> "BankAccount":
        account = cls()
        for event in events:
            account.apply(event)
        return account

    def deposit(self, amount: float) -> None:
        if amount <= 0:
            raise ValueError("Amount must be positive")
        event = MoneyDeposited(
            aggregate_id=self.id,
            amount=amount,
            version=self.version + 1,
        )
        self.apply(event)
        self._pending_events.append(event)

    @property
    def pending_events(self) -> list[Event]:
        return list(self._pending_events)
```

Event sourcing works well with CQRS — events update read models via projections. Use an event store (EventStoreDB, PostgreSQL with append-only tables) for persistence.

### Q70. What is the `__subclasshook__` method and how does it work with ABCs?

**Answer:**

`__subclasshook__` is a classmethod on ABCs that customizes `isinstance` and `issubclass` checks without requiring explicit registration or inheritance. It enables "virtual subclass" detection based on structural compatibility.

```python
from abc import ABC, abstractmethod


class Closeable(ABC):
    @abstractmethod
    def close(self) -> None: ...

    @classmethod
    def __subclasshook__(cls, C):
        if cls is Closeable:
            if hasattr(C, "close") and callable(getattr(C, "close")):
                return True
        return NotImplemented


class FileWrapper:
    def close(self) -> None:
        pass
```

### Q76. How do you implement database sharding strategies in Python?

**Answer:**

Sharding distributes data across multiple database instances. Common strategies:

```python
import hashlib
from typing import Any


class ShardRouter:
    def __init__(self, shard_count: int, connections: dict[int, Any]) -> None:
        self._shard_count = shard_count
        self._connections = connections

    def get_shard(self, shard_key: str) -> int:
        hash_val = int(hashlib.sha256(shard_key.encode()).hexdigest(), 16)
        return hash_val % self._shard_count

    def get_connection(self, shard_key: str) -> Any:
        shard_id = self.get_shard(shard_key)
        return self._connections[shard_id]

    def scatter_gather(self, query_fn, **kwargs) -> list:
        results = []
        for conn in self._connections.values():
            results.extend(query_fn(conn, **kwargs))
        return results
```

Considerations:
- **Shard key selection**: Choose a key with even distribution (user_id, tenant_id).
- **Cross-shard queries**: Expensive; use scatter-gather or denormalize.
- **Rebalancing**: Adding shards requires data migration — use consistent hashing to minimize movement.
- **Transactions**: Cross-shard transactions are complex; prefer designs where transactions stay within one shard.

### Q77. How do you implement retry policies with jitter for distributed systems?

**Answer:**

Retries with exponential backoff and jitter prevent thundering herd problems when a dependency recovers:

```python
import asyncio
import random
from collections.abc import Awaitable, Callable
from typing import TypeVar

T = TypeVar("T")


async def retry_with_backoff(
    fn: Callable[..., Awaitable[T]],
    *args,
    max_retries: int = 5,
    base_delay: float = 0.1,
    max_delay: float = 30.0,
    jitter_factor: float = 0.5,
    retryable_exceptions: tuple[type[Exception], ...] = (Exception,),
    **kwargs,
) -> T:
    last_exception: Exception | None = None

    for attempt in range(max_retries + 1):
        try:
            return await fn(*args, **kwargs)
        except retryable_exceptions as e:
            last_exception = e
            if attempt == max_retries:
                break

            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = delay * jitter_factor * random.random()
            actual_delay = delay + jitter

            await asyncio.sleep(actual_delay)

    assert last_exception is not None
    raise last_exception
```

**Jitter strategies**:
- **Full jitter**: `random(0, base * 2^attempt)` — most spread, lowest collision.
- **Equal jitter**: `base * 2^attempt / 2 + random(0, base * 2^attempt / 2)` — balanced.
- **Decorrelated jitter**: `min(cap, random(base, prev_delay * 3))` — good practical choice.

### Q78. How do you implement a leader election mechanism in Python?

**Answer:**

Leader election ensures only one instance performs certain tasks (scheduled jobs, singleton processing) in a distributed environment:

```python
import time
import threading
from uuid import uuid4


class RedisLeaderElection:
    def __init__(self, redis_client, lock_name: str, ttl: int = 30) -> None:
        self._redis = redis_client
        self._lock_name = lock_name
        self._ttl = ttl
        self._instance_id = str(uuid4())
        self._is_leader = False
        self._renewal_thread: threading.Thread | None = None

    def try_acquire(self) -> bool:
        acquired = self._redis.set(
            self._lock_name,
            self._instance_id,
            nx=True,
            ex=self._ttl,
        )
        if acquired:
            self._is_leader = True
            self._start_renewal()
        return bool(acquired)

    def release(self) -> None:
        if self._is_leader:
            current = self._redis.get(self._lock_name)
            if current and current.decode() == self._instance_id:
                self._redis.delete(self._lock_name)
            self._is_leader = False

    def _start_renewal(self) -> None:
        def renew():
            while self._is_leader:
                self._redis.expire(self._lock_name, self._ttl)
                time.sleep(self._ttl // 3)

        self._renewal_thread = threading.Thread(target=renew, daemon=True)
        self._renewal_thread.start()

    @property
    def is_leader(self) -> bool:
        return self._is_leader
```

### Q79. How do you implement a distributed cache with Python?

**Answer:**

Multi-level caching with cache-aside pattern:

```python
from typing import Any, Callable
import time
import json


class MultiLevelCache:
    def __init__(self, local_cache, redis_client) -> None:
        self._local = local_cache
        self._redis = redis_client

    def get(self, key: str) -> Any | None:
        value = self._local.get(key)
        if value is not None:
            return value

        raw = self._redis.get(key)
        if raw is not None:
            value = json.loads(raw)
            self._local.set(key, value, ttl=60)
            return value

        return None

    def set(self, key: str, value: Any, ttl: int = 300) -> None:
        self._local.set(key, value, ttl=min(ttl, 60))
        self._redis.setex(key, ttl, json.dumps(value, default=str))

    def invalidate(self, key: str) -> None:
        self._local.delete(key)
        self._redis.delete(key)

    def get_or_set(
        self,
        key: str,
        factory: Callable[[], Any],
        ttl: int = 300,
    ) -> Any:
        value = self.get(key)
        if value is not None:
            return value
        value = factory()
        self.set(key, value, ttl)
        return value
```

Patterns:
- **Cache-aside**: Application manages cache reads/writes.
- **Write-through**: Writes go to cache and DB together.
- **Write-behind**: Writes go to cache first, DB asynchronously.
- **Cache stampede prevention**: Use locks or probabilistic early expiration.

### Q82. How do you implement the Decorator pattern (GoF) differently from Python decorators?

**Answer:**

The GoF Decorator pattern wraps objects to add behavior while maintaining the same interface — different from Python's `@decorator` syntax which wraps functions/classes at definition time.

```python
from typing import Protocol


class Notifier(Protocol):
    def send(self, message: str) -> None: ...


class EmailNotifier:
    def send(self, message: str) -> None:
        print(f"Email: {message}")


class SlackNotifierDecorator:

    def __init__(self, wrapped: Notifier) -> None:
        self._wrapped = wrapped
```

- GoF Decorator maintains interface compatibility; Python decorators can change signatures.

### Q83. How do you implement connection pool management for async Python?

**Answer:**

```python
import asyncio
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator


class AsyncConnectionPool:
    def __init__(
        self,
        factory,
        min_size: int = 5,
        max_size: int = 20,
        max_idle_time: float = 300,
    ) -> None:
        self._factory = factory
        self._min_size = min_size
        self._max_size = max_size
        self._max_idle_time = max_idle_time
        self._pool: asyncio.Queue = asyncio.Queue(maxsize=max_size)
        self._size = 0
        self._lock = asyncio.Lock()

    async def initialize(self) -> None:
        for _ in range(self._min_size):
            conn = await self._factory()
            await self._pool.put(conn)
            self._size += 1

    @asynccontextmanager
    async def acquire(self) -> AsyncIterator:
        conn = await self._get_connection()
        try:
            yield conn
        except Exception:
            await self._discard(conn)
            raise
        else:
            await self._return(conn)

    async def _get_connection(self):
        try:
            return self._pool.get_nowait()
        except asyncio.QueueEmpty:
            async with self._lock:
                if self._size < self._max_size:
                    self._size += 1
                    return await self._factory()
            return await asyncio.wait_for(
                self._pool.get(), timeout=10.0
            )

    async def _return(self, conn) -> None:
        if await self._is_healthy(conn):
            await self._pool.put(conn)
        else:
            await self._discard(conn)

    async def _discard(self, conn) -> None:
        async with self._lock:
            self._size -= 1
        await conn.close()

    async def _is_healthy(self, conn) -> bool:
        try:
            await conn.execute("SELECT 1")
            return True
        except Exception:
            return False
```

### Q85. How do you implement a write-ahead log (WAL) in Python?

**Answer:**

A WAL ensures durability by writing operations to a log before applying them to the main data store. If the process crashes, the WAL is replayed to recover state:

```python
import json
import os
from dataclasses import dataclass, asdict
from datetime import datetime, timezone
from pathlib import Path
from typing import Any
from uuid import uuid4


@dataclass
class WALEntry:
    sequence: int
    operation: str
    data: dict[str, Any]
    timestamp: str
    checksum: str = ""


class WriteAheadLog:
    def __init__(self, path: Path) -> None:
        self._path = path
        self._sequence = 0
        self._file = None

    def open(self) -> None:
        self._path.parent.mkdir(parents=True, exist_ok=True)
        self._file = open(self._path, "a", encoding="utf-8")
        self._sequence = self._count_entries()

    def append(self, operation: str, data: dict) -> int:
        self._sequence += 1
        entry = WALEntry(
            sequence=self._sequence,
            operation=operation,
            data=data,
            timestamp=datetime.now(tz=timezone.utc).isoformat(),
        )
        line = json.dumps(asdict(entry))
        self._file.write(line + "\n")
        self._file.flush()
        os.fsync(self._file.fileno())
        return self._sequence

    def replay(self):
        with open(self._path, "r", encoding="utf-8") as f:
            for line in f:
                yield WALEntry(**json.loads(line.strip()))

    def truncate(self, up_to_sequence: int) -> None:
        remaining = []
        for entry in self.replay():
            if entry.sequence > up_to_sequence:
                remaining.append(json.dumps(asdict(entry)))

        with open(self._path, "w", encoding="utf-8") as f:
            for line in remaining:
                f.write(line + "\n")

    def _count_entries(self) -> int:
        if not self._path.exists():
            return 0
        with open(self._path, "r", encoding="utf-8") as f:
            return sum(1 for _ in f)

    def close(self) -> None:
        if self._file:
            self._file.close()
```

Custom linter rules enforce project-specific conventions. Using AST visitors:

```python
import ast
import sys
from pathlib import Path


class NoGlobalMutableDefaults(ast.NodeVisitor):
    def __init__(self) -> None:
        self.errors: list[tuple[int, str]] = []

    def visit_FunctionDef(self, node: ast.FunctionDef) -> None:
        for default in node.args.defaults + node.args.kw_defaults:
            if default is None:
                continue
            if isinstance(default, (ast.List, ast.Dict, ast.Set)):
                self.errors.append((
                    default.lineno,
                    f"Mutable default argument in {node.name}()"
                ))
        self.generic_visit(node)

    visit_AsyncFunctionDef = visit_FunctionDef


def check_file(path: Path) -> list[tuple[int, str]]:
    source = path.read_text(encoding="utf-8")
    tree = ast.parse(source, filename=str(path))
    checker = NoGlobalMutableDefaults()
    checker.visit(tree)
    return checker.errors


if __name__ == "__main__":
    for path in Path("src").rglob("*.py"):
        errors = check_file(path)
        for lineno, msg in errors:
            print(f"{path}:{lineno}: {msg}")
```

### Q88. How do you implement data migration scripts safely?

**Answer:**

```python
import logging
from contextlib import contextmanager
from datetime import datetime, timezone

log = logging.getLogger(__name__)


class Migration:
    def __init__(self, version: str, description: str) -> None:
        self.version = version
        self.description = description

    def up(self, session) -> None:
        raise NotImplementedError

    def down(self, session) -> None:
        raise NotImplementedError


class MigrationRunner:
    def __init__(self, session_factory) -> None:
        self._session_factory = session_factory

    def run(self, migration: Migration, dry_run: bool = False) -> None:
        log.info(f"Running migration {migration.version}: {migration.description}")

        session = self._session_factory()
        try:
            if dry_run:
                migration.up(session)
                session.rollback()
                log.info("Dry run complete — rolled back")
            else:
                migration.up(session)
                self._record_migration(session, migration)
                session.commit()
                log.info(f"Migration {migration.version} applied")
        except Exception:
            session.rollback()
            log.exception(f"Migration {migration.version} failed")
            raise
        finally:
            session.close()

    def _record_migration(self, session, migration: Migration) -> None:
        session.execute(
            "INSERT INTO migration_history (version, description, applied_at) "
            "VALUES (?, ?, ?)",
            (migration.version, migration.description,
             datetime.now(tz=timezone.utc)),
        )
```

Safety checklist:
- **Always have a rollback** (`down` method) tested independently.
- **Dry run mode**: Execute against production data without committing.
- **Batch processing**: For large tables, process in chunks with progress logging.
- **Backward compatibility**: Ensure old code works with new schema during deployment.
- **Monitoring**: Alert on migration duration and failure.

### Q89. How do you implement a Python application that handles millions of WebSocket connections?

**Answer:**

Scaling WebSocket connections requires careful architecture:

```python
import asyncio
from collections import defaultdict
from weakref import WeakSet


class ConnectionManager:
    def __init__(self) -> None:
        self._connections: dict[str, WeakSet] = defaultdict(WeakSet)
        self._user_connections: dict[str, WeakSet] = defaultdict(WeakSet)

    async def connect(self, websocket, channel: str, user_id: str) -> None:
        self._connections[channel].add(websocket)
        self._user_connections[user_id].add(websocket)

    async def disconnect(self, websocket, channel: str, user_id: str) -> None:
        self._connections[channel].discard(websocket)
        self._user_connections[user_id].discard(websocket)

    async def broadcast(self, channel: str, message: str) -> None:
        dead = []
        for ws in self._connections[channel]:
            try:
                await ws.send_text(message)
            except Exception:
                dead.append(ws)
        for ws in dead:
            self._connections[channel].discard(ws)

    async def send_to_user(self, user_id: str, message: str) -> None:
        for ws in self._user_connections.get(user_id, set()):
            try:
                await ws.send_text(message)
            except Exception:
                pass
```

Scaling strategy:
- **Single server**: ~10K-50K connections per process with uvloop.
- **Horizontal scaling**: Use Redis Pub/Sub or Kafka for cross-instance messaging.
- **Connection distribution**: Load balance WebSocket upgrades across nodes.
- **Heartbeats**: Detect dead connections; clean up resources.
- **Memory**: Each connection uses ~5-20KB; 1M connections ≈ 5-20GB RAM.

### Q91. How do you design Python applications for multi-tenancy?

**Answer:**

Multi-tenancy strategies:

1. **Shared database, shared schema**: Tenant ID column on every table. Simplest but requires careful query scoping.
2. **Shared database, separate schemas**: PostgreSQL schemas per tenant. Better isolation, moderate complexity.
3. **Separate databases**: Maximum isolation. Complex management but strongest security guarantees.

```python
from contextvars import ContextVar
from typing import Any

current_tenant: ContextVar[str] = ContextVar("current_tenant")


class TenantMiddleware:
    async def __call__(self, request, call_next):
        tenant_id = request.headers.get("X-Tenant-ID")
        if not tenant_id:
            raise HTTPException(status_code=400, detail="Missing tenant ID")

        token = current_tenant.set(tenant_id)
        try:
            response = await call_next(request)
            return response
        finally:
            current_tenant.reset(token)


class TenantAwareRepository:
    def __init__(self, session) -> None:
        self._session = session

    def query(self, model_class, **filters):
        tenant_id = current_tenant.get()
        return (
            self._session.query(model_class)
            .filter_by(tenant_id=tenant_id, **filters)
        )
```

Use `contextvars` to propagate tenant context through async call stacks. Apply query filtering at the repository layer as a safety net, regardless of what callers pass.

### Q93. How do you implement hot module reloading for development?

**Answer:**

Hot reloading detects source changes and restarts or reloads the application without manual intervention:

```python
import importlib
import sys
import time
from pathlib import Path
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler


class ModuleReloader(FileSystemEventHandler):
    def __init__(self, watch_modules: list[str]) -> None:
        self._modules = watch_modules

    def on_modified(self, event):
        if event.src_path.endswith(".py"):
            module_name = self._path_to_module(event.src_path)
            if module_name and module_name in sys.modules:
                try:
                    importlib.reload(sys.modules[module_name])
                    print(f"Reloaded: {module_name}")
                except Exception as e:
                    print(f"Reload failed: {module_name}: {e}")

    def _path_to_module(self, path: str) -> str | None:
        p = Path(path)
        parts = list(p.with_suffix("").parts)
        if "src" in parts:
            idx = parts.index("src")
            return ".".join(parts[idx + 1:])
        return None
```

Production frameworks (Uvicorn `--reload`, Flask debug mode) use similar file-watching mechanisms. For production, never enable hot reloading — use rolling deployments instead.

```python
from dataclasses import dataclass

class ErrorCode(Enum):
```

### Q95. How do you implement data partitioning strategies for time-series data in Python?

**Answer:**

Time-series data grows continuously and requires partitioning for query performance and storage management:

```python
from datetime import datetime, timezone
from pathlib import Path


class TimePartitionedStore:
    def __init__(self, base_path: Path) -> None:
        self._base = base_path

    def _partition_key(self, timestamp: datetime) -> str:
        return timestamp.strftime("%Y/%m/%d")

    def _partition_path(self, timestamp: datetime) -> Path:
        return self._base / self._partition_key(timestamp)

    def write(self, timestamp: datetime, data: bytes) -> None:
        path = self._partition_path(timestamp)
        path.mkdir(parents=True, exist_ok=True)
        filename = f"{timestamp.strftime('%H%M%S')}_{id(data)}.bin"
        (path / filename).write_bytes(data)

    def read_range(
        self, start: datetime, end: datetime
    ) -> list[tuple[Path, bytes]]:
        results = []
        current = start.replace(hour=0, minute=0, second=0)
        while current <= end:
            path = self._partition_path(current)
            if path.exists():
                for f in sorted(path.iterdir()):
                    results.append((f, f.read_bytes()))
            current = current.replace(
                day=current.day + 1
            ) if current.month == (current + __import__('datetime').timedelta(days=1)).month else current
        return results
```

Database-level partitioning:
- **PostgreSQL**: Declarative partitioning by range on timestamp columns.
- **TimescaleDB**: Automatic time-based hypertables.
- **ClickHouse**: Partition by month/day for efficient pruning.
- **File-based**: Parquet files partitioned by date in S3/GCS for data lake patterns.

### Q98. How do you implement chaos engineering practices with Python?

**Answer:**

Chaos engineering intentionally introduces failures to verify system resilience:

```python
import asyncio
import random
from functools import wraps
from typing import Any


class ChaosMonkey:
    def __init__(self, enabled: bool = False) -> None:
        self.enabled = enabled
        self.latency_injection_rate = 0.1
        self.error_injection_rate = 0.05
        self.max_latency_ms = 2000

    def inject_chaos(self, func):
        @wraps(func)
        async def wrapper(*args: Any, **kwargs: Any) -> Any:
            if not self.enabled:
                return await func(*args, **kwargs)

            if random.random() < self.error_injection_rate:
                raise ConnectionError("Chaos: simulated failure")

            if random.random() < self.latency_injection_rate:
                delay = random.uniform(0.1, self.max_latency_ms / 1000)
                await asyncio.sleep(delay)

            return await func(*args, **kwargs)
        return wrapper


chaos = ChaosMonkey(enabled=False)


@chaos.inject_chaos
async def call_payment_api(order_id: str) -> dict:
    # Real implementation
    return {"status": "success"}
```

Practices:
- **Start in staging**: Never run chaos experiments in production without circuit breakers and rollback capability.
- **Hypothesis-driven**: "If Redis goes down, we expect graceful degradation with cached data."
- **Blast radius control**: Limit to a percentage of traffic or specific service instances.
- **Automated rollback**: Stop experiments if error rates exceed thresholds.

### Q100. How do you evaluate and adopt new Python features for a production codebase?

**Answer:**

A structured approach to adopting new language features:

**Evaluation criteria**:
1. **Readability impact**: Does it make code clearer? (e.g., pattern matching for complex dispatch, `X | Y` union syntax).
2. **Performance benefit**: Measurable improvement? (e.g., `TaskGroup` for structured concurrency, specializing adaptive interpreter).
3. **Ecosystem readiness**: Do key dependencies, linters, and type checkers support it?
4. **Team familiarity**: Is the team comfortable? Training cost?
5. **Migration cost**: How much existing code needs changing?

**Adoption process**:
1. **Experiment**: Try in a branch or side project; benchmark if performance-motivated.
2. **Minimum version bump**: Update `requires-python` only when the feature is worth the version floor increase.
3. **Gradual rollout**: Use in new code first; refactor existing code opportunistically.
4. **Linter enforcement**: Configure `ruff` target-version to enable/require new syntax.
5. **Documentation**: Update team style guide with examples and rationale.

**Recent features worth evaluating (3.11-3.13+)**:
- **`TaskGroup` and `ExceptionGroup`** (3.11): Structured concurrency — adopt for async services.
- **`tomllib`** (3.11): TOML parsing in stdlib — replace `tomli` dependency.
- **`type` statement and `class Foo[T]`** (3.12): Cleaner generics — adopt when minimum version allows.
- **`sys.monitoring`** (3.12): Low-overhead instrumentation — adopt for production profiling.
- **Free-threaded mode** (3.13+): Experimental — evaluate for CPU-bound workloads but don't depend on yet.
- **JIT compiler** (3.13+): Experimental tier — monitor benchmarks but no code changes needed.

The key principle: adopt features that improve **correctness and clarity** first, **performance** second, and **novelty** never as a sole reason.
