### Q1. What is Django?

Django is a high-level Python web framework that encourages rapid development and clean, pragmatic design. It was originally developed by Adrian Holovaty and Simon Willison at the Lawrence Journal-World newspaper in 2003 and was publicly released in 2005.

Key characteristics:
- **Open source** — released under the BSD license.
- **Batteries included** — ships with an ORM, authentication system, admin panel, form handling, template engine, and more.
- **Follows the DRY (Don't Repeat Yourself) principle** — minimizes code duplication.
- **Secure by default** — protects against SQL injection, XSS, CSRF, and clickjacking out of the box.

Django is maintained by the Django Software Foundation and has a large, active community.

### Q2. What is the MTV architecture pattern in Django?

Django follows the **Model-Template-View (MTV)** pattern, which is Django's interpretation of the classic MVC (Model-View-Controller) pattern.

| MTV Component | MVC Equivalent | Responsibility |
|---------------|----------------|----------------|
| **Model** | Model | Defines data structure, database schema, and business logic related to data. |
| **Template** | View | Handles the presentation layer — HTML with Django Template Language (DTL). |
| **View** | Controller | Contains the business logic, processes requests, interacts with models, and returns responses. |

The flow works like this:
1. User sends an HTTP request.
2. Django's URL dispatcher routes the request to the appropriate **View**.
3. The View interacts with the **Model** to fetch or manipulate data.
4. The View passes data to a **Template** which renders the HTML response.
5. The response is sent back to the user's browser.

### Q3. Explain the Django project structure.

When you run `django-admin startproject myproject`, the following structure is created:

```
myproject/
├── manage.py
└── myproject/
    ├── __init__.py
    ├── settings.py
    ├── urls.py
    ├── asgi.py
    └── wsgi.py
```

- **`manage.py`** — A command-line utility that lets you interact with the Django project (run the server, make migrations, create apps, etc.).
- **`__init__.py`** — Marks the directory as a Python package.
- **`settings.py`** — Contains all project configuration: database settings, installed apps, middleware, template configuration, static files, etc.
- **`urls.py`** — The root URL configuration. Maps URL patterns to views.
- **`asgi.py`** — Entry point for ASGI-compatible web servers (for async support).
- **`wsgi.py`** — Entry point for WSGI-compatible web servers (traditional synchronous deployment).

### Q4. What is the difference between a Django project and a Django app?

| Aspect | Project | App |
|--------|---------|-----|
| **Definition** | A collection of configurations and apps that together form a web application. | A self-contained module that performs a specific functionality. |
| **Creation** | `django-admin startproject myproject` | `python manage.py startapp myapp` |
| **Scope** | One project per web application. | A project can contain multiple apps. |
| **Reusability** | Not reusable across other projects. | Designed to be reusable and pluggable. |
| **Contains** | `settings.py`, root `urls.py`, `wsgi.py`, `asgi.py`. | `models.py`, `views.py`, `urls.py`, `admin.py`, `tests.py`. |

**Example:** A project called `ecommerce` might have apps like `products`, `orders`, `users`, and `payments`.

### Q8. What is the Django request-response cycle?

The complete lifecycle of an HTTP request in Django:

1. **Browser sends HTTP request** → reaches the web server (Nginx/Apache).
2. **WSGI/ASGI handler** receives the request and creates an `HttpRequest` object.
3. **Middleware (request phase)** — each middleware processes the request in order (e.g., session, authentication, security checks).
4. **URL Resolver** — Django matches the URL against patterns defined in `urls.py`.
5. **View function/class** — the matched view is called with the `HttpRequest` object.
6. **View processes business logic** — queries the database via models, processes data.
7. **Template rendering** — the view renders a template with context data (or returns JSON for APIs).
8. **Middleware (response phase)** — each middleware processes the response in reverse order.
9. **`HttpResponse` is returned** to the browser.

### Q9. What is WSGI and ASGI in Django?

**WSGI (Web Server Gateway Interface):**
- The traditional Python standard (PEP 3333) for communication between web servers and Python applications.
- Synchronous — handles one request at a time per worker.
- Used with servers like Gunicorn, uWSGI.
- Django's `wsgi.py` provides the WSGI application callable.

**ASGI (Asynchronous Server Gateway Interface):**
- The async successor to WSGI, designed for modern async Python.
- Supports HTTP, WebSockets, and long-lived connections.
- Used with servers like Daphne, Uvicorn, Hypercorn.
- Django's `asgi.py` provides the ASGI application callable.
- Django 3.0+ supports ASGI.

```python
# wsgi.py
application = get_wsgi_application()

# asgi.py
application = get_asgi_application()
```

### Q11. What is Django middleware?

Middleware is a framework of hooks that process requests and responses globally before they reach the view or after the view returns a response. Middleware is defined as a list in `settings.py`.

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
]
```

**Execution order:**
- **Request phase:** Top to bottom.
- **Response phase:** Bottom to top.

Each middleware can implement these methods:
- `__init__(self, get_response)` — called once at startup.
- `__call__(self, request)` — called on every request.
- `process_view(self, request, view_func, view_args, view_kwargs)` — called before the view.
- `process_exception(self, request, exception)` — called if the view raises an exception.
- `process_template_response(self, request, response)` — called after the view if the response has a `render()` method.

### Q12. What is the `urls.py` file and how does URL routing work?

`urls.py` maps URL patterns to views. Django uses a URL dispatcher to match incoming requests to the appropriate view.

```python
# project/urls.py
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('blog/', include('blog.urls')),
]

# blog/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path('', views.index, name='blog-home'),
    path('<int:pk>/', views.detail, name='blog-detail'),
    path('<slug:slug>/', views.by_slug, name='blog-slug'),
]
```

**Key functions:**
- `path()` — defines a URL pattern with built-in converters (`int`, `str`, `slug`, `uuid`, `path`).
- `re_path()` — uses regex for complex URL patterns.
- `include()` — includes URL patterns from another app.
- `name` parameter — gives the URL pattern a unique name for reverse URL resolution.

### Q13. What is the difference between `path()` and `re_path()`?

| Feature | `path()` | `re_path()` |
|---------|----------|-------------|
| **Syntax** | Simple, readable | Regular expressions |
| **Import** | `from django.urls import path` | `from django.urls import re_path` |
| **Converters** | Built-in: `int`, `str`, `slug`, `uuid`, `path` | Regex groups: `(?P<name>pattern)` |
| **Readability** | More readable | Less readable |
| **Flexibility** | Less flexible | Highly flexible |

```python
# path()
path('articles/<int:year>/', views.year_archive)

# re_path() - equivalent
re_path(r'^articles/(?P<year>[0-9]{4})/$', views.year_archive)
```

**Guideline:** Use `path()` for most cases. Use `re_path()` only when you need regex-level control over URL matching.

### Q15. What is the `HttpResponse` object?

`HttpResponse` is what a Django view must return. It contains the response content, status code, and headers.

```python
from django.http import HttpResponse, JsonResponse, HttpResponseRedirect

# Basic text response
def hello(request):
    return HttpResponse("Hello, World!", content_type="text/plain")

# HTML response
def home(request):
    return HttpResponse("<h1>Home Page</h1>")

# JSON response
def api_data(request):
    return JsonResponse({"name": "Django", "version": "5.x"})

# Redirect
def old_page(request):
    return HttpResponseRedirect('/new-page/')
```

Common response classes:

| Class | Status Code | Purpose |
|-------|-------------|---------|
| `HttpResponse` | 200 | Standard response. |
| `JsonResponse` | 200 | JSON response (sets `Content-Type: application/json`). |
| `HttpResponseRedirect` | 302 | Temporary redirect. |
| `HttpResponsePermanentRedirect` | 301 | Permanent redirect. |
| `HttpResponseNotFound` | 404 | Not found. |
| `HttpResponseForbidden` | 403 | Forbidden. |
| `HttpResponseBadRequest` | 400 | Bad request. |
| `HttpResponseServerError` | 500 | Server error. |

### Q16. What is a Django view?

A view is a Python function (or class) that takes an `HttpRequest` and returns an `HttpResponse`. Views contain the business logic of your application.

**Function-Based View (FBV):**
```python
from django.http import HttpResponse

def hello(request):
    return HttpResponse("Hello, World!")
```

**Class-Based View (CBV):**
```python
from django.views import View
from django.http import HttpResponse

class HelloView(View):
    def get(self, request):
        return HttpResponse("Hello, World!")
```

---

### Q17. What is the difference between Function-Based Views and Class-Based Views?

| Feature | FBV | CBV |
|---------|-----|-----|
| **Syntax** | Simple Python functions | Python classes inheriting from `View` or generic views |
| **HTTP Methods** | Handled via `if request.method == 'GET'` | Separate methods: `get()`, `post()`, `put()`, `delete()` |
| **Reusability** | Limited; requires decorators/mixins | Highly reusable via inheritance and mixins |
| **Complexity** | Simple, explicit | More abstract, steeper learning curve |
| **Decorators** | Applied directly | Use `method_decorator` or `dispatch` override |
| **Best For** | Simple views, one-off logic | CRUD operations, standard patterns |

```python
# FBV
def post_list(request):
    if request.method == 'GET':
        posts = Post.objects.all()
        return render(request, 'posts/list.html', {'posts': posts})

# CBV
from django.views.generic import ListView

class PostListView(ListView):
    model = Post
    template_name = 'posts/list.html'
    context_object_name = 'posts'
```

### Q18. What are Django Generic Views?

Generic views are pre-built class-based views that handle common patterns. They reduce boilerplate code significantly.

| Generic View | Purpose |
|-------------|---------|
| `TemplateView` | Renders a template with optional context. |
| `ListView` | Displays a list of objects from a queryset. |
| `DetailView` | Displays a single object. |
| `CreateView` | Displays a form and creates a new object. |
| `UpdateView` | Displays a form and updates an existing object. |
| `DeleteView` | Confirms and deletes an object. |
| `FormView` | Displays a form and handles submission. |
| `RedirectView` | Redirects to a given URL. |

```python
from django.views.generic import ListView, DetailView

class ArticleListView(ListView):
    model = Article
    template_name = 'articles/list.html'
    context_object_name = 'articles'
    paginate_by = 10

class ArticleDetailView(DetailView):
    model = Article
    template_name = 'articles/detail.html'
    context_object_name = 'article'
```

### Q19. What is the `render()` function?

`render()` is a shortcut function that combines a template with a context dictionary and returns an `HttpResponse` with the rendered text.

```python
from django.shortcuts import render

def home(request):
    context = {
        'title': 'Home Page',
        'items': Item.objects.all(),
    }
    return render(request, 'home.html', context)
```

It is equivalent to:

```python
from django.template.loader import get_template
from django.http import HttpResponse

def home(request):
    template = get_template('home.html')
    html = template.render({'title': 'Home Page'}, request)
    return HttpResponse(html)
```

**Parameters:**
- `request` — the HTTP request object.
- `template_name` — path to the template file.
- `context` — dictionary of data available in the template (optional).
- `content_type` — MIME type (defaults to `text/html`).
- `status` — HTTP status code (defaults to 200).

### Q20. What is the Django Template Language (DTL)?

DTL is Django's built-in template engine for embedding dynamic content in HTML.

**Variables:**
```html
{{ variable }}
{{ user.username }}
{{ article.title|upper }}
```

**Tags (logic):**
```html
{% if user.is_authenticated %}
    <p>Welcome, {{ user.username }}</p>
{% else %}
    <p>Please log in.</p>
{% endif %}

{% for item in items %}
    <li>{{ item.name }}</li>
{% empty %}
    <li>No items found.</li>
{% endfor %}
```

**Filters (transform output):**
```html
{{ name|lower }}
{{ text|truncatewords:30 }}
{{ date|date:"Y-m-d" }}
{{ price|floatformat:2 }}
{{ html_content|safe }}
{{ items|length }}
```

**Comments:**
```html
{# This is a single-line comment #}

{% comment %}
    This is a multi-line comment.
{% endcomment %}
```

### Q21. What is template inheritance in Django?

Template inheritance allows you to build a base "skeleton" template that contains common elements and define blocks that child templates can override.

**Base template (`base.html`):**
```html
<!DOCTYPE html>
<html>
<head>
    <title>{% block title %}My Site{% endblock %}</title>
</head>
<body>
    <nav>Common Navigation</nav>
    <main>
        {% block content %}{% endblock %}
    </main>
    <footer>Common Footer</footer>
</body>
</html>
```

**Child template (`home.html`):**
```html
{% extends "base.html" %}

{% block title %}Home - My Site{% endblock %}

{% block content %}
    <h1>Welcome to the Home Page</h1>
    <p>This content replaces the content block.</p>
{% endblock %}
```

**Rules:**
- `{% extends %}` must be the first tag in the child template.
- Child templates can only override blocks defined in the parent.
- Use `{{ block.super }}` to include the parent block's content.

### Q24. What is the `{% url %}` tag?

The `{% url %}` tag generates a URL by reversing a named URL pattern, avoiding hardcoding URLs in templates.

```python
# urls.py
urlpatterns = [
    path('articles/<int:pk>/', views.article_detail, name='article-detail'),
]
```

```html
<!-- Template -->
<a href="{% url 'article-detail' pk=article.id %}">Read More</a>
```

This renders to something like `<a href="/articles/5/">Read More</a>`.

**In Python code**, the equivalent is `reverse()`:
```python
from django.urls import reverse
url = reverse('article-detail', kwargs={'pk': 5})
```

### Q25. What is the `{% csrf_token %}` tag and why is it important?

`{% csrf_token %}` inserts a hidden form field containing a CSRF (Cross-Site Request Forgery) token. This protects your application from CSRF attacks where a malicious site tricks a user's browser into making unwanted requests.

```html
<form method="POST">
    {% csrf_token %}
    <input type="text" name="title">
    <button type="submit">Submit</button>
</form>
```

This renders as:
```html
<input type="hidden" name="csrfmiddlewaretoken" value="random_token_value">
```

**How it works:**
1. Django generates a unique token for each user session.
2. The token is included in every POST form.
3. When the form is submitted, Django verifies the token matches.
4. If the token is missing or invalid, Django returns a 403 Forbidden error.

**For AJAX requests**, include the token in headers:
```javascript
const csrfToken = document.querySelector('[name=csrfmiddlewaretoken]').value;
fetch('/api/endpoint/', {
    method: 'POST',
    headers: {
        'X-CSRFToken': csrfToken,
        'Content-Type': 'application/json',
    },
    body: JSON.stringify(data),
});
```

### Q26. What is the `{% include %}` tag?

`{% include %}` renders another template within the current template. It is used for reusable components.

```html
<!-- header.html -->
<header>
    <h1>{{ site_name }}</h1>
    <nav>...</nav>
</header>

<!-- home.html -->
{% include "header.html" %}
<main>Content here</main>

<!-- Pass variables to included template -->
{% include "header.html" with site_name="My Blog" %}
```

**Difference from `{% extends %}`:**
- `{% extends %}` — full template inheritance (one parent, child overrides blocks).
- `{% include %}` — partial template inclusion (inserts a snippet).

### Q27. How do you serve static files in Django?

Static files (CSS, JS, images) are served differently in development vs. production.

**Development setup:**

1. Add `django.contrib.staticfiles` to `INSTALLED_APPS`.
2. Configure `settings.py`:
```python
STATIC_URL = '/static/'
STATICFILES_DIRS = [BASE_DIR / 'static']
```
3. Use in templates:
```html
{% load static %}
<link rel="stylesheet" href="{% static 'css/style.css' %}">
<img src="{% static 'images/logo.png' %}">
<script src="{% static 'js/main.js' %}"></script>
```

**Production setup:**

1. Set `STATIC_ROOT`:
```python
STATIC_ROOT = BASE_DIR / 'staticfiles'
```
2. Run `python manage.py collectstatic` to gather all static files.
3. Configure your web server (Nginx/Apache) to serve from `STATIC_ROOT`.

### Q28. What is the difference between `STATIC_URL`, `STATICFILES_DIRS`, and `STATIC_ROOT`?

| Setting | Purpose | Example |
|---------|---------|---------|
| `STATIC_URL` | The URL prefix browsers use to request static files. | `'/static/'` |
| `STATICFILES_DIRS` | A list of additional directories where Django looks for static files during development. | `[BASE_DIR / 'static']` |
| `STATIC_ROOT` | The single directory where `collectstatic` copies all static files for production deployment. | `BASE_DIR / 'staticfiles'` |

**Important:** `STATICFILES_DIRS` should never contain `STATIC_ROOT`.

### Q29. What is the `{% load %}` tag?

`{% load %}` loads a custom template tag library so you can use its tags and filters in your template.

```html
{% load static %}
{% load humanize %}
{% load custom_tags %}

<!-- Use loaded tags -->
<img src="{% static 'img/logo.png' %}">
<p>{{ large_number|intcomma }}</p>
```

Common libraries to load:
- `static` — for the `{% static %}` tag.

### Q31. What is a Django model?

A model is a Python class that represents a database table. Each attribute of the class maps to a database column.

```python
from django.db import models
```

### Q34. What are Django migrations?

Migrations are Django's way of propagating changes you make to your models (adding a field, deleting a model, etc.) into the database schema.

**Workflow:**
1. **Make changes** to your models in `models.py`.
2. **Create migrations:** `python manage.py makemigrations` — generates migration files.
3. **Apply migrations:** `python manage.py migrate` — executes the SQL against the database.

**Useful commands:**

| Command | Purpose |
|---------|---------|

Migration files are stored in each app's `migrations/` directory and should be committed to version control.

### Q35. What is the Django ORM?

ORM stands for **Object-Relational Mapping**. Django's ORM allows you to interact with your database using Python code instead of writing raw SQL.

```python
# Create
article = Article.objects.create(title="Hello", content="World")

# Read
all_articles = Article.objects.all()
article = Article.objects.get(pk=1)
published = Article.objects.filter(is_published=True)

# Update
article.title = "Updated Title"
article.save()
# or bulk update
Article.objects.filter(is_published=False).update(is_published=True)

# Delete
article.delete()
Article.objects.filter(is_published=False).delete()
```

**Benefits:**
- Database-agnostic — switch between SQLite, PostgreSQL, MySQL without changing code.
- Security — automatically parameterizes queries, preventing SQL injection.
- Pythonic — work with Python objects instead of SQL strings.

### Q36. What is a QuerySet?

A QuerySet is a collection of database queries that Django will execute. QuerySets are **lazy** — they don't hit the database until they are evaluated.

```python
# This does NOT hit the database yet
qs = Article.objects.filter(is_published=True)

# These trigger database evaluation:
list(qs)              # Converting to list
for item in qs:       # Iterating
    pass
qs[0]                 # Indexing
len(qs)               # Getting length
bool(qs)              # Boolean evaluation
repr(qs)              # Printing/repr
```

**QuerySet methods that return new QuerySets (chainable):**
- `filter()`, `exclude()`, `order_by()`, `distinct()`, `values()`, `values_list()`, `select_related()`, `prefetch_related()`, `annotate()`, `aggregate()`, `reverse()`, `defer()`, `only()`

**QuerySet methods that do NOT return QuerySets:**
- `get()`, `create()`, `count()`, `exists()`, `first()`, `last()`, `aggregate()`, `update()`, `delete()`

### Q37. What is the difference between `filter()` and `exclude()`?

- `filter(**kwargs)` — returns objects that **match** the given conditions.
- `exclude(**kwargs)` — returns objects that **do not match** the given conditions.

```python
# Get published articles
published = Article.objects.filter(is_published=True)

# Get articles that are NOT drafts
non_drafts = Article.objects.exclude(status='draft')

# Equivalent queries:
Article.objects.filter(is_published=True)
Article.objects.exclude(is_published=False)
```

### Q38. What are QuerySet lookups (field lookups)?

Field lookups are keyword arguments to `filter()`, `exclude()`, and `get()`. They use double underscores (`__`) to specify the lookup type.

| Lookup | SQL Equivalent | Example |
|--------|---------------|---------|
| `exact` | `= 'value'` | `title__exact='Hello'` |
| `iexact` | `ILIKE 'value'` | `title__iexact='hello'` |
| `contains` | `LIKE '%value%'` | `title__contains='django'` |
| `icontains` | `ILIKE '%value%'` | `title__icontains='django'` |
| `startswith` | `LIKE 'value%'` | `title__startswith='Hello'` |
| `endswith` | `LIKE '%value'` | `title__endswith='World'` |
| `in` | `IN (...)` | `id__in=[1, 2, 3]` |
| `gt` | `>` | `price__gt=100` |
| `gte` | `>=` | `price__gte=100` |
| `lt` | `<` | `price__lt=50` |
| `lte` | `<=` | `price__lte=50` |
| `range` | `BETWEEN ... AND ...` | `price__range=(10, 100)` |
| `isnull` | `IS NULL` | `author__isnull=True` |
| `date` | Cast to date | `created__date=date.today()` |
| `year` | Extract year | `created__year=2025` |
| `regex` | `REGEXP` | `title__regex=r'^[A-Z]'` |

### Q39. What are model relationships in Django?

Django supports three types of relationships:

**1. One-to-Many (ForeignKey):**
```python
class Author(models.Model):
    name = models.CharField(max_length=100)

class Book(models.Model):
    title = models.CharField(max_length=200)
    author = models.ForeignKey(Author, on_delete=models.CASCADE, related_name='books')
```
One author can have many books. Access: `author.books.all()`.

**2. Many-to-Many (ManyToManyField):**
```python
class Tag(models.Model):
    name = models.CharField(max_length=50)

class Article(models.Model):
    title = models.CharField(max_length=200)
    tags = models.ManyToManyField(Tag, related_name='articles')
```
An article can have many tags, and a tag can be on many articles.

**3. One-to-One (OneToOneField):**
```python
class User(models.Model):
    username = models.CharField(max_length=100)

class Profile(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    bio = models.TextField()
```
Each user has exactly one profile.

### Q40. What is the `on_delete` parameter in ForeignKey?

`on_delete` specifies what happens when the referenced object is deleted.

| Option | Behavior |
|--------|----------|
| `CASCADE` | Delete the related objects too. |
| `PROTECT` | Prevent deletion by raising `ProtectedError`. |
| `RESTRICT` | Similar to PROTECT but allows deletion if also referenced by CASCADE. |
| `SET_NULL` | Set the foreign key to `NULL` (requires `null=True`). |
| `SET_DEFAULT` | Set to the field's default value. |
| `SET(value)` | Set to a specific value or callable. |
| `DO_NOTHING` | Do nothing (risks database integrity errors). |

```python
class Comment(models.Model):
    post = models.ForeignKey(Post, on_delete=models.CASCADE)
    author = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
    category = models.ForeignKey(Category, on_delete=models.PROTECT)
```

### Q44. What is the difference between `null=True` and `blank=True`?

| Aspect | `null=True` | `blank=True` |
|--------|-------------|--------------|
| **Level** | Database level | Validation/form level |
| **Effect** | Allows `NULL` in the database column. | Allows empty values in forms. |
| **Use with strings** | Avoid (use `blank=True` with `default=''`). | Preferred for optional string fields. |
| **Use with non-strings** | Use for optional `DateField`, `IntegerField`, etc. | Use alongside `null=True`. |

```python
class Profile(models.Model):
    bio = models.TextField(blank=True, default='')
    birth_date = models.DateField(null=True, blank=True)
    phone = models.CharField(max_length=20, blank=True, default='')
```

**Rule of thumb:** For string-based fields (`CharField`, `TextField`), use `blank=True` with `default=''` instead of `null=True` to avoid having two possible "empty" values (`NULL` and `''`).

### Q48. How does form validation work in Django?

Django's form validation happens in a specific order:

1. **`to_python()`** — Converts the raw value to a Python type.
2. **`validate()`** — Field-specific validation (e.g., checks email format).
3. **`run_validators()`** — Runs any additional validators.
4. **`clean_<fieldname>()`** — Custom per-field validation you define.
5. **`clean()`** — Cross-field validation (validates fields together).

```python
class RegistrationForm(forms.Form):
    username = forms.CharField(max_length=50)
    password = forms.CharField(widget=forms.PasswordInput)
    confirm_password = forms.CharField(widget=forms.PasswordInput)
    email = forms.EmailField()

    def clean_username(self):
        username = self.cleaned_data['username']
        if User.objects.filter(username=username).exists():
            raise forms.ValidationError("Username already taken.")
        return username

    def clean(self):
        cleaned_data = super().clean()
        password = cleaned_data.get('password')
        confirm = cleaned_data.get('confirm_password')
        if password and confirm and password != confirm:
            raise forms.ValidationError("Passwords do not match.")
        return cleaned_data
```

### Q54. What is `commit=False` in `form.save()`?

`commit=False` creates the model instance but does NOT save it to the database. This allows you to modify the instance before saving.

```python
def create_post(request):
    if request.method == 'POST':
        form = PostForm(request.POST)
        if form.is_valid():
            post = form.save(commit=False)
            post.author = request.user
            post.slug = slugify(post.title)
            post.save()
            form.save_m2m()  # Required to save ManyToMany fields
            return redirect('post-detail', pk=post.pk)
```

**When to use `commit=False`:**
- You need to set fields not in the form (like the current user).
- You want to do additional processing before saving.
- You need to generate computed fields (like slugs).

**Important:** If the form has ManyToMany fields, you must call `form.save_m2m()` after `post.save()`.

### Q66. What is Django's authentication system?

Django's `django.contrib.auth` provides a complete authentication system:

- **User model** — stores username, password, email, and permissions.
- **Password hashing** — uses PBKDF2 by default, supports bcrypt and Argon2.
- **Login/Logout** — built-in views and forms.
- **Permissions** — per-object and per-model permissions.
- **Groups** — collections of permissions assigned to multiple users.
- **Decorators** — `@login_required`, `@permission_required`.

```python
from django.contrib.auth import authenticate, login, logout

def login_view(request):
    if request.method == 'POST':
        username = request.POST['username']
        password = request.POST['password']
        user = authenticate(request, username=username, password=password)
        if user is not None:
            login(request, user)
            return redirect('home')
        else:
            # Return "Invalid username or password" to prevent enumeration
            messages.error(request, "Invalid username or password.")
    return render(request, 'login.html')
```

### Q68. How does Django hash passwords?

Django never stores passwords in plain text. It uses a configurable system of **password hashers**.

Default configuration in `settings.py`:
```python
PASSWORD_HASHERS = [
    'django.contrib.auth.hashers.PBKDF2PasswordHasher',
    'django.contrib.auth.hashers.PBKDF2SHA1PasswordHasher',
    'django.contrib.auth.hashers.Argon2PasswordHasher',
    'django.contrib.auth.hashers.BCryptSHA256PasswordHasher',
]
```

A stored password looks like:
```
pbkdf2_sha256$600000$salt$hash
```

Format: `algorithm$iterations$salt$hash`

**Checking passwords:**
```python
from django.contrib.auth.hashers import make_password, check_password

hashed = make_password('my_password')
is_correct = check_password('my_password', hashed)
```

**Best practice:** Use Argon2 by installing `argon2-cffi` and moving `Argon2PasswordHasher` to the top of `PASSWORD_HASHERS`.

### Q78. What is the `collectstatic` command?

`collectstatic` copies all static files from each app's `static/` directory and from `STATICFILES_DIRS` into the single `STATIC_ROOT` directory.

```bash
python manage.py collectstatic
```

This is required for production deployment where a web server (Nginx) serves static files directly.

**Flow:**
1. Each app has its own `static/` directory.
2. `STATICFILES_DIRS` lists additional global static directories.
3. `collectstatic` copies everything to `STATIC_ROOT`.
4. Nginx is configured to serve `STATIC_ROOT` at `STATIC_URL`.

### Q86. What is the `DEBUG` setting and why should it be `False` in production?

`DEBUG = True`:
- Shows detailed error pages with stack traces, local variables, and SQL queries.
- Serves static files automatically.
- Disables some security checks.

`DEBUG = False`:
- Shows generic error pages (requires `404.html` and `500.html` templates).
- Requires proper static file serving configuration.
- Enables security features (CSRF, HSTS, etc.).

**If `DEBUG = True` in production:**
- Attackers see your source code, database structure, and settings.
- Memory usage increases (Django tracks SQL queries in debug mode).
- Static files may not be served properly.

### Q87. What is `ALLOWED_HOSTS`?

`ALLOWED_HOSTS` is a list of hostnames/domains that Django can serve. It protects against HTTP Host header attacks.

```python
# Development
ALLOWED_HOSTS = ['*']  # Only acceptable in development

# Production
ALLOWED_HOSTS = ['example.com', 'www.example.com', '192.168.1.100']
```

If a request comes in with a `Host` header not in this list, Django returns a 400 Bad Request. This setting is enforced when `DEBUG = False`.

### Q88. What is `SECRET_KEY` and why is it important?

`SECRET_KEY` is a random string used for cryptographic signing in Django:
- CSRF token generation.
- Session data signing.
- Password reset tokens.
- Message framework signatures.

**Best practices:**
- At least 50 characters of random text.
- Never commit to version control.
- Different value for each environment (dev, staging, prod).
- Store in environment variables or a secrets manager.

```python
import os
SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY')
```

If compromised, an attacker can forge sessions, CSRF tokens, and password reset links. Rotate immediately if exposed.

### Q95. What are Django management commands and how do you create custom ones?

Management commands are scripts you run via `manage.py`. You can create custom commands for tasks like data imports, cleanup, or batch processing.

```python
# myapp/management/commands/cleanup_expired.py
from django.core.management.base import BaseCommand
from django.utils import timezone
from myapp.models import Session

class Command(BaseCommand):
    help = 'Deletes expired sessions'

    def add_arguments(self, parser):
        parser.add_argument('--days', type=int, default=30)

    def handle(self, *args, **options):
        days = options['days']
        cutoff = timezone.now() - timezone.timedelta(days=days)
        count, _ = Session.objects.filter(expires_at__lt=cutoff).delete()
        self.stdout.write(self.style.SUCCESS(f'Deleted {count} expired sessions'))
```

```bash
python manage.py cleanup_expired --days 7
```

Directory structure:
```
myapp/
└── management/
    ├── __init__.py
    └── commands/
        ├── __init__.py
        └── cleanup_expired.py
```

### Q97. What are signals in Django?

Signals allow certain senders to notify a set of receivers when specific actions occur. They enable decoupled communication between components.

```python
from django.db.models.signals import post_save, pre_save
from django.dispatch import receiver
from django.contrib.auth import get_user_model

User = get_user_model()

@receiver(post_save, sender=User)
def create_user_profile(sender, instance, created, **kwargs):
    if created:
        Profile.objects.create(user=instance)
```

Common built-in signals:

| Signal | Fired When |
|--------|-----------|
| `pre_save` | Before `model.save()`. |
| `post_save` | After `model.save()`. |
| `pre_delete` | Before `model.delete()`. |
| `post_delete` | After `model.delete()`. |
| `m2m_changed` | When a ManyToMany relationship changes. |
| `request_started` | When an HTTP request begins. |
| `request_finished` | When an HTTP request completes. |

### Q99. What is the difference between `get()` and `filter()` in QuerySets?

| Feature | `get()` | `filter()` |
|---------|---------|-----------|
| **Returns** | A single model instance. | A QuerySet (can be empty). |
| **No match** | Raises `DoesNotExist` exception. | Returns an empty QuerySet. |
| **Multiple matches** | Raises `MultipleObjectsReturned`. | Returns all matches. |
| **Use case** | When you expect exactly one result. | When you expect zero or more results. |

```python
# get() - expects exactly one result
try:
    article = Article.objects.get(pk=1)
except Article.DoesNotExist:
    pass
except Article.MultipleObjectsReturned:
    pass

# filter() - always safe
articles = Article.objects.filter(status='published')
```

**Best practice:** Use `get_object_or_404()` in views:
```python
from django.shortcuts import get_object_or_404

article = get_object_or_404(Article, pk=1)
```

### Q100. What are the key differences between Django and Flask?

| Feature | Django | Flask |
|---------|--------|-------|
| **Type** | Full-stack, batteries-included. | Micro-framework, minimalist. |
| **ORM** | Built-in (Django ORM). | None (use SQLAlchemy, etc.). |
| **Admin** | Built-in admin panel. | None. |
| **Template Engine** | Django Template Language. | Jinja2. |
| **Authentication** | Built-in auth system. | None (use Flask-Login, etc.). |
| **Forms** | Built-in form handling. | None (use WTForms, etc.). |
| **Flexibility** | Opinionated — follows conventions. | Flexible — you choose everything. |
| **Learning Curve** | Steeper initially. | Gentle. |
| **Best For** | Large, full-featured web applications. | Small APIs, microservices, prototypes. |
| **Community** | Large, mature. | Large, growing. |

**Choose Django when:** You need a full-featured web app with admin, auth, ORM, and want rapid development with conventions.

**Choose Flask when:** You want maximum flexibility, are building a microservice or API, or need a lightweight framework.
