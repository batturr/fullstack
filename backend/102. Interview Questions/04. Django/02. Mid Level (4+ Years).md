### Q1. What is the N+1 query problem and how do you solve it in Django?

The N+1 problem occurs when Django executes one query to fetch N objects, then executes one additional query per object to fetch related data — resulting in N+1 total queries.

**Problem:**
```python
articles = Article.objects.all()  # 1 query
for article in articles:
    print(article.author.name)   # N queries (one per article)
```

**Solution 1: `select_related()` for ForeignKey/OneToOne (SQL JOIN):**
```python
articles = Article.objects.select_related('author').all()  # 1 query with JOIN
for article in articles:
    print(article.author.name)  # No additional query
```

`select_related` performs a SQL JOIN and includes the related object's data in the same query. Use it for ForeignKey and OneToOneField relationships.

**Solution 2: `prefetch_related()` for ManyToMany and reverse ForeignKey (separate query):**
```python
articles = Article.objects.prefetch_related('tags').all()
# Query 1: SELECT * FROM articles
# Query 2: SELECT * FROM tags WHERE id IN (...)
for article in articles:
    for tag in article.tags.all():  # No additional query — cached
        print(tag.name)
```

`prefetch_related` performs a separate lookup for each relationship and does the joining in Python. Use it for ManyToManyField and reverse ForeignKey relationships.

**Solution 3: `Prefetch` object for custom querysets:**
```python
from django.db.models import Prefetch

articles = Article.objects.prefetch_related(
    Prefetch('comments', queryset=Comment.objects.filter(is_approved=True).select_related('author'))
)
```

### Q2. Explain `select_related()` vs `prefetch_related()` in detail.

| Aspect | `select_related()` | `prefetch_related()` |
|--------|-------------------|---------------------|
| **SQL strategy** | Single query with SQL JOIN | Separate queries, joined in Python |
| **Relationships** | ForeignKey, OneToOneField | ManyToManyField, reverse ForeignKey, GenericRelation |
| **Performance** | Faster for single-valued relations | Better for multi-valued relations |
| **Result** | Single large query | Multiple smaller queries |
| **Customization** | Limited (follows FK chain) | Highly customizable via `Prefetch` objects |

**When to use each:**
- Use `select_related` when following a foreign key to a single object (the "one" side of a many-to-one).
- Use `prefetch_related` when fetching multiple related objects (the "many" side).

**Chaining both:**
```python
Article.objects.select_related('author', 'category').prefetch_related(
    'tags',
    Prefetch('comments', queryset=Comment.objects.select_related('author'))
)
```

### Q3. What are `F()` expressions and when do you use them?

`F()` expressions reference a model field's value within a query, allowing the database to perform operations without pulling data into Python.

**Use cases:**

**1. Referencing field values in filters:**
```python
from django.db.models import F

# Products where sale_price < original_price
Product.objects.filter(sale_price__lt=F('original_price'))
```

**2. Atomic updates (avoid race conditions):**
```python
# Increment view count atomically
article.views = F('views') + 1
article.save(update_fields=['views'])

# Bulk atomic update
Product.objects.filter(category='electronics').update(price=F('price') * 1.1)
```

**3. Annotating with calculated values:**
```python
from django.db.models import F, ExpressionWrapper, DecimalField

Product.objects.annotate(
    discount=ExpressionWrapper(
        F('original_price') - F('sale_price'),
        output_field=DecimalField()
    )
)
```

**Why F() matters:** Without `F()`, the update reads the current value into Python, modifies it, and writes it back — which is not thread-safe. `F()` pushes the operation to the database, making it atomic.

### Q4. What are `Q()` objects and how are they used?

`Q()` objects enable complex queries with OR, AND, and NOT conditions. By default, `filter()` conditions are ANDed together.

```python
from django.db.models import Q

# OR condition
Article.objects.filter(Q(status='published') | Q(author=request.user))

# AND condition (same as chaining filters)
Article.objects.filter(Q(status='published') & Q(category='tech'))

# NOT condition
Article.objects.filter(~Q(status='draft'))

# Complex combinations
Article.objects.filter(
    Q(status='published') & (Q(category='tech') | Q(category='science')),
    ~Q(is_archived=True)
)
```

**Dynamic query building:**
```python
query = Q()
if search_term:
    query &= Q(title__icontains=search_term) | Q(content__icontains=search_term)
if category:
    query &= Q(category=category)
if author:
    query &= Q(author=author)

articles = Article.objects.filter(query)
```

### Q5. Explain `annotate()` and `aggregate()` with examples.

**`aggregate()`** — returns a dictionary of aggregated values over the entire queryset (terminal operation).

```python
from django.db.models import Count, Avg, Sum, Max, Min

result = Product.objects.aggregate(
    total=Count('id'),
    avg_price=Avg('price'),
    max_price=Max('price'),
    total_revenue=Sum('price'),
)
# {'total': 150, 'avg_price': 45.50, 'max_price': 999.99, 'total_revenue': 6825.00}
```

**`annotate()`** — adds a computed column to each object in the queryset.

```python
# Count comments per article
articles = Article.objects.annotate(comment_count=Count('comments'))
for article in articles:
    print(f"{article.title}: {article.comment_count} comments")

# Authors with their article count, ordered by most prolific
authors = Author.objects.annotate(
    article_count=Count('articles')
).order_by('-article_count')

# Products with average rating
products = Product.objects.annotate(
    avg_rating=Avg('reviews__rating')
).filter(avg_rating__gte=4.0)
```

**Combining with `values()` for GROUP BY:**
```python
# Articles per category
Article.objects.values('category__name').annotate(
    count=Count('id'),
    avg_views=Avg('views')
).order_by('-count')
```

### Q6. What are Django model managers and when would you create a custom one?

A Manager is the interface through which database queries are provided to Django models. Every model gets at least one Manager (`objects`).

**Custom Manager for reusable query logic:**
```python
class PublishedManager(models.Manager):
    def get_queryset(self):
        return super().get_queryset().filter(
            status='published',
            published_date__lte=timezone.now()
        )

    def featured(self):
        return self.get_queryset().filter(is_featured=True)

    def by_author(self, author):
        return self.get_queryset().filter(author=author)

class Article(models.Model):
    title = models.CharField(max_length=200)
    status = models.CharField(max_length=20)
    is_featured = models.BooleanField(default=False)

    objects = models.Manager()      # Keep the default
    published = PublishedManager()  # Custom manager
```

```python
Article.objects.all()              # All articles
Article.published.all()            # Only published articles
Article.published.featured()       # Published and featured
```

**Custom QuerySet (even more flexible):**
```python
class ArticleQuerySet(models.QuerySet):
    def published(self):
        return self.filter(status='published')

    def featured(self):
        return self.filter(is_featured=True)

class Article(models.Model):
    objects = ArticleQuerySet.as_manager()

# Chainable
Article.objects.published().featured().order_by('-created')
```

### Q7. What is `values()` vs `values_list()` vs `only()` vs `defer()`?

| Method | Returns | Use Case |
|--------|---------|----------|
| `values()` | QuerySet of dictionaries | When you need dict-like results for specific fields |
| `values_list()` | QuerySet of tuples | When you need lightweight iterable results |
| `only()` | Model instances with only specified fields loaded | When you need model instances but want fewer fields |
| `defer()` | Model instances with specified fields deferred | When you need model instances but want to skip heavy fields |

```python
# values() — returns dictionaries
Article.objects.values('title', 'author__name')
# [{'title': 'Hello', 'author__name': 'John'}, ...]

# values_list() — returns tuples
Article.objects.values_list('title', 'id')
# [('Hello', 1), ('World', 2), ...]

# values_list(flat=True) — returns flat list (single field only)
Article.objects.values_list('title', flat=True)
# ['Hello', 'World', ...]

# only() — loads only specified fields, defers the rest
articles = Article.objects.only('title', 'author')
articles[0].content  # Triggers additional DB query

# defer() — loads everything except specified fields
articles = Article.objects.defer('content', 'html_content')
articles[0].content  # Triggers additional DB query
```

### Q8. How do you write raw SQL queries in Django?

**Method 1: `raw()` — returns model instances:**
```python
articles = Article.objects.raw('SELECT * FROM blog_article WHERE views > %s', [100])
for article in articles:
    print(article.title)  # Full model instance
```

**Method 2: Direct database cursor — returns raw data:**
```python
from django.db import connection

with connection.cursor() as cursor:
    cursor.execute("SELECT title, views FROM blog_article WHERE views > %s", [100])
    rows = cursor.fetchall()
    # [('Hello', 500), ('World', 300)]

    columns = [col[0] for col in cursor.description]
    results = [dict(zip(columns, row)) for row in cursor.fetchall()]
```

**Always use parameterized queries** (the `%s` and list pattern) to prevent SQL injection. Never concatenate user input into SQL strings.

### Q9. What are database transactions in Django?

A transaction is a group of database operations that must all succeed or all fail together (atomicity).

**`transaction.atomic()`:**
```python
from django.db import transaction

def transfer_money(from_account, to_account, amount):
    with transaction.atomic():
        from_account.balance = F('balance') - amount
        from_account.save(update_fields=['balance'])

        to_account.balance = F('balance') + amount
        to_account.save(update_fields=['balance'])

        TransferLog.objects.create(
            from_account=from_account,
            to_account=to_account,
            amount=amount,
        )
    # If any operation fails, ALL changes are rolled back
```

**As a decorator:**
```python
@transaction.atomic
def create_order(request):
    order = Order.objects.create(user=request.user)
    for item in cart_items:
        OrderItem.objects.create(order=order, product=item.product)
    return order
```

**Savepoints (nested atomics):**
```python
with transaction.atomic():  # Outer transaction
    user.save()
    try:
        with transaction.atomic():  # Savepoint
            risky_operation()
    except Exception:
        pass  # Savepoint rolled back, outer transaction continues
    safe_operation()  # Still runs
```

**`transaction.on_commit()`:**
```python
with transaction.atomic():
    order = Order.objects.create(user=request.user)
    transaction.on_commit(lambda: send_order_confirmation.delay(order.id))
```

### Q10. What are Django model signals? When should and shouldn't you use them?

Signals allow decoupled components to respond to events (model saves, deletions, etc.).

**Common signals:**
```python
from django.db.models.signals import pre_save, post_save, pre_delete, post_delete
from django.dispatch import receiver

@receiver(post_save, sender=User)
def create_profile(sender, instance, created, **kwargs):
    if created:
        Profile.objects.create(user=instance)

@receiver(pre_save, sender=Article)
def generate_slug(sender, instance, **kwargs):
    if not instance.slug:
        instance.slug = slugify(instance.title)
```

**When to use signals:**
- Creating related objects (profile for new user).
- Logging/auditing changes.
- Cache invalidation.
- Sending notifications for model changes triggered from multiple places.

**When NOT to use signals (prefer overriding `save()` or explicit calls):**
- Business logic that should be traceable in code flow.
- Complex workflows with multiple dependencies.
- When the logic is specific to one code path (not every save).
- Performance-critical paths (signals add overhead).

**Problems with signals:**
- Hard to debug — implicit flow.
- Can create circular dependencies.
- Run synchronously — slow signals block the request.
- Can be missed if `bulk_create()` or `update()` is used (they bypass signals).

### Q11. What are proxy models in Django?

A proxy model inherits from an existing model but does NOT create a new database table. It lets you change Python-level behavior (methods, managers, ordering, Meta) without altering the schema.

```python
class Article(models.Model):
    title = models.CharField(max_length=200)
    status = models.CharField(max_length=20)
    is_featured = models.BooleanField(default=False)

class PublishedArticle(Article):
    class Meta:
        proxy = True
        ordering = ['-published_date']

    objects = PublishedManager()

    def is_recent(self):
        return self.published_date >= timezone.now() - timedelta(days=7)

class FeaturedArticle(Article):
    class Meta:
        proxy = True

    objects = FeaturedManager()
```

**Use cases:**
- Different default ordering or managers for the same table.
- Adding methods without altering the original model.
- Different admin configurations for the same underlying data.
- Separating concerns in large codebases.

### Q12. What is abstract model inheritance vs multi-table inheritance?

**Abstract Base Class — no database table:**
```python
class TimestampedModel(models.Model):
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        abstract = True

class Article(TimestampedModel):
    title = models.CharField(max_length=200)
    # Gets created_at and updated_at fields
    # Only ONE table: blog_article
```

**Multi-Table Inheritance — each model gets its own table:**
```python
class Place(models.Model):
    name = models.CharField(max_length=50)
    address = models.CharField(max_length=80)

class Restaurant(Place):
    serves_pizza = models.BooleanField(default=False)
    # TWO tables: places_place, places_restaurant
    # Linked by an implicit OneToOneField
```

| Aspect | Abstract | Multi-Table | Proxy |
|--------|----------|-------------|-------|
| **DB Tables** | None for parent | One per model | Same as parent |
| **Query parent** | Not possible | Yes (`Place.objects.all()`) | Same as parent |
| **Use case** | Shared fields/methods | True IS-A relationship | Different behavior, same data |
| **Performance** | Best (no joins) | Slower (implicit JOINs) | Same as parent |

**Recommendation:** Use abstract models for shared fields. Avoid multi-table inheritance unless you truly need polymorphic queries.

### Q13. What is `Subquery` and `OuterRef`?

`Subquery` and `OuterRef` allow correlated subqueries — a subquery that references a field from the outer query.

```python
from django.db.models import Subquery, OuterRef

# Get the latest comment date for each article
latest_comment = Comment.objects.filter(
    article=OuterRef('pk')
).order_by('-created_at')

articles = Article.objects.annotate(
    latest_comment_date=Subquery(latest_comment.values('created_at')[:1])
)

# Get articles where the author's total article count > 10
prolific_authors = Author.objects.filter(
    article_count__gt=10
).values('id')

articles = Article.objects.filter(
    author_id__in=Subquery(prolific_authors)
)
```

**`Exists()`** — for checking existence efficiently:
```python
from django.db.models import Exists, OuterRef

comments = Comment.objects.filter(article=OuterRef('pk'))
articles = Article.objects.annotate(has_comments=Exists(comments))
# articles[0].has_comments → True/False
```

### Q14. How do you use `bulk_create()` and `bulk_update()`?

These methods perform batch operations in a single query, dramatically improving performance.

```python
# bulk_create — inserts multiple rows in one query
articles = [
    Article(title=f"Article {i}", content=f"Content {i}")
    for i in range(1000)
]
Article.objects.bulk_create(articles, batch_size=100)

# bulk_update — updates multiple rows in one query
articles = Article.objects.filter(status='draft')
for article in articles:
    article.status = 'published'
Article.objects.bulk_update(articles, ['status'], batch_size=100)
```

**Limitations:**
- `bulk_create` does not call `save()` or fire `pre_save`/`post_save` signals.
- `bulk_create` does not set `auto_now`/`auto_now_add` fields.
- `bulk_update` does not call `save()` or fire signals.
- Custom `save()` methods are not called.
- `batch_size` controls how many objects are processed per query.

### Q15. What is `django.db.models.JSONField` and how do you query it?

`JSONField` stores JSON-encoded data. It's natively supported on PostgreSQL, MySQL 5.7+, SQLite 3.9+, and Oracle.

```python
class Product(models.Model):
    name = models.CharField(max_length=200)
    metadata = models.JSONField(default=dict)

# Create
Product.objects.create(
    name="Laptop",
    metadata={"brand": "Dell", "specs": {"ram": 16, "storage": 512}}
)

# Query by key
Product.objects.filter(metadata__brand="Dell")

# Query nested keys
Product.objects.filter(metadata__specs__ram__gte=16)

# Contains a key
Product.objects.filter(metadata__has_key="brand")

# Contains any of the keys
Product.objects.filter(metadata__has_any_keys=["brand", "model"])

# Contains exact value
Product.objects.filter(metadata__contains={"brand": "Dell"})

# Contained by
Product.objects.filter(metadata__contained_by={"brand": "Dell", "color": "black"})
```

### Q16. Explain the CBV method resolution order (MRO) and dispatch flow.

When a request arrives at a CBV:

1. **`as_view()`** — class method that returns a callable view function.
2. **`setup()`** — initializes attributes (`self.request`, `self.args`, `self.kwargs`).
3. **`dispatch()`** — routes to `get()`, `post()`, `put()`, `delete()` based on HTTP method.
4. **HTTP method handler** — your `get()`, `post()`, etc.

```python
class MyView(View):
    def dispatch(self, request, *args, **kwargs):
        # Called before the actual method handler
        # Good place for per-request setup or permission checks
        return super().dispatch(request, *args, **kwargs)

    def get(self, request):
        return HttpResponse("GET request")

    def post(self, request):
        return HttpResponse("POST request")
```

**MRO for multiple inheritance:**
```python
class LoginRequiredMixin:
    def dispatch(self, request, *args, **kwargs):
        if not request.user.is_authenticated:
            return redirect('login')
        return super().dispatch(request, *args, **kwargs)

class MyView(LoginRequiredMixin, TemplateView):
    # MRO: MyView → LoginRequiredMixin → TemplateView → ... → View
    pass
```

Mixins should be listed **before** the main view class.

### Q17. What are Django mixins and how do you create them?

Mixins are classes that provide specific functionality to be mixed into views via multiple inheritance.

```python
class StaffRequiredMixin:
    def dispatch(self, request, *args, **kwargs):
        if not request.user.is_staff:
            raise PermissionDenied
        return super().dispatch(request, *args, **kwargs)

class TitleMixin:
    title = ''

    def get_context_data(self, **kwargs):
        context = super().get_context_data(**kwargs)
        context['title'] = self.title
        return context

class PaginationMixin:
    paginate_by = 20

class ArticleListView(StaffRequiredMixin, TitleMixin, PaginationMixin, ListView):
    model = Article
    title = 'Article List'
    template_name = 'articles/list.html'
```

**Mixin design rules:**
- Mixins should call `super()` to preserve the MRO chain.
- Keep mixins focused on a single responsibility.
- List mixins before the base view class.
- Avoid mixins that override the same method without calling `super()`.

### Q18. How does `get_queryset()`, `get_context_data()`, and `get_object()` work in CBVs?

```python
class ArticleListView(ListView):
    model = Article
    template_name = 'articles/list.html'

    def get_queryset(self):
        """Customize which objects are listed."""
        qs = super().get_queryset()
        category = self.request.GET.get('category')
        if category:
            qs = qs.filter(category__slug=category)
        return qs.select_related('author')

    def get_context_data(self, **kwargs):
        """Add extra data to template context."""
        context = super().get_context_data(**kwargs)
        context['categories'] = Category.objects.all()
        context['total_articles'] = self.get_queryset().count()
        return context

class ArticleDetailView(DetailView):
    model = Article

    def get_object(self, queryset=None):
        """Customize how a single object is retrieved."""
        obj = super().get_object(queryset)
        obj.views = F('views') + 1
        obj.save(update_fields=['views'])
        return obj
```

### Q19. What is Django REST Framework (DRF)?

DRF is a powerful toolkit for building Web APIs in Django. It provides serializers, viewsets, routers, authentication, permissions, throttling, and a browsable API.

```python
# pip install djangorestframework

# serializers.py
from rest_framework import serializers

class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ['id', 'title', 'content', 'author', 'created_at']

from rest_framework import viewsets

class ArticleViewSet(viewsets.ModelViewSet):

    def perform_create(self, serializer):

# urls.py
from rest_framework.routers import DefaultRouter

router = DefaultRouter()
router.register(r'articles', ArticleViewSet)
urlpatterns = router.urls
```

### Q20. What is the difference between `Serializer` and `ModelSerializer`?

| Feature | `Serializer` | `ModelSerializer` |
|---------|-------------|------------------|
| **Field definition** | Manual — define every field | Automatic — generated from model |
| **Validation** | Manual | Includes model-level validation |
| **`create()`/`update()`** | Must implement manually | Built-in implementations |
| **Flexibility** | Maximum | Some constraints |
| **Use case** | Non-model data, complex APIs | Standard CRUD APIs |

```python
# Serializer — manual
class ArticleSerializer(serializers.Serializer):
    title = serializers.CharField(max_length=200)
    content = serializers.CharField()

    def create(self, validated_data):
        return Article.objects.create(**validated_data)

    def update(self, instance, validated_data):
        instance.title = validated_data.get('title', instance.title)
        instance.save()
        return instance

# ModelSerializer — automatic
class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ['id', 'title', 'content', 'author']
```

### Q22. What are DRF ViewSets and Routers?

**ViewSets** combine logic for multiple related views into a single class.

```python
from rest_framework import viewsets

class ArticleViewSet(viewsets.ModelViewSet):
    # Provides list, create, retrieve, update, partial_update, destroy
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
```

| ViewSet | Provides |
|---------|---------|
| `ViewSet` | No default actions — define your own |
| `GenericViewSet` | Base with `get_queryset()`, `get_object()`, etc. |
| `ModelViewSet` | Full CRUD: list, create, retrieve, update, destroy |
| `ReadOnlyModelViewSet` | Read-only: list, retrieve |

**Routers** automatically generate URL patterns from ViewSets:
```python
from rest_framework.routers import DefaultRouter

router = DefaultRouter()
router.register(r'articles', ArticleViewSet, basename='article')
# Generates:
# GET    /articles/          → list
# POST   /articles/          → create
# GET    /articles/{pk}/     → retrieve
# PUT    /articles/{pk}/     → update
# PATCH  /articles/{pk}/     → partial_update
# DELETE /articles/{pk}/     → destroy
```

### Q28. How do you implement custom actions in DRF ViewSets?

The `@action` decorator adds custom routes to a ViewSet.

```python
from rest_framework.decorators import action
from rest_framework.response import Response

class ArticleViewSet(viewsets.ModelViewSet):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

    @action(detail=True, methods=['post'], permission_classes=[IsAuthenticated])
    def publish(self, request, pk=None):
        article = self.get_object()
        article.status = 'published'
        article.published_date = timezone.now()
        article.save()
        return Response({'status': 'published'})

    @action(detail=False, methods=['get'])
    def recent(self, request):
        recent = self.get_queryset().order_by('-created_at')[:10]
        serializer = self.get_serializer(recent, many=True)
        return Response(serializer.data)

# Generates:
# POST /articles/{pk}/publish/
# GET  /articles/recent/
```

- `detail=True` — operates on a single object (URL includes `{pk}`).
- `detail=False` — operates on the collection (list endpoint).

### Q29. What is serializer validation in DRF?

DRF serializer validation follows a specific order:

1. **Field-level validation** — `validate_<field>(self, value)`
2. **Object-level validation** — `validate(self, data)`
3. **Validators** — `validators` on Meta class or field

```python
class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = ['title', 'content', 'status']
        validators = [
            UniqueTogetherValidator(
                queryset=Article.objects.all(),
                fields=['title', 'author'],
            )
        ]

    def validate_title(self, value):
        if 'spam' in value.lower():
            raise serializers.ValidationError("Title contains forbidden words.")
        return value

    def validate(self, data):
        if data.get('status') == 'published' and not data.get('content'):
            raise serializers.ValidationError("Published articles must have content.")
        return data
```

### Q31. How do you write custom middleware in Django?

```python
import time
import logging

logger = logging.getLogger(__name__)

class RequestTimingMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        start_time = time.monotonic()

        response = self.get_response(request)

        duration = time.monotonic() - start_time
        response['X-Request-Duration'] = f"{duration:.3f}s"
        logger.info(

            extra={
                'path': request.path,
                'method': request.method,
                'status': response.status_code,
                'duration_ms': round(duration * 1000, 2),
            }
        )
        return response

    def process_exception(self, request, exception):
        logger.error(f"Unhandled exception on {request.path}", exc_info=True)
        return None
```

### Q32. What is the middleware execution order and how does it affect request processing?

```
Request comes in
    ↓
SecurityMiddleware.__call__()        ← Top of MIDDLEWARE list
    ↓
SessionMiddleware.__call__()
    ↓
CommonMiddleware.__call__()
    ↓
CsrfViewMiddleware.__call__()
    ↓
AuthenticationMiddleware.__call__()
    ↓
MessageMiddleware.__call__()
    ↓
XFrameOptionsMiddleware.__call__()   ← Bottom of MIDDLEWARE list
    ↓
    VIEW FUNCTION CALLED
    ↓
XFrameOptionsMiddleware              ← Response goes BOTTOM to TOP
    ↓
MessageMiddleware
    ↓
AuthenticationMiddleware
    ↓
CsrfViewMiddleware
    ↓
CommonMiddleware
    ↓
SessionMiddleware
    ↓
SecurityMiddleware
    ↓
Response sent to client
```

**Key rules:**
- `SecurityMiddleware` should be first (sets security headers early).
- `SessionMiddleware` must be before anything that uses sessions.
- `AuthenticationMiddleware` must be after `SessionMiddleware`.
- `CsrfViewMiddleware` must be before any view that uses CSRF.

### Q40. How do you implement soft delete in Django?

```python
    def get_queryset(self):
```

### Q41. How do you implement audit trails/logging for model changes?

```python
from django.contrib.auth import get_user_model

User = get_user_model()

class AuditLog(models.Model):
    ACTION_CHOICES = [
        ('create', 'Create'),
        ('update', 'Update'),
        ('delete', 'Delete'),
    ]
    model_name = models.CharField(max_length=100)
    object_id = models.PositiveIntegerField()
    action = models.CharField(max_length=10, choices=ACTION_CHOICES)
    changes = models.JSONField(default=dict)
    user = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
    timestamp = models.DateTimeField(auto_now_add=True)

class AuditMixin:
    def save(self, *args, **kwargs):
        is_new = self.pk is None
        if not is_new:
            old = type(self).objects.get(pk=self.pk)
            changes = {}
            for field in self._meta.fields:
                old_val = getattr(old, field.name)
                new_val = getattr(self, field.name)
                if old_val != new_val:
                    changes[field.name] = {'old': str(old_val), 'new': str(new_val)}
        super().save(*args, **kwargs)
        AuditLog.objects.create(
            model_name=type(self).__name__,
            object_id=self.pk,
            action='create' if is_new else 'update',
            changes=changes if not is_new else {},
        )
```

For production, consider `django-auditlog` or `django-simple-history`.

### Q42. What is `select_for_update()` and when do you use it?

`select_for_update()` locks the selected rows until the end of the transaction, preventing other transactions from modifying them. This prevents race conditions.

```python
from django.db import transaction

@transaction.atomic
def purchase_ticket(user, event_id):
    event = Event.objects.select_for_update().get(pk=event_id)
    if event.tickets_remaining > 0:
        event.tickets_remaining -= 1
        event.save(update_fields=['tickets_remaining'])
        Ticket.objects.create(user=user, event=event)
        return True
    return False
```

**Options:**
- `select_for_update(nowait=True)` — raises `DatabaseError` instead of waiting.
- `select_for_update(skip_locked=True)` — skips locked rows.
- `select_for_update(of=('self',))` — locks only specific tables in a join.

### Q46. How does Django's CSRF protection work internally?

1. Middleware generates a CSRF token and stores it in a cookie (`csrftoken`).
2. The `{% csrf_token %}` tag embeds the token in forms as a hidden field.
3. On POST/PUT/PATCH/DELETE, middleware compares the cookie token with the submitted token.
4. If they don't match, Django returns a 403 Forbidden response.

**For AJAX/API calls:**
```javascript
const csrfToken = document.cookie
    .split('; ')
    .find(row => row.startsWith('csrftoken='))
    ?.split('=')[1];

fetch('/api/data/', {
    method: 'POST',
    headers: {
        'X-CSRFToken': csrfToken,
        'Content-Type': 'application/json',
    },
    body: JSON.stringify(data),
});
```

**Exempting views from CSRF (use sparingly):**
```python
from django.views.decorators.csrf import csrf_exempt

@csrf_exempt
def webhook(request):
    pass
```

```python
            ip = self.get_client_ip(request)
            cache_key = f'rate_limit:{hashlib.sha256(ip.encode()).hexdigest()}'
            requests_count = cache.get(cache_key, 0)

            if requests_count >= 100:
                return HttpResponseTooManyRequests("Rate limit exceeded.")

            cache.set(cache_key, requests_count + 1, timeout=3600)

        return self.get_response(request)

    def get_client_ip(self, request):
        x_forwarded = request.META.get('HTTP_X_FORWARDED_FOR')
        if x_forwarded:
            return x_forwarded.split(',')[0].strip()
        return request.META.get('REMOTE_ADDR')
```

For production, use `django-ratelimit` or implement at the reverse proxy (Nginx) level.

### Q65. What are database indexes and how do you use them in Django?

**When NOT to add indexes:**
- Small tables.
- Fields rarely used in queries.
- Tables with heavy write operations (indexes slow writes).

### Q74. What is `QuerySet.iterator()` and when should you use it?

`iterator()` evaluates the queryset and returns results one at a time, without caching the entire result set in memory.

```python
# Without iterator — loads ALL rows into memory
for article in Article.objects.all():
    process(article)

# With iterator — fetches rows in chunks
for article in Article.objects.all().iterator(chunk_size=2000):
    process(article)
```

**Use when:**
- Processing millions of rows.
- Memory is a concern.
- You only need to iterate once (no re-use).

**Don't use when:**
- You need to access the queryset multiple times.
- You need `len()` or indexing on the queryset.

### Q81. What is `setUpTestData` vs `setUp` in Django tests?

| Method | Execution | Scope | Use Case |
|--------|-----------|-------|----------|
| `setUpTestData()` | Once per test class | Shared, read-only data | Creating test data that doesn't change |
| `setUp()` | Before every test method | Fresh per test | Data that may be modified by tests |

```python
class ArticleTest(TestCase):
    @classmethod
    def setUpTestData(cls):
        # Called ONCE for the entire TestCase class
        cls.user = User.objects.create_user(username='test', password='pass')

    def setUp(self):
        # Called before EACH test method
        self.client.login(username='test', password='pass')
```

`setUpTestData` is faster because it creates data once and wraps each test in a transaction that rolls back.

### Q83. What is `TransactionTestCase` vs `TestCase`?

| Feature | `TestCase` | `TransactionTestCase` |
|---------|-----------|----------------------|
| **Isolation** | Wraps each test in a transaction (rolled back) | Truncates tables after each test |
| **Speed** | Fast | Slow |
| **Transaction support** | Cannot test real commits | Can test `transaction.on_commit()`, signals, etc. |
| **Use case** | Most tests | Tests involving real transactions |

```python
class TransactionTest(TransactionTestCase):
    def test_on_commit_callback(self):
        with self.captureOnCommitCallbacks(execute=True) as callbacks:
            Order.objects.create(user=self.user)
        self.assertEqual(len(callbacks), 1)
```

### Q94. How do you handle database migrations in a team environment?

1. Always commit migration files to version control.
2. Run `makemigrations` locally, never on production.
3. If migrations conflict, use `makemigrations --merge`.
4. Squash old migrations periodically: `squashmigrations app_name 0001 0010`.
5. Test migrations with `migrate --plan` before applying.
6. Use `RunPython` for data migrations.

```python
# Data migration
from django.db import migrations

def populate_slugs(apps, schema_editor):
    Article = apps.get_model('blog', 'Article')
    for article in Article.objects.filter(slug=''):
        article.slug = slugify(article.title)
        article.save(update_fields=['slug'])

class Migration(migrations.Migration):
    dependencies = [('blog', '0005_article_slug')]
    operations = [
        migrations.RunPython(populate_slugs, migrations.RunPython.noop),
    ]
```

### Q96. What is Django Channels and when do you use it?

Django Channels extends Django to handle WebSockets, HTTP2, and other async protocols.

```python
# pip install channels channels-redis

# routing.py
from django.urls import re_path
from . import consumers

websocket_urlpatterns = [
    re_path(r'ws/chat/(?P<room_name>\w+)/$', consumers.ChatConsumer.as_asgi()),
]

# consumers.py
from channels.generic.websocket import AsyncWebsocketConsumer
import json

class ChatConsumer(AsyncWebsocketConsumer):
    async def connect(self):
        self.room_name = self.scope['url_route']['kwargs']['room_name']
        self.room_group_name = f'chat_{self.room_name}'
        await self.channel_layer.group_add(self.room_group_name, self.channel_name)
        await self.accept()

    async def receive(self, text_data):
        data = json.loads(text_data)
        await self.channel_layer.group_send(
            self.room_group_name,
            {'type': 'chat_message', 'message': data['message']}
        )

    async def chat_message(self, event):
        await self.send(text_data=json.dumps({'message': event['message']}))
```

### Q98. What are Django's `Prefetch` objects and when do you use them?

`Prefetch` objects give you fine-grained control over `prefetch_related`.

```python
from django.db.models import Prefetch

articles = Article.objects.prefetch_related(

        to_attr='approved_comments',
    ),
```

### Q99. How do you implement health checks in Django?

```python
# views.py
from django.http import JsonResponse
from django.db import connection

def health_check(request):
    checks = {}
    try:
        with connection.cursor() as cursor:
            cursor.execute("SELECT 1")
        checks['database'] = 'ok'
    except Exception as e:
        checks['database'] = f'error: {e}'

    try:
        from django.core.cache import cache
        cache.set('health_check', 'ok', timeout=10)
        checks['cache'] = 'ok' if cache.get('health_check') == 'ok' else 'error'
    except Exception as e:
        checks['cache'] = f'error: {e}'

    overall = all(v == 'ok' for v in checks.values())
    return JsonResponse(
        {'status': 'healthy' if overall else 'unhealthy', 'checks': checks},
        status=200 if overall else 503,
    )

# urls.py
path('health/', health_check, name='health-check'),
```
