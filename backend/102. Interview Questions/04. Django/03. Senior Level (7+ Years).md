### Q1. Walk through Django’s request–response lifecycle at the framework level: middleware ordering, exception propagation, and how responses are finalized.

Django’s HTTP handling begins when the WSGI or ASGI server invokes the application callable with an environ (or ASGI scope) and a start-response callback. The core `WSGIHandler` (or `ASGIHandler`) builds a `HttpRequest`, then walks the **middleware chain in order** for the request phase. Each middleware’s `process_request` runs sequentially; the first middleware that returns a non-`None` response short-circuits the chain and skips the view entirely—this is how authentication redirects or maintenance pages are often implemented without touching views. If no middleware returns early, the URL resolver matches `request.path` against `urlpatterns`, instantiates the view callable (function or `View.as_view()`), and invokes it. Middleware `process_view` hooks run just before the view; they receive `(request, view_func, view_args, view_kwargs)` and can again return a response to abort the view.

When the view executes, it may raise `Http404`, `PermissionDenied`, or any other exception. Unhandled exceptions propagate back through middleware in reverse order: first `process_exception` on each middleware (innermost to outermost), giving middleware a chance to convert failures into HTTP responses. If none handle it, Django’s default exception handling maps known exceptions to status codes or falls through to the 500 handler. If the view returns normally, the response object flows through `process_response` **in reverse order** of middleware registration—this is the phase where you typically set security headers, compress bodies, or attach cookies. Because `process_response` always receives the final `HttpResponse` (or subclass), ordering matters: outer middleware runs last on the way “out,” which is why `SecurityMiddleware` is often placed first in `MIDDLEWARE` so its response processing runs late and can still wrap the fully built response.

Production teams rely on this contract for observability: open tracing spans in `process_request`, close them in `process_response`, and use `process_exception` to record error fingerprints. Misunderstanding the order causes subtle bugs—for example, putting a middleware that assumes the session is loaded before `SessionMiddleware`, or expecting `process_response` to run when an earlier middleware returned a response (it will, because the short-circuit still unwinds the stack). Async views under ASGI follow the same conceptual phases but require async-capable middleware; synchronous middleware is wrapped via `sync_to_async`, which has thread-pool implications under load.

```python
# settings.py — order matters: first listed = outer request / inner response
MIDDLEWARE = [
    "django.middleware.security.SecurityMiddleware",
    "django.contrib.sessions.middleware.SessionMiddleware",
    "django.middleware.common.CommonMiddleware",
    "django.middleware.csrf.CsrfViewMiddleware",
    "django.contrib.auth.middleware.AuthenticationMiddleware",
    "django.middleware.clickjacking.XFrameOptionsMiddleware",
]

# middleware/example.py
class RequestIdMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        request.request_id = request.META.get("HTTP_X_REQUEST_ID", "")
        response = self.get_response(request)
        if request.request_id:
            response["X-Request-ID"] = request.request_id
        return response
```

### Q2. How does Django’s ORM translate a `QuerySet` into SQL, and what role do lazy evaluation, the SQL compiler, and query expressions play?

A `QuerySet` is deliberately **lazy**: constructing `User.objects.filter(is_active=True)` does not hit the database until the query is evaluated—by iteration, `list()`, slicing that forces evaluation, `count()` in some paths, or truthiness checks that require existence. Internally, `QuerySet` holds a `django.db.models.sql.Query` object (often called the “query object”) that accumulates filters, annotations, joins, ordering, and select/prefetch hints. When evaluation happens, Django’s **SQL compiler** (`SQLCompiler` and database-specific subclasses) walks that `Query` tree and emits parameterized SQL plus a list of positional parameters bound separately from the SQL string—this separation is what makes Django resistant to classic string-concatenation SQL injection when you use ORM APIs.

The compiler pipeline resolves field lookups (`user__profile__bio__icontains`) into join paths and `WHERE` clauses, applies table aliases, and may rewrite expressions for databases that lack certain features. **Query expressions** (`F`, `Func`, `Subquery`, aggregates) implement `as_sql()` methods so they participate in the same compilation process rather than being pre-evaluated in Python. That is why `F('salary') * 1.1` becomes SQL arithmetic instead of fetching rows into Python. `QuerySet` chaining returns new `QuerySet` instances (or clones) so immutability semantics stay predictable—each `.filter()` adds `WHERE` nodes without mutating prior query objects you may still hold references to.

In production, understanding laziness prevents accidental query explosions: storing a `QuerySet` in a view and evaluating it multiple times may re-execute identical SQL unless you cache results. Understanding compilation helps when `annotate()`/`order_by()` interact with `GROUP BY` rules on PostgreSQL versus MySQL, or when `distinct()` is required to stabilize results after joins. Profiling with `connection.queries` (in development) or APM tools shows the final SQL; `str(qs.query)` is a useful approximation but not identical to every database backend’s emitted SQL.

```python
from django.db import connection
from django.db.models import F, Count, Q

qs = (
    User.objects.filter(Q(is_staff=True) | Q(is_superuser=True))
    .annotate(order_count=Count("orders"))
    .filter(order_count__gte=1)
    .order_by(F("date_joined").desc(nulls_last=True))
)

# Force compilation / execution
list(qs[:10])

if connection.settings_dict.get("DEBUG"):
    print(connection.queries[-1]["sql"])
```

### Q3. Explain Django’s model metaclass machinery: `ModelBase`, model registration, and the app registry.

Every Django model class is constructed by `ModelBase`, a metaclass that runs at **class creation time**—not at import time for instances, but when Python executes `class MyModel(models.Model): ...`. `ModelBase` collects `Field` objects from the class body, handles inheritance (including abstract base models and multi-table inheritance), and wires up default managers, `Meta` options, and the `_meta` `Options` object that powers introspection (`MyModel._meta.get_field('email')`). Fields are assigned to the model class with descriptors that implement the attribute protocol so that access on instances triggers database I/O only when needed, while access on the class yields field objects for the ORM.

Registration happens through `apps.register_model(app_label, model)`, invoked as part of model preparation. Django’s **application registry** (`django.apps.apps`) tracks installed applications, their `AppConfig` instances, and all models. The registry enforces uniqueness of `(app_label, model_name)` and resolves string references like `'myapp.User'` lazily after all models are loaded—this is what makes forward references and circular imports tractable if you use string relations (`ForeignKey('otherapp.Thing')`) or `apps.get_model`. The loading process is orchestrated in three phases (`django.setup()`): populate installed apps, import models modules, then run `ready()` signals on each `AppConfig`.

Senior engineers care because import cycles, duplicate model definitions, and `AppRegistryNotReady` errors almost always trace back to module import side effects that touch the ORM too early—for example, calling `get_user_model()` at module import before `django.setup()` in scripts. The fix is deferring lookups into methods or using `apps.get_model` inside functions. Custom metaclasses that compose with `ModelBase` are rare but possible; you must inherit `ModelBase` correctly or break model discovery entirely.

```python
# apps.py — safe place for initialization that needs the registry
from django.apps import AppConfig

class AccountsConfig(AppConfig):
    default_auto_field = "django.db.models.BigAutoField"
    name = "accounts"

    def ready(self):
        # Import signal modules here so models are registered first
        from . import signals  # noqa: F401

# models.py — string reference avoids circular import at class body time
class Profile(models.Model):
    user = models.OneToOneField(
        "auth.User",
        on_delete=models.CASCADE,
        related_name="profile",
    )
```

### Q4. How does Django’s migration framework work internally, and how do state, operations, and squashing interact?

Migrations are not raw SQL scripts; they are **Python modules** containing a `Migration` subclass whose `operations` list describes declarative schema changes (`CreateModel`, `AlterField`, `RunSQL`, etc.). Django maintains two parallel concepts: **migration history** (the ordered chain of applied migration files on disk) and **project state**—a pure Python representation of models after sequentially applying operations. When you run `makemigrations`, Django diffs your current models against the latest frozen state and emits new operations. When you run `migrate`, the migration executor builds a plan by walking the dependency graph (each migration declares `dependencies`), acquires a schema editor for your database backend, and applies operations transactionally where supported.

The **dependency graph** ensures correctness across apps: cross-app foreign keys introduce edges so that tables exist before keys reference them. Squashing (`optimizesquash` / squash migrations) merges many migrations into one file for speed of `migrate` on fresh databases, while keeping old migration files for existing environments until you transition. Squashed migrations carry `replaces` and carefully rewritten dependencies so the graph remains acyclic. **State vs. database** can diverge if someone applies manual SQL or edits databases by hand; `inspectdb` and `sqlmigrate` help reconcile reality with code.

Production pitfalls include long-running `AlterField` on huge tables (locking), **backwards-incompatible** renames that `makemigrations` cannot prove are renames (data loss risk), and multi-database setups where routers must permit migration writes on the correct alias. Senior teams often pair migrations with expand/contract patterns for zero-downtime deploys: add nullable column, dual-write in application code, backfill, then enforce `NOT NULL` in a later migration.

```python
from django.db import migrations, models

class Migration(migrations.Migration):
    dependencies = [
        ("shop", "0007_auto_20240401_1200"),
    ]

    operations = [
        migrations.SeparateDatabaseAndState(
            state_operations=[
                migrations.AddField(
                    model_name="order",
                    name="legacy_ref",
                    field=models.CharField(max_length=32, null=True),
                ),
            ],
            database_operations=[
                # Custom SQL for CONCURRENT index on PostgreSQL, etc.
            ],
        ),
```

### Q5. Describe ASGI support in Django: how async views integrate with the event loop, and what limitations apply.

Django’s ASGI entrypoint (`django.core.handlers.asgi.ASGIHandler`) consumes ASGI messages (`http.request`, `http.disconnect`, etc.) and constructs `HttpRequest` objects similarly to WSGI, but the execution model can be **async** end-to-end when your view is a coroutine function. If an async view awaits I/O—HTTP calls to other services, non-blocking database drivers where available, or `asyncio` primitives—those waits yield control back to the event loop, improving throughput for I/O-bound workloads compared to blocking threads. Synchronous views and middleware are supported via adapters that run sync code in a thread pool (`sync_to_async`), which avoids blocking the loop but is not free: under high concurrency, thread pools can become the bottleneck.

ORM access from async views has historically required `sync_to_async` because Django’s database layer is largely synchronous; recent Django versions have introduced **asynchronous ORM interfaces** (`aget`, `acreate`, etc.) on a growing surface, but not every API is async-safe yet. Misuse patterns include calling blocking ORM methods directly inside `async def` views, which stalls the entire event loop for that worker. Template rendering and many third-party libraries remain synchronous; wrapping them correctly matters. Deployment implications include choosing ASGI servers (Uvicorn, Daphne, Hypercorn) and understanding **per-request scope** isolation—long-lived connections (WebSockets) require consumers beyond classic request/response.

Teams adopt async views selectively for microservice gateways, fan-out aggregation, or endpoints that spend most of their time waiting on external APIs. CPU-heavy work still belongs in task queues (Celery, RQ), not in async views. Testing async views uses `AsyncClient` from Django’s test framework.

```python
# views.py
import httpx
from django.http import JsonResponse
from asgiref.sync import sync_to_async

async def aggregate_status(request):
    async with httpx.AsyncClient() as client:
        r = await client.get("https://status.example.com/api", timeout=5.0)
    return JsonResponse(r.json())

@sync_to_async
def _count_users():
    return User.objects.count()

async def user_count(request):
    n = await _count_users()
    return JsonResponse({"users": n})
```

### Q6. How does Django’s class-based view (CBV) dispatch mechanism work, including MRO, `as_view()`, and mixin interaction?

`View.as_view()` returns a closure that is itself a WSGI/ASGI-callable-friendly function; on each request it instantiates the view **per request** (`self.request`, `self.args`, `self.kwargs`) and calls `dispatch()`. `dispatch()` examines `request.method.lower()` and looks up a handler attribute (`get`, `post`, etc.), defaulting to `http_method_not_allowed` if missing. This design favors explicitness: CBVs are classes for reuse via inheritance, but instance state is not shared across requests—avoid storing mutable request-correlated data on `self` beyond the lifetime of a single dispatch.

Method resolution order (MRO) follows Python’s C3 linearization. Django’s generic CBVs compose behavior through **mixins** (`LoginRequiredMixin`, `PermissionRequiredMixin`, `ListView` + `MultipleObjectMixin`, etc.). Ordering matters: mixins that short-circuit (`dispatch` overrides that redirect) should appear **leftmost** in the inheritance list so their `dispatch` runs before wider `View.dispatch` behavior—though in practice many mixins call `super()` and cooperative multiple inheritance relies on consistent `super()` chains. Conflicts arise when two mixins override the same method without collaborating, or when `get_context_data`/`get_queryset` forget to call `super()`, silently dropping data from sibling classes.

Senior debugging often involves reading `django.views.generic` source: `TemplateView` provides `get_template_names`, `DetailView` wires `SingleObjectMixin` with `BaseDetailView`. Understanding `as_view(**initkwargs)` lets you inject class-level configuration safely (only whitelisted attributes are copied onto the instance). For APIs, Django REST framework’s `APIView` layers renderer/permission negotiation atop similar ideas—worth comparing.

```python
from django.contrib.auth.mixins import LoginRequiredMixin, PermissionRequiredMixin
from django.views.generic import ListView

class InvoiceListView(LoginRequiredMixin, PermissionRequiredMixin, ListView):
    permission_required = "billing.view_invoice"
    model = Invoice
    paginate_by = 25

    def get_queryset(self):
        qs = super().get_queryset()
        return qs.filter(tenant_id=self.request.user.tenant_id)
```

### Q7. Explain how Django’s template language is lexed, parsed, evaluated, and rendered, including how context resolves variables.

Django’s template engine processes source text through a **lexer** that tokenizes tags, variables, and text into structured tokens. The **parser** consumes tokens and builds a tree of `Node` objects—compiled representation—not raw strings. Tags like `{% if %}`, `{% for %}`, and `{% extends %}` register implementations via decorators (`@register.tag`) that parse their arguments and return specialized `Node` subclasses whose `render()` methods orchestrate child nodes. Variable resolution (`{{ user.email }}`) uses a pluggable expression pipeline (`FilterExpression`) that handles dotted lookups, optional filters (`|default:"n/a"`), and lazy callables.

Rendering walks the node tree with a **Context** (or `RequestContext` if you use context processors) that supports layered dict-like scopes (`push`/`pop`). Resolution uses attribute then key then method call rules, with `_()`-aware behavior for autoescaping. Templates are **not** Python; they intentionally constrain logic to reduce coupling, pushing heavy computation into views or custom template tags. Caching (`cached_template_loader` in production) avoids re-parsing files on every request; `Template` objects can be compiled once and reused.

Production issues include expensive context processors that run globally, accidental quadratic behavior in deep `{% include %}` chains, and autoescaping mistakes (`|safe` overuse). Alternative engines (Jinja2) integrate through `BACKEND` configuration but change extension ecosystems; migrating requires retesting tags/filters.

```python
# templatetags/pricing.py
from django import template

register = template.Library()

@register.filter
def cents(value):
    try:
        cents = int(value)
    except (TypeError, ValueError):
        return ""
    return f"{cents / 100:.2f}"

# usage: {{ amount|cents }}
```

### Q8. What are Django signals, how are they implemented, and when should you avoid them?

Signals implement a lightweight **observer pattern**: `Signal` objects maintain lists of receivers (callables), connected via `receiver()` decorator or `signal.connect`. When `signal.send` or `send_robust` fires, Django iterates receivers—`send_robust` catches exceptions per receiver so one bad handler does not break others. Internally, weak references may be used to avoid retaining receivers forever if connections are not explicitly managed, depending on connection options—important in long-running processes where leaks amplify.

Signals are **synchronous** and run in-process during the same request/transaction scope unless dispatched to a task queue manually. They are not a substitute for durable workflows: if the process crashes after your model save but before a signal side effect completes, you have no at-least-once guarantee. They complicate testing because implicit couplings hide dependencies—tests may need to disconnect receivers or use `captureOnCommitCallbacks` patterns when transactions matter. They also make stack traces harder to follow than explicit service functions.

Senior teams prefer **explicit application services** for core business logic (`create_order()` calls billing, inventory, notifications in one place), reserving signals for orthogonal concerns like cache invalidation, denormalized index updates, or audit hooks where coupling must stay loose. If you use `pre_save`/`post_save`, understand transaction boundaries: `on_commit` is often safer for external side effects.

```python
from django.db.models.signals import post_save
from django.dispatch import receiver
from django.db import transaction
from .models import Profile, User

@receiver(post_save, sender=User)
def ensure_profile(sender, instance, created, **kwargs):
    def _create():
        Profile.objects.get_or_create(user=instance)

    transaction.on_commit(_create)
```

### Q9. How does Django’s system check framework operate, and how do you implement deployment-aware checks?

`manage.py check` (also run implicitly before many commands) loads settings and executes **registered checks**: built-ins (`C001`–`C999` style tags) validate configuration, models, security settings, and common deployment footguns. Checks are plain functions decorated with `@register_check` or tagged via `Tags`, optionally constrained to **deployment** only (`deploy=True`) so development environments stay quiet while CI/staging/production fail fast on insecure `SECRET_KEY`, `ALLOWED_HOSTS`, or missing HTTPS settings.

Custom checks should be **idempotent**, fast, and side-effect free—they are diagnostics, not migrations. Return `Error` for hard failures, `Warning` for risks, `Info` for guidance. Use checks to encode organizational policies: ensuring `CORS_ALLOWED_ORIGINS` is non-wildcard in prod, verifying required environment variables for feature flags, or asserting database connection pooling parameters are present. Pair checks with CI gates (`python manage.py check --deploy`) so bad configs never reach orchestrators.

Checks differ from middleware: middleware runs per request; checks run at process startup (or management command time). Avoid importing heavy modules globally if it slows every CLI invocation; lazy-import inside the check function when necessary.

```python
# checks.py
from django.core.checks import Error, register, Tags

@register(Tags.security, deploy=True)
def check_session_cookie_secure(app_configs, **kwargs):
    from django.conf import settings

    if not settings.SESSION_COOKIE_SECURE:
        return [
            Error(
                "SESSION_COOKIE_SECURE must be True in production.",
                id="myapp.E001",
            )
        ]
    return []
```

### Q10. Describe Django’s application loading sequence: `AppConfig`, `INSTALLED_APPS`, `ready()`, and import hazards.

At startup, Django iterates `INSTALLED_APPS`, resolves dotted paths to `AppConfig` classes (default `AppConfig` if unspecified), and constructs instances. It imports each application’s `models` module (unless configured otherwise) to register models, building the registry discussed earlier. The **`ready()`** hook fires once per `AppConfig` after the model registry is fully populated—this is the sanctioned place to import signal modules, register checks, or warm caches that require `apps.get_model`.

Import order hazards stem from Python’s eager execution: referencing models at module import time before the registry finishes causes `AppRegistryNotReady`. Similarly, defining forms or serializers that instantiate querysets at import time executes database calls during startup—sometimes before connections are configured—leading to fragile tests and accidental production queries. The remedy is lazy evaluation: `ModelChoiceField(queryset=User.objects.none())` plus setting queryset in `__init__`, or using `get_user_model()` inside methods.

`default_app_config` is legacy; modern Django uses `AppConfig` subclasses referenced in `INSTALLED_APPS` like `'myapp.apps.MyAppConfig'`. Namespace packages and duplicate `app_label` collisions surface here; keeping app labels unique and explicit prevents mystifying migration graphs.

```python
# apps.py
from django.apps import AppConfig

class CatalogConfig(AppConfig):
    name = "catalog"
    verbose_name = "Product Catalog"

    def ready(self):
        from . import signals  # noqa: F401

# Avoid at import time:
# bad_qs = Product.objects.all()  # triggers DB / registry assumptions
```

### Q11. How do custom `Manager`/`QuerySet` pairs compose, and what does `as_manager()` change about inheritance and chaining?

Django managers expose ORM entry points (`User.objects...`). Subclassing **`QuerySet`** lets you define domain-specific filters (`active()`, `for_tenant()`, `published()`) that return `self` or cloned querysets to preserve chainability. Pairing with a **custom `Manager`** that overrides `get_queryset()` ensures those methods are always available on `Model.objects`. The `QuerySet.as_manager()` class method synthesizes a `Manager` whose `get_queryset()` returns instances of that `QuerySet` subclass—reducing boilerplate while keeping a single class to maintain.

Composition matters when combining mixins: multiple abstract models might each add managers; using `RelatedManager` for reverse relations repeats custom queryset behavior for `parent.children`. Without careful naming, you can shadow default managers or break migrations that expect `_base_manager` semantics. Using `from_queryset()` lets you reuse queryset classes across models with explicit manager classes.

In production, encapsulating access patterns in querysets prevents duplicated `.filter()` logic scattered across views and tasks, easing refactors when schema evolves. Unit tests target the queryset methods directly, independent of HTTP.

```python
class ProductQuerySet(models.QuerySet):
    def available(self):
        return self.filter(is_active=True, stock__gt=0)

    def for_store(self, store_id):
        return self.filter(store_id=store_id)

class Product(models.Model):
    store = models.ForeignKey(Store, on_delete=models.CASCADE)
    is_active = models.BooleanField(default=True)
    stock = models.PositiveIntegerField()

    objects = ProductQuerySet.as_manager()
```

### Q12. Explain Django’s query expression API: how `F`, `Q`, `Subquery`, `OuterRef`, and `Exists` compose for advanced SQL.

`Q` objects encode boolean logic for `WHERE` clauses with `&`, `|`, and `~`, enabling OR conditions and negations that keyword-only `.filter()` calls cannot express cleanly. `F` references column values on the database side, enabling atomic updates (`F('views') + 1`) and comparisons between columns without race-prone read-modify-write in Python. **`Subquery` and `OuterRef`** correlate nested selects: `OuterRef` points outward to an enclosing query’s columns, letting you annotate each row with aggregates or lookups from related tables in one SQL roundtrip—critical for dashboards that would otherwise N+1.

`Exists` is optimized to SQL `EXISTS (...)` semantics, often better than `COUNT` subqueries when you only need a boolean. Pairing `Subquery` with `.values()` projections controls selected columns and avoids loading full models. Database backends differ slightly in allowed correlations and `GROUP BY` interactions; PostgreSQL’s planner handles many patterns gracefully, while SQLite has stricter limitations.

Production teams use these constructs for authorization-aware annotations (`Exists` subquery against join tables for roles), soft-delete aware aggregates, and avoiding Python-side filtering of large datasets.

```python
from django.db.models import Exists, OuterRef, Subquery, F, Q

newest_comment = Comment.objects.filter(post=OuterRef("pk")).order_by("-created_at")

Post.objects.annotate(
    has_mod_comment=Exists(
        Comment.objects.filter(
            post=OuterRef("pk"),
            Q(user__is_staff=True) | Q(body__icontains="[mod]"),
        )
    ),
    latest_comment_id=Subquery(newest_comment.values("id")[:1]),
    score=F("upvotes") - F("downvotes"),
)
```

### Q13. How do window functions work in Django’s ORM, and what patterns suit ranking, running totals, and partitioning?

Window functions compute values across a **partition** of rows related to the current row, without collapsing groups like simple `GROUP BY`. Django exposes them via `Window` expressions combined with functions like `RowNumber`, `Rank`, `DenseRank`, `NthValue`, and aggregates with the `filter` argument inside partitions. Define `partition_by` on keys that scope each window (e.g., per `user_id`), and `order_by` to control sequence within the partition.

Use **ranking** queries to deduplicate “latest row per group” problems in SQL instead of Python grouping. **Running totals** use `Sum` over an ordered window; check your database for `ROWS BETWEEN` framing support—Django’s API maps closely to SQL:2011 window features on PostgreSQL. Performance-wise, window functions can be faster than correlated subqueries for large tables but may still require appropriate indexes on `(partition keys, order keys)`.

Edge cases include NULL ordering affecting ranks, and databases where window functions cannot appear in `WHERE`—you typically wrap in subqueries or filter on annotated aliases in outer queries (`WHERE rn = 1` pattern via `Subquery`).

```python
from django.db.models import Window, F
from django.db.models.functions import RowNumber, Sum

qs = (
    LedgerEntry.objects.annotate(
        running_balance=Window(
            expression=Sum("amount"),
            partition_by=[F("account_id")],
            order_by=F("posted_at").asc(),
        ),
        rn=Window(
            expression=RowNumber(),
            partition_by=[F("account_id")],
            order_by=F("posted_at").desc(),
        ),
    )
)
```

### Q14. When should you drop to raw SQL or database functions, and how do you escape and structure this safely?

Raw SQL is justified when the ORM cannot express a query plan you need (complex CTE pipelines, vendor-specific optimizations), or when bulk maintenance tasks dwarf ORM overhead. Django offers `raw()` for model instantiation from SQL with typed parameters, `cursor.execute()` with placeholders, and `Func`/`database functions` to stay partially ORM-integrated. **Never** interpolate user input into SQL strings; always bind parameters via the DB-API placeholders so the driver escapes values correctly.

Use **database functions** (`Lower`, `Coalesce`, `Now`, or custom `Func` subclasses) when computation belongs in SQL for performance or correctness (timezone-aware timestamps). For migrations, `RunSQL` with `elidable=True` or separate operational scripts may apply DDL unavailable through schema editors. Operational risk includes bypassing ORM signals and soft-delete managers—raw updates can desynchronize denormalized fields unless you update them explicitly.

Teams document raw SQL in repositories, add tests asserting row counts and invariants, and run `EXPLAIN ANALYZE` in staging mirroring production volumes.

```python
from django.db import connection

def refresh_stats():
    sql = """
        UPDATE stats_daily AS s
        SET revenue = d.revenue
        FROM (
            SELECT day, SUM(amount) AS revenue
            FROM orders_order
            WHERE status = %s
            GROUP BY day
        ) AS d
        WHERE s.day = d.day
    """
    with connection.cursor() as cursor:
        cursor.execute(sql, ["paid"])
```

### Q15. How does multi-database routing work in Django, and what are the limitations around replicas and cross-database relations?

The `DATABASE_ROUTERS` setting lists router classes implementing `db_for_read`, `db_for_write`, `allow_relation`, and `allow_migrate`. Django asks routers which alias to use for each model operation; first non-`None` wins. This enables **primary/replica** splits: writes go to `default`, reads to `replica`, with sticky-session or latency caveats—replication lag means a read-your-writes test might fail briefly after registration unless you route critical reads to primary.

`allow_relation` prevents accidental `ForeignKey` between models that live on different databases—Django cannot join across connections, so ORM-level joins spanning aliases are not supported. `allow_migrate` keeps migrations scoped: some apps only exist on analytics warehouses. Testing requires `DATABASES` mirrors and `--database` flags for commands.

Production complexities include connection pool sizing per alias, secrets rotation per cluster, and ensuring management commands target the correct writer. Cross-database work uses application-level IDs and APIs (REST/events), not database foreign keys.

```python
class PrimaryReplicaRouter:
    read_db = "replica"
    write_db = "default"

    def db_for_read(self, model, **hints):
        return self.read_db

    def db_for_write(self, model, **hints):
        return self.write_db

    def allow_relation(self, obj1, obj2, **hints):
        return True

    def allow_migrate(self, db, app_label, model_name=None, **hints):
```

### Q16. Explain database-level constraints in Django: `UniqueConstraint`, `CheckConstraint`, `ExclusionConstraint`, and partial indexes.

Schema-level constraints move invariants closer to data: **`UniqueConstraint`** enforces uniqueness on column tuples or expressions, optionally **conditional** (`condition=Q(...)`) for partial unique indexes—common for “one primary email per user” while allowing multiple secondary emails. **`CheckConstraint`** encodes boolean SQL predicates (`check=Q(age__gte=18)`), catching application bugs that bypass validation. On PostgreSQL, **`ExclusionConstraint`** (via `django.contrib.postgres.constraints`) prevents overlapping ranges—ideal for reservations, scheduling, and preventing double-booking with GiST/SP-GiST indexes.

Django declares these in `Meta.constraints` and generates migrations; databases enforce them concurrently with transactions, unlike Python-only validation. Trade-offs: constraints can complicate bulk loads (order matters), and migrations may need `NOT VALID` patterns on PostgreSQL for large tables (supported through `AddConstraintNotValid` workflows in advanced migration operations or raw SQL). Partial indexes reduce write amplification and index size by indexing only hot subsets (`WHERE tenant_id = ...`).

Senior engineers pair ORM `validate_constraints()` (where applicable) with database checks so race conditions cannot insert invalid rows between `clean()` and `save()`.

```python
from django.conf import settings
from django.db import models
from django.db.models import Q, UniqueConstraint

class UserEmail(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    email = models.EmailField()
    is_primary = models.BooleanField(default=False)

    class Meta:
        constraints = [
            UniqueConstraint(
                fields=["user"],
                condition=Q(is_primary=True),
                name="uniq_one_primary_email_per_user",
            ),
        ]
```

### Q17. Compare optimistic and pessimistic concurrency control in Django applications: `select_for_update`, versioning, and trade-offs.

**Pessimistic** control uses database locks: `select_for_update()` inside `transaction.atomic()` locks matching rows until the transaction ends, preventing concurrent writers from reading stale state and overwriting each other—essential for ledger-like updates or inventory decrements with strong isolation. Downsides: lock contention, deadlocks if lock order is inconsistent across code paths, and shorter transactions are mandatory to avoid throughput collapse.

**Optimistic** control adds a **version field** (`version = PositiveIntegerField(default=0, help_text="for optimistic locking")`) and checks it in updates: `UPDATE ... WHERE id=%s AND version=%s` with `version=version+1`, failing if row counts show zero rows updated—signals a concurrent change, typically retried with user feedback or exponential backoff. This scales better under low conflict rates but pushes conflict detection to commit time.

Isolation levels matter: `READ COMMITTED` (PostgreSQL default) still allows phantom reads unless you design queries carefully; `SERIALIZABLE` adds safety at a performance cost. Production services choose per use case: financial transfers may combine `select_for_update` with idempotency keys; profile edits may use optimistic versioning.

```python
from django.db import transaction
from django.db.models import F

def allocate_stock(order_id):
    with transaction.atomic():
        item = InventoryItem.objects.select_for_update().get(sku="ABC123")
        if item.quantity < 1:
            raise ValueError("out of stock")
        item.quantity = F("quantity") - 1
        item.save(update_fields=["quantity"])
```

### Q18. Describe Django’s transaction management: `atomic()`, savepoints, `on_commit`, and isolation.

`transaction.atomic()` demarcates a database transaction, optionally **nested** via savepoints on backends that support them—inner failures roll back to the savepoint without aborting outer work. This is how you implement partial batch processing: process N records per savepoint, continue on benign errors, commit outer scope only if overall success criteria hold. Decorator and context manager forms exist; pick context managers when you need dynamic boundaries.

`transaction.on_commit` schedules callable hooks after successful commit—ideal for enqueueing Celery tasks or emitting events so workers see committed state. Without this, tasks may run early and read missing rows. Be mindful: long transactions delay `on_commit` callbacks and hold locks—keep atomic sections tight.

Database **isolation** is configured at the connection level (`OPTIONS` with `isolation_level` on some backends) or via raw SQL `SET TRANSACTION`. Higher isolation reduces anomalies but increases retries and deadlocks. Testing should exercise concurrency with threads or `ThreadPoolExecutor` where relevant.

```python
from django.db import transaction

def place_order(order):
    with transaction.atomic():
        order.save()
        transaction.on_commit(lambda: notify_warehouse.delay(order.id))
```

### Q19. What are the semantics and production patterns for bulk ORM operations (`bulk_create`, `bulk_update`) on very large datasets?

`bulk_create` issues multi-row `INSERT` statements, drastically reducing roundtrips versus per-row `save()`. Caveats: many-to-many relations are not saved, auto `ManyToMany` through tables are untouched, and primary keys may not populate on some databases unless `ignore_conflicts`/`update_conflicts` strategies and `returning_fields` (PostgreSQL) are used appropriately. **`bulk_update`** sets multiple rows with a CASE/WHERE style update (backend-dependent), requiring you to specify `fields` to mutate; it skips `save()` signals and model `save()` overrides—by design for speed, risky if business invariants live in overrides.

For **millions** of rows, chunking is mandatory: iterate source data in batches (e.g., 1k–10k rows), use `iterator()` on querysets to avoid loading everything into memory, and consider raw `COPY`/`LOAD` on PostgreSQL for ingest beats. Wrap chunks in transactions per batch to balance durability and lock duration. For idempotent re-runs, use natural keys and `update_conflicts` where supported.

Observability includes measuring rows/sec, WAL growth on PostgreSQL, and replica lag if bulk writes replicate. Background tasks perform heavy bulk work off request threads.

```python
def import_users(rows, batch_size=2000):
    for i in range(0, len(rows), batch_size):
        batch = rows[i : i + batch_size]
        User.objects.bulk_create(
            [User(username=r["username"], email=r["email"]) for r in batch],
            batch_size=batch_size,
            ignore_conflicts=True,
        )
```

### Q21. How do custom authentication backends work in Django, and how would you implement LDAP, OAuth, token-based auth, and multi-backend resolution order in production?

Django authenticates users through `AUTHENTICATION_BACKENDS`, a list of dotted paths to classes implementing `authenticate(request, **credentials)` and `get_user(user_id)`. Each backend is tried in order until one returns a non-`None` user (or all return `None`). The first successful backend “wins” for that login attempt. `get_user` is used during session restoration to rehydrate the user from the primary key stored in the session; backends must implement `get_user` so that subsequent requests can load the user without re-presenting credentials.

For LDAP, you typically implement a backend that binds to the directory with the submitted username/password (or uses a service account + user search), maps LDAP attributes to Django `User` fields, and creates or updates a local `User` row on first successful bind. You must handle connection pooling, TLS, referral chasing, and account lockout policies at the LDAP layer. OAuth “login” is usually not a classic password backend: you exchange an authorization code for tokens in a view, then map the external subject identity to a local user (create-on-first-login or link by verified email). You might still register a lightweight backend if you need `RemoteUser`-style flows, but most OAuth flows authenticate in a view and call `login(request, user)` after user resolution.

Token-based APIs often use DRF authentication classes rather than session backends, but if you want a session-style backend for token verification, you can validate a bearer token in `authenticate`, look up or create the user, and return it. Multi-backend order matters: put `ModelBackend` last if you want LDAP to try first, or put `ModelBackend` first if local superusers must always work even when directory is down—document the failover behavior. Use `AUTHENTICATION_BACKENDS` ordering plus monitoring: if LDAP is first and slow, login latency spikes for everyone.

```python
# settings.py — order is significant
AUTHENTICATION_BACKENDS = [
    "myauth.backends.LdapOrModelBackend",
    "django.contrib.auth.backends.ModelBackend",
]

# myauth/backends.py
from django.contrib.auth.backends import ModelBackend
from django.contrib.auth import get_user_model

User = get_user_model()

class LdapOrModelBackend(ModelBackend):
    def authenticate(self, request, username=None, password=None, **kwargs):
        if not username or not password:
            return None
        # 1) Try LDAP bind + attribute mapping
        ldap_user = self._ldap_validate_and_map(username, password)
        if ldap_user:
            user, _ = User.objects.update_or_create(
                username=ldap_user["uid"],
                defaults={"email": ldap_user.get("mail", ""), "is_active": True},
            )
            return user if self.user_can_authenticate(user) else None
        # 2) Fall through: let next backend handle (not reached if ModelBackend is separate)
        return None

    def get_user(self, user_id):
        try:
            return User.objects.get(pk=user_id)
        except User.DoesNotExist:
            return None
```

In production, isolate secrets (LDAP bind passwords, OAuth client secrets) in the environment or a secrets manager, use timeouts on directory calls, and add circuit breakers so a degraded IdP does not hang every request. Log authentication backend used (without passwords) for audit trails.

### Q22. Explain Django’s permission framework internals: how `has_perm` works, permission caching, and custom permissions.

Permissions are rows in `auth_permission` tied to a `ContentType`; users get them through `User.user_permissions` or through groups. `ModelBackend` implements `has_perm(user_obj, perm, obj=None)` by checking superuser short-circuit, then resolving `perm` as `"app_label.codename"`, then testing direct permissions, group permissions, and (for staff) implicit `django.contrib.auth` admin access patterns. The `obj` argument is ignored by the stock backend but is the extension point for object-level backends.

`User.get_all_permissions(obj)` resolves and caches effective permissions on the user instance (`_perm_cache`), so repeated `has_perm` checks after a warm cache avoid repeated joins through `user_user_permissions` and `auth_group_permissions`. Anonymous users short-circuit to an empty set. Custom permissions on models are declared in `Meta.permissions` as `(codename, human_name)` tuples; migrations create `auth_permission` rows for those codenames. Admin-heavy views that call `has_perm` in tight loops benefit from ensuring the cache is populated once per request (for example via `get_all_permissions()` early) rather than triggering partial queries repeatedly.

Superusers return `True` for any `has_perm` without consulting per-model rows for routine checks. In replicated database topologies, permission grants must be visible on the connection handling the request before `has_perm` can reflect role changes; read-your-writes consistency matters when permissions change during active sessions.

```python
# models.py
class Invoice(models.Model):
    class Meta:
        permissions = [
            ("approve_invoice", "Can approve invoices"),
        ]

# Custom backend adding object-level checks
from django.contrib.auth.backends import BaseBackend

class InvoiceBackend(BaseBackend):
    def has_perm(self, user_obj, perm, obj=None):
        if perm == "myapp.approve_invoice" and isinstance(obj, Invoice):
            return user_obj.is_authenticated and obj.owner_id == user_obj.id
        return None  # None = not handled; other backends may decide
```

For production, combine explicit `auth_permission` grants for coarse roles with object-level logic in a dedicated backend or django-guardian. Invalidate or avoid stale permission caches when you change group membership mid-request (rare); typically you reload the user from DB after role changes in admin workflows.

### Q23. How do you implement row-level or object-level permissions in Django, and what are the trade-offs of django-guardian versus custom approaches?

Row-level security means `has_perm("change_invoice", invoice)` depends on attributes of `invoice`, not only global roles. `django-guardian` stores per-object grants in `GuardianUserObjectPermission` and `GuardianGroupObjectPermission` tables, linking users/groups to generic foreign keys. Trade-offs: extra joins and tables; at scale, millions of object permissions can bloat the DB and complicate backups; you must index `(content_type_id, object_pk)` appropriately.

Custom approaches include encoding ownership on the model (`owner = ForeignKey(User)`) and enforcing checks in views/serializers, or using PostgreSQL row security policies (RLS) with a DB role per request—powerful but operational complexity. # django-guardian style filtering in a viewset
```python
from guardian.shortcuts import get_objects_for_user

    return qs
```

Production guidance: prefer ownership columns for simple “only my rows” cases; use guardian when arbitrary sharing/grants per object are required; use RLS when the database must enforce policy even if application code regresses. ### Q24. Describe CSRF protection internals in Django: token generation, double-submit cookie pattern, SameSite cookies, and CSRF with AJAX/SPAs.

Django’s `CsrfViewMiddleware` validates unsafe methods (POST, PUT, PATCH, DELETE) by comparing a token bound to the user session (or rotating secret) with a token submitted in POST data or a header (`X-CSRFToken` / `X-CSRFTOKEN`). The cookie `csrftoken` holds the client-visible value; the hidden form field or header must match. For `CSRF_USE_SESSIONS`, the secret lives server-side in the session; otherwise a signed cookie-based mask is used. The token is not automatically secret from JavaScript in same-origin pages—mitigation is SameSite cookies and not exposing XSS.

The “double-submit cookie” pattern appears in some frameworks as cookie+header equality without server state; Django instead ties CSRF to session (or signed cookie machinery), which is stronger against certain subdomain attacks when configured with correct `CSRF_TRUSTED_ORIGINS` and cookie flags. `CSRF_COOKIE_SAMESITE = "Lax"` or `"Strict"` reduces cross-site request risk; `"None"` requires `Secure` and is needed for cross-site embedded flows—then you rely more on exact token validation and Origin checks.

```javascript
// SPA: read cookie set by Django (ensure JS can read it — not HttpOnly)
const csrftoken = document.cookie
  .split("; ")
  .find((row) => row.startsWith("csrftoken="))
  ?.split("=")[1];

fetch("/api/action/", {
  method: "POST",
  credentials: "include",
  headers: {
    "Content-Type": "application/json",
    "X-CSRFToken": csrftoken,
  },
  body: JSON.stringify({}),
});
```

For AJAX APIs that use token auth without cookies, CSRF often does not apply to the same threat model; DRF with `SessionAuthentication` still needs CSRF for browser sessions. Production checklist: set `CSRF_TRUSTED_ORIGINS` for HTTPS hosts, avoid wildcard origins, use `CSRF_COOKIE_HTTPONLY` carefully (if true, JS cannot read cookie—then use explicit CSRF from template context), and test login flows with subdomains.

### Q30. How do you implement Content Security Policy (CSP) in Django using nonces, reporting, and integration patterns?

```python
# Middleware sketch
import secrets
from django.utils.deprecation import MiddlewareMixin

class CSPMiddleware(MiddlewareMixin):

        request.csp_nonce = secrets.token_urlsafe(16)

        policy = (
            "default-src 'self'; "
            f"script-src 'self' 'nonce-{nonce}'; "
            "style-src 'self' 'unsafe-inline'; "
            "report-uri /csp-report/"
        )
        response["Content-Security-Policy-Report-Only"] = policy
        return response
```

### Q31. Explain DRF serializer internals: field binding, the validation pipeline, and `to_representation` versus `to_internal_value`.

Serializers orchestrate `Field` instances: on `Serializer.__init__`, fields bind to `source` paths on the model or input dict (`source='profile.bio'` traverses attributes). `to_internal_value` runs per-field `Field.get_value` → `Field.run_validation` (including `validators=` and `validate_<field>` on serializer), then `validate` for cross-field checks. `to_representation` walks fields for output, calling `Field.get_attribute` (traversing ORM attributes) and `Field.to_representation`.

Understanding this order explains common bugs: `SerializerMethodField` is read-only and bypasses normal attribute fetch; nested serializers may trigger extra queries if not prefetched. Overriding `to_internal_value` is for wholesale input shaping; overriding `validate` is for business rules after fields are coerced types.

```python
class UserSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ["id", "email", "profile"]

    def validate(self, attrs):
        if attrs.get("email", "").endswith("@temp.com"):
            raise serializers.ValidationError("Temporary emails not allowed")
        return attrs

    def to_representation(self, instance):
        data = super().to_representation(instance)
        data["display_name"] = instance.get_full_name()
        return data
```

Production: split read and write serializers when validation differs from output; avoid heavy work in `SerializerMethodField` (see Q36); use `context={"request": request}` for hyperlinks and permission-aware fields.

### Q32. How do custom DRF authentication classes work for JWT rotation, API key scoping, and per-request authentication?

JWT rotation: issue short-lived access tokens and refresh tokens; on refresh endpoint, validate refresh token, detect reuse (revocation list), issue new pair. Implement `BaseAuthentication` subclass parsing `Authorization: Bearer`, verifying signature (asymmetric keys preferred), checking `exp`, `aud`, `iss`, and attaching scopes as `auth` tuple second element or on `request.auth`.

API keys: store hashed keys in DB with metadata (`scopes`, `organization_id`); authenticate by header `X-API-Key`; map to a service user or custom principal; enforce scopes in permission classes reading `request.auth`. Per-request authentication: same class can branch on path or method using `view` in `authenticate(self, request)`—for example, read-only key for GET, write key for POST.

```python
from rest_framework.authentication import BaseAuthentication
from rest_framework.exceptions import AuthenticationFailed

class ApiKeyAuthentication(BaseAuthentication):
    def authenticate(self, request):
        key = request.headers.get("X-API-Key")
        if not key:
            return None
        token = ApiKey.objects.select_related("user").filter(key_hash=hash_key(key)).first()
        if not token or not token.is_valid():
            raise AuthenticationFailed("Invalid API key")
        return (token.user, token.scopes)
```

Production: never log raw keys; rotate keys with overlap windows; bind JWT `jti` to Redis for revocation; use asymmetric signing so resource servers verify without shared secret material beyond public keys.

### Q33. Describe DRF viewset and router architecture: custom actions, nested routers, and URL namespace management.

ViewSets combine CRUD into single classes; `Router` introspects `basename` and registers list/detail routes. `@action(detail=True|False, methods=[...])` adds routes under the prefix; `url_path` and `url_name` customize URLs. `as_view({'get': 'list'})` still works without routers for custom dispatch.

```python
from rest_framework import viewsets
from rest_framework.decorators import action

class ProjectViewSet(viewsets.ModelViewSet):

        project.archived = True
        project.save()
        return Response({"status": "archived"})
```

Production: keep `get_queryset` scoped to user to avoid IDOR; use `basename` that matches model ambiguity when multiple viewsets target one model; document custom actions in OpenAPI schema via `drf-spectacular` or similar.

### Q34. What are practical API versioning strategies in DRF (URL, header, namespace, Accept-Version), and what are the trade-offs?

URL path versioning (`/api/v1/users/`) is explicit, cache-friendly, and easy to route at the edge; downside is proliferating routes. Header versioning (`API-Version: 2024-01-01`) keeps URLs stable; requires disciplined clients and gateway rules. Namespace versioning ties to Django URL namespaces—good for server-rendered docs, same trade-offs as URL. `Accept` header custom media types (`application/vnd.myapp+json; version=2`) is REST-purist but awkward for browser tools.

DRF does not mandate one approach: version negotiation classes (`DEFAULT_VERSIONING_CLASS`) like `URLPathVersioning`, `NamespaceVersioning`, `AcceptHeaderVersioning` set `request.version`. You then branch serializers/viewsets or use separate modules per version.

```python
# settings.py
REST_FRAMEWORK = {
    "DEFAULT_VERSIONING_CLASS": "rest_framework.versioning.URLPathVersioning",
    "ALLOWED_VERSIONS": ["v1", "v2"],
    "DEFAULT_VERSION": "v1",
}

# urls.py
urlpatterns = [
    path("api/<version>/users/", include("users.urls")),
]
```

Trade-offs: URL versioning is simplest for public APIs; header versioning reduces churn in hypermedia links; never break v1 silently—deprecate with sunset headers and telemetry. Maintain compatibility layers or adapters between internal models and v1 serializers.

### Q35. Explain DRF throttling internals, custom throttle classes, and distributed rate limiting with Redis.

Throttling runs after authentication; `check_throttles` iterates `DEFAULT_THROTTLE_CLASSES` and view-level `throttle_classes`. Each throttle implements `allow_request(self, request, view)` and `wait()`. Built-ins use caches (`ScopedRateThrottle` keys include `scope` + user id or IP). Custom throttles should build stable cache keys (user id preferred over IP for NAT).

```python
from rest_framework.throttling import SimpleRateThrottle

class BurstThrottle(SimpleRateThrottle):
    scope = "burst"

    def get_cache_key(self, request, view):
        if request.user and request.user.is_authenticated:
            ident = request.user.pk
        else:
            ident = self.get_ident(request)
        return self.cache_format % {"scope": self.scope, "ident": ident}
```

Distributed: set `CACHES` to Redis with atomic `INCR`/`DECR` or sliding window via Lua scripts; avoid per-process Django locmem. Align throttle rates with edge/WAF limits. For billing-tier APIs, combine throttles with quotas stored in Postgres updated asynchronously.

### Q36. How do you optimize DRF serializer performance: `SerializerMethodField` pitfalls, read vs write serializers, and prefetch strategies?

`SerializerMethodField` executes Python per row—O(n) callbacks—and hides N+1 queries if methods touch related fields. Replace with annotated fields (`Serializer(FloatField(source='avg_score')`) after `queryset.annotate()`), or `prefetch_related`/`select_related` plus ordinary fields. Split serializers: `UserReadSerializer` with nested shallow data; `UserWriteSerializer` with `PrimaryKeyRelatedField` to avoid writing nested complexity on every update.

```python
from django.db.models import Count, Prefetch
from rest_framework import serializers

class OrderSerializer(serializers.ModelSerializer):
    line_count = serializers.IntegerField(read_only=True)

    class Meta:
        model = Order
        fields = ["id", "reference", "line_count"]

    @classmethod
    def setup_eager_loading(cls, queryset):
        return queryset.annotate(line_count=Count("lines")).prefetch_related(
            Prefetch("lines", queryset=Line.objects.select_related("product"))
        )
```

Call `setup_eager_loading` from `get_queryset` in viewsets. Use `only()`/`defer()` cautiously with serializers that touch many fields—premature `defer` can cause extra queries. Cache expensive computed outputs at the queryset layer, not in serializers.

### Q39. How do you implement real-time features with Django Channels: WebSocket consumers, channel layers, and group messaging?

Channels replaces synchronous request-only Django with ASGI: `ProtocolTypeRouter` routes HTTP vs WebSocket. Consumers are async classes handling `connect`, `receive`, `disconnect`. The channel layer (Redis) backs fan-out: `async_to_sync(channel_layer.group_add)(group, channel_name)` on connect, `group_send` to push events. Auth: parse session or JWT in `connect` and reject unauthorized.

```python
from channels.generic.websocket import AsyncWebsocketConsumer

class NotifyConsumer(AsyncWebsocketConsumer):
    async def connect(self):
        self.group_name = f"user_{self.scope['user'].id}"
        await self.channel_layer.group_add(self.group_name, self.channel_name)
        await self.accept()

    async def disconnect(self, close_code):
        await self.channel_layer.group_discard(self.group_name, self.channel_name)

    async def notify(self, event):
        await self.send(text_data=event["message"])
```

Production: run Daphne/Uvicorn workers behind a WebSocket-capable load balancer with sticky sessions if needed; scale Redis channel layer; handle reconnect backoff on clients; cap message rates; validate payloads. For cross-process events (e.g., model save → push), call `group_send` from signals via `async_to_sync`.

### Q40. Compare API pagination strategies in DRF: cursor vs offset, and keyset pagination for large datasets.

Offset/limit (`PageNumberPagination`) is simple but unstable under concurrent writes (rows shift, duplicates/skips) and degrades on large offsets because databases scan skipped rows. Cursor pagination (`CursorPagination`) uses opaque ordering on indexed fields (often `created_at`, `id`), giving stable iteration forward; poor for arbitrary jumps to page N.

Keyset (seek) pagination uses `WHERE (created_at, id) > (:last_created, :last_id)` with composite index—efficient for infinite scroll feeds. DRF cursor pagination approximates this when ordering is fixed.

```python
from rest_framework.pagination import CursorPagination

class MessageCursorPagination(CursorPagination):
    page_size = 20
    ordering = "-created_at"
    cursor_query_param = "cursor"
```

Production: choose cursor/keyset for high-churn timelines; use offset only for admin grids with stable data; ensure unique, monotonic sort keys; document that `count` may be absent or approximate for huge tables; add database indexes matching `ordering`. For exports, use background tasks and files, not giant paginated API walks.

### Q61. How do you design, implement, and test custom Django management commands for production use?

Custom management commands extend `BaseCommand` (or `LabelCommand` / `AppCommand` when you need positional app labels). Production-grade commands separate argument parsing from business logic: use `add_arguments(self, parser)` with `argparse`-style options, typed choices, mutually exclusive groups, and `nargs` for repeatable inputs. Prefer `parser.add_argument("--dry-run", action="store_true")` for destructive operations and default to safe behavior when stdin is not a TTY.

For output, use `self.stdout.write()` and `self.stderr.write()` with Django’s `Style` API (`self.style.SUCCESS`, `WARNING`, `ERROR`) rather than `print()`, so commands behave correctly when stdout is piped or captured. Wrap the main work in `try/except CommandError` for user-facing failures (Django maps these to exit code 1 and suppresses tracebacks unless `--traceback` is passed). For unexpected exceptions, log with the module logger and re-raise or wrap in `CommandError` depending on whether operators need a clean message.

Testing uses `call_command()` from `django.core.management` inside tests, optionally capturing stdout/stderr with `io.StringIO`. Assert on database side effects, not just console text. For commands that hit the network or filesystem, patch at boundaries. Example:

```python
# myapp/management/commands/reindex_search.py
from django.core.management.base import BaseCommand, CommandError
from django.db import transaction

class Command(BaseCommand):
    help = "Rebuild search index for a tenant."

    def add_arguments(self, parser):
        parser.add_argument("tenant_id", type=int)
        parser.add_argument("--batch-size", type=int, default=500)
        parser.add_argument("--dry-run", action="store_true")

    def handle(self, *args, **options):
        tenant_id = options["tenant_id"]
        if tenant_id < 1:
            raise CommandError("tenant_id must be positive")
        self.stdout.write(self.style.NOTICE(f"Starting reindex for tenant {tenant_id}"))
        try:
            if options["dry_run"]:
                self.stdout.write(self.style.WARNING("Dry run — no writes"))
                return
            with transaction.atomic():
                # ... reindex work ...
                pass
        except Exception as exc:
            self.stderr.write(self.style.ERROR(str(exc)))
            raise CommandError("Reindex failed") from exc
```

```python
# tests/test_management.py
from io import StringIO
from django.core.management import call_command
from django.test import TestCase

class ReindexCommandTests(TestCase):
    def test_dry_run_does_not_write(self):
        out = StringIO()
        call_command("reindex_search", "1", dry_run=True, stdout=out)
        self.assertIn("Dry run", out.getvalue())
```

### Q63. How do you configure Django’s logging pipeline for production: `LOGGING` dict, custom handlers, structured logs, and aggregation?

Django’s logging is stdlib-based: you configure **loggers**, **handlers**, **filters**, and **formatters** in `settings.LOGGING`. In production, avoid `DEBUG`-style console spam; instead route application logs to **structured JSON** (for ELK, Datadog, CloudWatch) via a formatter that outputs `message`, `level`, `logger`, `request_id`, and `user_id` as fields. Use `django.utils.log.AdminEmailHandler` sparingly—it can email-bomb you on traffic spikes; prefer exception trackers (see Q64).

A common pattern is a root logger at `INFO`, a `django` logger for framework noise, and per-app loggers (`myapp.services.payment`) at `DEBUG` only in staging. Custom handlers might subclass `logging.Handler` to push to syslog, or use `watchtower`/`python-json-logger` for cloud sinks. Correlation IDs belong in **middleware**: generate or propagate `X-Request-ID`, stash it on `request`, and add a `logging.Filter` that injects it into `LogRecord`.

```python
# settings/production.py (excerpt)
LOGGING = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "json": {
            "()": "pythonjsonlogger.jsonlogger.JsonFormatter",
            "format": "%(levelname)s %(name)s %(message)s",
        },
    },
    "filters": {
        "request_id": {"()": "myapp.logging.RequestIdFilter"},
    },
    "handlers": {
        "stdout": {
            "class": "logging.StreamHandler",
            "formatter": "json",
            "filters": ["request_id"],
        },
    },
    "loggers": {
        "django.request": {"handlers": ["stdout"], "level": "ERROR", "propagate": False},
        "myapp": {"handlers": ["stdout"], "level": "INFO"},
    },
    "root": {"handlers": ["stdout"], "level": "INFO"},
}
```

Aggregation systems then index JSON lines; alerts query on `level:ERROR` and `transaction_id`. The trade-off is log volume versus cost—sample or rate-limit debug logs, and never log secrets or full PII in clear text.

### Q65. What strategies do you use to test database migrations safely: forward/backward runs, data migrations, and CI?

Migrations are code that runs in production; bugs can corrupt data or lock tables for minutes. **Forward testing**: on CI, spin up a disposable database, apply migrations from an empty schema or from a known `N-1` dump, then run the app’s test suite or smoke imports. **Backward testing**: run `migrate app_label migration_name` to the previous node and assert the schema matches expectations—especially critical before destructive operations.

**Data migrations** deserve unit-style tests: create minimal rows in `setUp`, run `MigrationExecutor` or call the migration’s `RunPython` functions in isolation, and assert invariants (`assert Model.objects.filter(...).count() == expected`). For zero-downtime concerns, split additive migrations (add nullable column) from backfills (data migration) from constraint additions (separate deploy).

```python
from django.db import connection
from django.db.migrations.executor import MigrationExecutor
from django.test import TransactionTestCase

class OrderMigrationTests(TransactionTestCase):
    def test_0012_adds_column_and_rolls_back_cleanly(self):
        executor = MigrationExecutor(connection)
        app_label, migration_name = "orders", "0012_order_external_ref"
        executor.migrate([(app_label, migration_name)])
        with connection.cursor() as cursor:
            cursor.execute(
                "SELECT column_name FROM information_schema.columns "
                "WHERE table_name=%s AND column_name=%s",
                ["orders_order", "external_ref"],
            )
            self.assertIsNotNone(cursor.fetchone())
        executor.migrate([(app_label, "0011_previous")])
        # Assert column absent or follow expand/contract expectations for rollback
```

In large teams, **migration lint** tools (e.g., `django-migration-linter`) block dangerous operations (`ALTER TABLE` with full rewrites on huge MySQL tables). The trade-off is CI time: parallelize DB jobs or use containers with volume snapshots for faster replay.

```bash
pip-audit
python manage.py check --deploy
```

Deeper tools: **Semgrep** rules for Django/DRF, **Snyk** for container images. The trade-off is false positives—tune severities and document exceptions in-repo. Security is not a single scan; it is **threat modeling** plus **defense in depth** (CSP, CSRF, ORM over raw SQL).

### Q71. Describe a production Django deployment topology: Gunicorn/uWSGI, Nginx, and process supervision.

A typical production stack places **Nginx** at the edge as a reverse proxy: TLS termination, HTTP/2, buffering, rate limits, and static file serving (or offload to CDN). Nginx forwards to **Gunicorn** (or uWSGI) Unix sockets or TCP ports. Gunicorn runs multiple **worker processes** (`workers = (2 * CPU) + 1` is a starting heuristic) each loading your Django WSGI app; workers are restarted to mitigate memory leaks (`max_requests`).

**uWSGI** offers more knobs (routing, caching, emperor mode) but more complexity; Gunicorn is simpler and common for pure WSGI. **ASGI** deployments (Channels, async views) use **Uvicorn/Daphne/Hypercorn** behind Nginx, often with separate worker counts for sync vs async workloads.

Process management uses **systemd** unit files or **supervisor** to restart crashed workers, set `EnvironmentFile`, and enforce `LimitNOFILE`. Health checks hit Nginx → app `/healthz` (see Q76). Trade-offs: too few workers → request queuing; too many → CPU thrash and DB connection exhaustion—tune with load tests and DB **connection pooling** (PgBouncer).

```ini
# gunicorn.conf.py
bind = "unix:/run/gunicorn.sock"
workers = 5
worker_class = "sync"
timeout = 60
graceful_timeout = 30
max_requests = 1000
max_requests_jitter = 50
```

### Q72. How do you Dockerize Django for production: multi-stage builds, health checks, configuration, and secrets?

**Multi-stage builds** keep final images small: a `builder` stage installs compilers and `pip install`, a `runtime` stage copies only wheels/site-packages and app code. Use **non-root** users, read-only filesystem where possible, and **distroless** or slim base images after auditing libc compatibility.

**Health checks** in Compose/Kubernetes should hit a lightweight endpoint that validates process liveness; readiness should include DB/cache (see Q76). **Configuration** injects via environment variables (12-factor); never bake secrets into images—use **Docker secrets**, Kubernetes secrets, or cloud parameter stores mounted at runtime.

```dockerfile
FROM python:3.12-slim AS builder
WORKDIR /app
RUN pip install poetry && poetry export -f requirements.txt -o requirements.txt --without-hashes
RUN pip wheel --no-cache-dir -r requirements.txt -w /wheels

FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /wheels /wheels
RUN pip install --no-cache /wheels/*
COPY . .
USER appuser
CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000"]
```

**Secrets**: prefer runtime injection (`DATABASE_URL` from vault) over `.env` in images. Scan images with Trivy in CI.

### Q73. How do you manage Django settings across environments with django-environ, split modules, and 12-factor compliance?

**12-factor** settings treat the codebase as one artifact configured by environment. Split `settings/` into `base.py`, `local.py`, `staging.py`, `production.py`, and select via `DJANGO_SETTINGS_MODULE`. **django-environ** parses typed values (`env.int()`, `env.db()`, `env.cache_url()`) and fails fast on missing required vars.

Avoid `from .production import *` anti-patterns that hide overrides; prefer explicit imports or small composition functions. **Feature parity**: staging should mirror production flags (DEBUG off, same middleware) to catch issues early.

```python
# settings/base.py
import environ
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent
env = environ.Env(
    DJANGO_DEBUG=(bool, False),
    DATABASE_URL=(str, f"sqlite:///{BASE_DIR}/db.sqlite3"),
)
environ.Env.read_env(BASE_DIR / ".env", overwrite=True)

SECRET_KEY = env("DJANGO_SECRET_KEY")
DEBUG = env("DJANGO_DEBUG")
DATABASES = {"default": env.db()}
```

Document every env var in `README` or generated docs. Trade-off: strict parsing breaks sloppy local setups—provide `.env.example` with dummy values.

### Q74. How do you plan zero-downtime Django deployments with migrations, blue-green, and rolling updates?

Zero-downtime requires **backward-compatible schema changes** across at least two deploy steps: **expand** (add nullable column), **migrate data**, **contract** (add constraints, remove old column)—often spanning multiple releases. Running **destructive migrations** during traffic risks locks; use `CONCURRENTLY` indexes on PostgreSQL via `SeparateDatabaseAndState` or raw SQL migrations scheduled off-peak.

**Blue-green**: two full stacks; switch load balancer when green is healthy; instant rollback by switching back. **Rolling updates** (Kubernetes) replace pods gradually—ensure old and new code coexist during the roll; hence **compatible migrations first** is non-negotiable.

Combine with **maintenance windows** only when unavoidable (e.g., major table rewrites). Senior teams track **migration duration** in staging with production-sized data and use **statement timeouts** to fail fast rather than stall the site.

Deployment **ordering** matters: run migrations in a controlled job or one-off command **before** scaling up new application versions that depend on new columns, or ensure new code tolerates both old and new schema during the overlap window. For long-running data backfills, offload to **Celery** or batch workers with checkpoints rather than blocking `migrate`. If you use **database routers** or read replicas, verify migrations apply to the writer and replication lag does not expose readers to partially migrated shapes.

### Q75. How do you run Django on Kubernetes: Deployments, probes, HPA, and migration jobs?

Package the app as a container with **Deployments** controlling ReplicaSets. **Liveness** probes restart wedged workers; **readiness** probes remove pods from Service endpoints until dependencies are ready—avoid probing deep DB work every few seconds on every pod (cache results briefly or split `/live` vs `/ready`).

**Horizontal Pod Autoscaler** scales on CPU/memory or custom metrics (request rate, queue depth). **Init containers** can wait for DB DNS or run one-off setup, but **schema migrations** are safer as **Kubernetes Jobs** triggered in CI/CD before switching traffic, with backoff and logs—avoid concurrent migration runners.

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: web
          image: myregistry/django:abc123
          ports: [{ containerPort: 8000 }]
          readinessProbe:
            httpGet: { path: /readyz, port: 8000 }
            initialDelaySeconds: 10
          livenessProbe:
            httpGet: { path: /livez, port: 8000 }
```

Use **PodDisruptionBudgets** during node drains. Resource requests/limits prevent noisy neighbors; align Gunicorn workers with pod CPU.

### Q76. How do you implement `/healthz`, `/livez`, and `/readyz` in Django for orchestrators and load balancers?

Split endpoints: **liveness** answers “should this process be killed?”—cheap checks like “Python import works, threads not deadlocked.” **Readiness** answers “should traffic be sent here?”—database `SELECT 1`, cache `PING`, critical external APIs with short timeouts. **Startup** probes (Kubernetes 1.16+) cover slow Django initialization.

Implement as thin views or ASGI/WSGI middleware-free routes to minimize middleware side effects. Use **connection pool** health: catch `OperationalError` and return 503. Avoid logging every probe at INFO—sample or DEBUG.

```python
from django.http import JsonResponse
from django.db import connection
from django.core.cache import cache

def livez(request):
    return JsonResponse({"status": "ok"})

def readyz(request):
    try:
        with connection.cursor() as c:
            c.execute("SELECT 1")
        cache.set("healthcheck", "1", timeout=5)
        assert cache.get("healthcheck") == "1"
    except Exception as exc:
        return JsonResponse({"status": "unready", "error": str(exc)}, status=503)
    return JsonResponse({"status": "ready"})
```

Document SLAs: readiness failures should **not** page on transient blips unless sustained—pair with **SRE** burn rates.

### Q77. Explain `collectstatic`, production static hosting, WhiteNoise, CDNs, and object storage backends.

`collectstatic` gathers files from `STATICFILES_DIRS` and app `static/` into `STATIC_ROOT`. In production, **never** serve user uploads from `collectstatic`; use `MEDIA_ROOT` with object storage. **WhiteNoise** serves compressed static files from the same Gunicorn process—simple for small/medium sites; add far-future cache headers and `manifest_strict` when using `ManifestStaticFilesStorage` for cache-busted filenames.

**S3/GCS** backends (`django-storages`) offload static to buckets fronted by **CloudFront/CDN** for global latency and bandwidth savings. IAM roles grant bucket access without static keys in settings. **Cache invalidation** uses versioned filenames via manifest storage or CDN purge on deploy.

```python
STATIC_URL = "https://cdn.example.com/static/"
STATICFILES_STORAGE = "django.contrib.staticfiles.storage.ManifestStaticFilesStorage"
# or django-storages S3Boto3Storage with custom domain
```

Trade-off: WhiteNoise reduces moving parts; S3+CDN scales better for large bundles and global users.

CI/CD should run `collectstatic` during the **build** or release phase (not on every pod start) and upload artifacts to the bucket or bake them into the image layer used by Gunicorn when using WhiteNoise. Misconfigured `STATIC_URL`/`MEDIA_URL` in split environments is a common production bug—validate with integration tests that render templates and assert asset URLs resolve. For hashed filenames, ensure **manifest** mismatches fail fast in deployment rather than at user runtime.

### Q78. How do you approach observability for Django: ELK, Prometheus metrics, and OpenTelemetry tracing?

**Logs**: ship JSON logs (Q63) to Elasticsearch/OpenSearch via Filebeat/Fluent Bit; dashboards in Kibana/Grafana Loki. Correlate with trace IDs. **Metrics**: expose Prometheus endpoints with `django-prometheus` or OpenTelemetry metrics—track request latency histograms, DB pool usage, queue depth, Celery task rates.

**Tracing**: OpenTelemetry SDK instruments WSGI/ASGI, Django, Redis, psycopg; export OTLP to Jaeger/Tempo/Honeycomb. Sample production traffic (e.g., 1–10%) to control cost. Tie **RED** metrics (Rate, Errors, Duration) to SLOs.

```python
from prometheus_client import Counter, Histogram
REQUEST_LATENCY = Histogram("http_request_latency_seconds", "Latency", ["method", "path"])
REQUEST_COUNT = Counter("http_requests_total", "Requests", ["method", "status"])
```

Avoid **cardinality explosion**: do not label Prometheus metrics with unbounded `user_id`. Use exemplars linking metrics to traces where supported.

### Q79. How do you implement blue-green vs canary deployments with traffic splitting, flags, and rollback?

**Blue-green** switches 100% traffic between environments; **canary** gradually shifts percentage (5% → 25% → 100%) while monitoring error rates and latency. Implement canaries at **load balancer** (weighted pools), **service mesh** (Istio/Linkerd), or **API gateway** traffic rules. **Feature flags** (Q69) canary **code paths** inside one version—different from infra canary but combinable (deploy safe code, then enable behavior).

Rollback strategies: **instant** revert of traffic split; **database** rollback is often impossible—invest in **forward-fix** migrations and backups. **Kill switches** disable features without redeploying. Observability gates promotion: if 5-minute error budget burns, auto-revert canary.

Document **RTO/RPO** for rollback drills. Senior teams run **game days** simulating partial failures during canaries.

**Automation** can promote canaries when SLIs stay green: error rate below baseline, p95 latency within budget, saturation stable. Tie releases to **deployment markers** in APM (Q64, Q78) so regressions map to the exact traffic slice. When databases are shared across blue and green, schema must remain compatible for both—again favor expand/contract (Q80) over big-bang DDL with each color swap.

### Q80. What database migration strategies support zero-downtime schema evolution (add columns, renames, backfills)?

**Adding columns**: add **nullable** or **default-backed** columns without locking full table rewrites when possible; PostgreSQL 11+ can add columns with defaults without rewriting in some cases—verify per version. **Renaming** columns breaks running code—use multi-step: add new column, dual-write, backfill, switch reads, remove old column.

**Renaming in Django**: `RenameField` is a single operation; in production, prefer **AddField + data migration + gradual cutover + RemoveField** across releases. **Indexes**: create **concurrently** on PostgreSQL to avoid long writes locks.

**Data migrations**: batch updates (`UPDATE ... WHERE id BETWEEN ...`) to avoid single huge transactions; throttle for replication lag. **Expand–contract** pattern is the mental model: expand schema compatibility, deploy code that handles both, migrate data, deploy code that uses new only, contract schema.

```python
from django.db import migrations, models

class Migration(migrations.Migration):
    operations = [
        migrations.SeparateDatabaseAndState(
            state_operations=[
                migrations.RenameField("Order", "old_name", "new_name"),
            ],
            database_operations=[
                # Custom SQL: add new column, copy, then swap — if needed
            ],
        ),
    ]
```

Coordinate with **DBA** for lock timeouts, `lock_timeout`, and **autovacuum** on high-churn tables. The hardest part is not Django syntax—it is **ordering operations** across app versions so old workers never crash on new schema assumptions.

On **MySQL**, `ALGORITHM=INPLACE` and `LOCK=NONE` metadata matter for large InnoDB tables; on **PostgreSQL**, prefer additive DDL and concurrent index builds. Django’s migration autodetector does not know your table sizes—operators must override operations with `RunSQL` or split migrations when the framework-generated SQL is unsafe at scale. Always rehearse on a restored production snapshot: measure lock duration, rows examined, and replica lag during the change window.
