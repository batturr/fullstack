## Q1. What is Flask?

**Answer:**

Flask is a lightweight, open-source web framework for Python that helps you build web applications and APIs with minimal boilerplate. It follows the WSGI standard and is often described as a micro framework because its core is small and extensible rather than including every feature by default. You add functionality through extensions for databases, forms, authentication, and more as your project grows. Flask provides routing, request and response handling, templating via Jinja2, and development tools while leaving architectural decisions largely to you. It is widely used for REST APIs, small to medium web apps, and prototypes because it is easy to learn and flexible. The framework is maintained by the Pallets project and has a large ecosystem of third-party packages.

```python
from flask import Flask

app = Flask(__name__)

@app.route("/")
def home():
    return "Hello, Flask!"
```

---

## Q2. Why is Flask called a micro framework?

**Answer:**

Flask is called a micro framework because its core intentionally ships with only the essentials needed to handle HTTP requests, routing, and responses, rather than bundling an ORM, admin panel, authentication system, and other heavy components out of the box. The term micro refers to the size and scope of the core library, not to limitations on what you can build; large production applications are common when extensions and patterns like application factories and blueprints are used. This design keeps the learning curve gentle and avoids forcing a single stack on every project. Developers choose and plug in only the pieces they need, which can reduce complexity for small apps and improve flexibility for larger ones. In practice, a full Flask app often includes many extensions, so the final deployed system is not necessarily small.

## Q3. What are the main features of Flask?

**Answer:**

Flask provides a built-in development server and debugger for local work, though these are not intended for production use. It includes a flexible URL routing system with variable rules, HTTP method handling, and URL building through `url_for`. Template rendering is handled by Jinja2, which supports inheritance, filters, and macros for reusable UI pieces. The framework exposes convenient access to incoming data through the `request` object and helpers for cookies, sessions, redirects, and JSON responses. Static file serving is supported from configurable folders, and you can extend Flask with blueprints for modular applications. Together these features cover most web development needs while keeping the core API small and understandable.

## Q4. How do you install Flask?

**Answer:**

Flask is distributed on PyPI and is typically installed into a virtual environment so project dependencies stay isolated from your system Python. You create a virtual environment with `python -m venv .venv`, activate it, then run `pip install flask` to install the latest stable release. For reproducible deployments, teams often pin versions in a `requirements.txt` file generated with `pip freeze` or written by hand. Installing inside a venv avoids permission issues and version conflicts between projects. You verify the install by running `python -c "import flask; print(flask.__version__)"` or starting a minimal app.

```bash
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install flask
```

## Q5. What is WSGI and how does Flask use it?

**Answer:**

WSGI, the Web Server Gateway Interface, is a Python standard (PEP 3333) that defines how a web server passes HTTP requests to a Python application and how the application returns responses. It acts as a contract between servers like Gunicorn, uWSGI, or Waitress and your Python code, so you can swap servers without rewriting your app. Flask builds a WSGI application object (`app`) that servers call with an environ dict and a start_response callable. When you run `flask run` or use a production WSGI server, the server invokes that callable for each request. Understanding WSGI helps when configuring deployment, middleware, or ASGI alternatives for async workloads.

```python
# Conceptually, your Flask app is WSGI-callable
from flask import Flask
app = Flask(__name__)

# Production: gunicorn module:app
```

## Q6. What is Jinja2 in Flask?

**Answer:**

Jinja2 is a fast, designer-friendly templating language for Python that Flask uses by default to generate HTML (or any text) by combining static structure with dynamic data. Templates live in a `templates` folder and can use variables, control structures like loops and conditionals, template inheritance, and filters to format values safely. It auto-escapes HTML by default when rendering, which reduces cross-site scripting risks when displaying user content. Flask integrates Jinja2 through `render_template()` and allows you to register custom filters and global template functions. Separating presentation from Python logic keeps views cleaner and easier for front-end collaborators to edit.

```python
from flask import render_template

@app.route("/user/<name>")
def user(name):
    return render_template("user.html", name=name)
```

```html
<!-- templates/user.html -->
<h1>Hello, {{ name }}!</h1>
```

## Q7. What is Werkzeug in Flask?

**Answer:**

Werkzeug is a WSGI utility library that Flask depends on for low-level HTTP primitives such as request and response objects, URL routing, and debugging helpers. It provides data structures that parse query strings, form bodies, file uploads, and headers into easy-to-use Python objects. Werkzeug also includes development tools like the interactive debugger that appears when unhandled exceptions occur in debug mode. Flask composes these pieces into a higher-level API so you rarely import Werkzeug directly in simple apps, but it is there under the hood. Knowing Werkzeug helps when you need custom middleware, request subclasses, or advanced routing behavior.

## Q8. Write a minimal Flask application.

**Answer:**

A minimal Flask program creates an application instance, registers a route that maps a URL path to a Python function, and runs the development server when executed as the main module. The route function returns a string or response object that becomes the HTTP body. You should set `FLASK_APP` when using the CLI, or call `app.run()` for quick tests. Below is a complete, runnable example you can save as `app.py` and run with `python app.py` after installing Flask. This pattern is the starting point for almost every Flask project.

```python
from flask import Flask

app = Flask(__name__)

@app.route("/")
def index():
    return "<h1>Hello, World!</h1>"

if __name__ == "__main__":
    app.run(debug=True)
```

## Q9. What is the difference between Flask and Django?

**Answer:**

Django is a full-stack batteries-included framework with an ORM, admin site, authentication, migrations, and many conventions enforced by default, while Flask is minimal and expects you to choose extensions and structure yourself. Django uses its own project and app layout, URL configuration, and template language; Flask uses Jinja2 and flexible blueprints with less prescribed folder structure. Django suits large teams that want consistency and many built-in features; Flask suits developers who want fine-grained control, smaller learning surface, or API-first services. Both are mature and production-ready; the choice depends on team preference, project size, and how much convention you want. Many companies use Flask for microservices and Django for large content-driven sites.

## Q11. What is routing in Flask?

**Answer:**

Routing is the mechanism that maps incoming URL paths (and optionally HTTP methods) to Python view functions that handle the request and produce a response. When a client requests a URL, Flask compares the path against registered routes and dispatches to the first matching handler. Routes can include static segments and dynamic segments using variable rules and converters. Clear routing keeps URLs readable and helps organize application logic by feature or resource. Flask builds on Werkzeug's routing system, which supports ordering, defaults, and subdomain rules for more advanced setups.

```python
@app.route("/posts/<int:post_id>")
def show_post(post_id):
    return f"Post {post_id}"
```

## Q12. How do you define routes in Flask?

**Answer:**

You define routes by applying the `@app.route()` decorator to a view function, passing the URL rule as the first argument. You can register multiple decorators on the same function to bind several paths to one handler. For larger apps, the same pattern applies with `@blueprint.route()` on a `Blueprint` object before registering the blueprint on the app. The function name is used by `url_for` unless you pass `endpoint=` explicitly. Routes are registered when the module is imported, so import order and circular imports matter in complex projects.

```python
@app.route("/hello")
@app.route("/hi")
def greet():
    return "Hello!"
```

## Q13. What are variable rules in Flask routing?

**Answer:**

Variable rules let you capture parts of the URL as arguments to your view function, so one route can handle many dynamic values like user IDs or slugs. You declare them with angle brackets in the path, for example `/user/<username>`, and Flask passes the captured segment into the view. You can combine static and variable segments in one rule. Variable rules reduce duplication compared to parsing paths manually inside a single catch-all route. They also integrate with type converters for validation and automatic 404s on invalid formats.

```python
@app.route("/item/<name>")
def item(name):
    return f"Item: {name}"
```

## Q14. What are the different URL converters available in Flask?

**Answer:**

Flask supports built-in converters that parse and validate variable path segments before your view runs. The `string` converter (default) accepts any text except slashes; `int` accepts integers; `float` accepts floating-point numbers; `path` accepts slashes so you can match nested paths; and `uuid` accepts UUID strings. Custom converters can be registered on the URL map for domain-specific types. If conversion fails, Flask responds with 404, which gives you validation for free at the routing layer.

```python
@app.route("/users/<int:user_id>/posts/<int:post_id>")
def user_post(user_id, post_id):
    return f"User {user_id}, Post {post_id}"
```

## Q15. How do you handle multiple URLs for the same function?

**Answer:**

You stack multiple `@app.route()` decorators on a single view function, each with a different path (and optionally different methods). Flask registers each rule separately but they all invoke the same callable, which keeps shared logic in one place. Alternatively, you can register routes programmatically with `app.add_url_rule()` if you prefer not to use repeated decorators. This pattern is useful for legacy URL support, localized paths, or synonyms like `/login` and `/signin`. Ensure each rule is distinct to avoid ambiguous matches.

```python
@app.route("/about")
@app.route("/about-us")
def about():
    return "About page"
```

## Q16. What is url_for() in Flask?

**Answer:**

`url_for()` builds URLs for your application from endpoint names and optional path parameters instead of hardcoding strings in templates and redirects. It adapts automatically if you change URL prefixes, use blueprints, or run behind a reverse proxy with `APPLICATION_ROOT`. Passing `_external=True` generates absolute URLs including scheme and host, which is useful for links in emails or OAuth callbacks. Using `url_for` reduces broken links when refactoring routes. The endpoint defaults to the function name unless you set `endpoint=` on the route.

```python
from flask import url_for, redirect

@app.route("/profile/<username>")
def profile(username):
    return username

# Somewhere else:
redirect(url_for("profile", username="alice"))
```

## Q17. What is the difference between GET and POST methods in Flask?

**Answer:**

GET requests are intended to retrieve resources and should be idempotent; browsers may cache them and append parameters in the query string, which is visible in URLs and server logs. POST requests send data in the body and are used for actions that change server state, such as form submissions or file uploads; they are not assumed idempotent and should not be blindly retried without care. In Flask, `request.method` tells you which verb was used, and you restrict routes with `methods=["GET", "POST"]` as needed. Forms that modify data should use POST to avoid accidental resubmission via refresh. APIs often use additional verbs like PUT and DELETE for RESTful operations.

```python
from flask import Flask, request

app = Flask(__name__)

@app.route("/submit", methods=["GET", "POST"])
def submit():
    if request.method == "POST":
        return "Handled POST"
    return "Show form"
```

## Q18. How do you specify HTTP methods for a route?

**Answer:**

By default, a Flask route only accepts GET (and HEAD/OPTIONS added automatically in many cases); to accept others, pass `methods=[...]` to `@app.route()`. List every verb the same view should handle, such as `["GET", "POST"]` for a form page that both displays and processes input. You can branch inside the view using `request.method` or split logic into helpers for clarity. Specifying methods correctly prevents unintended access, for example blocking POST on a read-only endpoint. For APIs, you might register separate view functions per method on the same path using multiple routes or class-based resources.

```python
from flask import Flask, request

app = Flask(__name__)

@app.route("/api/item", methods=["GET", "POST", "PUT"])
def item_api():
    return request.method
```

## Q19. What happens when you access a URL that doesn't exist in Flask?

**Answer:**

If no registered route matches the requested path and method, Flask raises a `NotFound` exception (HTTP 404). In production, Flask renders a default simple 404 page unless you register a custom error handler with `@app.errorhandler(404)`. The development server may show a traceback-style page in debug mode for unhandled errors, but 404 is usually handled cleanly. Client applications should treat 404 as the resource being missing rather than a server failure. Custom handlers improve user experience and let you log missing routes for monitoring.

```python
@app.errorhandler(404)
def not_found(e):
    return "Page not found", 404
```

## Q20. What is URL building in Flask?

**Answer:**

URL building is the process of generating correct internal links and redirect targets from your application's routing table rather than concatenating strings manually. Flask's `url_for` consults registered endpoints and fills in path parameters and optional query arguments. It respects `SCRIPT_NAME` and blueprint URL prefixes when your app is mounted under a subpath. External URL building includes the public hostname, which matters for multi-tenant or CDN setups. Consistent URL building keeps applications maintainable when routes evolve.

```python
from flask import url_for

with app.test_request_context():
    u = url_for("profile", username="bob", _external=True)
```

## Q21. What is template rendering in Flask?

**Answer:**

Template rendering combines a Jinja2 template file with a context dictionary of variables to produce a final string, usually HTML, sent to the client. Flask looks for templates relative to the `templates` directory by default. Rendering happens server-side on each request (unless you cache output), so dynamic data stays fresh. The process separates presentation from controller logic, which improves readability and testing. You can render templates from view functions using `render_template()` or `render_template_string()` for small snippets.

## Q22. How do you render a template in Flask?

**Answer:**

Import `render_template` from `flask`, pass the template filename and keyword arguments for variables, and return the result from your route. Flask resolves paths under the configured template folder and raises `TemplateNotFound` if the file is missing. For AJAX or partial HTML, you might render a fragment template and return it as a string. Template names use forward slashes for subfolders, such as `admin/dashboard.html`. Always keep user-controlled data escaped unless you explicitly trust it.

```python
from flask import render_template

@app.route("/dashboard")
def dashboard():
    return render_template("dashboard.html", title="Home", items=[1, 2, 3])
```

## Q23. What is the Jinja2 template engine?

**Answer:**

Jinja2 is a text-based templating engine that compiles templates to Python bytecode for performance and supports inheritance, macros, includes, and filters. It is sandboxed to limit arbitrary code execution while still offering rich logic for views. Flask configures Jinja2 automatically and shares the application context where appropriate. Designers can work in HTML with placeholders without editing Python files. Custom filters and tests extend the language for formatting dates, currencies, or markdown.

```html
{% for item in items %}
  <li>{{ item }}</li>
{% endfor %}
```

## Q24. How do you pass variables to templates in Flask?

**Answer:**

You pass variables as keyword arguments to `render_template`, and they become names in the template context. Dictionaries can be unpacked with `**data` if you already built a context dict in your view. Avoid passing entire large objects unless needed; pass only fields required by the template to keep responsibilities clear. For shared values across many templates, consider context processors that inject globals like the current user. Sensitive data should never be placed in templates intended for caching on shared proxies unless appropriate controls exist.

```python
return render_template("page.html", user=user, settings=settings)
```

## Q25. What are template filters in Jinja2?

**Answer:**

Filters transform values inside `{{ }}` expressions using the pipe syntax, for example `{{ name|title }}` to capitalize words. Built-in filters include `default`, `length`, `join`, `round`, and `safe`. You can register custom filters on the Jinja environment via `@app.template_filter()` in Flask. Filters keep formatting logic out of views and reusable across templates. Chaining filters applies them left to right: `{{ text|trim|upper }}`.

```html
<p>{{ price|round(2) }}</p>
```

```python
@app.template_filter("currency")
def currency_filter(value):
    return f"${value:.2f}"
```

## Q26. What are template inheritance and blocks in Jinja2?

**Answer:**

Template inheritance lets you define a base layout with placeholders (`{% block %}`) that child templates override or extend with `{% extends "base.html" %}`. Blocks reduce duplication of headers, footers, and asset includes across pages. Child templates can call `{{ super() }}` inside a block to include parent content. This pattern mirrors object-oriented design and keeps branding consistent. Multiple blocks per page allow separate areas for title, content, and scripts.

```html
<!-- base.html -->
{% block head %}{% endblock %}
{% block body %}{% endblock %}
```

```html
<!-- child.html -->
{% extends "base.html" %}
{% block body %}<h1>Hi</h1>{% endblock %}
```

## Q27. How do you use if-else statements in Jinja2 templates?

**Answer:**

Jinja2 supports `{% if %}`, `{% elif %}`, and `{% endif %}` for branching inside templates, similar to Python but with slightly different syntax for tests. Conditions can use logical operators `and`, `or`, `not`, and comparisons like `==`, `!=`, `>`. Keep complex business rules in Python and pass simple booleans or enums to templates to avoid bloated templates. For optional sections, combine `if` with `defined` tests when variables may be missing. Always close control structures explicitly with `endif`.

```html
{% if user.is_authenticated %}
  <p>Welcome, {{ user.name }}</p>
{% else %}
  <a href="/login">Login</a>
{% endif %}
```

## Q28. How do you use loops in Jinja2 templates?

**Answer:**

Use `{% for item in sequence %} ... {% endfor %}` to iterate lists, tuples, or query results inside templates. The loop variable exposes `index`, `index0`, `first`, `last`, and `length` for styling stripes or separators. Use `{% else %}` on a for loop to render content when the sequence is empty. Avoid heavy computation in templates; prepare data in the view instead. For large datasets, paginate in the backend rather than rendering thousands of rows.

```html
<ul>
{% for name in names %}
  <li>{{ loop.index }}. {{ name }}</li>
{% else %}
  <li>No names</li>
{% endfor %}
</ul>
```

## Q29. What is the `safe` filter in Jinja2?

**Answer:**

The `safe` filter marks a string as safe HTML so Jinja2 will not escape it when output, which is necessary when you intentionally render trusted HTML such as rich text from your CMS. Using `safe` on untrusted user input creates cross-site scripting vulnerabilities, so it must never be applied to raw user content. Prefer storing markdown and converting server-side with a sanitizer library when you need formatting. If you must combine user text with markup, escape first, then mark specific fragments safe. Treat `safe` as an explicit security decision every time you use it.

```html
{{ trusted_html|safe }}
```

## Q30. How do you include one template inside another?

**Answer:**

The `{% include "partial.html" %}` directive renders another template at that point in the current template, sharing the current context unless you pass a limited context with `with context` or explicit variables. Includes are ideal for navbars, footers, and reusable form fragments. Unlike inheritance, includes do not override blocks; they are simple composition. You can include inside loops to render repeated partials. Keep included templates small and focused for readability.

```html
{% include "header.html" %}
<main>...</main>
{% include "footer.html" %}
```

## Q31. What is the request object in Flask?

**Answer:**

The `request` object is a thread-local proxy provided by Flask that represents the current HTTP request during handling. It exposes the URL, method, headers, cookies, query args, form fields, JSON body, and uploaded files. You import `request` from `flask` and access attributes inside view functions or hooks; it is only valid while handling a request. Understanding `request` is essential for building forms, APIs, and authentication checks. It is implemented on top of Werkzeug's `Request` class with Flask-specific integrations.

```python
from flask import request

@app.route("/search")
def search():
    q = request.args.get("q", "")
    return f"You searched for {q}"
```

## Q32. How do you access form data in Flask?

**Answer:**

For HTML forms submitted with `application/x-www-form-urlencoded` or `multipart/form-data`, use `request.form`, which behaves like an immutable multi-dict. Access single values with `request.form["key"]` or safely with `request.form.get("key", default)`. For multiple values under one name, use `request.form.getlist("items")`. File inputs appear in `request.files` rather than `request.form`. Always validate and sanitize form data on the server; never trust the client.

```python
from flask import request

username = request.form.get("username", "").strip()
```

## Q33. How do you access query parameters in Flask?

**Answer:**

Query string parameters appear in `request.args` for GET requests and can also be present on other methods if clients append a query string. Use `request.args.get("page", "1")` to supply defaults and avoid KeyError. For repeated keys, use `getlist`. Remember that query parameters are visible in URLs and logs, so avoid putting secrets there. Pagination and filtering are common use cases for query args.

```python
from flask import request

page = int(request.args.get("page", 1))
```

## Q34. How do you access JSON data from a request?

**Answer:**

For requests with `Content-Type: application/json`, call `request.get_json(silent=True)` or `request.json` (which uses `get_json` internally) to parse the body into Python dicts and lists. Use `silent=True` to avoid raising on invalid JSON, then check for `None`. For APIs, validate the structure with a schema library or manual checks before using values. Large payloads should be size-limited at the server or reverse proxy to prevent abuse.

```python
from flask import request

data = request.get_json(silent=True) or {}
name = data.get("name")
```

## Q35. What is the response object in Flask?

**Answer:**

In Flask, a route can return a string, tuple of `(body, status, headers)`, or a `Response` object; Flask normalizes these into a full HTTP response. The `Response` class from `werkzeug.wrappers` lets you set status, headers, and iterable body explicitly. Using `make_response()` is a convenient way to build or augment responses, for example attaching cookies. Response objects support mutating headers and setting mimetype. For fine-grained streaming, you can return a generator with appropriate headers.

```python
from flask import make_response

resp = make_response("OK", 201)
resp.headers["X-Custom"] = "1"
return resp
```

## Q36. How do you set response headers in Flask?

**Answer:**

After creating a response with `make_response()` or by capturing the return value of `render_template` in `make_response`, assign to `response.headers['Name']`. Alternatively return a tuple `(body, status, {"Header": "value"})` from the view. Headers control caching, security policies, content types, and custom metadata. Some headers have specific helper methods or Werkzeug utilities for cookies. Ensure header names and values conform to HTTP standards.

```python
from flask import make_response, jsonify

@app.route("/data")
def data():
    r = make_response(jsonify(ok=True))
    r.headers["Cache-Control"] = "no-store"
    return r
```

## Q37. How do you set cookies in Flask?

**Answer:**

Use `response.set_cookie(key, value, max_age=..., httponly=True, secure=True, samesite="Lax")` on a `Response` object to send `Set-Cookie` headers to the browser. Sensitive tokens should use `httponly` to reduce JavaScript access and `secure` in HTTPS environments. `max_age` or `expires` controls lifetime; omitting both makes a session cookie that disappears when the browser closes. Remember that cookies are sent on every matching request, so keep payloads small. Combine cookies with server-side sessions for larger or sensitive data.

```python
from flask import make_response

r = make_response("logged in")
r.set_cookie("session_id", "abc123", httponly=True, samesite="Lax")
return r
```

## Q38. How do you read cookies in Flask?

**Answer:**

Incoming cookies are available on `request.cookies`, which behaves like a dictionary interface. Use `request.cookies.get("name")` to avoid exceptions when a cookie is absent. Validate and verify cookie values because clients can tamper with them; signed cookies or server-side sessions mitigate this. For authentication, prefer frameworks like Flask-Login that manage secure session handling. Never trust raw cookie data as authorization without cryptographic verification.

```python
theme = request.cookies.get("theme", "light")
```

## Q39. What is the `make_response()` function?

**Answer:**

`make_response()` constructs a `Response` object from various convenient inputs such as strings, tuples, or existing responses, letting you modify headers, status codes, and cookies before returning. It is useful when you start from `render_template` or `jsonify` and need to attach cookies or custom headers. You can also wrap WSGI callables or file-like objects depending on configuration. Using `make_response` clarifies intent compared to manually instantiating `Response` for common cases. It integrates cleanly with Flask's return value processing.

```python
from flask import make_response, render_template

r = make_response(render_template("page.html"))
r.set_cookie("seen", "1")
return r
```

## Q40. How do you return JSON responses in Flask?

**Answer:**

Use `jsonify()` from Flask to serialize Python dicts and lists to JSON with the correct `Content-Type` and a UTF-8 encoded body. `jsonify` handles basic types and returns a `Response` with status 200 by default; pass `status=` or return tuples for other codes. For custom serialization, build a string and set `mimetype="application/json"` or use Flask's `json` provider hooks. JSON responses are standard for APIs consumed by SPAs and mobile apps. Ensure you do not double-encode already serialized strings.

```python
from flask import jsonify

@app.route("/api/user")
def api_user():
    return jsonify({"id": 1, "name": "Ada"})
```

## Q41. How do you serve static files in Flask?

**Answer:**

Place files under the `static` folder next your application module and reference them with `url_for('static', filename='css/style.css')` in templates. Flask serves them automatically in development from `/static/` by default. In production, a reverse proxy or CDN usually serves static assets for performance, but Flask can still do it if configured. You can change the static folder and URL path via `Flask(__name__, static_folder='public', static_url_path='/assets')`. Use cache-busting query strings or hashed filenames for long-term caching.

```html
<link rel="stylesheet" href="{{ url_for('static', filename='app.css') }}">
```

## Q43. How do you handle file uploads in Flask?

**Answer:**

HTML forms must use `method="POST"` and `enctype="multipart/form-data"` with `<input type="file">` fields. In Flask, uploaded files appear in `request.files`; retrieve with `request.files['field']` and check `filename` before saving. Use `werkzeug.utils.secure_filename` to strip dangerous paths from user-supplied names. Save files to disk with `save()` or process in memory with `stream` for small files. Enforce size and type limits to prevent abuse.

```python
from flask import Flask, request
from werkzeug.utils import secure_filename
```

## Q54. How do sessions work internally in Flask?

**Answer:**

By default Flask uses signed cookies: session data is serialized, timestamped if configured, signed with HMAC using the secret key, and sent to the browser as `Set-Cookie`. On subsequent requests, Flask verifies the signature and deserializes the payload into `session`. If verification fails, the session is discarded, treating the user as anonymous. You can replace the session interface with server-side storage by configuring custom session backends via extensions. Understanding signing explains why session data is visible to the client but not trustworthy to modify.

## Q55. What is the difference between cookies and sessions?

**Answer:**

Sessions managed by Flask include integrity protection when cookie-based, whereas raw cookies are just strings unless you add signing yourself. Choose sessions for server-controlled user state with tamper resistance.

## Q65. How do you perform CRUD operations with Flask-SQLAlchemy?

**Answer:**

## Q68. What is the difference between `db.session.add()` and `db.session.commit()`?

**Answer:**

`add()` stages new or updated objects in the SQLAlchemy session so they will be persisted on the next flush, but no SQL is guaranteed to hit the database until flush/commit occurs. `commit()` ends the transaction, applying all pending changes and releasing locks according to isolation level. Calling `commit()` implicitly flushes pending work first. If `commit()` fails, you should `rollback()` the session to avoid partial state. For read-only operations, you typically do not need `commit()`.

Set `static_folder` and `static_url_path` on the blueprint to serve assets under a blueprint-specific URL prefix. Reference files with `url_for('blueprint_name.static', filename='...')` when using endpoint names. For production, often a build tool bundles assets globally, but blueprint static folders help during development. Avoid duplicating large libraries across blueprints; centralize vendor files. Cache headers remain important regardless of folder layout.

```python
admin = Blueprint("admin", __name__, static_folder="static", static_url_path="/admin/static")
```

## Q89. What is `__init__.py` in a Flask project?

**Answer:**

`__init__.py` marks directories as Python packages so imports like `from app import create_app` work reliably. In Flask projects, `app/__init__.py` commonly defines the application factory and extension initialization. It can be empty in Python 3 namespace packages in some cases, but explicit packages are clearer for application code. Package initialization runs once on import, so avoid heavy side effects unless intentional. Keeping factories here centralizes app creation.

## Q90. How do you share data between Blueprints?

**Answer:**

Use the database, session, or Flask's `g` object for request-scoped data rather than global mutable module variables. For cross-cutting concerns, `before_request` handlers can populate `g` with user or tenant context. Application configuration in `app.config` holds shared settings. If modules need to call each other, depend on services or repositories rather than importing routes circularly. For loosely coupled communication, consider signals or message queues in larger systems.

```python
from flask import g

@app.before_request
def load_user():
    g.user = get_current_user()
```

## Q92. What is the `g` object in Flask?

**Answer:**

`g` is a namespace object attached to the application context for storing data during a single request lifecycle. It is suitable for caching per-request objects like database connections or the current user loaded once in `before_request`. Values do not persist across requests and should not be used as long-term storage. Use `g` to avoid repeated lookups within the same request handling chain. Clear or avoid leaking sensitive data if using middleware that might reuse contexts incorrectly in custom servers.

```python
from flask import g

def get_db():
    if "db" not in g:
        g.db = connect_db()
    return g.db
```

## Q93. What are before_request and after_request hooks?

**Answer:**

`@app.before_request` registers functions that run before each view, useful for authentication, logging, or setting `g` attributes. `@app.after_request` receives the response object and can modify headers or content before it is sent. `teardown_request` and `teardown_appcontext` clean up resources even if exceptions occur, depending on scope. Register hooks on blueprints for modular behavior. Order of execution follows registration order unless you use `Blueprint.before_app_request` variants for global hooks tied to a blueprint.

```python
import time

from flask import Flask, g

app = Flask(__name__)

@app.before_request
def start_timer():
    g.start = time.time()

@app.after_request
def add_header(response):
    response.headers["X-Elapsed"] = str(time.time() - g.start)
    return response
```

## Q94. How do you send emails in Flask?

**Answer:**

Use Flask-Mail or a transactional email API client to send messages from view functions or background tasks. Configure SMTP settings or API keys via environment variables, never hardcoded in source. Construct messages with subject, recipients, plain text, and HTML bodies as needed. For reliability, offload sending to a task queue like Celery so web requests return quickly. Handle failures with retries and monitoring rather than failing user-facing requests silently.

```python
from flask_mail import Mail, Message

mail = Mail(app)

msg = Message("Hello", recipients=["user@example.com"], body="Hi there")
mail.send(msg)
```

## Q95. What is Flask-Mail?

**Answer:**

Flask-Mail is an extension that configures a `Mail` object with Flask's configuration system and provides a simple `Message` API for SMTP email sending. It supports attachments, bulk mail, and connection pooling for performance. In modern apps, many teams prefer API-based providers with official SDKs, but SMTP via Flask-Mail remains common for simple deployments. ## Q96. How do you deploy a Flask application?

**Answer:**

Run the app behind a production WSGI server such as Gunicorn or uWSGI, often proxied by Nginx or another reverse proxy for TLS termination and static files. Set environment variables for secrets and bind the server to localhost interfaces, letting the proxy face the public internet. Use a process manager like systemd, supervisord, or container orchestration to restart failed workers. Database connections and logging must be configured for production reliability. Automate deployments with CI/CD pipelines that run tests before release.

```bash
gunicorn -w 4 -b 127.0.0.1:8000 "myapp:create_app()"
```

## Q97. What is Gunicorn and how is it used with Flask?

**Answer:**

Gunicorn is a mature WSGI HTTP server for Python that forks multiple worker processes to handle concurrent requests, unlike Flask's single-threaded development server. You point Gunicorn at your application callable, often `module:app` or `package:create_app()` with a factory flag depending on invocation style. Workers should match CPU cores and workload characteristics; blocking workloads may use threads or async workers with care. Gunicorn integrates with reverse proxies for HTTPS and load balancing. Monitor worker restarts and memory usage under load.

```bash
pip install gunicorn
gunicorn -w 3 mypackage.wsgi:app
```

## Q98. How do you use environment variables in Flask?

**Answer:**

Load configuration from the environment with `os.environ` or libraries like `python-dotenv` that read `.env` files during development. Mark sensitive values as never committed and document required variables in example files like `.env.example`. Flask's `from_prefixed_env` or custom loaders can map `FLASK_` prefixed vars to config keys in newer versions depending on setup. Environment-driven config separates dev, staging, and production settings cleanly. Validate required variables at startup to fail fast when misconfigured.

```python
import os

class Config:
    SECRET_KEY = os.environ["SECRET_KEY"]
    SQLALCHEMY_DATABASE_URI = os.environ.get("DATABASE_URL", "sqlite:///dev.db")
```

## Q99. What is Flask-RESTful?

**Answer:**

Flask-RESTful is an extension that provides abstractions for building REST APIs in Flask, including class-based resources with method dispatch for GET, POST, PUT, and DELETE. It offers request parsing helpers and structured error handling patterns common in APIs. Many teams now use plain Flask routes, Flask-RESTX, or FastAPI for APIs, but Flask-RESTful remains in existing codebases. It sits on top of Flask's routing and integrates with authentication extensions. Evaluate whether simpler `MethodView` classes meet your needs without extra dependencies.

```python
from flask_restful import Api, Resource

api = Api(app)

class Hello(Resource):
    def get(self):
        return {"hello": "world"}

api.add_resource(Hello, "/hello")
```

## Q100. What are the differences between `app.run()` parameters?

**Answer:**

The `app.run()` method accepts parameters that control the development server only, not production behavior. `host` defaults to `127.0.0.1`; set `0.0.0.0` to accept connections from other machines on the network during testing. `port` selects the TCP port to bind, avoiding conflicts with other services. `debug` enables the debugger and reloader as discussed earlier. `threaded` and `processes` adjust concurrency for the dev server but are still unsuitable for production loads compared to Gunicorn. `ssl_context` can enable ad hoc TLS for local HTTPS experiments.

```python
if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000, debug=True, threaded=True)
```
