## Q1. Explain the Flask application factory pattern in detail.

**Answer:**

The application factory pattern means you define a function (commonly `create_app`) that constructs and returns a fully configured `Flask` instance instead of creating a global `app` at import time. This avoids side effects during import, makes testing easier because each test can request a fresh app with isolated configuration, and allows multiple app instances (for example, different configs in the same process). Factories typically load configuration from objects or environment variables, initialize extensions by calling `ext.init_app(app)`, register blueprints, and attach CLI commands. You still need an entry point (WSGI server, `flask run`) that imports the factory and calls `create_app()` to obtain the application object. The pattern pairs well with the `FLASK_APP` variable pointing to `create_app` so the CLI discovers the factory automatically.

```python
# app/__init__.py
from flask import Flask

def create_app(config_name: str = "development") -> Flask:
    app = Flask(__name__)
    app.config.from_object(f"config.{config_name.capitalize()}Config")

    from .extensions import db, migrate
    db.init_app(app)
    migrate.init_app(app, db)

    from .api import api_bp
    app.register_blueprint(api_bp, url_prefix="/api")

    return app
```

```python
# wsgi.py
from app import create_app

app = create_app("production")
```

## Q2. How do you structure a large Flask application?

**Answer:**

Large Flask apps benefit from a package layout with clear boundaries: a top-level package (for example `app/`) containing `__init__.py` with `create_app`, subpackages for `api` or `views`, `models`, `services`, `repositories`, `schemas` or serializers, `tasks` for background work, and `cli` for custom commands. Blueprints group related routes and can own template/static folders when needed; each feature area can be a blueprint submodule. Shared infrastructure (database, cache, mail) lives in `extensions.py` with deferred `init_app` calls. Configuration is split per environment in `config.py` or a `config/` package. Tests mirror the app structure under `tests/`. This structure scales because imports stay localized, teams can own blueprints, and you avoid a single giant `views.py` file.

```
app/
  __init__.py       # create_app
  extensions.py     # db, migrate, cache (unbound until init_app)
  config.py
  api/
    __init__.py
    users.py        # blueprint or route modules
  models/
  services/
  repositories/
tests/
```

## Q3. What is the application context and request context in Flask?

**Answer:**

Flask uses two distinct context types implemented as thread-local (or task-local) stacks. The **application context** represents the active application for the current worker thread; it is pushed when handling a request or when you manually use `app.app_context()`. It makes `current_app` and `g` available. The **request context** is pushed per HTTP request (or when using the test client) and wraps request-specific data: `request`, `session`, and also ensures an app context exists (Flask pushes an app context if one is missing when the request context is entered). Understanding the split matters because `current_app` only works inside an app context, while `request` only works inside a request context. Background threads and Celery tasks do not automatically get either context; you must push `app.app_context()` (and often `request` is not applicable without faking it). Teardown functions run when the context pops, which is how you release DB connections or clean `g`.

```python
from flask import Flask, g, current_app

app = Flask(__name__)

@app.before_app_request
def bind_db():
    g.db = connect_db()

@app.teardown_appcontext
def close_db(exc):
    db = g.pop("db", None)
    if db is not None:
        db.close()
```

## Q4. How does Flask's context local work internally?

**Answer:**

Flask builds on Werkzeug's `LocalStack` and `LocalProxy` abstractions. A `LocalStack` holds a stack of context objects per greenlet or thread (depending on configuration), so concurrent requests do not overwrite each other’s data. When you access `current_app` or `request`, you are actually reading through a `LocalProxy` that resolves to the top of the appropriate stack at access time. Pushing a context duplicates the stack entry; popping removes it after the request finishes. This design is why you must not store the real `request` object in a global variable for later async callbacks without re-binding context. In async mode (Flask 2.0+ with async views), context propagation uses the same conceptual model adapted to the async runtime. Multiple contexts can be nested (for example, manual `app.app_context()` inside a test), so the stack model matters when debugging “working outside of application context” errors. Understanding locals also explains why global singletons that capture `request` are a recurring source of bugs under concurrency.

## Q5. What is the difference between `current_app` and `app`?

**Answer:**

`app` refers to the concrete `Flask` instance you created, typically available in the module where you called `Flask(__name__)` or returned from `create_app`. `current_app` is a proxy to the application bound to the active application context, and it is the idiomatic way to reach configuration and extensions from code that is imported in many places (for example, utility modules, models, or services) without threading the `app` object through every function signature. Using `current_app` outside an app context raises `RuntimeError`. In tests and scripts, use `app.app_context()` or pass `app` explicitly when you want to avoid implicit globals. Extension patterns like `db = SQLAlchemy()` then `db.init_app(app)` rely on `current_app` inside methods when no explicit app is passed.

```python
from flask import current_app

def get_setting(name: str) -> str:
    return current_app.config[name]
```

## Q6. How do you implement the repository pattern in Flask?

**Answer:**

The repository pattern hides persistence behind an interface so services and views depend on “how to load and save aggregates” rather than raw SQLAlchemy session details. In Flask you typically define a repository class per aggregate root (for example `UserRepository`) with methods like `get_by_id`, `list`, `add`, implemented with Flask-SQLAlchemy’s `db.session`. Repositories are plain Python classes; you instantiate them per request or inject them via your DI approach. This keeps transaction boundaries in a service layer while repositories focus on queries and mapping. Unit tests can substitute an in-memory or fake repository without touching Flask.

```python
from typing import Optional
from app.extensions import db
from app.models import User

class UserRepository:
    def get(self, user_id: int) -> Optional[User]:
        return db.session.get(User, user_id)

    def add(self, user: User) -> None:
        db.session.add(user)
```

## Q7. What is dependency injection in Flask and how do you implement it?

**Answer:**

Flask does not ship a full DI container; dependencies are usually constructor parameters, factory functions, or app extensions. Common approaches include: passing dependencies explicitly from `create_app` into blueprints via closures or accessor functions; using third-party libraries such as `injector` or `dependency-injector` to bind interfaces to implementations per app; or lightweight patterns like a `get_container()` that returns request-scoped services stored on `g` in `before_request`. For APIs, frameworks like Flask-RESTX or custom decorators can resolve parsers and services. The goal is the same as in other stacks: invert dependencies so routes stay thin and services stay testable without patching Flask globals.

```python
from flask import g

def get_user_service():
    if "user_service" not in g:
        g.user_service = UserService(UserRepository())
    return g.user_service
```

## Q8. How do you manage configuration for different environments (dev/staging/prod)?

**Answer:**

Use layered configuration: a base class with defaults, subclasses or objects per environment, and finally environment variables or `.env` files for secrets (never committed). `app.config.from_object`, `from_pyfile`, and `from_envvar` let you select which layer loads. Twelve-factor style apps read `DATABASE_URL`, `SECRET_KEY`, and feature flags from the environment in production. For local development, `python-dotenv` can load `.env` before `create_app` runs. Keep secrets out of source control; use your platform’s secret store in production. Validate required keys at startup so misconfiguration fails fast.

```python
import os

class BaseConfig:
    SECRET_KEY = os.environ.get("SECRET_KEY", "dev-only")

class DevelopmentConfig(BaseConfig):
    DEBUG = True

class ProductionConfig(BaseConfig):
    DEBUG = False

config_by_name = {
    "development": DevelopmentConfig,
    "production": ProductionConfig,
}
```

## Q9. What are Flask extensions and how do they work internally?

**Answer:**

A Flask extension is typically a class that follows the “init app” pattern: construct once without an app (`ext = SomeExtension()`), then call `ext.init_app(app)` inside `create_app` so the same extension object can serve multiple apps (tests, scripts). Internally, extensions stash state on `app.extensions` under a unique key and often use `current_app` or `app` references inside bound methods. They may register teardown handlers, CLI commands, and signals. Understanding `init_app` matters when you see circular import issues: import models after `db` is created but defer model imports that trigger queries until the app exists.

```python
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy()

def create_app():
    app = Flask(__name__)
    app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite://"
    db.init_app(app)
    return app
```

## Q10. How do you implement the service layer pattern in Flask?

**Answer:**

The service layer contains business rules and orchestrates repositories, external APIs, and domain logic. Flask views or REST resources should delegate to services after validating input, so HTTP concerns stay separate from domain rules. Services are plain classes or functions; you can instantiate them per request or register singletons. They own transaction boundaries: call `db.session.commit()` in one place after the service succeeds, or use a unit-of-work pattern. This makes it straightforward to reuse the same service from a CLI command, a Celery task, or a test without duplicating logic that would otherwise live in route handlers.

```python
class OrderService:
    def __init__(self, orders: OrderRepository, payments: PaymentClient):
        self._orders = orders
        self._payments = payments

    def place_order(self, user_id: int, cart: Cart) -> Order:
        order = Order(user_id=user_id, items=list(cart.items))
        self._orders.add(order)
        self._payments.charge(order.total)
        return order
```

## Q11. How do you implement custom URL converters?

**Answer:**

Flask’s routing uses Werkzeug converters; you can register custom ones with `app.url_map.converters`. Subclass `BaseConverter`, set `regex`, and implement `to_python` (string from URL to Python value) and `to_url` (Python value back to URL segment). Register the converter on the app map, then use it in `@app.route` rules. This is useful for UUIDs, slugs with validation, or encoded identifiers. Keep converters pure and fast because they run on every match attempt.

```python
from uuid import UUID
from werkzeug.routing import BaseConverter

class UUIDConverter(BaseConverter):
    regex = r"[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}"

    def to_python(self, value):
        return UUID(value)

    def to_url(self, value):
        return str(value)

def create_app():
    app = Flask(__name__)
    app.url_map.converters["uuid"] = UUIDConverter
    return app
```

## Q12. What are Flask middleware and how do you create one?

**Answer:**

Strictly speaking, Flask sits on WSGI; “middleware” often means either WSGI middleware wrapping the Flask app, or Flask hooks like `before_request` and `after_request` that behave similarly for many use cases. WSGI middleware is a callable `middleware(environ, start_response)` that wraps `app` and can inspect or mutate the environment and response. In Flask, prefer `before_request`/`after_request` for request/response augmentation tied to the Flask lifecycle, and WSGI middleware when integrating cross-cutting concerns at the server boundary (for example, custom metrics). `ProxyFix` from Werkzeug is a common WSGI middleware for trusting `X-Forwarded-*` headers behind a reverse proxy.

```python
class PrefixMiddleware:
    def __init__(self, app, prefix):
        self.app = app
        self.prefix = prefix

    def __call__(self, environ, start_response):
        if environ["PATH_INFO"].startswith(self.prefix):
            environ["PATH_INFO"] = environ["PATH_INFO"][len(self.prefix) :] or "/"
            return self.app(environ, start_response)
        return self.app(environ, start_response)
```

## Q13. How do you implement API versioning in Flask?

**Answer:**

Common strategies include URL prefix versioning (`/api/v1/...`), header versioning (`Accept: application/vnd.myapi.v2+json`), or query parameters (less RESTful). In Flask, URL versioning is straightforward: register blueprints with `url_prefix="/api/v1"` and add parallel blueprints for v2, sharing services but potentially different serializers. Header-based versioning can be implemented in a `before_request` handler that selects a view function or dispatches to different modules based on `request.accept_mimetypes`. Version deprecation should be communicated via headers (`Sunset`, `Deprecation`) and documentation. Keep shared business logic in services so multiple API versions do not duplicate core rules.

```python
from flask import Blueprint

v1 = Blueprint("api_v1", __name__, url_prefix="/api/v1")
v2 = Blueprint("api_v2", __name__, url_prefix="/api/v2")

def create_app():
    app = Flask(__name__)
    app.register_blueprint(v1)
    app.register_blueprint(v2)
    return app
```

## Q14. What is the `before_request`, `after_request`, and `teardown_request` lifecycle?

**Answer:**

`before_request` handlers run after the request context is pushed and before the view; they can short-circuit by returning a response (for example, authentication failure). `after_request` handlers receive the response object and must return a (possibly modified) response; they run in reverse registration order for the same phase. `teardown_request` runs after the response is fully processed, with access to the exception if any; use it for cleanup that must happen even if the view errors. `teardown_appcontext` is similar but tied to app context lifetime. Order matters: register teardown handlers for resources opened in `before_request` to avoid leaks.

```python
@app.before_request
def assign_request_id():
    g.request_id = uuid4().hex

@app.after_request
def add_header(response):
    response.headers["X-Request-ID"] = g.request_id
    return response

@app.teardown_request
def teardown(exc):
    pass
```

## Q15. How do you implement rate limiting in Flask?

**Answer:**

Rate limiting restricts how many requests a client can make in a window, typically keyed by IP address, API key, or user id. The `Flask-Limiter` extension integrates with Flask and supports Redis, Memcached, or in-memory backends. You decorate routes or apply defaults globally, specifying limits like `"100 per hour"`. For distributed deployments, use a shared Redis backend so all workers enforce one global counter. Combine with proper `429` responses and `Retry-After` headers where supported. Rate limits complement authentication; they do not replace it.

```python
from flask import Flask
from flask_limiter import Limiter
from flask_limiter.util import get_remote_address

app = Flask(__name__)
limiter = Limiter(get_remote_address, app=app, default_limits=["200 per day"])

@app.route("/expensive")
@limiter.limit("10 per minute")
def expensive():
    return {"ok": True}
```

## Q16. How do you handle CORS in Flask?

**Answer:**

Cross-Origin Resource Sharing is enforced by browsers; your API must emit `Access-Control-Allow-Origin` and related headers for allowed origins. The `flask-cors` extension can apply CORS globally or per blueprint with `CORS(app, resources={r"/api/*": {"origins": "https://app.example.com"}})`. For APIs, be explicit: allow credentials only with trusted origins, avoid `*` when using cookies. Preflight `OPTIONS` requests must return allowed methods and headers. Misconfigured CORS is a frequent source of “works in Postman, fails in browser” issues.

```python
from flask_cors import CORS

def create_app():
    app = Flask(__name__)
    CORS(app, resources={r"/api/*": {"origins": ["https://trusted.example"]}})
    return app
```

## Q17. What are method views and when should you use them?

**Answer:**

`MethodView` from `flask.views` maps HTTP methods to class methods (`get`, `post`, `delete`), reducing boilerplate when one URL supports multiple verbs with shared setup. You register with `app.add_url_rule` or use `MethodView.as_view` with a blueprint. Use them for REST-style resources where behavior diverges cleanly by verb. They pair well with class-based organization and mixins for authentication. For very small endpoints, a function view is simpler; for CRUD resources, `MethodView` often reads clearer.

```python
from flask.views import MethodView

class ItemAPI(MethodView):
    def get(self, item_id):
        return {"id": item_id}

    def delete(self, item_id):
        return "", 204

app.add_url_rule("/items/<int:item_id>", view_func=ItemAPI.as_view("item"))
```

## Q18. How do you implement URL prefixing with Blueprints?

**Answer:**

Blueprints accept `url_prefix` at creation or registration. `Blueprint("auth", __name__, url_prefix="/auth")` prefixes all routes in that blueprint. You can also pass `url_prefix` to `register_blueprint` to compose paths without editing the blueprint definition, which helps reuse the same blueprint under different mounts in tests or multi-tenant setups. Static and template paths can be blueprint-relative when configured. Remember that `url_for` needs the blueprint endpoint name including the blueprint’s name prefix when namespaced.

```python
from flask import Blueprint, render_template

bp = Blueprint("blog", __name__, url_prefix="/blog")

@bp.route("/post/<int:post_id>")
def show_post(post_id):
    return render_template("post.html", post_id=post_id)
```

## Q19. How do you handle trailing slashes in Flask URLs?

**Answer:**

Flask distinguishes `/path` and `/path/`; by default a rule defined without a trailing slash redirects the slash variant with `308` (or `301` in older behavior) to the canonical form, which can affect SEO and API clients. Define routes consistently and disable `strict_slashes` on a rule if you need both to match the same endpoint: `@app.route("/items/", strict_slashes=False)`. For APIs, pick one convention and document it. Reverse URL generation with `url_for` respects the rule definition. Proxies and load balancers should preserve paths when redirecting.

```python
@app.route("/items", strict_slashes=False)
def items():
    return {"items": []}
```

## Q20. How do you implement request validation middleware?

**Answer:**

Validation middleware typically runs early in the request lifecycle: parse JSON, check required fields, coerce types, and reject with `400` before the view runs. Implement as `before_request` on a blueprint, a decorator stack, or a base `MethodView` that calls a schema validation function. Libraries like Marshmallow, Pydantic, or `webargs` integrate cleanly with Flask. For OpenAPI-first designs, you might validate against a generated schema. Keep error messages consistent and machine-readable for API clients.

```python
from flask import request, jsonify, abort

@app.before_request
def validate_json():
    if request.method in ("POST", "PUT", "PATCH") and request.is_json:
        data = request.get_json(silent=True)
        if data is None:
            abort(400, description="Invalid JSON")
```

## Q21. How do you design a RESTful API with Flask?

**Answer:**

Model resources as nouns with stable URIs, use HTTP verbs semantically (`GET` read, `POST` create, `PUT`/`PATCH` update, `DELETE` remove), and return appropriate status codes (`201` with `Location` on create, `204` on delete, `409` on conflicts). Use Flask blueprints for versioning and resource grouping. Representations should be consistent JSON shapes; avoid leaking ORM models directly—use serializers. Support pagination, filtering, and sorting via query parameters. Document authentication (Bearer JWT, API keys) and error formats. HATEOAS is optional but document hypermedia if you use it.

## Q22. What is Flask-RESTful vs Flask-RESTX?

**Answer:**

**Flask-RESTful** provides `Api`, `Resource`, and request parsing helpers to build class-based REST endpoints with less boilerplate; it is mature but maintenance has slowed and Swagger integration is manual. **Flask-RESTX** is a fork that adds first-class OpenAPI/Swagger documentation, namespaces, and response marshalling helpers, making it attractive for documented corporate APIs. Both encourage resource classes similar to `MethodView`. Choose RESTX when automatic docs and models matter; choose plain Flask or Marshmallow when you want minimal dependencies and full control.

## Q23. How do you implement request parsing and validation in Flask APIs?

**Answer:**

Parse `request.get_json()` for JSON bodies; validate with Marshmallow `Schema.load`, Pydantic models, or `webargs` decorators that bind query, form, headers, and JSON in one place. Return validation errors as `422` or `400` with a structured `errors` field. Avoid manual `if "email" not in data` scattered in views—centralize schemas per endpoint or resource. For file uploads, validate MIME types and size limits before saving. Parsing should be deterministic and logged at debug level without logging secrets.

```python
from flask import request
from marshmallow import Schema, fields, ValidationError

class UserSchema(Schema):
    email = fields.Email(required=True)
    name = fields.Str(required=True)

@app.post("/users")
def create_user():
    try:
        data = UserSchema().load(request.get_json() or {})
    except ValidationError as err:
        return {"errors": err.messages}, 422
    return {"created": True, "data": data}, 201
```

## Q24. How do you handle API authentication with JWT?

**Answer:**

JSON Web Tokens encode claims (subject, expiry, scopes) signed with a server secret or asymmetric key. Clients send `Authorization: Bearer <token>` on each request. The server verifies signature and expiry, then loads the user or principal. Stateless JWTs scale horizontally but revocation requires short TTLs, refresh tokens, or server-side blocklists. Never put secrets or PII in payload claims unless encrypted (JWE). Use HTTPS everywhere. In Flask, libraries like Flask-JWT-Extended handle encoding, decoding, and refresh flows.

## Q25. What is Flask-JWT-Extended and how do you use it?

**Answer:**

Flask-JWT-Extended integrates JWT with Flask’s request lifecycle: decorators like `@jwt_required()` protect views, optional `@jwt_required(refresh=True)` for refresh endpoints, and `get_jwt_identity()` reads the subject. It supports cookie or header transport, fresh vs. access tokens, token blocklists for logout, and custom claims. Configure `JWT_SECRET_KEY`, token locations, and cookie security flags in production. It reduces boilerplate compared to hand-rolling PyJWT in every view while remaining flexible for custom unauthorized handlers.

```python
from flask import Flask
from flask_jwt_extended import JWTManager, create_access_token, jwt_required, get_jwt_identity

app = Flask(__name__)
app.config["JWT_SECRET_KEY"] = "change-me"
jwt = JWTManager(app)

@app.post("/login")
def login():
    access = create_access_token(identity="user-123")
    return {"access_token": access}

@app.get("/me")
@jwt_required()
def me():
    return {"user": get_jwt_identity()}
```

## Q26. How do you implement pagination in Flask APIs?

**Answer:**

Pagination avoids loading entire tables into memory and gives clients stable chunks. Accept `page` and `per_page` (or `cursor` for keyset pagination) as query parameters with sane maximums. Use SQLAlchemy’s `query.paginate` or `limit`/`offset` with `order_by` on a unique column. Return metadata: `total`, `page`, `pages`, `next`, `prev` URLs. Cursor-based pagination is better for large, frequently changing datasets because offsets become expensive. Include pagination links in the response body or `Link` header per RFC 5988.

```python
@app.get("/users")
def list_users():
    page = request.args.get("page", 1, type=int)
    per_page = min(request.args.get("per_page", 20, type=int), 100)
    q = User.query.order_by(User.id)
    p = q.paginate(page=page, per_page=per_page, error_out=False)
    return {
        "items": [u.to_dict() for u in p.items],
        "total": p.total,
        "page": p.page,
        "pages": p.pages,
    }
```

## Q27. How do you implement API response serialization?

**Answer:**

Serialization converts domain objects or ORM models into JSON-safe dicts. Options include Marshmallow `Schema.dump`, Pydantic `model_dump`, manual `to_dict` methods on models, or libraries like `flask-marshmallow`. Keep serializers separate from models to avoid circular imports and to version API output independently of the database schema. For collections, serialize in list comprehensions or use `many=True`. Consider excluding internal fields (`password_hash`) at the schema layer, not in ad hoc dict building. For datetime and Decimal values, define explicit serialization rules so clients always receive ISO 8601 strings or numbers rather than inconsistent defaults. When an endpoint exposes multiple representations (public vs admin), use separate schema classes or context flags rather than branching inside a single giant serializer.

```python
from marshmallow import Schema, fields

class UserPublicSchema(Schema):
    id = fields.Int()
    name = fields.Str()
    created_at = fields.DateTime(format="iso")

schema = UserPublicSchema()
return schema.dump(user), 200
```

## Q28. How do you handle API errors consistently?

**Answer:**

Register custom error handlers with `@app.errorhandler` for HTTP exceptions and Python exceptions. Return a uniform JSON body: `{"error": {"code": "...", "message": "...", "details": ...}}` with appropriate status codes. Map domain exceptions (for example `UserNotFound`) to HTTP responses in one place. Log server-side stack traces at `ERROR` level without exposing internals to clients in production. For validation, use `422` with field-level errors. Flask’s `abort()` raises `HTTPException` which your handlers can format.

```python
from flask import jsonify

class ApiError(Exception):
    def __init__(self, message, status=400, code="bad_request"):
        self.message = message
        self.status = status
        self.code = code

@app.errorhandler(ApiError)
def handle_api_error(err):
    return jsonify(error={"code": err.code, "message": err.message}), err.status
```

## Q29. How do you implement HATEOAS in Flask?

**Answer:**

HATEOAS embeds hypermedia links in representations so clients discover actions without hard-coding URLs. In Flask, build link dicts in serializers: include `self`, `next`, `related` URIs using `url_for(..., _external=True)` with appropriate endpoint names. This increases payload size and requires stable link relations (`rel`). For JSON APIs, HAL or JSON-LD are common formats. HATEOAS is less common in mobile SPAs that prefer fixed OpenAPI contracts, but it shines in evolvable public APIs.

```python
from flask import url_for

def user_hal(user):
    return {
        "_links": {
            "self": {"href": url_for("user_detail", user_id=user.id, _external=True)},
            "orders": {"href": url_for("user_orders", user_id=user.id, _external=True)},
        },
        "name": user.name,
    }
```

## Q30. How do you document Flask APIs with Swagger/OpenAPI?

**Answer:**

Use Flask-RESTX, `apispec` with Marshmallow plugins, or `flask-openapi3` to generate OpenAPI from decorators and models. Annotations describe paths, parameters, request bodies, and responses; the UI is served at `/swagger` or similar. Keep documentation close to code so it updates with changes. For hand-written specs, maintain `openapi.yaml` in repo and validate it in CI. Document auth schemes (`bearerAuth`), error shapes, and pagination consistently. In code-first workflows, review generated specs in pull requests the same way you review API behavior, because misleading docs are worse than missing docs. For internal microservices, publishing the spec to a portal and enforcing backward compatibility with breaking-change detection saves integration pain. Expose only the documentation endpoints in non-production environments if you worry about information disclosure.

## Q31. How do you implement database connection pooling in Flask?

**Answer:**

SQLAlchemy maintains a `QueuePool` of connections per engine by default (for DBAPIs that support pooling). ```python

```python
    "pool_recycle": 280,
```

## Q32. What are SQLAlchemy relationships (one-to-many, many-to-many) in detail?

**Answer:**

**One-to-many** links one parent row to many children via a foreign key on the child table; use `relationship(..., back_populates=...)` for bidirectional navigation. **Many-to-many** uses an association table with two foreign keys; `relationship` uses `secondary=association_table` or association object pattern if you need extra columns on the link. Loading strategies (`lazy`, `selectin`, `joinedload`) control when related rows are fetched. Cascades (`cascade="all, delete-orphan"`) propagate deletes carefully—misconfigured cascades can delete more than intended. Always define indexes on foreign keys for join performance.

```python
class User(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    posts = db.relationship("Post", back_populates="author")

class Post(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    user_id = db.Column(db.Integer, db.ForeignKey("user.id"))
    author = db.relationship("User", back_populates="posts")
```

## Q33. How do you implement soft deletes with Flask-SQLAlchemy?

**Answer:**

Soft delete marks rows as deleted (`deleted_at` timestamp or `is_deleted` flag) instead of removing them, preserving referential integrity and audit history. Filter queries with `.filter(Model.deleted_at.is_(None))` or use a custom query class / SQLAlchemy `with_loader_criteria` to apply the filter globally. Override `delete()` on the model to set the flag and commit. Remember to index `deleted_at` and adjust unique constraints to include the flag (partial unique indexes on PostgreSQL). Background jobs can purge old soft-deleted rows.

```python
class SoftMixin:
    deleted_at = db.Column(db.DateTime, index=True)

    def soft_delete(self):
        self.deleted_at = datetime.utcnow()
```

## Q34. What are SQLAlchemy events and how do you use them?

**Answer:**

SQLAlchemy emits events such as `before_insert`, `after_update`, `before_flush` on mappers and sessions. Listen with `@event.listens_for(Model, "before_insert")` to denormalize fields, validate invariants, or write audit rows. Events run synchronously inside the transaction; heavy work belongs in Celery, not blocking listeners. Use events when behavior must hold for all code paths touching the ORM, not just one service method.

```python
from sqlalchemy import event

@event.listens_for(User, "before_insert")
def hash_password(mapper, connection, target):

        target.password_plain = None
```

## Q35. How do you optimize database queries in Flask-SQLAlchemy?

**Answer:**

## Q36. How do you handle database transactions in Flask?

**Answer:**

Flask-SQLAlchemy scopes sessions to the request: `db.session.commit()` on success and implicit rollback on exceptions if you use patterns like try/except around commits, or rely on teardown. For multi-step operations, wrap in `with db.session.begin():` (SQLAlchemy 2.0 style) or explicit `try/finally` with `session.rollback()`. Do not hold transactions open across long external HTTP calls. Use savepoints for nested operations on databases that support them. Celery tasks need explicit `app.app_context()` and manual session handling.

```python
@app.post("/transfer")
def transfer():
    try:
        debit()
        credit()
        db.session.commit()
    except Exception:
        db.session.rollback()
        raise
    return {"ok": True}
```

## Q37. What is the N+1 query problem and how do you solve it in Flask?

**Answer:**

Solve with eager loading: `query.options(selectinload(User.posts))` or `joinedload` when a join is efficient. Profile with Flask-SQLAlchemy logging or tools like `sqlalchemy-utils` query counting in tests. Sometimes denormalization or a single aggregated query is better than ORM convenience.

```python
users = User.query.options(selectinload(User.posts)).all()
```

## Q42. How do you prevent SQL injection in Flask?

**Answer:**

Use ORM query APIs or parameterized `text()` statements so user input is never concatenated into SQL strings. Regular security reviews and static analysis rules that flag string-built SQL catch regressions. In reviews, treat any raw SQL touching request data as a mandatory second pair of eyes.

## Q43. How do you prevent XSS attacks in Flask?

**Answer:**

Jinja2 escapes variables by default in HTML contexts; use `|safe` only for trusted markup. For JSON APIs, frameworks return `application/json` without HTML interpretation, but if you reflect user input into HTML pages, stay disciplined with escaping and Content Security Policy. Defense in depth means combining output encoding, CSP, and secure cookie flags rather than relying on a single layer.

## Q46. How do you implement role-based access control (RBAC) in Flask?

**Answer:**

Model roles and permissions in the database (`roles`, `permissions`, association tables). After authentication, load the user’s roles into `g` or JWT claims. Decorate views with `@requires_roles("admin")` that checks membership before running the view. For fine-grained control, map permissions to endpoints or resources. Centralize checks to avoid inconsistent authorization logic. Test denied paths explicitly.

```python
from functools import wraps
from flask import abort, g

def requires_roles(*roles):
    def decorator(f):
        @wraps(f)
        def wrapped(*args, **kwargs):
            if g.user.role not in roles:
                abort(403)
            return f(*args, **kwargs)
        return wrapped
    return decorator
```

## Q50. How do you protect against clickjacking in Flask?

Celery is a distributed task queue using brokers (Redis/RabbitMQ) and workers executing Python callables. Share the Flask app instance by calling `create_app()` inside the worker process or using `celery.py` that imports the app for context. Tasks should push `app.app_context()` when they touch the database. Configure serialization (`json`), result backends, and task time limits. Monitor queues for backlog growth.

```python
from celery import Celery

def make_celery(app):
    celery = Celery(app.import_name)
    celery.conf.update(app.config["CELERY_CONFIG"])

    class ContextTask(celery.Task):
        def __call__(self, *args, **kwargs):
            with app.app_context():
                return self.run(*args, **kwargs)

    celery.Task = ContextTask
    return celery
```

## Q75. What is the difference between threading and multiprocessing in Flask?

**Answer:**

Threading shares memory within a process; good for I/O-bound concurrency but limited by the GIL for CPU-bound Python. Multiprocessing spawns separate Python interpreters with isolated memory, better for CPU-bound tasks but higher memory overhead. Gunicorn defaults to sync workers; `gevent`/`eventlet` patch for green threads; `gunicorn -w 4` uses multiple processes. Choose based on profiling, not assumptions. Thread-based models complicate C extensions that release the GIL unpredictably, while multiprocessing complicates shared state and startup time. For mixed workloads, split CPU-heavy work to Celery with prefork pools and keep the web tier I/O focused.

## Q78. How do you implement Server-Sent Events (SSE) in Flask?

**Answer:**

SSE is one-way server-to-client streaming over HTTP using `text/event-stream`. Implement a generator view that `yield`s `data: ...\n\n` lines and flush periodically. Use `Response(stream_with_context(gen()), mimetype="text/event-stream")`. Run under a server that does not buffer responses excessively. For auth, use cookies on same origin or tokens in query string with care (logs may capture them).

```python
from flask import Response, stream_with_context
import time

def event_stream():
    for i in range(3):
        yield f"data: {i}\n\n"
        time.sleep(1)

@app.get("/events")
def sse():
    return Response(stream_with_context(event_stream()), mimetype="text/event-stream")
```

## Q84. How do you implement health checks in Flask?

## Q91. How do you implement the circuit breaker pattern in Flask?

**Answer:**

A circuit breaker stops calling failing dependencies after a threshold of errors, failing fast for a cooldown period before retrying. Libraries like `pybreaker` wrap client calls used in services. Expose breaker state in metrics for operators. Combine with timeouts on HTTP clients. In Flask, implement around outbound calls in the service layer, not in every view.

```python
import pybreaker
import requests

api_breaker = pybreaker.CircuitBreaker(fail_max=5, reset_timeout=60)

@api_breaker
def call_remote():
    resp = requests.get("https://example.com/api", timeout=5)
    resp.raise_for_status()
    return resp.json()
```

## Q99. How do you implement ETags and conditional requests in Flask?

**Answer:**

ETags identify specific versions of a resource; clients send `If-None-Match` with the ETag on subsequent GETs. Flask’s `make_response` combined with manual ETag headers or `send_file(..., etag=True)` can enable `304 Not Modified` responses. Strong ETags assume byte-identical responses; weak ETags (`W/`) suit semantically equivalent content. Conditional requests save bandwidth for large static or API payloads that rarely change. Prefer stable hashes (for example SHA-256 of bytes) rather than Python’s built-in `hash()`, which varies across processes. For dynamic JSON, sometimes a revision column or `updated_at` timestamp is a simpler ETag than hashing the entire body.

```python
import hashlib
from flask import request, make_response

def build_doc(doc_id: int) -> str:
    return f"Document body {doc_id}"

@app.get("/doc/<int:doc_id>")
def doc(doc_id):
    body = build_doc(doc_id)
    payload = body if isinstance(body, bytes) else body.encode("utf-8")
    etag = hashlib.sha256(payload).hexdigest()
    if etag in request.if_none_match:
        return "", 304
    resp = make_response(body)
    resp.set_etag(etag)
    return resp
```

## Q100. What are signals in Flask and how do you use them?

**Answer:**

Signals are Blinker-based publish/subscribe hooks: `request_started`, `request_finished`, `appcontext_teardown`, etc. Connect with `@request_finished.connect_via(app)` or `.connect()`. Use signals for cross-cutting observability (metrics), not core business logic, to avoid hidden coupling. They run synchronously in the same thread as the request. Test signal handlers by dispatching in test apps or mocking Blinker. Disconnect handlers in tests when they accumulate across app instances to prevent duplicate metric emissions. Exceptions in signal handlers can affect the request lifecycle depending on Flask version and configuration, so keep handlers defensive. Prefer explicit hooks in small codebases if signals obscure control flow for your team.

```python
from flask import request_finished

def log_response(sender, response, **extra):
    sender.logger.info("status=%s", response.status_code)

def register_signals(app):
    request_finished.connect(log_response, app)
```
