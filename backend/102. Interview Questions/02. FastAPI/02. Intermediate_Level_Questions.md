### Q1. Explain async/await in FastAPI and when to use async def vs def?

**Answer:**

FastAPI runs on Starlette/ASGI, which is built around asyncio’s event loop. When you declare a route with `async def`, FastAPI awaits your coroutine directly on the loop, so you can use `await` for non-blocking I/O (HTTP clients, async DB drivers, Redis). When you use plain `def`, FastAPI runs the function in a thread pool so it does not block the event loop—use this for CPU-bound work that releases the GIL poorly, or legacy sync libraries (e.g. `requests`, sync SQLAlchemy) that would otherwise block all concurrent requests. Mixing styles is fine per endpoint; the rule of thumb is: async endpoint + async I/O stack, or sync endpoint + blocking libraries offloaded to threads.

```python
from fastapi import FastAPI
import httpx

app = FastAPI()

@app.get("/async-example")
async def read_async():
    async with httpx.AsyncClient() as client:
        r = await client.get("https://example.com")
    return {"status": r.status_code}

@app.get("/sync-example")
def read_sync():
    import requests
    r = requests.get("https://example.com", timeout=10)
    return {"status": r.status_code}
```

### Q2. How does FastAPI handle blocking I/O operations in async endpoints?

**Answer:**

If you put blocking I/O (disk, sync HTTP, sync DB) inside an `async def` route without offloading it, it runs on the event loop thread and stalls every other request until the call finishes. FastAPI does not magically make blocking calls concurrent; you must either switch to async libraries and `await`, or run blocking code in a thread pool using `asyncio.to_thread()` (Python 3.9+) or `loop.run_in_executor()`. For CPU-heavy work, prefer `ProcessPoolExecutor` or external workers (Celery) instead of threads.

```python
import asyncio
from fastapi import FastAPI

app = FastAPI()

def blocking_io():
    with open("/etc/hosts") as f:
        return f.read()

@app.get("/safe-blocking")
async def safe():
    data = await asyncio.to_thread(blocking_io)
    return {"len": len(data)}
```

### Q3. What is the difference between concurrency and parallelism in FastAPI?

**Answer:**

Concurrency means many tasks are *in flight* and can make progress without each finishing before the next starts—on one CPU core, the event loop interleaves awaits so thousands of idle-on-network requests coexist cheaply. Parallelism means work runs *at the same physical time* on multiple CPU cores. FastAPI’s default deployment (one Uvicorn worker, one asyncio loop) gives high concurrency for I/O-bound handlers but not parallel CPU for pure Python. You add parallelism by running multiple worker processes, using thread pools for blocking sync routes, or process pools / external workers for CPU-heavy jobs. Confusing the two leads to putting CPU-bound loops in `async def` routes expecting “parallelism,” which actually blocks the whole loop. Instrument with latency under load: concurrent I/O scales until DB/CPU saturates; parallel CPU needs the right executor or more processes.

### Q4. How does FastAPI use threadpool for sync endpoints?

**Answer:**

When your route is declared with plain `def`, Starlette does not call it on the main asyncio thread. Instead it wraps execution in `anyio.to_thread.run_sync` (historically `run_in_threadpool`), which submits the callable to a worker thread from a bounded pool. That way synchronous ORM calls or `requests.get` do not freeze the event loop for other *async* routes—though they still consume thread-pool slots. Pool size and behavior follow AnyIO’s defaults (typically `min(32, os.cpu_count()+4)` for the default asyncio backend—verify for your version). If every route is sync and blocks for hundreds of milliseconds, threads become the bottleneck and you should add Uvicorn workers or migrate hot paths to async drivers. Observability: thread-pool starvation shows up as rising latency while CPU stays moderate.

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/sync")
def read_sync():
    # Runs in a thread pool worker, not on the event loop thread.
    return {"mode": "threadpool"}
```

### Q5. How to run CPU-bound tasks without blocking the event loop?

**Answer:**

Do not run heavy pure-Python CPU work on the asyncio loop. Options: (1) `ProcessPoolExecutor` for CPU-bound tasks; (2) offload to Celery/RQ with a broker; (3) push work to a compiled extension or Rust service. `asyncio.to_thread` helps blocking I/O in threads but does not bypass the GIL for CPU-bound Python.

```python
import asyncio
from concurrent.futures import ProcessPoolExecutor
from fastapi import FastAPI

app = FastAPI()
executor = ProcessPoolExecutor(max_workers=2)

def heavy(n: int) -> int:
    return sum(i * i for i in range(n))

@app.get("/cpu")
async def cpu(n: int = 10_000_000):
    loop = asyncio.get_running_loop()
    result = await loop.run_in_executor(executor, heavy, n)
    return {"result": result}
```

### Q6. What is asyncio.gather() and how to use it in FastAPI?

**Answer:**

`asyncio.gather(*aws, return_exceptions=True)` schedules multiple awaitables concurrently and waits for all to finish. In FastAPI it is the idiomatic way to parallelize independent async I/O (e.g. fan-out to several microservices or DB queries). Use `return_exceptions=True` to avoid one failure cancelling the whole batch; handle exceptions per result.

```python
import asyncio
import httpx
from fastapi import FastAPI

app = FastAPI()

@app.get("/aggregate")
async def aggregate():
    async with httpx.AsyncClient() as client:
        results = await asyncio.gather(
            client.get("https://api.a/data"),
            client.get("https://api.b/data"),
            return_exceptions=True,
        )
    return {"a": results[0], "b": results[1]}
```

### Q7. How to implement background tasks in FastAPI?

**Answer:**

Use `BackgroundTasks` from FastAPI for work that should run after the response is sent (e.g. send email, write audit log). Tasks run in the same process; they are not durable queues. For reliability at scale, use a real task queue. Pass callables with arguments; FastAPI will execute them after the response.

```python
from fastapi import BackgroundTasks, FastAPI

app = FastAPI()

def notify(user_id: int, msg: str):
    ...

@app.post("/signup")
async def signup(bg: BackgroundTasks, user_id: int):
    bg.add_task(notify, user_id, "welcome")
    return {"ok": True}
```

### Q8. What is BackgroundTasks and how does it work?

**Answer:**

`BackgroundTasks` is injected like any other dependency (`BackgroundTasks` type annotation). You call `add_task(fn, *args, **kwargs)` to queue work that must run *after* the response body is ready to send—useful for emails, analytics, cache invalidation. Starlette stores tasks on the request scope and executes them in order once the route coroutine completes successfully enough to produce a response. Async callables are awaited; sync callables are offloaded to the thread pool. Exceptions in background tasks do not roll back the HTTP response (the client may already have received 200), so you need explicit logging, retries, or a durable queue for critical side effects. Tasks are not distributed across processes; they die with the worker. For idempotency, design task payloads so retries are safe.

### Q9. How to use asyncio.create_task() in FastAPI?

**Answer:**

`asyncio.create_task(coro())` schedules a coroutine as a Task on the current loop immediately; unlike `BackgroundTasks`, it is not tied to response lifecycle. Use it for fire-and-forget work that must run concurrently with the request, but you must handle exceptions (e.g. `task.add_done_callback`) and avoid referencing request-scoped objects after the request ends unless cloned.

```python
import asyncio
from fastapi import FastAPI

app = FastAPI()

async def side_effect():
    await asyncio.sleep(1)

@app.get("/fire")
async def fire():
    asyncio.create_task(side_effect())
    return {"status": "accepted"}
```

### Q10. What happens when you use time.sleep() vs asyncio.sleep() in async endpoints?

**Answer:**

`time.sleep(n)` blocks the entire thread—including the event loop—so no other coroutines run for `n` seconds. `await asyncio.sleep(n)` yields control back to the loop, allowing other requests to progress. Never call `time.sleep` in `async def` routes unless wrapped in `asyncio.to_thread`.

```python
import asyncio
import time
from fastapi import FastAPI

app = FastAPI()

@app.get("/bad")
async def bad():
    time.sleep(2)  # blocks whole server in this worker
    return {"ok": False}

@app.get("/good")
async def good():
    await asyncio.sleep(2)
    return {"ok": True}
```

### Q11. How to handle concurrent database connections in async FastAPI?

**Answer:**

Use an async driver (asyncpg, aiosqlite) with SQLAlchemy 2 async engine or databases like Tortoise/Beanie. Create one engine/pool at startup (lifespan), not per request. Sessions should be short-lived per request via dependencies with `async with session` or equivalent. Bound pool size to expected concurrency; too many connections overwhelm the DB.

```python
DATABASE_URL = "postgresql+asyncpg://user:pass@localhost/db"
engine = create_async_engine(DATABASE_URL, pool_size=20, max_overflow=10)
SessionLocal = async_sessionmaker(engine, expire_on_commit=False, class_=AsyncSession)
```

### Q12. What is the event loop in FastAPI context?

**Answer:**

The event loop is asyncio’s scheduler: it runs a single thread that executes coroutines until they hit an `await`, at which point control returns so other ready tasks can run. Uvicorn (with asyncio loop) creates one loop per worker process; all `async def` endpoints and middleware chains share it. Blocking calls on that thread—`time.sleep`, sync file I/O, heavy CPU—prevent the loop from polling sockets or waking other coroutines, so tail latency for *all* requests on that worker spikes. Libraries like `httpx.AsyncClient` and async database drivers integrate with the loop via selectors (`epoll`/`kqueue`). When debugging “hangs,” check for accidental blocking and for deadlocks between `await` and locks held across I/O.

### Q13. How to use async context managers in FastAPI?

**Answer:**

Use `async with` for resources that support async setup/teardown (HTTP clients, async DB sessions, locks). In dependencies, `yield` inside `async def` pairs naturally with async context managers for request-scoped cleanup.

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup
    yield
    # shutdown

app = FastAPI(lifespan=lifespan)
```

### Q14. How to implement async generators in FastAPI?

**Answer:**

An async generator is an object defined with `async def` that uses `yield`; iterating it produces values across `await` points, which fits streaming HTTP bodies where data arrives over time (logs, ML tokens, large exports). `StreamingResponse` accepts an async iterator and sends chunks as they are produced, which keeps memory bounded compared to building a giant `bytes` in memory. You can also use `EventSourceResponse` (via `sse-starlette`) for Server-Sent Events with proper `text/event-stream` framing. Be careful with exception handling: a failure mid-stream may leave clients with partial data unless you document abort semantics. For very large streams, consider chunk size and timeouts at the reverse proxy.

```python
import asyncio
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

async def byte_stream():
    for i in range(5):
        yield f"chunk-{i}\n".encode()
        await asyncio.sleep(0.1)

@app.get("/stream")
async def stream():
    return StreamingResponse(byte_stream(), media_type="text/plain")
```

### Q15. What are the common pitfalls of async programming in FastAPI?

**Answer:**

Experienced teams still hit: (1) using sync libraries (`requests`, blocking boto3) inside `async def` without `to_thread`; (2) forgetting `await` so coroutines never run; (3) shared global mutable caches updated from concurrent requests without locks; (4) assuming `gather` parallelizes work that still uses one connection or one lock; (5) fire-and-forget `create_task` that raises silently; (6) long-lived transactions held across awaits, increasing deadlock risk; (7) `@lru_cache` on async functions (wrong—cache sync wrappers instead); (8) testing only with `TestClient` synchronously and missing async integration bugs. Mitigation: linters/runtimes that warn on blocking calls, structured logging, explicit timeouts on external calls, and load tests that mirror production concurrency.

### Q16. How to implement dependency injection with database sessions?

**Answer:**

Define an async generator dependency that yields a session and closes it in `finally`, or uses SQLAlchemy’s `async_sessionmaker` pattern. Inject `Depends(get_db)` into routes. This keeps session lifecycle tied to the request and avoids leaks.

```python
from typing import AsyncGenerator
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession

async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with SessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()

async def read_items(db: AsyncSession = Depends(get_db)):
    ...
```

### Q17. What are yield dependencies and how do they handle cleanup?

**Answer:**

Dependencies that `yield` run code before `yield` as setup and code after `yield` as teardown (similar to context managers). FastAPI guarantees teardown runs after the request, even on exceptions—use `try/finally` inside the generator for guaranteed resource release.

```python
async def get_resource():
    res = await open_resource()
    try:
        yield res
    finally:
        await res.close()
```

### Q18. How to create parameterized dependencies?

**Answer:**

You need a dependency that behaves differently per route (e.g. required role, pagination defaults). Patterns: (1) a class whose `__init__` takes parameters—FastAPI treats the class as callable if you use `Depends()` with a factory; (2) a function `make_dep(param: str)` returning an inner async function (less common); (3) `Annotated` + `Depends` with a subclass of `APIKeyHeader` per tenant. The cleanest idiom is a class with stored config and an async `__call__` or method referenced via `Depends`. FastAPI resolves `Depends(MyClass("admin"))` by inspecting the callable’s signature for sub-dependencies.

```python
from fastapi import Depends, FastAPI, HTTPException

app = FastAPI()
# Assume User model and async def get_current_user(...) are defined elsewhere.

class RoleChecker:
    def __init__(self, allowed: str):
        self.allowed = allowed

    async def __call__(self, user=Depends(get_current_user)):
        if user.role != self.allowed:
            raise HTTPException(status_code=403)
        return user

@app.get("/admin")
async def admin_only(user=Depends(RoleChecker("admin"))):
    return {"ok": True, "user_id": user.id}
```

### Q20. How to chain multiple dependencies?

**Answer:**

Declare multiple parameters each with `Depends()`. FastAPI builds a dependency graph, caches per-request results for the same dependency, and resolves in topological order. Sub-dependencies can depend on other dependencies.

```python
async def get_current_user(token: str = Depends(oauth2_scheme)):
    ...

async def require_admin(user=Depends(get_current_user)):
    if not user.is_admin:
        raise HTTPException(403)
    return user

@app.get("/admin")
async def admin_panel(user=Depends(require_admin)):
    ...
```

### Q21. What is the scope of dependencies in FastAPI (request-scoped vs singleton)?

**Answer:**

FastAPI does not use a DI container with named scopes like some enterprise frameworks. Each time a route needs `Depends(foo)`, `foo` is invoked according to the dependency graph, with caching *within the same request* for identical dependency callables (same object identity). Two different routes do not share a cached dependency result unless you put shared state on `app.state`, module-level singletons, or a connection pool. That means database sessions should be created per request via yield dependencies, not stored globally. “Singleton” resources—Redis clients, engines—belong in lifespan and `request.app.state`. Anti-pattern: a global `db_session` variable reused across requests, which breaks transaction isolation and is not thread/async safe.

### Q22. How to use dependencies for authentication and authorization?

**Answer:**

**Authentication** answers “who are you?”—dependencies parse bearer tokens, session cookies, or API keys, validate signatures/expiry, and load a `User` record (or service principal). **Authorization** answers “what may you do?”—separate dependencies check roles, permissions, resource ownership, or call policy engines. Layer them: `get_current_user` (401 if missing/invalid) then `require_admin` (403 if wrong role). Benefits: policies live in one place, OpenAPI can document security, and tests override only auth deps. Avoid querying the DB twice per request by caching the user on `request.state` if multiple deps need it. For microservices, a gateway may terminate JWT validation and forward user headers—still validate trust boundaries.

```python
from fastapi import Depends, HTTPException

async def get_current_user(token: str = Depends(oauth2)) -> User:
    user = await verify_token(token)
    if not user:
        raise HTTPException(status_code=401, detail="Invalid credentials")
    return user
```

### Q23. How to create reusable dependency classes with __call__?

**Answer:**

A class instance can be used with `Depends()` if it implements `__call__` with the same signature FastAPI needs, or use class methods. Common pattern: `PermissionChecker("items:read")` returning a callable.

```python
class PermissionChecker:
    def __init__(self, perm: str):
        self.perm = perm

    async def __call__(self, user: User = Depends(get_current_user)):
        if self.perm not in user.permissions:
            raise HTTPException(403)

@app.get("/x", dependencies=[Depends(PermissionChecker("read"))])
async def x():
    return {}
```

### Q24. How to handle dependency injection with async dependencies?

**Answer:**

Declare dependencies as `async def` whenever they perform I/O that has an async API (database, Redis, HTTP to auth service). FastAPI awaits them directly on the event loop. If you mix sync and async in one chain, sync `def` dependencies may run in a thread pool, which adds overhead and can obscure blocking behavior. Keep dependency trees shallow and avoid awaiting inside tight loops. For async generators (`yield`), the teardown after `yield` still runs in an async context—use `await` in `finally` for closing connections. Type hints should reflect return types (`AsyncSession`, not `Session`) so callers know they must await repository methods.

### Q25. How to use dependencies at application level vs router level?

**Answer:**

`FastAPI(dependencies=[Depends(...)])` applies to all routes on that app. `APIRouter(dependencies=[...])` applies to all routes mounted on that router. Route-level `dependencies=[]` adds more. Useful for global auth logging or per-module RBAC.

```python
router = APIRouter(prefix="/api", dependencies=[Depends(common_headers)])
```

### Q28. How to handle errors in dependencies properly?

**Answer:**

Dependencies should fail fast with `HTTPException` for authn/authz and validation-style problems so the route never runs with invalid context. For infrastructure errors (DB unreachable), either let the exception propagate to a global handler that returns 503 with a safe body, or wrap and map to a typed app exception. In yield dependencies, code after `yield` runs during teardown—even when the route raised—so put rollback/cleanup in `try/finally`. Log structured errors with correlation IDs from middleware; never return stack traces to clients in production. If a dependency catches an exception, re-raise or convert deliberately; swallowing hides bugs. For async cleanup failures, log at ERROR because they may leave partial state.

### Q29. How to integrate SQLAlchemy with FastAPI?

**Answer:**

Create engine and `sessionmaker`, inject sessions via dependencies, use declarative models, and commit/rollback per request. For sync SQLAlchemy use thread pool–compatible sessions; for async stack use 2.0 style async engine. Keep model definitions separate from routes (repository/service layers).

```python
from sqlalchemy.orm import sessionmaker, Session
from sqlalchemy import create_engine

engine = create_engine(DATABASE_URL, pool_pre_ping=True)
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

### Q31. What is the repository pattern and how to implement it in FastAPI?

**Answer:**

Repositories encapsulate data access (CRUD queries) behind an interface; services contain business logic; routes orchestrate. This simplifies testing (mock repos) and keeps SQL out of controllers. In Python, a `UserRepository` class taking `AsyncSession` in `__init__` is typical.

```python
class UserRepository:
    def __init__(self, db: AsyncSession):
        self.db = db

    async def get_by_id(self, user_id: int) -> User | None:
        return await self.db.get(User, user_id)
```

### Q32. How to handle database migrations with Alembic in FastAPI?

**Answer:**

Alembic tracks schema revision history in your database (`alembic_version` table). After `alembic init`, configure `env.py` to import your SQLAlchemy `Base.metadata` so `autogenerate` can diff models against the DB. Workflow: change models → `alembic revision --autogenerate -m "add users table"` → review the generated script (autogenerate is not perfect) → `alembic upgrade head` in CI/staging then production. For async SQLAlchemy, use async Alembic templates or run migrations synchronously with a sync URL. In Kubernetes, run migrations as an init Job or pre-deploy hook, not inside every pod’s startup, to avoid races. Keep migrations backward compatible when doing blue/green deploys (expand/contract pattern).

### Q33. How to implement CRUD operations with SQLAlchemy?

**Answer:**

Use session `add`, `delete`, `merge`, and queries via `select()`. For async, `await session.execute(select(Model).where(...))`, `await session.commit()`. Return Pydantic schemas, not ORM objects directly, using `model_validate` or `from_attributes=True`.

```python
async def create_user(db: AsyncSession, data: UserCreate) -> User:
    user = User(**data.model_dump())
    db.add(user)
    await db.commit()
    await db.refresh(user)
    return user
```

### Q34. How to use database sessions properly with dependency injection?

**Answer:**

Bind session lifetime to the HTTP request: open at the start (or first use), commit or rollback once at the boundary, close in `finally`. Never stash `AsyncSession` on `request.state` across middleware that might spawn tasks without clear ownership. Avoid long transactions that span multiple external API calls—split into shorter transactions or use sagas. For read-only endpoints, some teams use `session.begin_nested()` or explicit read-only flags depending on the DB. If you use lazy loading, remember async sessions may require explicit `await session.refresh()` after commit when `expire_on_commit=True`. Connection leaks often come from exceptions before `close()`—yield dependencies handle this if written carefully.

### Q35. What is SQLModel and how does it differ from SQLAlchemy + Pydantic?

**Answer:**

SQLModel builds on SQLAlchemy 2 and Pydantic v2: one class can define table columns (`Field(sa_column=...)`) and request/response shapes, reducing duplicate field definitions. It fits CRUD-heavy services and rapid prototyping. Compared to **SQLAlchemy + Pydantic separately**, you gain less mapping code but may blur layers—DTOs often diverge from persistence (e.g. `password_hash` never exposed). Advanced SQLAlchemy (complex relationships, custom loaders, multiple schemas) sometimes maps more cleanly with explicit ORM models. Migration tooling (Alembic) works the same because underneath it is SQLAlchemy metadata. Teams choosing SQLModel should still use repositories/services to avoid business logic leaking into route handlers.

### Q36. How to implement database connection pooling?

**Answer:**

SQLAlchemy’s `QueuePool` keeps a pool of open DB connections to avoid TCP+auth overhead per request. Key settings: `pool_size` (baseline), `max_overflow` (burst), `pool_timeout` (wait for a connection), `pool_recycle` (avoid stale connections behind firewalls), `pool_pre_ping` (validate before use). For async engines, semantics align but use the async dialect. Capacity planning: if you run 4 Uvicorn workers each with `pool_size=5` and `max_overflow=5`, you might need up to 40 connections at peak—must stay under Postgres `max_connections` minus admin overhead. Serverless or high-churn clients sometimes prefer `NullPool` or small pools with external poolers (PgBouncer). Monitor “pool exhausted” errors and queue time.

### Q37. How to handle database transactions in FastAPI?

**Answer:**

Wrap logical units in `async with session.begin()` or explicit `try/commit/except/rollback`. Nested transactions use savepoints. For multiple repositories sharing one transaction, pass the same session and commit once at the route or unit-of-work service boundary.

```python
async with session.begin():
    await repo_a.step()
    await repo_b.step()
```

### Q38. How to use multiple databases in FastAPI?

**Answer:**

Declare separate `create_async_engine` instances per database URL—e.g. primary OLTP and read replica, or operational vs reporting warehouse. Expose `get_session_primary` and `get_session_replica` dependencies; services accept the session(s) they need. Use distinct `Base` declarative metadata per bounded context if schemas differ completely, or one `Base` with `__table_args__ = {"schema": "analytics"}`. Be explicit in tests: which DB gets migrations, which is SQLite. For read-your-writes consistency, avoid routing the same user’s write-then-read to different replicas until replication lag is acceptable. Configuration via settings objects keeps URLs out of route code.

### Q40. How to use async database drivers (asyncpg, aiosqlite)?

**Answer:**

`asyncpg` is a high-performance protocol implementation for PostgreSQL used by SQLAlchemy’s async dialect; it speaks PostgreSQL’s binary protocol without blocking the event loop. `aiosqlite` wraps SQLite for async usage—fine for tests and small apps, not for high-concurrency production write loads. For managed clouds (RDS, Cloud SQL), enable SSL parameters in `connect_args`. When you need features asyncpg does not expose through SQLAlchemy, you can drop to raw connections from `engine.connect()` for specific queries—keep those paths small and well-tested.

### Q41. How to implement soft deletes in FastAPI?

**Answer:**

Soft delete preserves rows for audit and recovery while hiding them from normal queries. Add `deleted_at: datetime | None` or `is_deleted: bool` with partial indexes for performance. Every `select` path must include the filter—ORM events or a custom query API can reduce mistakes. Unique constraints often need partial uniqueness (`WHERE deleted_at IS NULL`) so a new row can reuse a “deleted” business key. FastAPI layer: DELETE endpoints set `deleted_at` instead of removing rows; GET by id returns 404 if deleted unless admin query param. Background jobs can purge rows after retention. Be careful with foreign keys: soft-deleting a parent may leave orphans unless you cascade soft-delete or use ON DELETE SET NULL.

### Q45. What is OAuth2PasswordBearer and how does it work?

**Answer:**

`OAuth2PasswordBearer` subclasses OpenAPI’s OAuth2 security scheme: constructor args include `tokenUrl` (relative path shown in Swagger as where to obtain tokens) and optional `scopes`. The Swagger UI “Authorize” dialog uses `tokenUrl` to know where to POST `application/x-www-form-urlencoded` username/password for the password flow; your `/token` route must exist separately. In production, prefer short-lived JWTs and rotate secrets; document that OAuth2PasswordBearer is only transport/extraction, not full authz.

### Q49. How to implement refresh token mechanism?

**Answer:**

Access tokens stay short (minutes) to limit exposure; refresh tokens are long-lived but revocable. Store only a hash of the refresh token in DB (like passwords) keyed by device/session id. On `/refresh`, validate signature/expiry, load user, check not revoked, then issue new access token and optionally rotate refresh token (detect reuse: if an old refresh token appears twice, revoke all sessions for that user). HttpOnly `Secure` `SameSite` cookies reduce XSS stealing refresh tokens compared to localStorage. For SPAs, consider BFF pattern or backend-for-frontend token exchange. Logout deletes the refresh row; password change invalidates all refresh tokens for the account. Rate-limit `/refresh` to slow brute force.

```python
from fastapi import Body, FastAPI, HTTPException

app = FastAPI()

@app.post("/refresh")
async def refresh(refresh_token: str = Body(..., embed=True)):
    row = await validate_refresh_token_hash(refresh_token)
    if not row:
        raise HTTPException(401)
    new_access = create_access_token(row.user_id)
    new_refresh = rotate_refresh_token(row)
    return {"access_token": new_access, "refresh_token": new_refresh}
```

### Q50. How to hash and verify passwords using passlib?

**Answer:**

Use bcrypt or argon2 via passlib. Hash on registration/login verification only; never store plaintext. Constant-time compare avoids timing leaks.

```python
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

def hash_password(p: str) -> str:
    return pwd_context.hash(p)

def verify_password(p: str, hashed: str) -> bool:
    return pwd_context.verify(p, hashed)
```

### Q54. How to prevent common security vulnerabilities (XSS, CSRF, SQL injection) in FastAPI?

**Answer:**

**SQL injection**: use SQLAlchemy Core/ORM with bound parameters; never `f"SELECT ... {user_input}"`. Raw SQL should use `:name` binds. **XSS**: JSON REST responses are not HTML—risk moves to SPAs consuming `dangerouslySetInnerHTML`; if you render Jinja2, autoescape and CSP `default-src 'self'`. **CSRF**: attacks trick browsers into sending cookies; mitigate with `SameSite=Lax/Strict`, CSRF tokens for cookie sessions, or avoid cookies for APIs (Bearer header in JS memory has XSS trade-offs). **SSRF**, **path traversal** on file APIs, and **mass assignment** (never `model_dump()` straight into ORM without a schema) are adjacent FastAPI concerns. Pydantic v2 validates types/ranges; combine with explicit allowlists for enums and string patterns.

### Q57. How to implement IP whitelisting in FastAPI?

**Answer:**

Raw `request.client.host` is the immediate TCP peer—often 127.0.0.1 behind a reverse proxy, so you must use `ProxyHeadersMiddleware` or read `X-Forwarded-For` **only** when you trust the proxy chain (otherwise clients spoof IPs). Parse the left-most untrusted hop carefully; use `ipaddress` module for CIDR checks (`IPv4Network`). Combine allowlists with VPN or private network ranges for internal admin APIs. Return 403 without body detail to reduce reconnaissance. Prefer enforcing at load balancer/ingress for efficiency; duplicate in app for defense in depth. IPv6 requires normalized comparison.

```python
import ipaddress
from fastapi import Request
from fastapi.responses import JSONResponse

ALLOWED = {ipaddress.ip_network("10.0.0.0/8")}

@app.middleware("http")
async def ip_whitelist(request: Request, call_next):
    peer = ipaddress.ip_address(request.client.host)
    if not any(peer in net for net in ALLOWED):
        return JSONResponse({"detail": "forbidden"}, 403)
    return await call_next(request)
```

### Q58. How to secure WebSocket connections?

**Answer:**

The WebSocket handshake is an HTTP Upgrade request: you can reject before `accept()` based on cookies, query `token`, or subprotocol negotiation. Browsers historically could not attach arbitrary headers to `new WebSocket()`, so JWT-in-query or cookie sessions are common—treat query tokens as secrets in logs (redact). Always use `wss://` with valid TLS in production; mixed content is blocked. After `accept()`, validate every message’s size and schema to prevent DoS (memory bombs, huge JSON). Apply per-connection and per-IP rate limits at the gateway. Map authenticated user id to connection registry; on token revocation, force-close with policy-defined codes. Consider periodic server pings and idle timeouts.

```python
@app.websocket("/ws")
async def ws(websocket: WebSocket, token: str | None = None):
    if not token or not verify_jwt(token):
        await websocket.close(code=4401)
        return
    await websocket.accept()
```

### Q71. How to implement WebSocket endpoints in FastAPI?

### Q76. What is the difference between HTTP and WebSocket in FastAPI?

**Answer:**

HTTP/REST in FastAPI maps cleanly to caching layers, CDN, retries, and OpenAPI documentation—each request is independent and idempotency is well understood. WebSockets upgrade a single TCP connection to full-duplex binary/text frames: lower per-message overhead for high-frequency updates, but no standard HTTP caching, harder load balancing (sticky sessions or shared pub/sub), and stateful connection management on the server. Use HTTP for CRUD and WebSockets when the server must push many updates without client polling. Hybrid designs (SSE for one-way server push over HTTP) are simpler operationally when you do not need client→server streaming.

### Q79. How to implement request logging middleware?

**Answer:**

Log method, path, client IP, user id from scope, latency from `time.perf_counter()`. Use structured JSON logging; avoid logging secrets/PII. Correlate with `X-Request-ID` generated or propagated.

```python
import time

async def access_log(request: Request, call_next):

    response = await call_next(request)
    log.info(
        "%s %s %s %.2fms",
        request.method,
        request.url.path,
        response.status_code,
        (time.perf_counter() - start) * 1000,
    )
    return response
```

### Q81. How to implement request rate limiting middleware?

**Answer:**

Middleware runs early: extract client identity (IP from trusted forwarded chain, API key, JWT `sub`), build Redis key `rl:{bucket}:{id}`, increment with `INCR` + `EXPIRE` for fixed window, or use Lua for sliding window atomicity. If over limit, return `JSONResponse(status_code=429, headers={"Retry-After": "60"})` **without** calling `call_next` to save CPU. Skip health checks and metrics paths via path prefix. Combine with per-route weights (expensive endpoints lower quota). Distributed systems need Redis; in-memory dicts only work single-process. Watch memory if storing many keys—TTL aggressively.

### Q82. How to measure request processing time with middleware?

**Answer:**

Use `time.perf_counter()` (monotonic, high resolution) immediately before `await call_next(request)` and immediately after—this measures **server processing until response start** for many response types, not necessarily full body download for streaming responses. For end-to-end including consumer reading body, you need lower-level hooks. Add `X-Process-Time: 0.042` header in seconds or milliseconds consistently. Log alongside `request.method`, path template (if available), status code. Sample high-cardinality paths in production to control log volume. Correlate with DB query timing from ORM listeners for deep dives.

### Q85. How to handle exceptions in middleware?

**Answer:**

Middleware sits outside route-level `HTTPException` handling unless you delegate. Pattern: `try: return await call_next(request) except Exception: log.exception(...); return JSONResponse(500, ...)` for outer catch-all—use sparingly to avoid masking bugs. Often you **re-raise** after logging so Starlette’s `exception_handler` converts to consistent JSON. Ordering: exception handlers registered on `app` apply to raised exceptions from routes; middleware try/except runs in middleware order. For `HTTPException`, you may convert to response directly. Never leak tracebacks to clients; include `request_id` in JSON body for support correlation.

### Q87. How to implement the service layer pattern?

**Answer:**

Services encapsulate business rules, accept primitives or DTOs, call repositories. No FastAPI types in services for easier reuse from CLI/tasks. Raises domain exceptions mapped to HTTP in routers or exception handlers.

```python
class UserService:
    def __init__(self, repo: UserRepository):
        self.repo = repo

    async def register(self, data: UserCreate) -> User:
        if await self.repo.email_exists(data.email):
            raise ConflictError("email taken")
        return await self.repo.create(data)
```

### Q89. How to implement caching in FastAPI (Redis)?

**Answer:**

Use `redis.asyncio` client in dependencies or a decorator that builds cache keys from path+params. Set TTL; invalidate on writes. For HTTP caching, add `Cache-Control` headers. Watch stampede with locks or single-flight pattern.

```python
import json
import redis.asyncio as redis

r = redis.from_url("redis://localhost")

@app.get("/expensive")
async def expensive(q: str):
    key = f"exp:{q}"
    cached = await r.get(key)
    if cached:
        return json.loads(cached)
    data = await compute(q)
    await r.setex(key, 60, json.dumps(data))
    return data
```

### Q92. How to implement pagination with cursor-based and offset-based approaches?

**Answer:**

**Offset/limit** is trivial (`OFFSET (page-1)*size LIMIT size`) and works with arbitrary jumps, but PostgreSQL still scans skipped rows for large offsets—O(n) cost for deep pages—and concurrent inserts cause **duplicates/skips** between pages without snapshot isolation. **Keyset (cursor)** pagination uses a stable sort: `WHERE (sort_col, id) < ($cursor_sort, $cursor_id) ORDER BY sort_col DESC, id DESC LIMIT n+1` to detect `has_next`. Encode opaque cursor (base64url JSON) with fields you filter on; validate types. Expose `next_cursor` in response. Use offset for back-office tools with low page numbers; cursor for infinite-scroll mobile feeds and high-volume APIs.

```python
# Cursor idea (pseudo-SQLAlchemy)
# stmt = select(Item).where(tuple_(Item.created_at, Item.id) < cursor_tuple).order_by(...)
```

### Q98. How to use FastAPI events (startup/shutdown/lifespan)?

**Answer:**

Prefer `@asynccontextmanager` `lifespan` on `FastAPI(lifespan=...)` over deprecated `@app.on_event` for async resource init (engines, Redis). Yield separates startup/shutdown; ensures cleanup on SIGTERM.

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.redis = await redis.from_url(settings.redis_url)
    yield
    await app.state.redis.close()

app = FastAPI(lifespan=lifespan)
```

### Q100. How to generate OpenAPI schema customization?

**Answer:**

Set `app = FastAPI(title=..., version=..., openapi_tags=..., openapi_url=...)`. Customize `app.openapi` function to merge vendor extensions, add servers, security schemes. For field-level examples use Pydantic `json_schema_extra` and `Field(examples=[...])`.

```python
from fastapi import FastAPI
from fastapi.openapi.utils import get_openapi

app = FastAPI()

def custom_openapi():
    if app.openapi_schema:
        return app.openapi_schema
    schema = get_openapi(
        title=app.title,
        version=app.version,
        routes=app.routes,
    )
    schema["info"]["x-logo"] = {"url": "https://example.com/logo.png"}
    app.openapi_schema = schema
    return app.openapi_schema

app.openapi = custom_openapi
```
