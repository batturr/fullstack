### Q1. How would you architect a large-scale microservices system using FastAPI?

**Answer:**

Design starts with clear bounded contexts and ownership: each service exposes a narrow API contract (OpenAPI-first), owns its data store, and communicates asynchronously for cross-domain workflows. FastAPI fits as an edge-friendly ASGI service: place it behind an API gateway (Kong, Envoy, AWS API Gateway) for auth termination, rate limits, and routing; use a service mesh (Istio/Linkerd) for mTLS, retries, and traffic policies between internal services.

Inside each service, structure modules by domain—not by technical layer only—so that routers are thin, use cases/services encapsulate business rules, and repositories isolate persistence. Prefer async I/O end-to-end (async SQLAlchemy, async Redis, httpx.AsyncClient) to maximize throughput on I/O-bound workloads, but keep CPU-heavy work off the event loop (thread/process pool or separate workers).

Operational concerns drive the architecture: centralized structured logging with trace context, OpenTelemetry export, health/readiness endpoints, graceful shutdown hooks, and idempotent consumers for messaging. Version contracts (URL or header versioning), publish backward-compatible schemas, and run consumer-driven contract tests in CI. Scale horizontally with stateless pods, externalized session storage, and partitioned queues/topics to avoid single hot brokers.

```python
# Thin router delegating to application service (keeps HTTP layer dumb)
from fastapi import APIRouter, Depends

router = APIRouter(prefix="/orders", tags=["orders"])

def get_create_order_uc() -> "CreateOrderUseCase":
    ...

@router.post("", status_code=201)
async def create_order(body: OrderCreate, uc: CreateOrderUseCase = Depends(get_create_order_uc)):
    return await uc.execute(body)
```

### Q2. What design patterns are essential for production FastAPI applications?

**Answer:**

**Dependency Injection (FastAPI `Depends`)** is the backbone: bind interfaces to implementations per request scope (DB session, current user, feature flags). **Repository + Unit of Work** isolate persistence and transactions. **Factory/Provider** patterns register clients (HTTP, Kafka producer) with connection pooling and lifecycle hooks (startup/shutdown). **Strategy** swaps algorithms (payment providers, storage backends) without changing routers.

**Decorator/middleware patterns** implement cross-cutting concerns: request ID injection, auth, metrics, timeout enforcement. **Circuit breaker and retry** belong at integration boundaries (httpx, Redis clients), not scattered in route handlers. **Outbox** for reliable publishing and **Saga/Orchestration** for multi-step distributed workflows. **CQRS** where read models differ materially from writes.

Anti-patterns to avoid: fat routers with SQL, global mutable singletons without locks, and `async def` routes that call blocking libraries without `run_in_executor`. Production code favors explicit interfaces, typed DTOs (Pydantic v2), and clear failure modes (domain exceptions mapped to HTTP status).

```python
from typing import Annotated, AsyncGenerator
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker

async def get_session() -> AsyncGenerator[AsyncSession, None]:
    async with session_factory() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise

SessionDep = Annotated[AsyncSession, Depends(get_session)]
```

### Q3. How to implement Domain-Driven Design (DDD) with FastAPI?

**Answer:**

DDD separates **domain model** (entities, value objects, aggregates, domain services) from **application layer** (use cases, orchestration) and **infrastructure** (ORM mappings, messaging adapters). FastAPI sits in the **interface/adapter** layer: Pydantic models are *DTOs* for HTTP, not your domain entities—map between them explicitly to keep domain rules independent of serialization.

Define **aggregates** with invariants enforced in constructors/factories; repositories load/save aggregates by ID; avoid anemic models where validation only lives in Pydantic. **Domain events** represent facts that occurred inside a transaction; raise them from entities and dispatch after commit (via Outbox) to integrate bounded contexts.

Ubiquitous language should appear in module names, class names, and error messages surfaced to clients (without leaking internals). For complex domains, consider **specifications** for reusable query rules and **policies** for cross-aggregate coordination.

```python
# Domain entity (no FastAPI imports)
class Order:
    def __init__(self, order_id: str, customer_id: str):
        self.id = order_id
        self.customer_id = customer_id
        self._lines: list[OrderLine] = []
        self._events: list[DomainEvent] = []

    def add_line(self, sku: str, qty: int) -> None:
        if qty <= 0:
            raise DomainError("qty must be positive")
        self._lines.append(OrderLine(sku, qty))
        self._events.append(OrderLineAdded(self.id, sku, qty))
```

### Q4. How to implement Clean Architecture / Hexagonal Architecture in FastAPI?

**Answer:**

**Ports** are interfaces your domain/application depends on (e.g., `PaymentGateway`, `OrderRepository`). **Adapters** implement those ports: SQLAlchemy adapter, Stripe adapter, fake adapters for tests. FastAPI routers/controllers depend **inward** on application services/use cases, never on infrastructure directly—wire dependencies in composition root (`main.py` or DI container).

Organize packages as `domain/`, `application/`, `adapters/inbound/http/` (FastAPI), `adapters/outbound/persistence/`, `adapters/outbound/messaging/`. Configuration and framework glue live at the outermost ring. Testing becomes tractable: unit-test domain without HTTP; integration-test adapters; contract-test HTTP layer.

Hexagonal thinking also applies to **time, randomness, and IDs**: inject `Clock` and `IdGenerator` ports to make behavior deterministic in tests.

```python
from typing import Protocol

class OrderRepository(Protocol):
    async def get(self, order_id: str) -> Order | None: ...
    async def save(self, order: Order) -> None: ...

class CreateOrderHandler:
    def __init__(self, repo: OrderRepository):
        self._repo = repo

    async def handle(self, cmd: CreateOrderCommand) -> str:
        order = Order.new(cmd.customer_id)
        await self._repo.save(order)
        return order.id
```

### Q5. How to design a multi-tenant SaaS application with FastAPI?

**Answer:**

Choose a **tenant isolation model**: (1) **separate databases** per tenant (strongest isolation, higher ops cost), (2) **separate schemas** in one DB, or (3) **shared schema** with `tenant_id` on every row (most common at scale). The decision balances compliance (SOC2, HIPAA), noisy-neighbor risk, and migration complexity.

At the HTTP layer, resolve tenant from **subdomain**, **JWT claim**, or **header** early in middleware; reject missing/unknown tenants before business logic runs. Propagate `tenant_id` through `ContextVar` or explicit parameter objects—avoid thread-local in async code unless you know the implications.

Enforce tenant scoping in **one place**: repository base class or SQLAlchemy event that injects `WHERE tenant_id = :tid` for every query. Row-Level Security (PostgreSQL RLS) adds defense-in-depth against developer mistakes. Rate limits and quotas should be per-tenant; cache keys must include tenant.

```python
from contextvars import ContextVar

tenant_ctx: ContextVar[str] = ContextVar("tenant_id")

async def tenant_middleware(request: Request, call_next):
    tid = request.headers.get("X-Tenant-ID")
    if not tid:
        raise HTTPException(400, "missing tenant")
    token = tenant_ctx.set(tid)
    try:
        return await call_next(request)
    finally:
        tenant_ctx.reset(token)
```

### Q6. How to implement the CQRS pattern in FastAPI?

**Answer:**

**CQRS** splits **commands** (mutations, transactional consistency) from **queries** (read-optimized models). In FastAPI, expose separate route modules or even separate deployable read APIs when scale diverges. Commands validate invariants, emit domain events, and update write store; an **async projector** builds read models in Elasticsearch/Redis/SQL materialized views.

Avoid premature CQRS: use it when reporting/search needs differ radically from write schema or when you need independent scaling. **Eventual consistency** must be explicit in API docs (e.g., “search index updates within seconds”). For simple CRUD, a single model suffices.

Implementation sketch: command handlers return IDs/ack; query handlers return DTOs from read DB. Use Pydantic models per side (`OrderWriteCmd`, `OrderListItem`). Transaction boundaries wrap command pipeline only.

```python
@router.post("/orders", status_code=202)
async def create_order(cmd: CreateOrderCommand, bus: CommandBus = Depends()):
    order_id = await bus.dispatch(cmd)  # transactional write + outbox event
    return {"id": order_id}

@router.get("/orders/search")
async def search_orders(q: str, qh: OrderQueryService = Depends()):
    # hits read model / ES — eventually consistent
    return await qh.search(q)
```

### Q7. How to implement Event Sourcing with FastAPI?

**Answer:**

**Event sourcing** stores an append-only log of domain events as the system of record; aggregates are rebuilt by **replaying** events or snapshots. FastAPI endpoints append commands that translate to events; reads can be **projections** built from the stream. You need **versioning** for event schema evolution (upcasters), **idempotency** on command handling, and tooling for **replays** and debugging.

Infrastructure often uses Kafka, EventStoreDB, or relational `events` table with monotonic sequence. Snapshots prevent replaying thousands of events per request—store periodic snapshots + event offset.

Expose operational endpoints carefully: replays are admin-only and audited. Testing uses **given events → when command → then new events** style assertions.

```python
async def load_order(stream_id: str, store: EventStore) -> Order:
    snapshot = await store.load_snapshot(stream_id)
    from_seq = snapshot.version + 1 if snapshot else 0
    events = await store.read_stream(stream_id, from_sequence=from_seq)
    agg = snapshot.aggregate if snapshot else Order.empty(stream_id)
    for ev in events:
        agg = agg.apply(ev)
    return agg
```

### Q8. How to design an API gateway using FastAPI?

**Answer:**

An **API gateway** centralizes cross-cutting ingress: authentication (JWT/OAuth2 introspection), authorization coarse checks, rate limiting, request validation, routing to upstream services, response aggregation (BFF-style), and protocol translation (REST→gRPC). FastAPI can implement a **lightweight gateway** when traffic is moderate; at very high scale, prefer Envoy/Nginx + control plane.

Key design points: **stateless** gateway pods, **connection pooling** to upstreams (`httpx` limits), **timeouts and retries with jitter**, **bulkheads** (separate connection pools per upstream), and **circuit breakers** when upstream error rates spike. Propagate **W3C trace context** and **correlation IDs**.

Avoid embedding heavy business rules in the gateway—keep orchestration thin. Cache public keys/JWKS with TTL. Use OpenAPI for the gateway’s public contract even if upstreams differ.

```python
import httpx
from fastapi import APIRouter, Request

router = APIRouter()

@router.api_route("/proxy/{path:path}", methods=["GET", "POST", "PUT", "PATCH", "DELETE"])
async def proxy(path: str, request: Request, client: httpx.AsyncClient = Depends(get_upstream_client)):
    url = f"{settings.USERS_SERVICE_URL}/{path}"
    body = await request.body()
    resp = await client.request(request.method, url, content=body, headers=filter_hop_by_hop(request.headers))
    return Response(content=resp.content, status_code=resp.status_code, headers=dict(resp.headers))
```

### Q9. What is the Strangler Fig pattern and how to use it to migrate to FastAPI?

**Answer:**

The **Strangler Fig** incrementally replaces a legacy system by routing slices of traffic/functionality to a new implementation until the old system is “strangled.” For FastAPI migration, place a **facade** (proxy, API gateway, or reverse proxy rules) in front of the monolith; implement **one bounded context** at a time in FastAPI with compatible routes and schemas.

Techniques: **parallel run** (shadow traffic) comparing responses, **feature flags** to flip cohorts, **database strangler** (sync via CDC, dual writes with reconciliation, or read from old/write to new with careful consistency). Measure latency/error budgets continuously.

Risk mitigation includes contract tests, canary releases, and rollback paths. Teams should align on **definition of done** per slice: observability parity, SLOs, and data migration validation.

### Q10. How to implement the Saga pattern for distributed transactions?

**Answer:**

**Sagas** coordinate long-running business transactions across services without two-phase commit. **Choreography** (events only) is simpler early but hard to reason about; **orchestration** (central coordinator) fits complex flows with explicit state machines.

Each step must be **compensatable**: define undo operations (`reserve_inventory` ↔ `release_inventory`). FastAPI services expose idempotent step endpoints; orchestrator stores saga state (pending/completed/compensating) durably (DB) and retries with backoff.

Use **outbox** to emit “step completed” reliably; idempotency keys tie client retries to same saga instance. Observability: correlate all logs/metrics with `saga_id`.

```python
class SagaOrchestrator:
    async def run(self, cmd: PlaceOrderSaga) -> None:
        pay = await self._payments.authorize(cmd.payment, idempotency_key=cmd.id)
        try:
            ship = await self._shipping.schedule(cmd.shipment, idempotency_key=cmd.id)
        except Exception:
            await self._payments.cancel(pay.id, idempotency_key=f"{cmd.id}:cancel")
            raise
```

### Q11. How to design idempotent API endpoints?

**Answer:**

**Idempotency** means repeating the same request has the same effect as once (for HTTP: safe methods are naturally idempotent; POST needs keys). Accept **`Idempotency-Key`** header (Stripe-style) for mutating operations: hash `(tenant, key, route)` to a storage record storing response body/status or “in progress” lock with TTL.

Implementation uses **atomic compare-and-set** in Redis/DB: first request wins; duplicates return stored response **replay**. For payments, tie keys to external PSP references. Expire keys after legal/compliance windows.

Document client responsibilities: keys must be unique per logical operation, stable across retries. Server must handle concurrency (two parallel posts with same key).

```python
async def with_idempotency(key: str, fn):
    existing = await store.get(key)
    if existing:
        return existing
    async with distributed_lock(key):
        existing = await store.get(key)
        if existing:
            return existing
        result = await fn()
        await store.put(key, result, ttl=86400 * 7)
        return result
```

### Q12. How to implement circuit breaker pattern in FastAPI services?

**Answer:**

A **circuit breaker** stops calling failing dependencies after a threshold, failing fast and allowing recovery. States: **closed** (normal), **open** (reject calls), **half-open** (trial calls). Use libraries (`pybreaker`, Tenacity with circuit, or resilience4j-like wrappers for Python).

Apply at **client boundaries** (httpx session wrapper, DB pool health), not per-request randomly. Combine with **timeouts** and **bulkheads**. Emit metrics: state transitions, failure rates, rejected calls.

In FastAPI, inject a resilient client via `Depends`. On open circuit, return **503** with `Retry-After` if appropriate, or degraded responses if business allows.

```python
class ResilientClient:
    def __init__(self, inner: httpx.AsyncClient, cb: CircuitBreaker):
        self._inner = inner
        self._cb = cb

    async def get(self, url: str):
        return await self._cb.call(self._inner.get, url)
```

### Q13. How to handle eventual consistency in FastAPI microservices?

**Answer:**

**Eventual consistency** is unavoidable in distributed systems; APIs must **communicate uncertainty**: return **202 Accepted** with status URLs, expose **read-your-writes** where needed via sticky routing/session, or provide **version tokens** for clients to poll until visible.

Mitigations: **Saga compensations**, **idempotent consumers**, **at-least-once** delivery with dedupe keys, **outbox** for atomic write+publish. For UX, use **optimistic UI** with rollback paths or show “processing” states.

Testing requires **state models** and **jepsen-style** chaos (message reordering, duplicates). Document **SLAs** for convergence time on projections.

### Q14. How to implement the Outbox pattern for reliable event publishing?

**Answer:**

The **outbox** writes business data and an `outbox` row in the **same DB transaction**. A separate **relay process** polls or tails the log, publishes to Kafka/RabbitMQ, and marks rows published—achieving **exactly-once effect** relative to DB state (at-least-once to broker + idempotent consumers).

In FastAPI, the request handler only commits domain changes + outbox events; never publish directly in-request to avoid dual-write inconsistency. Scale relay horizontally with **SKIP LOCKED** row claiming.

Schema: `id, aggregate_id, type, payload(json), created_at, published_at`. Use **transactional inbox** on consumer side for dedupe.

```python
async def create_order(..., session: AsyncSession, bus: OutboxWriter):
    order = Order.new(...)
    session.add(order_orm(order))
    await bus.enqueue("OrderCreated", order.id, payload=order.to_event())
    await session.commit()  # atomic
```

### Q17. How to design API contracts and backward compatibility?

**Answer:**

Treat OpenAPI as the **contract**: semver for breaking vs additive changes. **Additive**: optional fields, new endpoints, new enum values (careful with clients that reject unknowns). **Breaking**: removing fields, changing types, tightening validation—require **version bump** (`/v2` or header negotiation).

Patterns: **expand-contract** for DB migrations; **parallel fields** during transition (`legacy_id` + `id`); **deprecation headers** with sunset dates. Consumer-driven **Pact** tests catch accidental breaks.

Never rename silently—use aliases in Pydantic `Field(alias="oldName")` during migration windows.

### Q18. How to implement the Backend for Frontend (BFF) pattern with FastAPI?

**Answer:**

A **BFF** serves a specific client (web/mobile) and aggregates calls to downstream microservices, shaping responses to UI needs. Implement as a FastAPI app per client team, with **typed clients**, **parallel `asyncio.gather`**, **timeouts per dependency**, and **caching** of stable fragments.

BFF is **not** a second monolith if boundaries stay client-specific; still avoid business rules duplication—delegate to domain services when logic is shared. Authentication often ends at BFF; service-to-service uses mTLS or internal JWT.

```python
@router.get("/dashboard")
async def dashboard(
    users: UsersClient = Depends(),
    orders: OrdersClient = Depends(),
):
    u, o = await asyncio.gather(users.me(), orders.recent(limit=5))
    return {"profile": u, "orders": o}
```

### Q19. How to profile and optimize FastAPI application performance?

**Answer:**

Start with **metrics**, not guesses: request latency histograms (p50/p95/p99), saturation (CPU, event loop lag), GC pauses, DB pool waits. Use **OpenTelemetry** + **py-spy** or **scalene** for CPU profiling; **memray** for memory. For async, measure **event loop blocked time**—often blocking calls in `async def`.

Optimize hot paths: Pydantic v2 model validation, ORM N+1 queries (`selectinload`), JSON serialization (`orjson`), and unnecessary middleware. Load test with **k6/Locust** with realistic payloads.

Use **profiling middleware** sparingly in staging. Consider **Cython/Rust** extensions only after profiling proves CPU-bound hotspots.

```python
@app.middleware("http")
async def add_server_timing(request: Request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    dur = (time.perf_counter() - start) * 1000
    response.headers["Server-Timing"] = f"app;dur={dur:.2f}"
    return response
```

### Q20. How to implement connection pooling strategies for high-throughput APIs?

**Answer:**

**HTTP**: single shared `httpx.AsyncClient` with limits (`max_connections`, `max_keepalive_connections`) created in lifespan—**never** per request.**DB**: SQLAlchemy `QueuePool` sized to `(expected_concurrency * avg_query_time / target_latency)` bounded by DB `max_connections` divided among replicas/pods.**Redis**: connection pools with health checks.

Tune **pool overflow**, **recycle** (avoid stale connections behind NAT/LB), and **pre-ping**. Monitor **pool checkout time**—if high, scale DB or reduce connection churn. For multi-tenant heavy workloads, **shard pools** or use **PgBouncer** in transaction mode cautiously with async ORMs.

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http = httpx.AsyncClient(
        limits=httpx.Limits(max_connections=100, max_keepalive_connections=20),
        timeout=httpx.Timeout(5.0),
    )
    yield
    await app.state.http.aclose()
```

### Q21. How to implement multi-level caching (L1/L2) in FastAPI?

**Answer:**

**L1**: in-process LRU/TTL for hot keys—fast but not shared across pods.**L2**: Redis/Memcached for cross-instance consistency.**CDN** for static/public GETs. Choose **cache aside**: read-through on miss, populate on write invalidation or TTL.

**Key design**: namespaced keys (`tenant:entity:id`), **TTL jitter** against thundering herd, **stale-while-revalidate** for tolerable staleness. **Never** cache user-specific data without auth in key.

Use **ETag/If-None-Match** for HTTP caching semantics on safe endpoints.

```python
async def get_product(pid: str, l1: TTLCache, redis: Redis):
    if (v := l1.get(pid)):
        return v
    if (v := await redis.get(f"p:{pid}")):
        l1.set(pid, json.loads(v))
        return json.loads(v)
    v = await db.load(pid)
    payload = json.dumps(v)
    await redis.setex(f"p:{pid}", 300, payload)
    l1.set(pid, v)
    return v
```

### Q23. How to implement request coalescing/deduplication?

**Answer:**

**Request coalescing** (singleflight) ensures only one backend computation runs for identical in-flight keys—others await the same future. Useful for cache stampedes on popular keys. Implement with **asyncio locks per key** or libraries; cap key cardinality to avoid memory blowups (LRU of in-flight futures).

Combine with **short-lived debouncing** for bursty identical admin refreshes. Document **failure propagation**: if leader fails, waiters should see same error.

```python
class SingleFlight:
    def __init__(self):
        self._inflight: dict[str, asyncio.Task] = {}
        self._lock = asyncio.Lock()

    async def do(self, key: str, factory):
        async with self._lock:
            if key in self._inflight:
                task = self._inflight[key]
            else:
                task = asyncio.create_task(factory())
                self._inflight[key] = task
        try:
            return await task
        finally:
            async with self._lock:
                self._inflight.pop(key, None)
```

### Q24. How to handle thundering herd problem in FastAPI caches?

**Answer:**

Thundering herd: many requests miss cache simultaneously and hammer origin.**Mitigations**: **probabilistic early expiration** (jitter TTL), **singleflight**, **mutex per key** in Redis (`SETNX` + short lock), **prefetch** on cron for known hot keys, **stale-while-revalidate** serving slightly old data while async refresh.

At edge, **CDN shielding** absorbs bursts. Monitor **origin concurrency** spikes correlating with TTL alignment—fix by spreading TTL baselines.

### Q25. How to implement async streaming for large datasets?

**Answer:**

Use **streaming responses** to avoid buffering entire payloads—`StreamingResponse` with async generator reading DB **server-side cursors** or object storage chunks. Set correct **Content-Type** (`text/csv`, `application/x-ndjson`). For CSV/JSON lines, flush periodically to start transfer early.

Back-pressure: slow consumers naturally apply via TCP backpressure; still **timebox** DB cursors and **heartbeat** long streams behind proxies (some need periodic chunks).

```python
from fastapi.responses import StreamingResponse
import io

async def row_gen():
    async for row in db.iter_rows("SELECT ..."):
        yield (json.dumps(row) + "\n").encode()

@router.get("/export")
async def export():
    return StreamingResponse(row_gen(), media_type="application/x-ndjson")
```

### Q26. How to optimize serialization/deserialization performance?

Prefer **Pydantic v2** (Rust core) and **`model_validate_json`** on raw bytes. Use **`orjson`** for generic JSON. Avoid serializing ORM objects directly—map to **slim DTOs**. For bulk APIs, consider **binary formats** (Protobuf, CBOR) internally between services.

```python
import orjson
from fastapi.responses import Response
```

### Q27. How to implement database read replicas with FastAPI?

**Answer:**

Configure **two engines**: primary for writes, replica(s) for reads. Route queries explicitly in repositories (`using_replica=True`). Be aware of **replication lag**: critical reads-after-write may need **read-your-writes** via primary or **consistency tokens** (Postgres LSN) to wait until caught up.

ORMs like SQLAlchemy support **bind routing** with `get_bind` hooks from context. **Load balancers** for replicas with health checks.

```python
async def get_session(primary: bool) -> AsyncSession:
    engine = write_engine if primary else read_engine
    async with async_sessionmaker(engine, expire_on_commit=False)() as s:
        yield s
```

### Q28. How to handle hot partitions and data sharding strategies?

**Answer:**

**Hot partitions** occur when a key (tenant, celebrity user) saturates a single shard.**Mitigations**: **shard by hash** of entity id, **sub-shard** hot tenants, **rate limit** per hot key, **async offload** to queues. For relational, **partition pruning** and **avoid monotonic inserts** into same index range (use random suffixes for object storage prefixes).

API layer: **detect** hot keys via metrics, **isolate** to dedicated pools. Data model: sometimes **duplicate read paths** (regional caches).

### Q39. How to implement API request signing and verification?

**Answer:**

Used for **server-to-server** trust without mTLS overhead (AWS SigV4 style): canonical string over method, path, query, headers, body hash; **HMAC** with shared secret or asymmetric keys. Clients send `Authorization: HMAC ...` + timestamp + nonce.

Server: **replay protection** via nonce store + skew clock check; **constant-time** compare. **Key rotation** via key id header. Libraries: **`httpsig`**, cloud SDKs.

Document **exact canonicalization**—small differences break interoperability.

```python
import hmac, hashlib

def verify_signature(secret: bytes, body: bytes, sig: str, ts: str) -> bool:
    msg = ts.encode() + body
    expected = hmac.new(secret, msg, hashlib.sha256).hexdigest()
    return hmac.compare_digest(expected, sig)
```

### Q40. How to protect against SSRF attacks in FastAPI?

**Answer:**

**SSRF** tricks your server into requesting internal URLs (`http://169.254.169.254`, internal Redis). Mitigations: **allowlist** outbound URL schemes/hosts, **block** link-local/metadata IPs, resolve DNS and verify **resolved IP** against safe ranges, **disable redirects** or validate redirect targets.

For user-supplied URLs (webhooks), require **registration** of callback URLs per tenant. Use **`ipaddress`** module checks after resolution.

```python
import ipaddress, socket

BLOCKED = [
    ipaddress.ip_network("169.254.0.0/16"),
    ipaddress.ip_network("127.0.0.0/8"),
]

def host_allowed(hostname: str) -> bool:
    infos = socket.getaddrinfo(hostname, None)
    for fam, _, _, _, sockaddr in infos:
        ip = ipaddress.ip_address(sockaddr[0])
        if any(ip in n for n in BLOCKED):
            return False
    return True
```

### Q96. How do you evaluate when to choose FastAPI over other frameworks?

**Answer:**

Choose **FastAPI** when you want **async I/O**, **automatic OpenAPI**, **Pydantic validation**, and **modern Python typing**—ideal for **API-first microservices** and high-concurrency I/O workloads. Prefer **Django** when a **batteries-included** admin/ORM ecosystem and synchronous patterns dominate and a large monolith migration would be costly. Prefer **Flask** for **minimal** frameworks and maximum flexibility—though FastAPI often wins for greenfield APIs.

**Performance** differences are usually dominated by **app code/DB**, not framework—evaluate **team skills**, **ecosystem** (ASGI servers), and **long-term maintainability**. If you need **heavy server-side HTML**, Django/Jinja may still win; for **pure JSON APIs**, FastAPI shines.

### Q100. What are the known limitations of FastAPI and how to work around them?

**Answer:**

**Ecosystem maturity**: fewer batteries than Django—accept you'll assemble **auth**, **admin**, **tasks**.**Async complexity**: easy to **block event loop** with sync DB drivers or CPU work—use **async-native** libs or **`to_thread`**.**OpenAPI generation** edge cases with exotic typing—sometimes explicit `response_model`/`model_config` needed.

**WebSockets** scaling requires **sticky sessions** or **pub/sub** backplane.**GraphQL** is via third-party (Strawberry)—extra ops burden.**Large file uploads** should use **direct-to-object-storage** patterns.

**Workarounds**: invest in **observability** for loop lag, **strict code review** for async discipline, **load testing** early, and **composition** with specialized services rather than expecting FastAPI to do everything.

```python
import asyncio

async def cpu_bound(x: bytes) -> bytes:
    return await asyncio.to_thread(heavy_transform, x)
```
