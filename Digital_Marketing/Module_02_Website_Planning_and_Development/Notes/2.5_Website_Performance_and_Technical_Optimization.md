## 2.5.1 Page Speed Optimization

### Why Speed Matters

Page speed is not a nice-to-have. It directly impacts revenue, rankings, and user satisfaction.

**The Data:**
- **53%** of mobile visitors leave a page that takes longer than 3 seconds to load (Google)
- **A 1-second delay** in page load time reduces conversions by 7% (Akamai)
- **A 100ms improvement** increased Amazon's revenue by 1% (~$4.8B/year)
- **Google uses Core Web Vitals** as ranking factors — slow pages rank lower
- **Pinterest** reduced perceived load time by 40% → 15% increase in signups
- **BBC** found they lost 10% of users for every additional second of page load time

**Speed Impact by Industry:**

| Industry | Page Speed Impact | Revenue at Risk |
|---|---|---|
| E-commerce | 1s delay = 7% conversion loss | $100K/day site → $7K/day lost |
| Lead Generation | 1s delay = 10% fewer leads | 100 leads/day → 90 leads/day |
| Media/Publishing | 1s delay = 11% fewer pageviews | Less ad revenue, lower engagement |
| SaaS | 1s delay = 15% lower trial signups | $50/trial → $7.50 lost per visit |

### Core Web Vitals

Google's Core Web Vitals are the three metrics that Google uses to measure real-world user experience. They are ranking factors since June 2021.

| Metric | What It Measures | Good | Needs Work | Poor |
|---|---|---|---|---|
| **LCP** (Largest Contentful Paint) | Loading performance — when the main content appears | ≤2.5s | 2.5-4s | >4s |
| **INP** (Interaction to Next Paint) | Interactivity — response time to user interaction | ≤200ms | 200-500ms | >500ms |
| **CLS** (Cumulative Layout Shift) | Visual stability — how much the layout shifts during load | ≤0.1 | 0.1-0.25 | >0.25 |

**What Causes Poor Scores:**

| Metric | Common Causes | Fixes |
|---|---|---|
| **LCP** | Large unoptimized images, slow server response, render-blocking CSS/JS, no CDN | Optimize images, preload LCP element, use CDN, improve server response |
| **INP** | Heavy JavaScript execution, long tasks blocking main thread, complex event handlers | Code split, defer non-critical JS, optimize event handlers, use web workers |
| **CLS** | Images without dimensions, dynamic content injection, web fonts causing FOIT/FOUT | Set width/height on images, reserve space for ads/embeds, use font-display |

```
VISUAL: How CLS Happens

Before Load:         After Load:
┌──────────┐        ┌──────────┐
│  Header  │        │  Header  │
├──────────┤        ├──────────┤
│  Text    │        │  AD      │ ← Ad loads, pushes content down
│  Content │        ├──────────┤
│          │        │  Text    │ ← User was reading here
│          │        │  Content │    and content shifted
└──────────┘        │          │
                    └──────────┘

Fix: Reserve space for the ad before it loads.
```

**Real-Time Example — The Washington Post:**
- Rebuilt their entire website (PageBuilder Fusion) focused on Core Web Vitals
- Achieved 88% of pages passing Core Web Vitals (up from 52%)
- LCP improved from 3.8s to 2.1s
- Result: Pages with good CWV scores had 25% lower bounce rates

### Image Optimization

**Images are the #1 performance bottleneck** — they account for 50%+ of page weight on average.

| Format | Best For | Compression | Browser Support |
|---|---|---|---|
| **WebP** | Most web images (photos + graphics) | 25-35% smaller than JPEG | 97%+ browsers |
| **AVIF** | Next-gen format, best compression | 50% smaller than JPEG | 92%+ browsers |
| **JPEG** | Photos (fallback) | Good compression, universal | 100% |
| **PNG** | Graphics with transparency | Large files, lossless | 100% |
| **SVG** | Icons, logos, simple graphics | Scalable, tiny file size | 100% |

**Image Optimization Checklist:**
1. **Use correct format:** WebP for photos, SVG for icons, AVIF for maximum compression
2. **Compress before uploading:** TinyPNG, ShortPixel, Squoosh
3. **Serve responsive sizes:** Don't serve a 4000px image to a 400px mobile screen
4. **Lazy load below-the-fold images:** Only load images when user scrolls to them
5. **Use CDN for image delivery:** Cloudflare, Cloudinary, imgix
6. **Set dimensions in HTML:** Prevents CLS — always include width and height attributes
7. **Use modern `<picture>` element:** Serve WebP with JPEG fallback

**Real-Time Example — Etsy:**
- Etsy has 100M+ product images
- They converted all images to WebP (with JPEG fallback)
- Result: 800TB/day less bandwidth, pages loaded 0.4s faster
- For a site with 90M+ monthly visitors, 0.4s faster = millions in additional revenue

### Code Minification

**What:** Removing unnecessary characters (whitespace, comments, formatting) from code files without changing functionality.

| File Type | Before Minification | After | Savings |
|---|---|---|---|
| HTML | 150KB | 110KB | 27% |
| CSS | 200KB | 140KB | 30% |
| JavaScript | 500KB | 300KB | 40% |

**Tools:**
| Tool | What It Minifies | Integration |
|---|---|---|
| Terser | JavaScript | Build tools (Webpack, Vite) |
| cssnano | CSS | PostCSS, build tools |
| html-minifier | HTML | Node.js, build tools |

### Browser Caching

**What:** Stores static assets (images, CSS, JS, fonts) in the user's browser so repeat visits don't re-download everything.

**Cache-Control Headers:**

| Asset Type | Recommended Cache Duration |
|---|---|
| CSS/JS (with hash in filename) | 1 year (immutable) |
| Images | 1 month to 1 year |
| HTML pages | No cache or short (5 minutes) |
| Fonts | 1 year |
| API responses | Depends on freshness needs |

**Impact:** Returning visitors experience 50-80% faster load times with proper caching.

### Critical CSS and Above-the-Fold Rendering

**What:** Extract only the CSS needed for above-the-fold content and inline it in the HTML. Load the rest of the CSS asynchronously.

**Why:** Normally, the browser must download ALL CSS before rendering anything. Critical CSS lets the visible portion render immediately.

```
WITHOUT Critical CSS:                 WITH Critical CSS:
1. Request HTML  ─── 200ms           1. Request HTML (includes critical CSS) ─── 200ms
2. Request CSS   ─── 300ms           2. Render above-fold ─── 0ms (CSS is inline!)
3. Render page   ─── 100ms           3. Load rest of CSS async ─── 300ms (background)
Total: 600ms                          Total: 200ms to first render
```

**Tools:** Critical (npm package), WP Rocket (WordPress), Critters (Webpack plugin)

### Font Optimization

| Strategy | What It Does | Impact |
|---|---|---|
| **font-display: swap** | Shows fallback font immediately, swaps when custom font loads | Prevents invisible text (FOIT) |
| **Preload key fonts** | Tells browser to fetch fonts early | Reduces font load delay |
| **Self-host fonts** | Load from your server, not Google Fonts CDN | Eliminates extra DNS lookup |
| **Subset fonts** | Include only characters you use (Latin, not CJK) | 80%+ file size reduction |
| **Use system fonts** | No font download needed at all | Fastest possible text rendering |
| **Limit font weights** | Use 2-3 weights max (regular, bold, italic) | Each weight = 20-50KB less |

**Real-Time Example:**
- Stack Overflow switched to system fonts (no custom web fonts)
- Font-related network requests: reduced to zero
- First Contentful Paint improved by 200ms
- For a site with 100M+ monthly visitors, this is significant

### Strategies

1. **Performance Budget:** Set a maximum page weight (500KB) and load time (2s). Every new feature must fit within the budget or something else gets removed.
2. **Image-First Optimization:** Optimizing images alone typically improves PageSpeed score by 20-30 points. Start here for the biggest quick win.
3. **CDN + Caching as Foundation:** These two changes (10 minutes of work) can improve performance by 30-50% for most sites.
4. **Progressive Loading:** Load critical content first (above the fold), then load the rest as users scroll. Users perceive the site as fast even if total load time is longer.

### Best Practices

1. **Target a PageSpeed score of 90+.** Anything below 50 is actively hurting your SEO and conversions.
2. **Optimize the critical rendering path.** What users see first should load first — hero image, headline, CTA.
3. **Test on real 3G/4G connections.** Your office WiFi doesn't represent your users' experience.
4. **Monitor after every deployment.** A single unoptimized image in a blog post can tank your site speed.
5. **Make performance someone's responsibility.** If nobody owns speed, speed degrades over time.
6. **Set up alerts.** If PageSpeed drops below 80, get notified immediately.

## 2.5.2 Website Security for Marketers

### Why Marketers Must Care About Security

- **64% of consumers** will avoid a business after a data breach (IBM)
- Google **flags insecure sites** with "Not Secure" warning — killing trust and conversion
- **Hacked websites can be de-indexed** by Google, destroying all SEO investment
- **43% of cyberattacks** target small businesses (Verizon)
- Average cost of a data breach: **$4.88M** (IBM, 2024)

### SSL/TLS Certificates

**Already covered in 2.1.3 — key reminder for marketers:**
- HTTPS is mandatory. No exceptions.
- Free SSL via Let's Encrypt (included with most hosting)
- HTTPS improves SEO ranking, trust, and conversion

### Common Threats Marketers Face

| Threat | What It Is | How It Happens | Impact |
|---|---|---|---|
| **Malware** | Malicious code injected into your site | Vulnerable plugins, weak passwords | Site defaced, visitors infected, Google de-indexes |
| **SQL Injection** | Attacker inserts database commands via forms | Unsanitized input fields | Data theft, data destruction |
| **XSS (Cross-Site Scripting)** | Attacker injects scripts that run in visitors' browsers | Unvalidated user input displayed on page | Session hijacking, phishing, data theft |
| **DDoS** | Overwhelms your server with fake traffic | Botnets send millions of requests | Site goes down, lost revenue during outage |
| **Brute Force** | Automated password guessing | Weak admin passwords, no rate limiting | Attacker gains admin access |
| **Phishing/Domain Spoofing** | Fake version of your site | Lookalike domain registration | Customer data theft, brand damage |
| **SEO Spam Injection** | Attacker adds hidden links/content for SEO | Compromised CMS, backdoor access | Google penalizes your site |

**Real-Time Example — British Airways Data Breach (2018):**
- Hackers injected malicious JavaScript (Magecart attack) into the payment page
- Skimmed credit card data from 380,000 customers
- BA was fined $28.7M under GDPR
- The attack was a simple script injection — could have been prevented with CSP headers and code integrity monitoring

### Security Best Practices for Marketers

**Daily/Weekly:**
- Monitor Google Search Console for security issues
- Review login activity on CMS admin panel
- Check that SSL certificate is valid and not expired

**Monthly:**
- Update ALL plugins, themes, and CMS core
- Review admin user accounts (remove former team members)
- Run malware scan (Wordfence, Sucuri)
- Review file change logs

**Quarterly:**
- Full security audit
- Review and update passwords
- Test backup restoration
- Check security headers

### Security Headers Every Site Should Have

| Header | What It Does | Value |
|---|---|---|
| **Strict-Transport-Security (HSTS)** | Forces HTTPS, prevents downgrade attacks | `max-age=31536000; includeSubDomains` |
| **Content-Security-Policy (CSP)** | Controls what resources can load | Restricts script/style/image sources |
| **X-Frame-Options** | Prevents clickjacking (embedding your site in an iframe) | `DENY` or `SAMEORIGIN` |
| **X-Content-Type-Options** | Prevents MIME-type sniffing | `nosniff` |
| **Referrer-Policy** | Controls what referrer info is sent | `strict-origin-when-cross-origin` |
| **Permissions-Policy** | Controls browser feature access | Disable camera, microphone, geolocation |

**Test your security headers:** securityheaders.com (free, instant scan)

### Implementation Steps

1. **Verify HTTPS is active** — check for padlock icon and "Not Secure" warnings
2. **Install a WAF** — Cloudflare free tier provides basic protection
3. **Update everything** — CMS, plugins, themes — all to latest versions
4. **Install security plugin** — Wordfence (WordPress) or platform equivalent
5. **Enable two-factor authentication** — for ALL admin accounts
6. **Set strong passwords** — 16+ characters, unique per service, use a password manager
7. **Set up automated backups** — daily backups to external storage
8. **Remove unused plugins/themes** — every unused plugin is an attack surface
9. **Configure security headers** — HSTS, CSP, X-Frame-Options
10. **Set up monitoring** — Google Search Console, UptimeRobot, Sucuri

### Tools

| Tool | Purpose | Cost |
|---|---|---|
| Cloudflare | WAF + DDoS protection + CDN | Free tier |
| Wordfence | WordPress security | Free / $119/yr |
| SecurityHeaders.com | Check security header configuration | Free |
| 1Password / Bitwarden | Password management for teams | Free / $3/mo |
| Google Search Console | Security issue notifications | Free |
| UptimeRobot | Downtime alerts (detect DDoS) | Free |

### Gap Analysis

| Gap | Symptom | Fix |
|---|---|---|
| No SSL/HTTPS | "Not Secure" browser warning | Install Let's Encrypt or provider's auto-SSL |
| Outdated CMS/plugins | Vulnerability exploits, malware | Update immediately, set up auto-updates |
| No WAF | Frequent attacks, spam submissions | Set up Cloudflare WAF (free) |
| Weak admin passwords | Brute force access, site defacement | Enforce strong passwords + 2FA |
| No backup strategy | Hack = total loss of website | Enable automated daily backups |
| No security headers | Vulnerable to clickjacking, XSS, downgrade attacks | Configure HSTS, CSP, X-Frame-Options |
| Shared admin credentials | Can't trace who did what, no accountability | Individual accounts per team member |

### Strategies

1. **Defense in Depth:** Layer security — SSL + WAF + security plugin + strong passwords + 2FA + backups. No single measure is sufficient.
2. **Principle of Least Privilege:** Give each user account only the permissions they need. Marketing interns don't need admin access.
3. **Automate Security Updates:** Use managed hosting (WP Engine, Kinsta) that auto-updates WordPress core and patches.
4. **Incident Response Plan:** Know what to do if hacked: isolate site, restore from backup, scan for malware, change all passwords, notify affected users.

### Best Practices

1. **Update everything, always.** 56% of WordPress hacks exploit outdated software.
2. **Use 2FA on every admin account.** Passwords alone are not enough.
3. **Back up before you update.** If an update breaks something, you can roll back immediately.
4. **Use a password manager.** Teams sharing passwords via Slack is a security disaster waiting to happen.
5. **Monitor Google Search Console.** Google will notify you if they detect malware or security issues.
6. **Remove what you don't use.** Deactivated plugins are still attack surfaces — delete them.

## 2.5.3 Website Maintenance & Management

### Why Ongoing Maintenance Matters

A website is not "set and forget." Without maintenance:
- Content becomes stale → SEO rankings drop
- Links break → user experience suffers, SEO is penalized
- Software becomes outdated → security vulnerabilities appear
- Performance degrades → load times increase, conversions fall
- Competitors improve → your relative position weakens

### Content Freshness

**Why Fresh Content Matters:**
- Google's "freshness" algorithm rewards recently updated content
- Users trust current information — a "2021 Guide" feels outdated in 2024
- Updated content can recapture lost rankings

**Content Update Schedule:**

| Content Type | Review Frequency | Update Action |
|---|---|---|
| Blog posts | Quarterly | Update statistics, add new sections, refresh examples |
| Landing pages | Monthly | Update testimonials, offers, CTAs |
| Product pages | As needed | Update images, descriptions, pricing, availability |
| About/Team page | Monthly | Add new team members, update bios |
| Pricing page | Quarterly | Adjust pricing, update feature lists |
| Case studies | Bi-annually | Add new case studies, update metrics |
| FAQ page | Quarterly | Add new questions from customer support data |

**Real-Time Example — HubSpot's Content Refresh Strategy:**
- HubSpot has 10,000+ blog posts
- They dedicate 60% of blog resources to UPDATING existing posts, not creating new ones
- Process: Identify posts losing traffic → update data and examples → re-optimize for current keywords → republish with current date
- Result: Updated posts recover 106% of original traffic on average, and some posts see 200-300% increases

### Broken Link Monitoring

**Why Broken Links (404s) Matter:**
- Each broken link is a dead end for users (frustration, bounce)
- Google crawl budget is wasted on 404s
- Internal broken links lose link equity (SEO value)
- External broken links signal neglect to both users and search engines

**Common Causes:**
- Pages deleted without redirects
- URL structure changes without 301 redirects
- External sites you link to go down or change URLs
- Typos in link URLs

**Monitoring Tools:**

|---|---|---|

5. **Use staging for all changes.** Never test on the live site. Period.
6. **Document everything.** If your webmaster gets hit by a bus, can someone else maintain the site?

## 2.5.4 Schema Markup & Structured Data

### What Is Schema Markup

**Schema markup** (structured data) is code you add to your website that helps search engines understand your content better. It doesn't change what users see — it helps Google understand WHAT your content is (a product, a recipe, a FAQ, an event, etc.) and display it with rich results in search.

**Why It Matters:**
- Rich results get **58% more clicks** than standard results (Search Engine Journal)
- **Rich snippets increase CTR** by an average of 30% (Ahrefs)
- Schema helps Google understand entity relationships (your brand, products, people)
- **Featured snippets, knowledge panels, and rich cards** all use schema data
- Schema is NOT a ranking factor but rich results dramatically improve visibility and click-through

### Types of Schema Markup

| Schema Type | What It's For | Rich Result Display |
|---|---|---|
| **Organization** | Company name, logo, contact, social profiles | Knowledge panel for brand searches |
| **Product** | Product name, price, availability, reviews | Price, availability, and stars in search |
| **FAQ** | Frequently asked questions | Expandable Q&A directly in search results |
| **HowTo** | Step-by-step instructions | Steps displayed in search |
| **Review/Rating** | Aggregate ratings | Star ratings in search results |
| **Event** | Date, location, performers, tickets | Event details with date, location, ticket links |
| **Article** | Author, date, headline, image | Article info in Google Discover and News |
| **BreadcrumbList** | Site navigation hierarchy | Breadcrumb path in search results |
| **LocalBusiness** | Address, hours, phone, reviews | Local pack, Google Maps |
| **Video** | Duration, thumbnail, description | Video thumbnails in search results |
| **Recipe** | Ingredients, cook time, nutrition, rating | Full recipe card in search results |
| **JobPosting** | Title, salary, location, company | Google for Jobs listing |
| **SoftwareApplication** | Name, rating, price, OS | App details in search |

### JSON-LD Implementation

**JSON-LD (JavaScript Object Notation for Linked Data)** is Google's preferred format for structured data. It's a `<script>` tag placed in the `<head>` or `<body>` of your page.

**Example — Organization Schema:**
```json
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "HubSpot",
  "url": "https://www.hubspot.com",
```

### Implementation Steps

1. **Identify relevant schema types** for your website (Organization, Product, FAQ, etc.)
2. **Generate JSON-LD code** using Google's Markup Helper or Merkle's generator
3. **Add JSON-LD to your pages** — in `<head>` or `<body>` (both work)
4. **Validate with Rich Results Test** — fix any errors or warnings
5. **Submit URL for indexing** in Google Search Console
6. **Monitor performance** — track rich result impressions and clicks in Search Console
7. **Add schema to all relevant pages** — not just the homepage
8. **Keep schema updated** — prices, availability, ratings must match page content
9. **Use a plugin** (WordPress) — Yoast, Rank Math, or Schema Pro automate much of this
10. **Review quarterly** — new schema types are added regularly by Google

### Gap Analysis

| Gap | Symptom | Fix |
|---|---|---|
| No schema markup at all | No rich results in search, plain blue links | Start with Organization + FAQ schema |
| Schema errors | Rich results not appearing despite markup | Validate with Rich Results Test, fix errors |
| Schema data doesn't match page | Google may remove rich results for mismatches | Ensure schema data reflects actual page content |
| Only homepage has schema | Missing rich results on product/blog/FAQ pages | Add relevant schema to ALL important pages |
| Using outdated schema format | Microdata or RDFa instead of JSON-LD | Migrate to JSON-LD (Google's preferred format) |
| No FAQ schema on FAQ pages | Missing expandable Q&A in search results | Add FAQ schema — high-impact, low-effort |

### Strategies

1. **FAQ Schema First:** Adding FAQ schema to your top landing pages is the highest-ROI schema implementation. It can double your search result real estate.
2. **Product Schema for E-Commerce:** Every product page should have Product schema with price, availability, and reviews. Rich results dramatically increase CTR.
3. **Local Business Schema:** If you have physical locations, LocalBusiness schema is essential for Google Maps and local pack visibility.
4. **Schema Cascading:** Apply Organization schema site-wide, Article schema to blog posts, Product schema to product pages, FAQ schema to FAQ sections — layer appropriately.

### Best Practices

1. **JSON-LD is the only format worth using.** Google explicitly prefers it over Microdata and RDFa.
2. **Schema MUST match visible content.** If your schema says price is $99 but the page shows $129, Google will penalize you.
3. **Don't schema everything.** Focus on types that produce rich results — FAQ, Product, Review, HowTo, Event.
4. **Test before publishing.** Always validate with Rich Results Test. One syntax error invalidates the entire block.
5. **Monitor in Search Console.** The "Enhancements" section shows which schema types are detected and any errors.
6. **Update schema when content changes.** If product price changes, schema must change too. Automate this with your CMS.
